Stream I "build kit" — entry point for coding sessions  ·  Stream I session I.1  ·  May 10, 2026

This document is the top of the coding-session reading stack. A coding session implementing module X reads this index, then the Stream I spec for X, then the READMEs of the modules X depends on. Coding sessions do not read original stream documents (Article XI § 2). The cross-module integration contracts live in the sibling doc stream_i_contracts_v1.html.

What's in the build kit

  • This document (stream_i_buildkit_index_v1.html) — the entry point. Reading order, module map, v1 acceptance criteria, implementation sequence with dependencies. Written by I.1.
  • Cross-module integration contracts (stream_i_contracts_v1.html) — TypeScript interfaces between every pair of modules. The canonical type surface. Written by I.1.
  • v1 project skeleton (argmap_v1_skeleton.zip) — drop-in repo with all build configuration, ESLint determinism gates, custom plugin scaffold, CI workflow, and module placeholders. Written by I.1; the user drops it into a fresh repo before I.2 begins.
  • Per-module Stream I spec docs — one per src/X/ module. Each gives TypeScript signatures internal to the module, function contracts, acceptance tests, and integration points to other modules. Written by I.2–I.9 (one session each, possibly more for UI).
  • v1 acceptance test plan — user-facing scenarios. Written by I.10.

How a coding session reads this

A coding session implementing module X loads exactly the following, in order:

  1. The Constitution and Protocol (automatic via project instructions; also mirrored at .claude/CLAUDE.md in the repo so Claude Code sessions inherit them).
  2. current_state.html — the canonical state.
  3. This document. Skim the Module map and v1 acceptance criteria; read the Implementation sequence row for X.
  4. stream_i_contracts_v1.html — read the contract section for X and for every module X depends on.
  5. stream_i_X_spec_v1.html — the per-module spec. Read in full.
  6. The README under each src/Y/ whose API X calls. Written by prior coding sessions.

The coding session does not read original stream documents (A–H). It does not read other modules' specs. It does not read other modules' source. The READMEs are the cross-session compression artifact; if a downstream session needs to know something about an upstream module, the answer lives in the README.

v1 acceptance criteria — high level

Detailed end-to-end scenarios are I.10's deliverable. The high-level criteria below let coding sessions know what the bar is.

  1. Determinism gate (Article II § 2). The CI iteration-order audit step passes on every PR. The runtime ESLint boundary passes on every PR. The runtime test suite runs under a frozen clock and produces byte-identical snapshot outputs on every run.
  2. Frame Building (legal + general modes). The user can create a frame, switch modes/flavor at frame creation, add every node type from the palette, create every edge type via the drag-from-handle gesture (filtered by valid pairs per V-EDGE-1), edit fields in the inspector, edit per-instance options_box on Checkpoint/Question/Interpretation nodes, save milestones, and use version history (preview, restore, structural diff). C5 strict validation runs on the Frame → Argument transition; errors block, warnings advise.
  3. Argument Running. The user can create a session against a FrameVersion, run the open-items interview in the deterministic order, answer Checkpoints (boolean / multiple_choice / graded), select Interpretations, attach Premises (with optional Authority refs in legal mode), see the three output views (path overlay default, decision tree, prose summary) update on every change, and see the status palette (5 states + sub-flags) consistently across canvas, outline tree, interview pane.
  4. Persistence (Stream H). Auto-save with the 5s/30s debounce works. Frame and ArgumentSession version histories are independent. Frame template deep-copy, session migration, and version restore are atomic. Search works across title/description/node text/tags/Conclusion statements. Cross-tab editing surfaces last-write-wins via BroadcastChannel without data loss in the worst case.
  5. Mode transitions (Stream D D3). Frame → Argument runs C5 strict and prompts for session choice on pass. Argument → Frame surfaces FrameVersion drift on return. Architectural mode change (legal ↔ general) groups blocking vs. advisory items and provides inline editors for blocking items. Flavor change commits immediately with an advisory list.
  6. LLM hooks (Stream G ship list). The hook contracts are wired and the Mock provider drives all hook tests deterministically. G1, G2, G3, G5, G6, G8, G9, G11, G13 ship with prompts; G7, G10, G12 ship interfaces with deterministic non-LLM fallbacks. Build-time hooks bake-and-commit; the audit log on LlmSettings.invocations populates correctly. The Anthropic provider works in dev when an API key is present; production builds gate it behind user opt-in.
  7. Onboarding (E7). First-time wizard works opt-out. Coachmarks fire once each, are dismissible, and are tracked per-user. Glossary tooltips and panel work in both modes.
  8. Export / import. A frame round-trips through FrameExport JSON without data loss. A session round-trips through ArgumentSessionExport with embedded frame.

"Ships" means: npm run ci + npm run test:e2e are green from a clean checkout, and a manual run-through of the I.10 acceptance plan succeeds.

Determinism — the implementation contract restated

Every coding session must understand this. Article II § 2 is non-negotiable. The implementation-level contract has three parts, all enforced by tooling shipped in I.1:

  1. Module boundary. src/runtime/ imports only @/schema and standard built-ins. Forbidden: @/persistence, @/ui, @/state, @/llm-hooks, @/modes, @/layout; React, Dexie, Zustand, ELK, anything that reads the clock or DOM. Enforced by no-restricted-imports in eslint.config.js.
  2. Sort before iterate. Every iteration over a Set, Map, or plain object inside src/runtime/ uses one of sortedIter, sortedKeys, sortedEntries from iteration-helpers.ts. Enforced by the custom argmap-determinism/no-unsorted-iteration rule.
  3. Frozen clock in tests. Every test in tests/runtime/ runs with fake timers and a frozen clock (configured in vitest.config.ts). Property tests (Stream I.3) run each case twice with different frozen times to surface incidental clock dependencies.

If a coding session encounters a case where one of these gates seems to forbid a necessary thing, the session must:

  • Look first for a way to refactor that respects the gate (almost always available).
  • If after analysis the gate appears to truly conflict with a needed capability, log an in-session-escalated flag (Article X § 1) explaining the conflict, and propose a refactor or an explicit, narrow-scope exception. Do not silently weaken any of the three gates. They implement Article II § 2 directly; a silent weakening is an Article V § 3(b) escalation trigger.

Token-economy reminders for spec authors (I.2–I.10)

Per Stream H consolidation in current_state.html "Token-optimization principles":

  • Each Stream I spec doc cites cross-module integration contracts by section name in stream_i_contracts_v1.html; do not re-state them.
  • Each spec doc commits its module's acceptance tests in Vitest form. The tests are compressed contracts — they bound implementation freedom and let the coding session implement-until-green.
  • The README under src/X/ is the cross-session compression artifact. Write it to be read, not exhaustive; it should answer "what does X expose, and what should I avoid breaking?" in one screen.
  • Each spec doc is read at most twice — once by I.1 author for cross-module ratification, once by the coding session that implements the module. Optimize accordingly.

I.1 self-review (Article VIII § 4)

Architectural commitments (Article II)

  • § 1 Frame/Argument Split. Honored. The contracts in stream_i_contracts_v1.html separate Frame/FrameVersion (frame layer) from ArgumentSession/ArgumentSessionVersion (argument layer); session embeds a frame snapshot. Module structure isolates frame-construction (modes D1, schema, validation) from argument-running (modes D2, runtime).
  • § 2 Determinism. Honored, and operationally enforced by three tooling gates shipping in the skeleton: ESLint runtime-boundary, ESLint no-unsorted-iteration, frozen-clock Vitest. The CI workflow exercises all three on every PR. The contract for compute() is fixed in stream_i_contracts_v1.html Contract 2 with the byte-identity guarantee.
  • § 3 Per-Node Satisfaction Criteria. Honored. SatisfactionPolicy and the 12-condition library are pinned in schema/satisfaction-policy per the contracts doc; options_box is on NodeBase per Stream B.
  • § 4 Independent Versioning. Honored. Repository.saveFrameVersion / saveSessionVersion are separate. Stream H atomicity contracts (frame-template deep-copy, session migration, restore) are exposed in the contracts doc as composite Repository operations.
  • § 5 Human-Readable Structure. Honored. The schema (per Stream B) carries the logic; no hidden ordering enters the runtime. Term.order is documented as a foreclosure-resolution + UI-default-presentation field only.
  • § 6 Use-Case Commitment. Honored. Both legal and general modes (with personal/academic flavors) are first-class. Mode-aware schema behavior is a contract in Stream B; mode-aware UI surface visibility is a Stream D and Stream E concern preserved in the buildkit's per-module references.

Quality criteria (Article III)

  • § 1 Correctness. The contracts re-state the canonical signatures from Streams B/C/G/H rather than re-derive them; the skeleton's tooling enforces the determinism contract that Correctness depends on for the runtime.
  • § 2 Legal Practice Fit. The buildkit preserves all legal-mode surfaces (Authority, jurisdiction, BINDING_IN, DISTINGUISHED_BY, burden_level, standard_of_review, F5/F6 stub) in the module map and contracts. F6-v2 wire-in path is documented as a deferred contract under Cross-cutting in the contracts doc.
  • § 3 Determinism and Auditability. Three tooling gates plus the HookInvocationRecord audit trail. Both load-bearing.
  • § 4 Clarity. The skeleton's module READMEs each fit on one screen; the buildkit index walks the full reading order in one place; the contracts doc is broken into eight per-module sections plus three cross-cutting sections.
  • § 5 Reversibility. Stream I docs are versioned (_v1); contracts changes bump to _v2 rather than rewriting. The skeleton's customizations (ESLint rules, the custom plugin) are local to the repo and replaceable. Dependency choices are minimal and replaceable behind their respective abstractions (Repository for storage, LlmProvider for LLMs).
  • § 6 Interface Fit. A coding session reads (1) Constitution, (2) current_state.html, (3) this index, (4) one contracts doc, (5) one spec doc, (6) zero-or-a-few dependent-module READMEs. Estimated reading load before useful work begins: well within one tractable session.

Article V § 3 escalation triggers — recheck

  • (a) Reverses a user-stated strategy. No. The buildkit follows the user's planning-session decisions (Stream I structure, two-tool split, token-optimization principles) without revision.
  • (b) Reverses or weakens an Article II commitment. No. The skeleton's tooling reinforces Article II § 2; nothing here weakens any Article II provision.
  • (c) Two Article II commitments conflict and the conflict is unresolvable from the Constitution. No. The contracts doc resolves all observed cross-stream interactions (e.g., G5 calibrated thresholds reaching compute() via FrameVersion projection rather than Frame-on-runtime; G6 prose rewrite living on ArgumentSessionVersion in a parallel field). Both have register-flag precedent (F-001, F-002).
  • (d) Forecloses a use case. No. Both modes preserved.
  • (e) Anthropic safety policies require user input. No.
  • (f) Constitution silent or contradictory on a substantial question. One discretionary call: putting cross-module contracts in a sibling doc (stream_i_contracts_v1.html) rather than inline in this index. The user explicitly delegated this choice ("at the agent's discretion"); Article V § 2(c) covers it; not a § 3 trigger.

In-session flags

None. No Stream B/C/D/E/H/G shape is altered; no register flag is generated. The discretionary sibling-doc choice is documented above and in Persistence; it is below the Article X § 1 escalation threshold (it does not reverse a prior stream decision and is not non-obvious).
