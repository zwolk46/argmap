stream_i_contracts_v2.html

Cross-module integration contracts  ·  Stream I session I.10  ·  May 11, 2026  ·  Supersedes stream_i_contracts_v1.html

This document is the canonical TypeScript boundary between modules. It supersedes stream_i_contracts_v1.html in full. Per-module Stream I specs (I.2–I.9d4) and the I.10 integration spec cite specific interfaces from this document by name; coding sessions implement to these signatures. The contracts are shape-only — they fix the type surface that crosses module boundaries, not the implementation. Stream-specific internal types live in the per-module specs and need not appear here.

Where a contract refers to a Stream-B-defined entity (Frame, FrameVersion, ArgumentSession, etc.), the canonical shape is in stream_b_schema_v1.html; this document re-states only the cross-module-relevant surface and points to Stream B for the rest. For new shapes introduced by per-module specs (FramePatch / SessionPatch unions, dispatch tables, selector view types, etc.), this document pins the public surface that other modules consume; the internal details remain in each per-module spec.

Module dependency graph

Allowed import directions (read "→" as "may import from"):

schema       ← runtime  ← persistence ← state ← ui
                   ↑                       ↑      ↑
                   └─────── modes ─────────┘      │
                                                  │
schema ← llm-hooks ← state ← ui                   │
schema ← layout ← state ← ui ─────────────────────┘
                                                  │
schema ← modes ← state ← ui                       │

Hard rules (unchanged from v1):

  • schema imports nothing from this codebase. Pure types and validators.
  • runtime imports only schema (Article II § 2; Stream H part 1; ESLint no-restricted-imports). It does not import persistence, ui, state, llm-hooks, modes, or layout. It does not import any third-party library that reads the clock or environment.
  • llm-hooks imports schema (and may import the Repository interface from persistence for hook contexts; see HookContext below). It is never imported by runtime.
  • persistence imports schema and runtime (the latter only to call compute on session restore; the runtime is itself pure).
  • modes imports schema, runtime, persistence, and llm-hooks. Modes orchestrates user-initiated actions; it is the layer where action surfaces (D1, D2) live.
  • layout imports schema only. Pure layout against frame structure; main thread speaks to the layout Web Worker via the bridge.
  • state imports everything except ui. Stores hold references to Repository, dispatch into modes, subscribe to runtime outputs.
  • ui imports state, schema, and primitives from layout and llm-hooks. No direct calls to persistence or runtime; routing through stores.
  • ui sub-modules may value-import from sibling ui sub-modules within the boundary established by the I.9a UI-internal discipline (e.g., @/ui/chrome imports @/ui/onboarding for OnboardingPreferencesSection; @/ui/home imports @/ui/onboarding for NewFrameWizard).

Notation conventions

  • Types written in TypeScript. Where Stream B uses a TypeScript-like notation (with ? for optional), this document follows TypeScript exactly.
  • NodeRef and EdgeRef are aliases for string (entity ids). They live in schema/identifiers.ts.
  • timestamp is string (ISO-8601). Stream B fixes this; reading it as Date happens in UI/state; runtime treats it as opaque.
  • Imports are written as @/<module>/<file>; the @ alias is wired in tsconfig.json and vite.config.ts.
  • Where a type is defined in another contract section of this document, it is referenced by name without re-stating its shape.
  • v2 convention: additions relative to v1 are flagged with the originating flag id (e.g., // F-007 #1) in inline comments. This is a documentation-aid; the production exports do not carry the comments.

Contract 2 — runtime → schema

The runtime module computes ComputeResult from a FrameVersion + ArgumentSessionVersion + a clock anchor. The output is byte-identical for byte-identical input (Article II § 2). Tooling gates: no-restricted-imports for the boundary; argmap-determinism/no-unsorted-iteration for sort-before-iterate; frozen clock in Vitest.

v1.5-deferred batch loader (F-006 #5)

The Compare view in the version-history pane currently loads two FrameVersions (or two ArgumentSessionVersions) with Promise.all over the existing single-version loaders. If a future polish pass surfaces a perceptible Compare-view flash from the two parallel reads, the Repository surface gains an additive pair-loader without restructuring:

// v1.5 (NOT shipped in v1)
loadFrameVersionPair(a_id: FrameVersionId, b_id: FrameVersionId): Promise<[FrameVersion, FrameVersion]>;

v1 ships without these methods. The deferral is recorded so the v1.5 surface is additive when added.

Contract 4 — modes → schema, runtime, persistence, llm-hooks

The modes module orchestrates user-initiated actions: D1 (Frame Building dispatch table), D2 (Argument Running dispatch table), D3 (mode transitions including orphan resolution), and the createFrame surface. Every dispatch reads deps.now() exactly once per call (the single-clock-read property required by Article II § 2; F-003 family discipline restated). Every dispatch is a pure transformation from (prior_state, patch, deps) to (next_state, change_summary); persistence side effects are performed by the action-runner in @/state wrapping the dispatch result.

Cross-cutting: validation result attachment

Three modules produce ValidationResult[]: schema (rule definitions), runtime (phase 1 evaluation), and llm-hooks (G4 gap-detection advisories with severity "warning"). All three flow into the state layer's validation projection, which the UI surfaces via the per-node chips, bottom drawer, and inspector block (E2). The contracts above pin ValidationResult in @/schema/validation-rules as the canonical shape. Dismissal: the user may dismiss individual advisories per F-004 #3; dismissed warnings live in AppState.dismissed_warnings and are surfaced through selectValidationDrawer's dismissal-aware projection. Dismissals are per-user and per-rule_id; they do not affect the underlying compute output.

Cross-cutting: HookInvocationRecord audit trail

Hook commits write into FrameVersion fields and append to Frame.llm_settings.invocations. The append happens through Repository.saveFrame in the same transaction as the FrameVersion write that the commit produces. The state layer renders the audit projection per Stream E; the UI never invokes hooks directly without going through llm-hooks contracts. Provenance discipline (D1 of UI determinism): every AI-attributed surface (G6 prose rewrite, G7 ranked premise reuse when enabled, G11 premise drafting, G12 advisories when enabled) carries an AIAttributionChip with verbatim fields from HookInvocationRecord. Deterministic fallbacks (v1 G7's rankPremiseReuse, v1 G10, v1 G12-off) carry no chip.

Cross-cutting: F5 (court hierarchy database) deferred contract

The F5 module under src/runtime/f5/ is stubbed in v1. Its eventual interface is fixed by Stream F's F5Database API surface. v1 ships F6-v1 only; the F5 wire-in session adds the F5-database-lookup branch to authority-binding.ts per F6-v2's specification, which widens the result envelope additively (kind, reasoning, database_version) without changing the function signature. The llm_settings_snapshot projection on FrameVersion has reserved field-slot space for whichever F5-determinism mechanism the wire-in session selects (per current_state.html open question).

Cross-cutting: integration-test harness expectations

Added in I.10 to pin the test-environment surface that scenario-grouped integration tests depend on. Each integration test mounts App with a RepositoryProvider whose repository is a Dexie instance backed by fake-indexeddb, whose autosave uses the Vitest fake-timers clock, and whose llm_settings_default.provider resolves to MockLlmProvider by default. The relevant environment seams the harness exposes (and that downstream coding sessions must preserve):

  • Deterministic clock anchor. Vitest's global frozen-clock default at 2026-05-10T00:00:00Z applies; integration tests may override per-scenario when they explicitly need a different anchor. The harness reads Date.now() only through RepositoryProvider's injected now() (used by AutosaveController for the debounce timer); state-layer dispatches read it through their own Deps.
  • Deterministic UUIDs. The harness injects a counter-backed uuid() ("00000000-0000-4000-8000-000000000001", then ...000002, etc.) via the same Deps seam. Tests that need explicit UUID values pin them; tests that snapshot output assert via stable JSON-key ordering and the counter sequence.
  • fake-indexeddb backing. The harness installs fake-indexeddb/auto in tests/integration/setup.ts and resets the database between scenarios via indexedDB.deleteDatabase("argmap") followed by Repository re-construction.
  • Mock LLM provider deterministic responses. Each test scenario that exercises an LLM-driven surface installs a MockLlmProvider whose invoke returns canonical responses keyed by input_sha256. The harness exposes a helper (mockLlmFor(scenario)) that builds the expected responses.
  • Layout-worker shim. The layout module's Web Worker is replaced in tests with a synchronous main-thread shim that delegates to the same pure computeLayout implementation. Worker round-trip timing is not part of the test contract; structural layout output is.

Those two placeholders never had real call sites — they were reserved by I.9b/I.9c with explicit "I.9d wires this" comments, and I.9d2 determined that the wiring is page-local rather than App-level. No downstream code depends on the removed shapes. Every other v1 surface is unchanged or has been extended with required-not-optional promotions or additive fields.
