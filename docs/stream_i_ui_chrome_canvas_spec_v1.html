stream_i_ui_chrome_canvas_spec_v1.html

Stream I per-module spec — UI substrate (chrome + canvas + primitives)  ·  Stream I session I.9a  ·  May 11, 2026

This document is the implementation source of truth for the shared UI substrate that src/ui/frame-building/ (I.9b), src/ui/argument-running/ (I.9c), and src/ui/version-history/ + src/ui/onboarding/ (I.9d) consume. It commits per-file TypeScript signatures and behavioral contracts for: the application shell and provider mounting (App.tsx); the typed routing primitive; the design-token stylesheet and the React Flow CSS-variable bindings; the primitives library (status badge, type icon, severity icon, AI-attribution chip, glossary tooltip, dialog, drawer, segmented toggle, pill); the top-bar chrome (operating-mode toggle, frame title, mode/flavor chip, validation indicator, version-history button, frame-settings button, help button, help/glossary slide-in pane); the React Flow canvas integration (the FrameCanvas component, the layout-result consumer, the per-node-type custom renderers, the per-edge-type custom renderers, the connector-handle and edge-creation gesture machinery, the canvas toolbar with zoom / fit / auto-arrange / search / FORECLOSES-layer toggle, the minimap, and the status-badge overlay system); the AI-suggestion lifecycle hook and the three-button confirmation drawer; and the keyboard-shortcut + cascade-confirmation hooks. The cross-module surface (what other modules import from @/ui) is pinned in stream_i_contracts_v1.html Contract 8; this spec defines the per-subdirectory exports and the internal types that I.9b/c/d consume.

The UI module is structurally limited to importing @/state (the integration surface), @/schema (the type vocabulary), @/layout (worker-backed layout), and @/llm-hooks (only for the SuggestionResult / ConfirmationDecision shapes the suggestion drawer renders). Imports from @/runtime, @/persistence, @/modes are forbidden per Contract 8's preamble; every action goes through a state store. This boundary is the I.9a coding session's responsibility to enforce in ESLint config (see "What this spec does not commit" below for the one item I.9a defers to the implementing session).

Cross-document resolution notes (for self-review)

The I.9a spec carries small refinements relative to stream_i_contracts_v1.html and prior Stream I specs. None is constitutionally novel; each is documented here so the self-review at the end can confirm the constitutional citations and the in-session-flag escalation that consolidates them into the register-flag queue.

  1. Suggestion-lifecycle additions to Contract 7 are treated as committed. The I.8 register flag (held in flag_pending_from_i8.html for merge) added FrameStore.invokeHook(hook_id, args), FrameStore.resolveSuggestion(decision), FrameStoreSnapshot.pending_suggestion?: SuggestionResult<unknown>, parallel additions on SessionStore, and a commit_hook_output dispatch entry on the modes layer. The I.9a spec references these as if they are present, on the same basis as I.4–I.8 referenced their pending refinements: the per-module spec is canonical for the consuming module, and downstream coding is not gated on the Contract-7 bump. See the suggestion-drawer and useAiSuggestion sections below.
  2. Argument-overlay projection. The FrameCanvas accepts an optional argumentOverlay projection that bundles the argument-layer edges (ANSWERS, SUPPORTS, CONTRADICTS) and the session's Premises + session Authorities for rendering when in Argument Running mode. The state layer does not currently export a single selector for this; useSessionStore with an inline selector is the call site, but the projection shape itself is named here so I.9c's session-store consumer is consistent. The shape is purely a UI-side projection; no state-layer signature changes.
  3. CheckpointOption virtual edges. Per D decision #3 and E1, CheckpointOption routing is a field on Checkpoint (options[i].target_node_id), not an edge in FrameVersion.edges. The FrameCanvas synthesizes virtual edges for each option with a target before passing edges to React Flow. The virtual edges are tagged with edge type checkpoint_option (a UI-only identifier not present in schema); the edge-validity computation in edge-validity.ts recognizes it.
  4. Routing primitive. v1 ships in-memory routing with URL-hash sync; no react-router dependency. The buildkit dependency set (stream_i_buildkit_index_v1.html) does not include a client router; v1 has three top-level routes (Home, FrameBuilding, ArgumentRunning) plus modal overlays that are UI-state, not route-state. The minimal routing.ts below handles this without adding a dependency. This is purely an implementation choice under Article V § 2(c); no contract changes.
  5. Cascade-confirmation dependency on modes. The cascade summary itself is computed by the modes layer (D1 cascade computation per stream_i_modes_spec_v1.html's cascade.ts). The UI consumes it as a selector through frame_store; the state layer wires the call. Naming the dependency here so the implementing session knows the chain. The state spec's selectors.ts exposes a useCascadeSummary(node_id) selector or equivalent; I.5 left this as agent-discretionary; this spec's useCascadeConfirmation hook calls it.
  6. One consolidated in-session-escalated register flag. The five items above are consolidated into a single register flag (same family as F-003 and the I.3–I.8 pending flags) recommending: Contract 8 (UI) gains the placeholder page exports for the I.9b/c/d split; Contract 7 (state) bump captures the suggestion-lifecycle additions inherited from the I.8 flag; the argumentOverlay projection type lands in the cross-module surface of @/state or as a UI-internal type (agent-discretionary at I.9c kickoff); and the useCascadeSummary selector is named in the state spec. Held in flag_pending_from_i9a.html for merging with the existing pending flags.

// src/ui/routing.ts

Typed in-memory routing with URL-hash mirroring. No client-router dependency in v1. The Route union is closed; adding a route is a schema-change-equivalent edit, not an extension point. URL-hash sync is for back-button support and link-share; the hash is the surface, not the state.

import * as React from "react";
import type { FrameId, SessionId } from "@/schema";

export type Route =
  | { kind: "home" }
  | { kind: "frame_building"; frame_id: FrameId }
  | { kind: "argument_running"; session_id: SessionId };

export const ROUTE_HASH_PATTERN = {
  home: "#/",
  frame_building: "#/frame/",            // suffix: frame_id
  argument_running: "#/session/",        // suffix: session_id
} as const;

export function routeFromHash(hash: string): Route;
export function hashFromRoute(route: Route): string;

export interface RouterContextValue {
  current: Route;
  navigate: (next: Route) => void;
}

export const RouterContext: React.Context<RouterContextValue | null>;

export function RouterProvider(props: { children: React.ReactNode }): React.ReactElement;

/** Hook returning the current Route. Throws RouterMissingError outside a RouterProvider. */
export function useRoute(): Route;

/** Imperative navigate. Throws RouterMissingError outside a RouterProvider. */
export function useNavigate(): (next: Route) => void;

export class RouterMissingError extends Error {
  constructor() {
    super("useRoute / useNavigate called outside RouterProvider.");
    this.name = "RouterMissingError";
  }
}

Behavioral contract. RouterProvider: on mount, reads window.location.hash through routeFromHash, seeds current; subscribes to hashchange events and re-derives current on each event. navigate(next) updates current and pushes hashFromRoute(next) to window.location.hash (which triggers hashchange, which re-derives — the round-trip is idempotent because the source of truth on dispatch is the next Route and the hashchange is a no-op when hash already matches). routeFromHash on an unrecognized hash returns { kind: "home" } — the v1 behavior is conservative; URL deep links into deleted frames land on Home gracefully.

// src/ui/error-boundary.tsx

A class-component error boundary surrounding the routed content. Catches render errors, displays a recovery panel ("Something went wrong. Reload the application to recover. Your work is auto-saved."), exposes a recovery button that triggers window.location.reload(). Per stream_e_uiux_v1.html E7 empty states, the panel uses neutral chrome — no celebration motion, no stack traces in production. Stack traces are visible only when import.meta.env.DEV.

import * as React from "react";
import type { ReactElement, ReactNode } from "react";

export interface AppErrorBoundaryProps {
  children: ReactNode;
  /** Optional override for the recovery action. Default: window.location.reload(). */
  onRecover?: () => void;
}

interface State {
  error: Error | null;
  componentStack?: string;
}

export class AppErrorBoundary extends React.Component<AppErrorBoundaryProps, State> {
  state: State = { error: null };
  static getDerivedStateFromError(error: Error): State {
    return { error };
  }
  componentDidCatch(error: Error, info: React.ErrorInfo): void {
    // No console.error in production; in dev, surface for debugging.
    if (import.meta.env.DEV) {
      // eslint-disable-next-line no-console
      console.error("AppErrorBoundary caught:", error, info);
    }
    this.setState({ error, componentStack: info.componentStack ?? undefined });
  }
  render(): ReactElement | ReactNode {
    if (!this.state.error) return this.props.children;
    return /* recovery panel JSX — see behavioral contract */ null;
  }
}

Determinism — UI-layer discipline

The UI is not on the Article II § 2 determinism boundary (only the runtime is). But the UI has obligations that protect the user's experience of the determinism guarantee:

  1. The canonical surface for every runtime output is rendered without alteration. The output viewer (I.9c) renders ConditionalOutput.prose_summary verbatim; the status badge renders NodeStatus.status verbatim; the path overlay highlights nodes from active_path verbatim. UI logic does not re-order, summarize, or interpret runtime outputs. This is the F-002 carry guidance applied uniformly.
  2. The AI-attribution chip is rendered wherever a field's current value came from a hook. Per Stream G, every AI-touched field surfaces its provenance. The attribution chip's tooltip lists hook id + prompt version + provider/model + timestamp — the same data HookInvocationRecord persists in LlmSettings.invocations.
  3. The interview pane (I.9c) renders items in the deterministic D2 order without re-sorting. The state layer's selector returns the open-items list in computed order; the UI iterates that list as-is. The "recommended next" highlight is the first item in the list, by construction; no UI-side reranking.
  4. Validation surfaces (I.9b) list rule firings in their canonical order. The state layer surfaces validation as an array; the UI iterates as-is. No client-side severity-based reordering changes the audit story.

The implementing session adds these rules to src/ui/README.md as the UI's "what we do not do" list. The boundary discipline (import restrictions) is mechanically enforced; the four rules above are reviewable through tests and code review.
