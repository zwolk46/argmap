stream_f_legal_mode_v1.html

Stream F — Legal Mode specifics  ·  Consolidation + F5/F6-v2 specification  ·  May 10, 2026  ·  Authoritative under Article VII § 1(d)

This document closes Stream F at the design-spec level. F1–F4 are consolidated by reference to the streams that absorbed them. F5 (court hierarchy database) is specified at the data-model and API-surface level; data collection itself is out of scope and runs as an independent sub-project. F6-v2 (binding determination using F5) is specified end-to-end. The output is shaped to let Stream I compile a self-contained v1 coding spec without F5 wired and to let a future F5 session and F6-v2 wire-in session proceed without re-litigating design.

Scope and method

Stream F was scoped in the master plan as the legal-mode specialization of the application: Authority schema (F1), jurisdiction modeling (F2), standard of review encoding (F3), burden of proof encoding (F4), court hierarchy database (F5), and binding determination (F6). The user's session brief for this work was: consolidate F1–F4 (which were absorbed by Streams A, B, C as those streams progressed), specify F5 at a level of detail sufficient for an independent data-collection sub-project, specify F6-v2 (the F5-aware binding algorithm), and shape the output to prepare Stream I for code implementation. F5 data collection itself is out of scope; the spec produces the shape that data must take and the API the data must answer to.

The dominant quality criterion for this work is Article III § 2 (Legal Practice Fit). The F5 model and the F6-v2 algorithm are written to match how practicing lawyers and courts actually reason about binding precedent, not how a tidy graph algorithm might prefer to model it. Where mathematical cleanliness and professional credibility conflict, professional credibility prevails.

F1 — Authority schema and behavior (consolidation)

The Authority node is fully specified at stream_b_schema_v1.html § B1.8. The shape supports both legal-mode use (citation, court, year, holding/dicta, binding, jurisdiction, binding_in) and academic-flavor use (author, venue, position) on the same node, with locus implicit from where it lives (frame layer when cited from CITES; argument layer when introduced via Premise's authority_ref). Edge types touching Authority are CITES (Authority → Interpretation, with strength), DISTINGUISHED_BY (Authority → Interpretation, with reasoning), and the conceptual-relation BINDING_IN (implemented as the embedded binding_in: Jurisdiction[] field on Authority rather than a row in the edges table). Validation rule V-FR-11 enforces that any Authority with is_binding: true in legal mode has a non-null jurisdiction. In personal flavor, the type is labeled "Source" in the UI but the schema name remains Authority for code consistency. "Scholarly consensus" is not a separate concept; it is an Authority node with position describing the consensus and a citation to the survey paper that establishes it.

The is_binding field is manually settable in v1 and is the manual-override input to F6 (overriding any algorithmic determination). When F6 computes binding via the algorithm, the field is read by display surfaces but does not get auto-written; the computed result lives in the runtime NodeStatus.via array ("binding_authority" / "persuasive_authority") and in the C6 result envelope.

F2 — Jurisdiction modeling (consolidation)

Jurisdiction is a value object — never a node, never an entity with its own id — embedded into Frame, Authority, and Interpretation where applicable. The full shape is at stream_b_schema_v1.html "Jurisdiction (Legal Mode)":

Jurisdiction { level: "federal" | "state" | "tribal" | "territory" | "international" region?: string // e.g., "9th Circuit", "California", "S.D.N.Y." court?: string // e.g., "Supreme Court of California" hierarchy_node_id?: string // reference into F5 court hierarchy when wired }

In v1 (without F5 wired), only level and region are required to be populated; court is optional free text; hierarchy_node_id is null. When F5 wires in, hierarchy_node_id becomes the canonical reference and a court picker (Stream E addition; see § "Integration with Stream E" below) populates it from F5; level, region, and court remain on the Jurisdiction value object as denormalized cache for display and for the F6 string-match fallback path.

Where Jurisdiction is used in the schema:

  • Frame.jurisdiction_default — legal-mode-only frame field; the default jurisdiction in which the frame's case is being heard. Used by F6 as the target for binding determination.
  • Authority.jurisdiction — where the authority issued from. Used by F6 as the source for binding determination.
  • Authority.binding_in: Jurisdiction[] — the conceptual BINDING_IN edge expressed as an embedded list. v1 is hand-curated by the user; v2 (post-F5) becomes derivable from the hierarchy database in many cases, though manual entries remain authoritative.
  • Interpretation.jurisdiction_override — overrides the frame default for the branch rooted at this Interpretation, allowing a frame to model jurisdictional pivots within a single argument (rare but legitimate in conflict-of-laws and choice-of-forum analyses).

Jurisdictional pre-checks (personal jurisdiction, subject-matter jurisdiction, etc.) are ordinary SubQuestions with is_jurisdictional: true per A8. They render first in interview ordering (Stream D) and are visually distinguished as a "Jurisdictional questions" section header (Stream E). Validation rule V-FR-7 warns — does not block — when a legal-mode frame has zero jurisdictional SubQuestions.

F3 — Standard of review encoding (consolidation)

Standard of review is encoded as part of the per-node satisfaction-criteria framework (A3's "options box") rather than as a separate concept. The mechanism:

  • Field carrier. Question and SubQuestion nodes carry an optional standard_of_review field (legal mode only) drawn from a fixed enum: de_novo, abuse_of_discretion, clear_error, arbitrary_and_capricious, substantial_evidence, plain_error. The full enum lives in stream_b_schema_v1.html.
  • Condition. The SatisfactionPolicy condition library includes standard_of_review_applied (no params), which evaluates true when the standard of review is set on the parent Question/SubQuestion and is consistent with the burden_met level on the current node. Implementation: it reads the parent's standard_of_review and the node's burden context and asserts they are coherent against a small fixed table.
  • LLM hook surface. G5 (standard-of-review threshold calibration) is the build-time hook for tuning the consistency table without violating runtime determinism; the calibrated thresholds project into FrameVersion via the llm_settings_snapshot field added by flag F-001.

In academic and personal flavors, standard_of_review is hidden and the condition is not offered in the options-box editor (per Stream D D5). The schema field remains for round-trip stability when toggling architectural mode.

F4 — Burden of proof encoding (consolidation)

Burden of proof is encoded analogously, also through the options-box framework:

  • Field carrier. Checkpoint nodes carry a burden_level field (legal mode only) drawn from a fixed enum: preponderance, clear_and_convincing, beyond_reasonable_doubt, scintilla, substantial_evidence.
  • Condition. The SatisfactionPolicy condition library includes burden_met with a level param. Default Checkpoint policy includes { name: "burden_met", level: "preponderance" }; per-instance overrides change the level.
  • Evaluation. Categorical (not stochastic). Per stream_c_runtime_v1.html, each supporting Premise contributes a weight by kind and edge weight; contradicting Premises subtract; the total is compared against a fixed threshold per burden level. The scoring table is in C, exposed in code as a single module for inspectability.
  • Flavor relabeling. In academic and personal flavors, the enum stays the same but the UI relabels for idiom (e.g., preponderance → "more likely than not"). The schema field is unchanged so that toggling architectural mode round-trips cleanly. Per Stream D, the burden-level UI in academic flavor also surfaces an epistemic-context advisory.

F5 — court hierarchy database (new specification)

Purpose and scope

F5 is the data layer that lets F6 answer the binding-determination question with reference to actual court relationships rather than the v1 string-match heuristic. F5 has two distinct deliverables:

  • F5-spec (this document): the data model courts must take, the API surface F6 calls, the versioning contract that preserves determinism, and the v1 coverage targets that constrain the data-collection effort. Belongs in Stream I's coding spec as a reserved module shape.
  • F5-data (independent sub-project): the actual list of US federal and state courts, their attributes, and their hierarchical relationships, conforming to the F5-spec. Sourcing strategy is open per the user's standing direction. Independent of Stream I.

The split lets Stream I compile a v1 coding spec that ships without F5-data — F6-v1 (string match, in C6) handles binding determination at v1 launch. F5-data is a follow-on deliverable that, when it lands, swaps F6-v1 out for F6-v2 by populating Jurisdiction.hierarchy_node_id for the courts the user selects from a picker (a new UI affordance described under "Integration with Stream E"). Existing frames continue to work via the string-match path until the user opts in by editing an Authority's jurisdiction and selecting a court from F5.

Data model: CourtNode

F5 is a graph of CourtNode records. The shape is designed to support the F6-v2 binding-determination algorithm directly while remaining recognizable to practitioners and inspectable in flat form.

CourtNode { id: string // stable opaque slug, e.g. "us-scotus", "us-ca-9", "us-dc-cda" name: string // "Supreme Court of the United States" short_label: string // "SCOTUS", "9th Cir.", "S.D. Cal." level: "federal" | "state" | "tribal" | "territory" | "international" region: string // "United States", "9th Circuit", "California", "S.D.N.Y." court_type: CourtType // see enum below system_id: string // hierarchy this court lives in: "us-federal", "us-state-ca", "us-state-ny", "us-tribal-navajo", ... tier: integer // rank within system: 1 = highest, ascending downward parent_court_ids: string[] // courts whose decisions bind this court (vertical stare decisis upward); transitively walked by F6 appellate_review_by_court_id?: string // the immediate appellate court over this one; informational, not used for binding binds_self: boolean // horizontal stare decisis: are this court's prior decisions binding on later panels of itself? binding_scope_court_ids?: string[] // optional explicit downward scope override; if absent, computed from descendants in same system not_binding_on_court_ids?: string[]// optional explicit exclusions for unusual cases is_specialized: boolean // e.g., Federal Circuit (patents-only nationwide), Tax Court, Bankruptcy specialty?: string // "patents", "tax", "bankruptcy", "claims_against_us", ... active_from?: string // ISO date; courts created after this date didn't bind before; informational in v1 active_until?: string // ISO date; defunct courts; informational in v1 notes?: string // free text for unusual cases; informational, not consumed by F6 } CourtType = | "supreme" // system high court (US SCOTUS, state supreme courts) | "intermediate_appellate" // US circuits, state intermediate courts of appeal | "trial_general" // US district courts, state superior/circuit/district courts | "trial_limited" // state limited-jurisdiction trial courts | "specialized" // Tax Court, Court of Federal Claims, Court of Chancery, etc. | "administrative" // agency tribunals; partial-inclusion, see "out of scope"

Notes on the model:

  • The hierarchy is implicit in parent_court_ids. F6's binding walk is "is the authority's court reachable from the target's court via parent_court_ids within the same system_id." That single check captures vertical stare decisis cleanly.
  • system_id is the unitary-hierarchy boundary. US federal is one system. Each state is its own system. Tribal courts are individual systems. Cross-system relationships are persuasive only by default; specific cross-system bindings (e.g., a state high court binding federal courts in diversity per Erie) are out of scope for F5 v1 and described under "out of scope" below.
  • tier is informational, not the binding driver — parent_court_ids is. Tier exists for UI display, sorting, and as a sanity check for data quality.
  • binds_self captures horizontal stare decisis, which varies by court (the 9th Circuit treats prior panels as binding absent en banc; some state courts of appeal treat panels as merely persuasive). For specialized cases, binding_scope_court_ids and not_binding_on_court_ids let the data layer override the algorithmic walk.
  • Specialized courts are flagged. The Federal Circuit, for example, is binding nationwide on patents and as a regional appellate court for limited subject matter. F6-v2 reads is_specialized + specialty and reports them as reasoning on its result; v1 does not condition the binding answer on the case's substantive subject matter (deferred — see "out of scope").
  • The id is a stable opaque slug. Slugs are human-readable for inspection and debug but are treated as opaque keys by F6; renaming a court's display name does not change its id. A renaming registry tracks display-name changes without affecting referential integrity. (Per Stream B's identifier model, opaque ids are the canonical reference; short_label changes are display-only.)

API surface (consumed by F6-v2)

F5 is consumed by F6-v2 through a small, pure interface. F5 is read-only at runtime; updates ship with app releases (see versioning, below). The interface lives at src/runtime/f5/ in the Stream H module structure (src/runtime/ is the determinism boundary; F5's data file is bundled and treated as a static input, not an external service call).

// Read interface type F5Database = { version: string // e.g., "f5-2026-05-10"; see "Versioning" getCourt(courtId: string): CourtNode | null isBinding(authorityCourtId: string, targetCourtId: string): F5BindingResult listCourts(filter?: F5CourtFilter): CourtNode[] // for the UI court picker // Reverse lookups for the UI searchCourts(query: string, limit?: number): CourtNode[] } type F5BindingResult = { binding: boolean kind: "vertical_stare_decisis" | "horizontal_stare_decisis" | "specialized_subject_matter" // reserved for v2 case-context-aware binding; not produced in v1 | "persuasive_cross_system" | "persuasive_same_system" // peer or sibling court; not vertical, not horizontal | "not_binding_unknown" reasoning: string // human-readable explanation, e.g., "9th Cir. is parent of S.D. Cal." database_version: string // echoes F5Database.version; consumed by F6 for the determinism contract } type F5CourtFilter = { level?: "federal" | "state" | "tribal" | "territory" | "international" system_id?: string region_substring?: string // case-insensitive substring match court_type?: CourtType is_specialized?: boolean }

The isBinding implementation is a deterministic graph walk over parent_court_ids within the same system_id, with explicit overrides consulted first. It is specified in pseudocode under "F6 — binding determination" below; F5 owns the data, F6 owns the algorithm, but the algorithm is colocated with the database for cohesion (the isBinding method on F5Database is the canonical implementation; F6 calls it).

The interface is intentionally small. It does not expose a "list ancestors of court X" query, an "is X above Y" query separate from binding, or any free-form graph traversal. Anything F6 or the UI needs is a named query. This keeps F5's contract narrow and lets the implementation choose any storage shape (in-memory map, IndexedDB store, embedded SQLite) without leaking through.

Versioning and the determinism contract

F5 is an immutable versioned dataset. Each release of F5 carries a version string of the form f5-YYYY-MM-DD (or finer, e.g., f5-YYYY-MM-DD-rN for revisions on the same day). Two releases may differ in court records (added, removed, reattributed) or in metadata, but a given version string always denotes the same byte-identical dataset.

The determinism contract is straightforward in stating what must hold and intentionally underdetermined in how the implementation preserves it:

  • What must hold. The runtime compute() pipeline must produce byte-identical outputs across F5 releases for a given (FrameVersion, ArgumentSessionVersion) pair. An F5 update must not silently change the binding determinations recorded in an existing session's outputs.
  • How — left open. The mechanism by which the application preserves byte-identity across F5 updates is deferred to the F5 wire-in session (or to Stream I if it elects to commit early). Two viable mechanisms are presented below; the choice depends on F5-data's actual size and update cadence, neither of which is known yet.

Two candidate mechanisms, both compatible with this spec:

  1. Bundled-pinned. F5 ships bundled with the app at a known version. The bundled version is a constant in the runtime. Determinism holds within an app version. App updates that bump F5 produce visible, expected drift (analogous to a library upgrade). ArgumentSessionVersion records the F5 version it was last computed against in an additive optional field f5_version_consulted; on reopening a session whose recorded version is older than the bundled version, the runtime surfaces a non-blocking "F5 hierarchy data has updated since this session was last computed" notice and offers a Recompute action that creates a new ArgumentSessionVersion. Pro: simplest, no historical-data retention burden. Con: byte-identity across app versions is not preserved; the determinism guarantee is "within an app version, plus visible drift notice across app versions."
  2. Snapshot-on-session. Each ArgumentSessionVersion records, in an additive optional field, the F5 lookups it consulted (inputs + outputs) — a lightweight cache of the form {authority_court_id, target_court_id, F5BindingResult}[]. On recompute, lookups are served from the cache; new lookups query the bundled F5. The result is byte-identical recomputation across F5 updates. Pro: strict determinism. Con: each session carries a small additional payload (typically a handful of entries; bounded by the number of Authorities cited in the frame).

Either mechanism requires an additive optional field on ArgumentSessionVersion (mechanism 1: f5_version_consulted: string?; mechanism 2: binding_determinations_cache: BindingDeterminationCacheEntry[]?). Both are precedented by the F-001 (FrameVersion llm_settings_snapshot) and F-002 (ArgumentSessionVersion output_overrides) patterns, both ship as in-session escalated flags when the F5 wire-in session occurs. Stream I should reserve one additive-field slot on ArgumentSessionVersion for whichever mechanism is selected, but does not need to commit to which one for the v1 coding spec, because v1 does not have F5 wired and therefore does not produce results that need to be preserved against F5 updates.

The v1 coding spec ships F6-v1 (string match, no F5), which has no determinism-across-F5-updates concern because there are no F5 updates. The first F5 release plus F6-v2 wire-in is the moment the determinism mechanism must be picked. This document marks that as an open design point for the F5 wire-in session.

Storage and packaging

F5 v1 is bundled with the application as a static JSON dataset at src/runtime/f5/data/f5-{version}.json. The runtime imports and indexes it on app boot into in-memory maps (byId, byShortLabel, bySystemId, byRegion) for O(1) lookups; isBinding's ancestor-walk is bounded by the depth of the system hierarchy (≤ 4 in the US federal and US state systems, by construction of tier).

Bundling rather than fetching is required by the determinism contract: a network fetch introduces uncontrollable inputs (latency, partial responses, eventual 404s). The dataset is small enough to bundle (a US federal-and-state F5 v1 is on the order of a few hundred court records and a few thousand parent-edge entries; well under a megabyte uncompressed).

The Stream H tech-stack module structure already reserves src/runtime/ as the determinism boundary with restricted imports. F5's data file is a JSON resource imported by the runtime; the import is allowed under the existing ESLint rule because it is data, not a third-party clock-reading library.

v1 coverage targets

F5 v1's data-collection scope is the US federal and US state court systems at the level needed for typical legal-analysis use cases. Specifically:

  • US federal courts (full coverage required for v1):
    • Supreme Court of the United States (1 record).
    • All thirteen Courts of Appeals: 1st through 11th Circuits, D.C. Circuit, Federal Circuit (13 records).
    • All ninety-four district courts (94 records).
    • Specialized federal courts: Court of Federal Claims, Court of International Trade, Tax Court, Court of Appeals for Veterans Claims, Court of Appeals for the Armed Forces (5 records). Bankruptcy courts: one record per district (94 records, attached to their respective district courts).
  • US state courts (full coverage required for v1):
    • Each state's highest court (50 records + DC + relevant territories).
    • Each state's intermediate appellate courts at the level the state actually has them (varies; some states are flat, some have geographic divisions like California's Courts of Appeal districts).
    • Each state's general-jurisdiction trial courts at the level of "California Superior Court" or "New York Supreme Court" — one synthetic node per state representing the trial-court level, even where the actual courts are county-by-county. County-level disaggregation is deferred to v2.
    • Notable specialized state courts: Delaware Court of Chancery, New York Surrogate's Court, Texas Court of Criminal Appeals (separate from the Texas Supreme Court), Oklahoma Court of Criminal Appeals (same), specialized juvenile and family courts where they are statewide rather than per-county.
  • DC and US territories — the District of Columbia courts as a state-equivalent system; Puerto Rico, Guam, U.S. Virgin Islands, Northern Mariana Islands (territory-level systems, with their connection to federal court review represented).

A v1 dataset meeting these targets is on the order of three hundred to four hundred CourtNode records, with the parent-edge structure encoded inline. The work is nontrivial but bounded.

Out of scope (v1)

  • Tribal courts. Tribal court systems are individual sovereigns with varying internal structures and varying relationships to state and federal courts. Modeling them faithfully is its own substantial sub-project. v1 ships without tribal coverage; the schema supports them (level: "tribal", per-tribe system_id) so v2 layers in without restructuring. v2
  • International and foreign national courts. Same rationale; the schema supports them. v2
  • Administrative law tribunals. NLRB, BIA, immigration courts, agency ALJs. Their decisions interact with Article III courts in nuanced ways (Chevron historically, post-Loper Bright reorientations). v1 may include a few high-traffic administrative bodies as court_type: "administrative" records but does not aim for completeness. v2
  • Erie-doctrine binding (state high court binding federal courts in diversity). Modeling Erie correctly requires F6 to know the case's "applicable substantive law," which is not currently an input. The v1 model treats cross-system as persuasive only; the Erie case is a known false-negative. A user can override by setting Authority.is_binding: true manually on the relevant state high court Authority. v2
  • Subsequent treatment (overruled, abrogated, vacated, questioned). Already deferred per stream_c_runtime_v1.html § "Reserved / deferred design space." F5 v1 models court status, not case status. v2
  • Per-case subject-matter conditioning. The Federal Circuit is binding nationwide on patents and as a regional appellate court for non-patent matters. F6-v1 and F6-v2 do not condition on the case's subject matter; the F5BindingResult kind: "specialized_subject_matter" value is reserved for the v2 implementation that does. v1 reports the Federal Circuit's is_specialized + specialty in reasoning so the user can interpret manually. v2
  • Court reorganizations and historical states. The schema carries active_from / active_until for documentation; F6-v1 and F6-v2 do not reason about whether a court was binding at the time the case was decided. Modeling that would require F6 to know the authority's date and the target case's date, plus a full historical hierarchy. v2
  • Interlocutory specialty (e.g., en banc effects, panel-versus-en-banc binding inside the same circuit). v1 treats a circuit as one node. v2

Sourcing constraints

The user's standing direction is that F5 sourcing is open and is to be decided when F5 begins. This document does not pick a sourcing strategy. Constraints on whatever strategy is later picked:

  • Output shape. The deliverable is a JSON file conforming to the CourtNode schema above. Intermediate formats (CSVs, scraped HTML, manually edited spreadsheets) are fine; the final form must be the JSON.
  • Determinism. Every record's id is a stable opaque slug. The slug must not change between F5 versions for the same court. (Renames of name and short_label are allowed; id is the referential anchor.) Deletion of a court (court abolished) requires a tombstone record with active_until set, not silent removal — the slug must remain resolvable for historical sessions.
  • Quality bar. The data must be correct enough to credit a practitioner. Article III § 2 (Legal Practice Fit) is the controlling criterion. A federal-courts-only first cut is acceptable as a milestone if the state-court coverage lags; partial coverage with explicit gaps is better than guessed data.
  • Provenance. Each CourtNode should be sourceable to its origin (court website, U.S. Courts directory, state judicial branch site, statutory creation reference). Provenance is recorded in a separate file (e.g., f5-provenance-{version}.json), not in the runtime CourtNode shape (where it would bloat the file without benefiting the algorithm).
  • License. Public-domain or permissively-licensed sources only. Government court directories are typically public domain in the US and are the natural primary source.
  • Reproducibility. Where possible, the F5 build process should be scripted (scrape + transform), not purely manual, so that updates can be re-run.

Testing strategy

F5 testing is in three layers, all colocated with the runtime tests under Stream H's Vitest setup:

  • Schema tests. Every CourtNode validates against the shape; required fields are present; level, court_type are within enum; system_id resolves to at least one other court (no orphan systems in v1); parent_court_ids all resolve to existing court ids; no cycles in parent_court_ids; tier is monotonic along parent edges within a system.
  • Algorithm tests. A fixture of well-known binding pairs from the legal canon: SCOTUS binds 9th Cir., 9th Cir. binds S.D. Cal., 9th Cir. does not bind 2d Cir., California Supreme Court binds California Court of Appeal, California Supreme Court does not bind 9th Cir. (cross-system; persuasive in F6-v1; would be Erie-binding in v2 with applicable-law context), Federal Circuit binds D. Mass. on patents (v2-only) — v1 reports specialized + persuasive_cross_system. The fixture is the canonical regression test for F5-data updates.
  • Determinism tests. isBinding is invoked under vi.useFakeTimers() with frozen clock; results are byte-identical across runs and across two different frozen times, per the Stream H determinism testing pattern.

F6 — binding determination

F6-v1 (resolved; recap)

F6-v1 is fully specified at stream_c_runtime_v1.html § C6. Recap: is_binding(authority, target_jurisdiction) returns { binding: bool, source: "manual_override" | "hierarchy_database" | "string_match" | "no_jurisdiction", certainty: "exact" | "heuristic" }. Resolution order is (1) manual override on Authority's is_binding, (2) hierarchy_database via F5 if both jurisdictions have hierarchy_node_id populated, (3) string-match on level + region with the SCOTUS-everywhere special case wired by name, (4) no-jurisdiction fallback. v1 ships without F5 wired, so step 2 is a no-op and step 3 is the working path for the algorithmic determination.

F6-v1 ships with v1 of the application. It runs inside the authority_binding condition during phase 4 of compute() and is consumed by the prose-summary generator in phase 5. It is a leaf-level utility, not a pipeline phase.

F6-v2 (specification)

F6-v2 is the F5-aware revision of F6-v1. It activates when F5-data has shipped and at least some Authorities have jurisdiction.hierarchy_node_id populated. The function signature and its consumers (the authority_binding condition, the prose-summary generator) are unchanged; the body adds the F5 lookup and enriches the result envelope.

Algorithm

def is_binding(authority: Authority, target: Jurisdiction) -> F6Result: # Step 1 — manual override (unchanged from v1) if authority.is_binding is not None: return F6Result( binding=authority.is_binding, source="manual_override", certainty="exact", kind=None, reasoning="user-set is_binding on this Authority", database_version=None, ) # Step 2 — hierarchy database lookup via F5 (NEW in v2) auth_court_id = authority.jurisdiction.hierarchy_node_id if authority.jurisdiction else None target_court_id = target.hierarchy_node_id if target else None if auth_court_id and target_court_id: f5_result = F5.isBinding(auth_court_id, target_court_id) return F6Result( binding=f5_result.binding, source="hierarchy_database", certainty="exact", kind=f5_result.kind, reasoning=f5_result.reasoning, database_version=f5_result.database_version, ) # Step 3 — string-match heuristic (unchanged from v1, now a fallback for un-upgraded Authorities) if authority.jurisdiction and target: result = _string_match(authority.jurisdiction, target) return F6Result( binding=result.binding, source="string_match", certainty="heuristic", kind=None, reasoning=result.reasoning, database_version=None, ) # Step 4 — no jurisdiction information (unchanged from v1) return F6Result( binding=False, source="no_jurisdiction", certainty="exact", kind=None, reasoning="authority or target has no jurisdiction information", database_version=None, )

The F5 isBinding implementation that F6-v2 calls is the deterministic graph walk:

def F5.isBinding(authority_court_id, target_court_id) -> F5BindingResult: auth = self.getCourt(authority_court_id) target = self.getCourt(target_court_id) if auth is None or target is None: return F5BindingResult(False, "not_binding_unknown", "court id not in F5 database", self.version) # Explicit overrides win, in this order: if target.id in (auth.not_binding_on_court_ids or []): return F5BindingResult(False, "not_binding_unknown", f"explicit not_binding_on entry: {auth.id} -> {target.id}", self.version) if target.id in (auth.binding_scope_court_ids or []): return F5BindingResult(True, "vertical_stare_decisis", f"explicit binding_scope entry: {auth.id} -> {target.id}", self.version) # Horizontal stare decisis (same court): if auth.id == target.id: if auth.binds_self: return F5BindingResult(True, "horizontal_stare_decisis", f"{auth.short_label} binds itself per binds_self=true", self.version) return F5BindingResult(False, "persuasive_same_system", f"{auth.short_label} does not bind itself per binds_self=false", self.version) # Cross-system: persuasive only (Erie deferred to v2; manual override available) if auth.system_id != target.system_id: return F5BindingResult(False, "persuasive_cross_system", f"{auth.short_label} ({auth.system_id}) is in a different system than {target.short_label} ({target.system_id})", self.version) # Same system: walk target's ancestors via parent_court_ids visited = set() frontier = list(sorted(target.parent_court_ids or [])) # sorted for determinism while frontier: cid = frontier.pop(0) if cid in visited: continue visited.add(cid) if cid == auth.id: return F5BindingResult(True, "vertical_stare_decisis", f"{auth.short_label} is an ancestor of {target.short_label} via parent_court_ids walk", self.version) ancestor = self.getCourt(cid) if ancestor is not None: frontier.extend(sorted(ancestor.parent_court_ids or [])) # Same system, neither self nor ancestor: peer or sibling return F5BindingResult(False, "persuasive_same_system", f"{auth.short_label} is in same system as {target.short_label} but is not an ancestor", self.version)

The walk is BFS over the parent-edge graph rather than DFS to keep the result's reasoning readable when multi-step ancestry chains appear (it surfaces the shortest-ancestry path first). Sorted-iteration on the parent_court_ids list satisfies the Stream H no-unsorted-iteration ESLint rule.

Specialized-court handling is intentionally not case-context-aware in v1. The Federal Circuit, by being in system_id: "us-federal" with is_specialized: true and specialty: "patents", gets the same answer as any other circuit-level court when asked about non-Federal-Circuit district courts: not an ancestor, so persuasive. F6-v2's F5BindingResult.reasoning includes the specialized-status note so a UI surface can flag it. The case-context-aware variant is the reserved kind: "specialized_subject_matter" value, produced only by a future v2 implementation that takes the case's subject matter as an additional input.

Result envelope (F6Result)

F6-v2 widens the F6-v1 result envelope additively:

F6Result = { binding: boolean source: "manual_override" | "hierarchy_database" | "string_match" | "no_jurisdiction" certainty: "exact" | "heuristic" // New in F6-v2 (optional in v1; populated when source == "hierarchy_database"): kind?: F5BindingKind // from F5BindingResult; null for non-hierarchy sources reasoning?: string // human-readable explanation for the surfacing UI database_version?: string // F5 dataset version consulted; null when source != "hierarchy_database" }

v1 ship of the application can include this enriched envelope as the canonical F6Result with the new fields documented as null-when-not-applicable. Stream I should specify F6Result with the v2 fields present, the F6-v1 body filling them as null. This way F6-v2 wire-in changes only the function body, not the call sites or schema.

v1 → v2 migration

The transition from F6-v1 to F6-v2 is gradual and per-Authority, not a global flip:

  • Existing Authorities continue to work via string-match. An Authority created at v1 launch has jurisdiction.hierarchy_node_id null; F6-v2 falls through to step 3 (string match) for it.
  • The user upgrades by editing. When F5 ships and the Authority editor (Stream E surface, see below) gains a court picker, the user can edit any Authority and select a court from F5; that populates hierarchy_node_id and the next compute pass uses the F5 path. Same for Frame.jurisdiction_default.
  • No silent data migration. The application does not attempt to auto-match string-based jurisdictions to F5 court ids on upgrade. The reason is Article III § 2 (Legal Practice Fit): an auto-match could quietly bind a frame to the wrong court and the user might not notice. The opt-in editor flow keeps the user in the loop.
  • A "modernize this frame" wizard is permitted as a future affordance that walks through every Authority and Frame default and offers a court picker pre-filtered to the closest matches. Whether to ship the wizard is a Stream E concern at the time F5 lands; this document does not commit to it.

The string-match heuristic remains in F6-v2's body (step 3) indefinitely. Even after F5 ships, users may create Authorities for jurisdictions not yet in F5 (foreign courts, tribal courts, administrative tribunals); string-match keeps those usable as best-effort. The "soft heuristic" flag on the result is still surfaced in the UI for those cases.

Integration with Stream E (UI surfaces for Authority and jurisdiction)

F5 wire-in adds two new UI affordances; both live in the inspector pane (Stream E E2) and the frame settings panel (Stream E E6). They are reserved here so Stream I knows what hooks the inspector needs:

  • Court picker on Authority.jurisdiction. When the user edits an Authority's jurisdiction, the inspector offers (in addition to the existing free-text fields for level, region, court) a "Pick from court hierarchy" affordance that opens a typeahead-searchable list of CourtNodes filtered by the level/region the user has typed. Determinism design point (carry-forward open question)

This is the explicit open design point that the F5 wire-in session must resolve before F5-data ships:

  • Question. When F5-data updates between app releases, how does the application preserve the byte-identity of recomputation for existing ArgumentSessionVersions? The two candidate mechanisms are bundled-pinned with a drift notice (mechanism 1 above) and snapshot-on-session caching (mechanism 2 above).
  • Why deferred. The choice depends on F5-data's actual size and update cadence, neither of which is known until F5-data has been collected at least once. Committing now would lock in a choice that may not be the right shape for the data we end up with.
  • What deferral does not foreclose. Both candidate mechanisms are additive-only on ArgumentSessionVersion (per the F-001 / F-002 pattern); the choice does not affect Stream I's v1 coding spec because v1 ships without F5-data and therefore without the determinism-across-F5-updates concern.
  • Where the decision lands. The F5 wire-in session escalates a flag (analogous to F-001 / F-002), picks a mechanism, and amends Stream B's ArgumentSessionVersion shape. The work is well-shaped; this document leaves it for the agent that has F5-data in hand.

Reserved / deferred design space

Beyond the v1 out-of-scope items already enumerated, the following are reserved for future work:

  • F5 search index. The court picker's typeahead may need a search index for performance with large jurisdiction lists. Stream H H1 already specifies a search index per Frame; an F5-specific index can layer on the same pattern. v2 if needed
  • Authority deduplication using F5. Two Authority nodes citing the same case in the same frame are not currently deduplicated. With F5 in place, a "merge duplicates" pass becomes more reliable (same court + same year + similar citation strings is a high-confidence merge). v2
  • F5-driven validation enrichments. V-FR-11 (Authority with is_binding: true has non-null jurisdiction) could be tightened to "has a hierarchy_node_id" once F5 is available, with a soft warning when string-match is used despite F5 being available. v2
  • F5-driven authority suggestions. When the user is creating an Authority, the citation field could be parsed to suggest a CourtNode (e.g., "Erie R.R. v. Tompkins, 304 U.S. 64 (1938)" → SCOTUS). This is a build-time LLM hook candidate; G2 already covers Interpretation suggestion, but a parallel "Authority court suggestion" hook (call it G14, escalated through Stream G's amendment process) is a natural extension. v2; not in current Stream G
  • F5 export and import. Power users may want to import their own court entries (e.g., for foreign or specialized jurisdictions). The Stream B JSON envelope already supports adding records; an F5-specific extension protocol would standardize it. v2

Quality criteria scoring (Article III)

  • § 1 Correctness. The F5 model is the standard "vertical and horizontal stare decisis with specialized exceptions" model, plus explicit overrides for unusual cases. It accurately reflects how binding works in the US system at the level v1 targets.
  • § 2 Legal Practice Fit. The dominant criterion for this work. The model is recognizable to a practitioner: courts as nodes, hierarchies as parent-edges, systems as boundaries, specialized courts as flagged. The deferrals (Erie, subsequent treatment, per-case subject matter, tribal courts) are the right deferrals — they are real, hard problems that v1 cannot adequately model and that practitioners would not expect a v1 tool to solve.   • § 5 Reversibility. The split between F5-spec and F5-data preserves all the reversibility we want: sourcing strategy is open, mechanism choice is open, scope expansion (tribal, international) is layerable.
