Project identity

A web-based argument mapping application that separates the LOGICAL FRAME of an argument from the PREMISES used to make it sound. The primary use case is legal analysis (appellate opinions, statutory interpretation, trial preparation), with a Legal Mode toggle that activates jurisdiction- and authority-aware features. The same frame/argument architecture serves general analytical discourse — academic argument mapping and analytical personal questions of an academic flavor. The user is a law student and aspiring attorney or judge who does not code; the user makes conceptual and natural-language decisions and delegates everything else to agents. The product is iterated over months to years; correctness and flexibility are favored over short-term convenience.

Architectural decisions — consolidated

Modes and flavors

Two architectural modes: legal and general. They differ in schema (Legal adds Authority binding, jurisdiction, court hierarchy hooks, standard-of-review and burden fields). Within general, a flavor — personal or academic — is metadata, not schema; it changes Authority node behavior, Premise kind vocabulary, and UI labels. In personal flavor, Authority is hidden by default but available per-frame and labeled "Source." Two operating modes, architecturally enforced: Frame Building (the realm of validity; constructs the logical structure without premises) and Argument Running (the realm of soundness; attaches premises and traces conclusions).

Node types and edge types

Node types: RootQuestion, SubQuestion, Term, Interpretation, Checkpoint, LogicalGate (AND / OR / NOT / IF-THEN / UNLESS — discriminated union with named slots), Conclusion, Premise, Authority. Edge types: DECOMPOSES_INTO, TURNS_ON, INTERPRETED_AS, LEADS_TO, FORECLOSES, GATES, ANSWERS, SUPPORTS, CONTRADICTS, CITES, BINDING_IN, DISTINGUISHED_BY. FORECLOSES is an edge only (not a logical gate). ALTERNATIVE_PATH_TO was removed (two paths to one Conclusion are expressed by two LEADS_TO edges). CITED_BY was renamed CITES for clarity. Every node carries an optional notes: text field (A12) for working thoughts, not used by path tracing.

Identifier model

Every persisted entity has a stable opaque id (UUID v4 from day one) plus an optional human-readable slug. References use id; slug is for logs, debug UIs, and exports. There is no slug-to-UUID migration step; the original migration plan was replaced with a config flag for whether to auto-generate slugs.

Per-node satisfaction criteria — the "options box" (A3)

Each node type carries a configurable SatisfactionPolicy drawn from a library of approximately twelve named conditions covering structural status (premise_attached, interpretation_selected, all_children_resolved, path_complete), quality (not_contradicted, premise_kind_in[list], burden_met(level)), authority (authority_required, authority_binding, not_distinguished), procedure (standard_of_review_applied), and the universal not_foreclosed. Composition is AND by default with at most one explicit AnyOf group; no nesting, no NOT (negations are explicit conditions). Default policy is per-node-type and editable at the frame level; Checkpoint, Question, and Interpretation nodes can override per-instance. Premise and Authority have no satisfaction policy (they are inputs/sources). Burden of proof and standard of review live inside this framework. Full library, default tables, and override resolution are in stream_a_resolutions_v1.html A3 and stream_b_schema_v1.html SatisfactionPolicy. Stream D specifies which conditions are offered in the options-box editor by mode/flavor (legal-only conditions are hidden in general mode; authority_required is opt-in for academic flavor). Stream E specifies the editor itself: a three-tier display (effective policy / source / edit) with a multi-select picker filtered by mode/flavor.

Status enum and precedence

Five statuses: open (≥1 condition unmet), satisfied (with sub-flags via_binding_authority / via_persuasive_authority exposed for legal-mode display), contested (premises on both sides, neither dispositive), foreclosed (the not_foreclosed condition failed), not_applicable (node lives on a path never reached). Precedence is fixed: foreclosed > not_applicable > contested > open > satisfied. Status assignment is a single deterministic pass.

Conclusion content (A1)

A Conclusion stores statement, direction (mode-aware enum: legal mode uses a fixed list including favors_plaintiff, affirm, reverse, remand, dismiss, etc.; general mode references a per-frame Position table), optional reasoning_summary, optional tags, optional remedy (legal mode only), optional notes. Confidence is computed from the session's status map and displayed on the Conclusion, never stored on the node. This prevents asserting certainty the structure does not support. Promotion of remedy to a separate node type is reserved for v2.

Checkpoint UX (A2)

Checkpoints have one of three answer types defined at frame-build time: boolean, multiple_choice (author-defined options each mapped to a target branch; default for new Checkpoints is multiple_choice with two options "yes"/"no"), or graded (three-state: met / not met / unclear, where the middle state routes to contested). Every answer requires the user to attach a Premise (created on the fly: text statement + optional source/Authority). In legal mode, Checkpoints can be flagged requires_authority: true, requiring the attached Premise to reference an Authority node. Checkpoints are atomic; multi-element legal tests (e.g., negligence: duty/breach/causation/damages) are modeled as multiple Checkpoints, not one Checkpoint with multiple parts. A non-deterministic freeform answer type is reserved for v2. Stream D specifies answer_type conversion rules (boolean ↔ multiple_choice ↔ graded) for use in Frame Building.

Premise typing (A11)

Premise.kind is an enum whose vocabulary depends on Frame mode/flavor (legal: stipulated/found/disputed/procedural; academic: empirical/definitional/normative/stipulated; personal: observation/value/assumption/stipulated). The kind feeds the contested status calculation and the display layer.

Term scope and reuse (A10)

Term.linked_to is a nullable term_id field. When a user creates a Term whose name matches an existing Term in the same frame, the UI offers a checkbox to share Interpretations with the existing Term. Linking is one-way and cannot create cycles. Linked Terms inherit Interpretations and selection state at runtime; they may have their own options_box override.

Resolution order semantics (A7)

Order is purely a foreclosure mechanism. Outside foreclosure, resolution order has no effect on what Interpretations are available for later Terms. Frame authors who want "Term X interpretation A constrains Term Y" must express that structurally — by routing through a SubQuestion containing Y with the constrained Interpretation set. Silent coupling is forbidden: the frame's logic must be readable from the graph, not from a hidden ordering variable. The order field on Term is retained for the foreclosure resolver and default UI presentation order.

Jurisdiction (A8, Legal Mode)

Jurisdictional pre-checks (personal, subject-matter, etc.) are ordinary SubQuestions with an is_jurisdictional: true flag. They use the same Term/Interpretation/Checkpoint machinery. They render first in UI ordering (Stream D's interview ordering algorithm honors this with a priority bubble; Stream E renders the bubble as a "Jurisdictional questions" section header in the interview pane) and are visually distinguished. The frame completeness checker warns — does not block — when a Legal-mode frame has no jurisdictional SubQuestions.

Cross-frame references (A6) — deferred to v2

Live cross-frame references are deferred to v2. The v1 substitute is frame templates: at frame creation the user can pick "start from existing frame" and get a deep copy. No live link, no version coupling. The id design is template-friendly (random UUIDs, no positional dependencies), so v2 cross-frame edges layer in without restructuring. Stream D specifies the create-from-template action (deep-copy regenerates every node and edge id, rewriting internal references in one pass); Stream E places it in the new-frame wizard (E7) and (per I.10) also exposes it on the Home page as "+ New frame from template"; Stream H wraps the deep-copy in a single Dexie transaction (H1 atomicity contract § 2). A symmetric "+ Blank frame" path is exposed by Repository.createBlankFrame (H1 atomicity contract § 4) per F-009.

Token-optimization principles

  1. Module isolation. Each coding session reads only its module's spec, dependent module READMEs, and the project knowledge minimums.
  2. Module READMEs as cross-session compression. Coding sessions write a README per module documenting the public API. Subsequent sessions read READMEs, not specs.
  3. Pre-built scaffolding. Stream I session I.1 produces the project skeleton so coding sessions don't waste tokens on configuration setup.
  4. Acceptance tests as contracts. Each Stream I module spec includes Vitest acceptance tests; coding sessions implement until tests pass. The tests are compressed contracts that bound implementation freedom.
  5. No re-reads of design-phase docs. current_state.html consolidates the design; coding sessions do not read original stream documents (Article XI § 2). Stream I per-module specs are the design surface for coding.
