Stream E — UI / UX Design

Companion to stream_a_resolutions_v1.html, stream_b_schema_v1.html, stream_c_runtime_v1.html, and stream_d_modes_v1.html  ·  Session date: May 10, 2026  ·  Version 1

This document specifies the user-facing surface of the application at the wireframe and interaction-flow level. It covers Stream E end-to-end: graph layout strategy (E1), Frame Building editor (E2), Argument Running interface (E3), status visualization (E4), version history (E5), mode toggles (E6), and onboarding (E7). It is the bridge between Stream D's action inventories and Stream H's eventual tech-stack selection. Visual aesthetics — exact color values, typography, animation curves — are agent-discretionary defaults here, intended to be revised by user direction in subsequent design work without affecting downstream streams. What this document does fix is the structural layout of every primary screen, the interaction model for every action surfaced by Stream D, and the visual semantics of the runtime status output from Stream C.

How to read this

This document is screen-and-flow oriented. Each section can be read in isolation: a future session needing only the version-history flow can skip to E5; one needing only the status palette can jump to E4. References to other streams use section ids: D1 = Frame Building actions, C5 = frame completeness checker, A9 = conditional output presentation. Where a behavior is fully specified by an upstream stream, this document specifies its UI surface only — which control invokes it, where the control lives, what feedback the user gets — without restating the underlying logic.

Conventions:

  • Screen = a primary application view occupying the main content area (e.g., Home, Frame Building editor, Argument Running session).
  • Pane = a major region within a screen (left, center, right, top bar, footer).
  • Surface = a localized UI feature within a pane (inspector field, validation chip, version-history tree).
  • Wireframe = a structural sketch of a screen using ASCII layout. Pixel dimensions are illustrative; actual sizing is a downstream concern.
  • Affordance = the visible cue that an action is available (a button, a drag handle, a hover state).

Decisions made in this session

The decisions below are agent-discretionary under Article V § 2(c) (implementation choices, framework selections at the UI level). Each is justified against Article III and Article II where applicable. None reverses a prior stream decision; none weakens an architectural commitment; none forecloses a use case.

  1. Top-down hierarchical (Sugiyama-style) graph layout is the default. Frames are DAGs rooted at a RootQuestion; legal and academic readers move top-down through a logical decomposition. Top-down layout matches that mental model and reads naturally on a wide screen. The frame's x, y presentation hints (B1) override the auto-layout per node when set, allowing the user to fine-tune.
  2. Frame Building uses a three-pane editor; Argument Running uses a two-pane session. Frame Building needs a node palette, a graph canvas, and an inspector — three regions of work. Argument Running uses only the frozen frame for context, so a wider output pane and a focused interview pane suffice. Distinct layouts also reinforce the operating-mode distinction visually (Article II § 1).
  3. The graph canvas is the canonical view in both modes. Other views (decision tree, prose summary, version history, validation panel) are toggleable overlays, drawers, or tabs that sit alongside or atop the graph. The graph never disappears as the spatial anchor for the user's understanding of the frame.
  4. Status colors are paired with status icons in every rendering. Color alone is insufficient (color-blindness, monochrome printouts, screen-reader contexts). The five statuses each get a distinct icon as well as a distinct hue.
  5. Every node's status is visible at a glance from the graph; details are one click away. Status badges sit on every node in both modes. Hovering reveals the failed_conditions list. Clicking opens the inspector with full status detail. This honors Article III § 3 (auditability): the runtime output is never hidden behind a separate "compute results" screen.
  6. Inline text editing for statement-like fields; inspector for everything else. RootQuestion statement, SubQuestion statement, Term name, Interpretation statement, Checkpoint question, Conclusion statement — all editable by double-clicking the node text. Other fields (mode-aware, enum-typed, structured) live in the inspector. This shortens the path for the most common edit (rephrasing) without making the inspector irrelevant.
  7. Edge creation is drag-from-source-to-target, with a type picker offering only valid types. Hover over a node, a small connector handle appears; drag from the handle to another node; on release a popup shows the edge types valid for that source/target pair per B2. This avoids guessing at compatibility and surfaces the schema's connection rules implicitly.
  8. The interview pane lists open items; the path overlay highlights them on the graph. The interview pane (E3) and the path overlay view (A9 third view) are coupled. Selecting an item in the list highlights its node on the graph; clicking the highlighted node opens the same item editor. The list and the graph reflect the same underlying state.
  9. Validation surfaces in three places, with consistent severity icons. (a) Per-node chips on the graph for node-scoped rules; (b) a validation panel listing all current rule firings; (c) the inspector for whichever node is selected. Errors and warnings use the same icons and language across all three.
  10. Version history is a side pane, not a modal. Modals interrupt; users frequently want to compare a draft state to an earlier one while continuing to think. A right-side pane preserves the working canvas while showing the history tree alongside.
  11. Operating-mode toggle is the leftmost top-bar control; architectural-mode change is in frame metadata. Operating-mode toggling (Frame Building ↔ Argument Running) is frequent; architectural-mode change (legal ↔ general) is rare and structural. Their placement reflects their frequency and weight.
  12. Onboarding is opt-out, not opt-in. A first-time user is enrolled in a guided first-frame walkthrough by default. The walkthrough is dismissible at any point and never blocks. Users who want to skip click a single "skip introduction" link.

E1 — Graph layout strategy

The frame is a directed acyclic graph rooted at a single RootQuestion (per V-FR-1). The layout strategy is the visual translation of that graph into something legal and academic readers can scan, navigate, and edit.

Layout algorithm

The default is a top-down hierarchical (Sugiyama-style) layout: ranks assigned by topological depth from RootQuestion, nodes within a rank ordered to minimize edge crossings, edges drawn as orthogonal or smoothed-bezier paths. Frame templates and freshly imported frames receive an auto-layout pass on first open. Each node carries optional x and y presentation hints (B1); when set, they override the auto-layout for that node. The user can drag any node to reposition it; the drag persists the new x, y values to that node's presentation hints, which causes that node to be respected on subsequent auto-layouts (the layout engine treats user-positioned nodes as fixed anchors).

An "auto-arrange" command in the canvas toolbar discards the user's x, y overrides for all nodes (with confirmation) and reruns the layout from scratch.

For very large frames (hundreds of nodes), the layout supports collapsing a SubQuestion's subtree via the collapsed presentation hint. A collapsed SubQuestion renders as a single node with a "+N nodes" badge; expanding restores full layout. Collapse state is per-user, persisted on the SubQuestion's presentation hint.

Node shapes by type

Each node type has a distinct shape so the user reads the frame's structure without consulting a legend. Shapes are chosen to reinforce the type's semantic role — questions are open (rounded), terms are operative (rectangular), conclusions are closed (double-bordered terminals), gates are diamond/special, authorities are document-shaped.

Edge labels and direction

Every edge type carries an inherent meaning shown by an icon at its midpoint (small, monochrome, only on hover for the most common types to reduce visual noise). User-set label fields (B2) render alongside the type icon when present. Direction is shown by the arrowhead at the target end. For CheckpointOption virtual edges, the option's user-facing label is always shown ("yes" / "no" / "unclear" / etc.).

Selection, hover, focus

Hovering a node lightly emphasizes it (background tint shift), surfaces the connector handle for edge creation, and shows the failed_conditions tooltip if status is open or contested. Hovering an edge brightens the line and shows its type tooltip. Selecting a node (click) outlines it, opens the inspector to that node's details, and dims the rest of the canvas slightly. Multi-select via shift-click or rubber-band drag; multi-select shows a bulk action toolbar (delete, edit shared fields).

Zoom, pan, minimap

Standard canvas affordances: pinch / scroll-wheel zoom, drag-to-pan, fit-to-screen button, 100% zoom button, minimap in the bottom-right corner showing the full frame at scale with the current viewport boxed. Keyboard shortcuts: +/- zoom, arrow keys pan, F fit-to-screen, 0 100%.

Argument-layer overlay rendering (Argument Running only)

In Argument Running, the frame canvas is the same hierarchical layout, but the argument-layer overlay activates: ANSWERS edges from Premises to Checkpoints, SUPPORTS / CONTRADICTS from Premises to Interpretations or Conclusions, and any session Authorities sit alongside their attached Premises. To prevent visual overload, argument-layer overlays use the argument inspector (a panel) as their primary surface; on the canvas, argument-layer attachment is signaled by a small "premise count" chip on each Checkpoint, Interpretation, or Conclusion. Clicking the chip filters the canvas to just that node's argument-layer attachments.

Left pane — node palette + outline tree

Two stacked sections. The palette on top: one button per node type, visibility filtered by mode/flavor (e.g., Authority hidden in personal flavor unless opted in). Buttons are draggable: drag onto the canvas to create a node at the drop location. Clicking instead of dragging creates a node at the canvas center with default fields, ready to be connected.

The outline tree below the palette: a hierarchical text tree of the frame, rooted at the RootQuestion, descending by structural edges (DECOMPOSES_INTO, TURNS_ON, LEADS_TO). Each row shows: node-type icon, statement preview, status badge (matching graph). Clicking a row selects the node on the canvas and scrolls it into view. The outline tree is the keyboard-accessible alternative to graph navigation for users who prefer linear reading.

Center pane — graph canvas

The canvas hosts E1's graph rendering. The canvas has its own toolbar at the top edge:

  • Zoom controls (zoom in, zoom out, fit-to-screen, 100%).
  • Auto-arrange button: discards x, y overrides and reruns layout (with confirmation).
  • Layer toggle for FORECLOSES edges (visible / dimmed / hidden) — useful for working on the structural skeleton without foreclosure noise.
  • Search box: full-text across node statements, names, citations, tags. Matches highlight on the canvas; clicking a result selects.

Right-clicking the canvas opens a context menu with "create node here" entries for each valid node type (filtered by mode/flavor). Right-clicking a node opens a node-specific menu (edit, duplicate, delete, edit notes, edit options_box, ...). Right-clicking an edge opens an edge-specific menu (edit fields, edit label, edit notes, delete).

Right pane — inspector

Context-sensitive. Default state (nothing selected): shows frame-level information — title, description, mode + flavor, default_satisfaction_policies, tag list. When a node is selected: shows that node's full editable field set, organized in sections (statement / structural fields / mode-aware fields / options_box / notes). When an edge is selected: shows the edge's type, source/target (read-only — they are immutable per D edge actions), label, type-specific fields (e.g., FORECLOSES scope, SUPPORTS weight), and notes. When multiple items are selected: shows shared fields where defined (e.g., bulk tag edit) and a count of items.

Mode-aware visibility (D4/D5) governs which fields appear. A field hidden by the current mode is not rendered; the schema's value persists silently. When the user changes mode, the previously hidden fields reappear (and may surface in the mode-change scan if they conflict).

Edge creation flow

  1. User hovers a node; a small connector handle appears on the node's edge (one handle that follows the cursor around the node's perimeter).
  2. User drags from the handle. A live edge previews from the source to the cursor.
  3. On hover over a candidate target, the target highlights green if at least one edge type is valid (per B2); red if none is valid (with tooltip explaining why).
  4. On release over a valid target, a popup appears showing the valid edge types as buttons. The user clicks one.
  5. On release over an invalid target or empty space, the preview cancels.

For LogicalGate slot connections (which are not edges but field assignments per B1.6), the same gesture is used; the popup offers the gate's slots ("set as antecedent" / "set as consequent" / "add to inputs[]" depending on gate_type) instead of edge types. For CheckpointOption routing (which is a field, not an edge), the same gesture is used from a Checkpoint to a target node; the popup offers "wire option X to this target," with one option per existing CheckpointOption.

Validation surface

Three coordinated places per decision #9 above:

  • Per-node chips: a small badge on any node implicated in a validation rule firing. Hover to read; click to open inspector with the field implicated highlighted.
  • Validation drawer at the bottom of the editor: collapsed by default, expanded by clicking the top-bar validation indicator. Lists every current rule firing with severity, rule id (e.g., V-FR-3), affected nodes (linked), and suggested fix.
  • Inspector validation block: when a node is selected, any rules firing on that node show in a "validation" section of the inspector with the same content.

Validation runs continuously in editing strictness (per D1). The pre-mode-transition strict pass runs only when the user clicks Switch to Argument Running; on failure, errors block transition and show in the validation drawer.

Bulk actions and deletion

Multi-select via shift-click in the outline tree, shift-click on the canvas, or rubber-band drag on the canvas. Selected items show in a floating action bar at the top of the canvas: delete, group-edit-options-box, set-shared-tag (where applicable). Deletion always opens a confirmation dialog summarizing the cascade set per D1's cascade-with-confirmation rule (decision #2). The dialog is structured: "Deleting SubQuestion: Duty will also remove 4 Terms, 11 Interpretations, 3 Checkpoints, and 2 Conclusions" with an expandable list of every node and edge that will be removed.

Editing options_box (SatisfactionPolicy)

The options_box is per-node-type-default, frame-overridable, and per-instance-overridable for Checkpoint, Question, and Interpretation (per A3). The inspector exposes it as an "options_box" section with three tiers shown:

  • Effective policy: the policy that actually applies to this node, computed from the override chain.
  • Source: where the effective policy comes from (per-instance override / frame default / library default).
  • Edit: the user can edit at this instance level (creating a per-instance override) or at the frame default level (changing the frame default for this node type).

The condition library renders as a multi-select picker with the conditions valid for the current mode/flavor (per D4/D5 — legal-only conditions hidden in general; authority_required opt-in for academic). Selected conditions show as a vertical AND list with at most one explicit AnyOf group permitted (per A3 composition rules: AND default, no nesting, no NOT). The user adds/removes/configures conditions through the picker; the visual structure of the policy is always linearly readable.

Frame Building entry surface

From Home, the user enters Frame Building by: opening an existing frame, clicking "New frame" (which runs the new-frame wizard from E7), or clicking "New frame from template." Once entered, the user remains in Frame Building until explicitly transitioning to Argument Running or navigating away.

Left pane — interview pane

The interview pane is the open-items list as derived by D2's algorithm. The list is rendered top-to-bottom, with the recommended-next item highlighted (background tint + larger weight + a "recommended next" label). Each row shows: status badge, node-type icon, statement preview, parent-context breadcrumb (e.g., "Negligence → Duty → Reasonableness").

Above the list, two affordances:

  • Filter (multi-select chips): by node type (Checkpoint / Term / Interpretation), by jurisdictional priority, by status. Default: all types and priorities, status open only.
  • Search: full-text across the open-items list for fast keyboard navigation.

The jurisdictional priority bubble (D2) renders as a section header above the rest of the list — "Jurisdictional questions" — with the jurisdictional items grouped under it. Non-jurisdictional items follow under "Merits." If no jurisdictional items exist (e.g., none in the frame, or all resolved), the section header is omitted and the list flows from the top.

Clicking a row opens the item editor (right side, drawer-style or in-place modal — see next section).

As the user resolves items, the list refreshes (instantly; recompute is automatic per D decision #6). A small "recompute" indicator pulses briefly to acknowledge the change, then settles. The recommended-next item updates accordingly.

Item editor — per node type

Each open-item type opens a focused editor specific to that type. The editor occupies the right portion of the screen, narrowing the output viewer; the user can dismiss it (Esc, click outside, X button) and reopen by clicking the item again.

Checkpoint editor

  1. Header: the Checkpoint's question, with parent-context breadcrumb. Editable in Frame Building only; here read-only.
  2. Answer selector: boolean / multiple_choice / graded as configured. Clicking selects an option; the selection highlights.
  3. Premise authoring: a text field for Premise statement, a kind picker (mode/flavor-aware vocabulary per A11), optional source, optional Authority attachment. The text field offers reuse suggestions inline as the user types, ranked by similarity and recency (per D D2 reuse rule). Selecting a suggestion attaches the existing Premise instead of creating a new one.
  4. Authority attachment (legal mode + Checkpoint.requires_authority = true): an Authority picker showing frame-layer Authorities first, then session-layer Authorities, then "create new session Authority" inline.
  5. Save / cancel. Save creates the ANSWERS edge and CheckpointResponse and closes the editor; the interview list refreshes.

Term editor (Interpretation selection)

  1. Header: Term name, parent-context breadcrumb. Editable in Frame Building only.
  2. Linked-target redirect: if the Term has linked_to set and the target is reachable, the editor displays a notice ("This Term inherits its selection from target Term") and a "go to target" button. Saving a selection here raises V-ARG-8 (warning) per Stream C.
  3. Interpretation picker: list of the Term's Interpretations (children via INTERPRETED_AS), each with statement, attached Authorities (CITES) and Premise count. Multi-select permitted (recording alternative paths).
  4. Optional supporting evidence: SUPPORTS / CONTRADICTS attachments for each selected Interpretation (per Stream C InterpretationSelection schema). Same Premise authoring affordances as the Checkpoint editor.
  5. Optional notes (free text per A12).
  6. Save / cancel. Save creates or updates the InterpretationSelection record; if any selected Interpretation carries FORECLOSES edges or the Term is dispositive, the foreclosure resolver runs and the user is shown a brief "these branches were foreclosed" summary.

Interpretation evidence editor

Reached when an Interpretation is in the open-items list (its options_box requires authority/premise support that's missing). Same Premise authoring + Authority attachment affordances as above, scoped to SUPPORTS/CONTRADICTS edges from new or reused Premises to this Interpretation.

Right pane — output viewer (three views)

Per A9, three tabs render the same underlying ConditionalOutput object (B5):

  • Path overlay (default if any path is at least partly active): the frame canvas, dimmed; the active path is rendered at full opacity with status colors; foreclosed branches are faded and strike-through; not_applicable nodes are dimmed further; the recommended-next node pulses softly. Clicking any node opens its editor (if open) or its inspector (if not). Hovering shows the node's status detail.
  • Decision tree: a collapsed view rooted at the recommended-next branch point, showing only nodes at unresolved branches and the terminal Conclusion under each branch. Each branch terminates in a Conclusion preview (statement + direction). Useful for "where could this go from here."
  • Prose: a deterministic prose summary generated by templating (no LLM). Reads "If Term X is interpreted as A, the conclusion is Y. If B, then conditional on Checkpoint Z: ..." For complete sessions, reads as a single conclusion paragraph. For incomplete sessions, reads conditionally with explicit "if/then" structure. Exportable (copy to clipboard, copy as Markdown).

Tab choice is per-user (A9) with per-frame override; the user's choice is preserved across sessions.

Bottom panel — Premise pool and Session Authorities

Collapsed by default; expandable by clicking the bottom edge. When expanded, two side-by-side lists:

  • Premise pool: every Premise in the session (the premises[] array). Each row: statement, kind chip, attached-edges count (e.g., "answers 1, supports 2"). Clicking a Premise opens its inline editor and highlights its attachments on the canvas. Useful for triaging orphaned Premises after a frame migration.
  • Session Authorities: every session-layer Authority (the session_authorities[] array). Same row pattern: citation, attached-edges count. Clicking opens the Authority editor.

"Add Premise" and "Add Authority" buttons in the bottom panel allow the user to create entries here proactively (e.g., paste in a fact pattern's worth of Premises to draw on later); these are unattached Premises until linked through an answer or evidence action.

Argument Running entry surface

From Frame Building's "Switch to Argument Running" button (subject to C5 strict gate), or from Home's session list, or by creating a new session against a frame. New-session entry: title required; description optional; opens immediately into the empty session with the interview pane populated by all currently-open items.

E4 — Status visualization

Status is the single most important runtime output: every node's status communicates whether the user has more work to do, whether a path is closed, and what the conclusion-so-far is. The visualization treats this with care.

Status palette

The five statuses (per Stream C) get a distinct color and a distinct icon, paired everywhere.

StatusColor (default)IconMeaning
satisfiedGreen (medium saturation)Checkmark in circleAll conditions met; the node contributes to a determinate path.
openNeutral grayEmpty circleOne or more conditions unmet; user input pending.
contestedAmberTwo arrows clashingPremises on both sides, neither dispositive.
foreclosedDark gray with red accentX with strikethroughPath is closed by FORECLOSES or dispositive Term.
not_applicableLight gray (faded)Dimmed circleNode lives on a path never reached.

Color values are agent-discretionary defaults; they may be revised in a subsequent visual-design pass without affecting downstream streams. The pairing rule (every color paired with an icon) is fixed.

Sub-flag display (legal mode)

Per Stream C, a satisfied NodeStatus carries a via sub-flag of via_binding_authority or via_persuasive_authority in legal mode. The visualization:

  • via_binding_authority: a small "⚖ B" chip on the node's status badge.
  • via_persuasive_authority: a small "⚖ P" chip.
  • In non-legal mode, no chip; the sub-flag is computed but not surfaced.

Indeterminate gates

A LogicalGate whose inputs are partly unresolved produces an indeterminate result (per Stream C). Its own NodeStatus is open with failed_conditions: ["inputs_indeterminate"]. Visualization: a dashed border on the gate node + the open status badge. The gate's output edge (GATES) is rendered with a dashed style to signal "this output is conditional on unresolved inputs." Hovering the gate shows: "Inputs indeterminate: Term X unselected, Checkpoint Y unanswered."

failed_conditions tooltip

Every open or contested node, on hover, shows a tooltip listing the conditions that failed:

  • premise_attached: "No Premise attached."
  • authority_required: "Authority required but not provided."
  • burden_met(preponderance): "Burden of proof not met (current: scintilla)."
  • not_distinguished: "Cited Authority is distinguished by another Authority."
  • ...and so on for the full library.

The tooltip text is generated by deterministic templating from the condition id and parameters. The same text appears in the inspector's status section when the node is selected.

Status transitions

When a recompute changes a node's status, the badge animates briefly: a soft fade-and-scale on the status icon (~150ms), to give feedback that something updated. No celebration animations on satisfaction; the application is for serious analysis. The animation is dismissable via a "reduce motion" accessibility preference.

Status legend

A "legend" affordance in the canvas toolbar opens a small popover showing every status icon and its meaning, plus the legal-mode sub-flags. Available in both modes. Useful for first-time users and as a printed reference.

Status in non-canvas contexts

The same status palette is reused everywhere status is shown: the outline tree, the interview pane, the validation drawer, the version-history diff view. Consistent visual semantics across surfaces is the rule.

E5 — Version history UI

Every save is preserved (Article II § 4). The version-history UI exposes the full history per Frame and per ArgumentSession, with milestone markers, preview, restore, and diff.

Entry point

A clock-rewind icon in the top bar of both Frame Building and Argument Running. Clicking opens the version-history side pane (decision #10 — pane, not modal).

Side pane layout

The side pane occupies the right ~30% of the screen, pushing the inspector aside (Frame Building) or narrowing the output viewer (Argument Running).

+----------------------------------------+ | Version history · Frame: Negligence | +----------------------------------------+ | [filter: milestones only / all] | +----------------------------------------+ | ● v8 · current · 2 min ago | | ● v7 · 12 min ago · auto-save | | ★ v6 · "Added causation branch" · 1d | | ● v5 · 2d ago · auto-save | | ● v4 · 2d ago · auto-save | | ★ v3 · "Initial complete frame" · 5d | | ● v2 · 5d ago · auto-save | | ● v1 · 5d ago · created | | | | (branches show as indented children | | under the version they forked from) | | | +----------------------------------------+ | [Selected: v6] | | [Preview] [Restore] [Compare to v8] | +----------------------------------------+

Each row: version number, timestamp (relative + absolute on hover), milestone star (if is_milestone: true), change_summary (if present, otherwise "auto-save"). The current version is marked. Branches (created by restoring an older version, which forks per A4 Article II § 4) render as indented children of the version they forked from.

Milestones vs. auto-saves

Milestones (★) are visually prominent. A "milestones only" filter chip at the top hides auto-saves so the user sees only the meaningful waypoints. Default view is "all" so the full history is visible.

Preview

Selecting a version and clicking Preview opens that version in a read-only canvas overlay. The current canvas dims and a banner reads "Previewing version 6 (read-only)." Clicking "Exit preview" returns to the working version. No state changes during preview.

Restore

Restoring an ancestor creates a new FrameVersion (or ArgumentSessionVersion) whose contents equal the chosen ancestor, with parent_version_id pointing at the ancestor and a change_summary noting "restored from version N." The new version becomes current. The history pane updates to show the branch. Per Article II § 4, no version is deleted by this operation.

Confirmation dialog: "Restoring version 6 will create a new version (v9) and switch to it. The current version (v8) and all later versions are preserved." Single OK button (the operation is reversible by restoring back).

Compare

Selecting a version and clicking "Compare to vX" opens a diff view in the side pane (replacing the version list temporarily). The diff is structured:

  • Nodes added / removed / edited: each node listed with its statement and a short summary of what changed.
  • Edges added / removed / edited: same.
  • Frame metadata changes: title, mode, flavor, default policies, jurisdiction_default, etc.

Clicking a diff entry navigates the canvas to the implicated node (highlighting it in both versions). The diff view does not show pixel-level visual diffs; it shows structural and field-level changes. A "back to history" button returns to the version list.

Per-mode scope

In Frame Building, the version history shows FrameVersions. In Argument Running, the history shows ArgumentSessionVersions; a second tab in the side pane shows the FrameVersion history (read-only, since frame edits don't happen in Argument Running). This second tab is how the user inspects what version of the frame the session was created against and what's changed since.

No deletion affordance

The version-history UI has no "delete version" action. Per Article II § 4, no save is deleted within a Frame's lifetime. The only path to losing version data is deleting the entire Frame itself (D1, gated on no live sessions). This is by design.

E6 — Mode toggles

Two distinct kinds of mode change with different visual weight: operating mode (Frame Building ↔ Argument Running, frequent, top-bar) and architectural mode (legal ↔ general, with flavor as a sub-choice within general; rare, in frame settings).

Operating-mode toggle

Top-bar leftmost control. A two-position segmented toggle: [Frame] [Argument]. The active mode is highlighted; clicking the inactive label triggers the transition.

DirectionBehavior
Frame → ArgumentRuns C5 strict (D3). On pass: prompts for session choice (open existing on current FrameVersion / open existing on older / create new); transitions to Argument Running. On warning: acknowledgment dialog, then proceeds. On error: blocks; opens validation drawer with errors and links.
Argument → FrameAlways allowed. Returns to Frame Building. The session is preserved; on return, drift behavior applies if the underlying Frame has advanced.

Visual differentiation between operating modes: a subtle chrome accent shift. Frame Building uses one accent hue (e.g., neutral / build-ish); Argument Running uses a slightly distinct accent (e.g., focused / session-ish). The shift is minimal — enough to confirm the mode at a glance, not enough to feel like two different applications. Specific hues are agent-discretionary defaults.

Architectural-mode change

Surfaced through the frame settings panel (gear icon → "Frame settings" → "Mode"). Not surfaced in the top bar; it is rare and consequential.

The settings panel shows the current mode (and flavor if general) and a "change mode" button. Clicking opens the mode-change dialog, which:

  1. Asks the user to confirm the target mode (legal ↔ general; if general, also flavor).
  2. Runs the D3 mode-change scan in the background.
  3. Presents the results: a list of items grouped into blocking (Conclusion direction mismatches, per D3) and advisory (other field changes, persisted as inactive metadata).
  4. Each blocking item has an inline editor: e.g., a Conclusion direction picker for the new mode. The user resolves blocking items inline.
  5. When all blocking items are resolved (or there are none), the "Commit mode change" button enables. Advisory items are listed for awareness; the user does not need to address them.
  6. On commit, the frame's mode field updates, all advisory items become inactive metadata, the editor refreshes with new mode-aware visibility, and a new FrameVersion is saved with change_summary "mode changed: legal → general."

The user can cancel at any point during the dialog; no changes are committed until "Commit mode change" is clicked.

Flavor change (within general)

Same surface (frame settings → "Flavor") with lighter mechanics: the change commits immediately (per D3), an advisory list shows what's affected, and the editor refreshes with the new flavor's visibility/labeling. No blocking step. Per A4, flavor is metadata, not schema.

Mode + flavor chip in top bar

For at-a-glance awareness, the top bar shows a chip: "legal" or "general · personal" or "general · academic". The chip is non-interactive in v1 (clicks open the frame settings panel rather than offering a quick toggle); a quick-toggle gesture is reserved for v2 once usage patterns clarify.

E7 — Onboarding and first-frame experience

Per decision #12, onboarding is opt-out, dismissible at any point, never blocks. It is the first thing a new user encounters and the source of conceptual context (frame/argument split, the flavor/mode system, the per-node satisfaction model).

First-time welcome screen

A single-screen introduction, shown the first time a user opens the application.

  • Title: "Argument mapping that separates structure from substance."
  • Three short sections: (a) "Build a frame" — the logical skeleton of an argument; (b) "Run a session" — apply premises to reach a conclusion; (c) "Reuse and refine" — frames are independent of any single argument.
  • Two buttons: "Start guided first frame" (default) and "Skip introduction."

Guided first-frame walkthrough

If the user starts the guided flow, a wizard walks through frame creation with explanatory captions:

  1. Pick a mode: Legal or General. Caption explains what each is for. Defaults to General if the user is uncertain.
  2. If General, pick a flavor: Personal or Academic. Caption explains the distinction.
  3. If Legal, set a default jurisdiction: simple picker (level + region) with a "skip for now" option.
  4. Title and description: required title, optional description.
  5. Optional: start from a template: a list of available frame templates (the user's own frames available as templates per A6, plus any starter templates we ship — none ship in v1 unless we add them). User can skip.
  6. Frame created: the user lands in Frame Building with a stub RootQuestion ready to be filled in. A coachmark on the RootQuestion explains: "Every frame starts with a Root Question. Click to edit; this is the question your analysis answers."

Coachmarks

Coachmarks are small overlays appearing the first time a user encounters a primary affordance:

  • The first time the user opens the inspector — coachmark on it: "This panel shows the fields for whatever you've selected. Mode-specific fields appear or hide automatically."
  • The first time the user creates an edge — coachmark on the connector handle: "Drag from this handle to another node to create an edge. Only valid edge types will be offered."
  • The first time the user clicks "Switch to Argument Running" — coachmark on validation: "If your frame has errors, the switch is blocked. Warnings let you proceed."
  • The first time the user opens version history — coachmark: "Every save is preserved. Star icons mark milestones you've named."
  • The first time the user enters Argument Running — coachmark on the interview pane: "These are the open items. Top is recommended next, but you can address any of them."

Each coachmark is dismissed by clicking "got it" or by acting on the affordance. Coachmarks are tracked per-user and never repeat. A "reset coachmarks" toggle in user settings restores them for testing or for users who want a refresher.

Empty-state guidance

Empty states throughout the application carry instructional copy:

  • Empty Home: "No frames yet. Create your first frame, or start from a template."
  • Empty session list (on a frame): "No sessions yet against this frame. Create one to start running premises."
  • Empty interview pane (a complete session): "All items resolved. Conclusion: X." with a "save milestone" button to mark the completion.
  • Empty validation drawer: "No issues. The frame is ready." (in editing strictness; pre-mode-transition strictness shows the same message when zero errors and zero warnings.)

Help and glossary surface

A help icon (?) in the top-right of every screen opens a slide-in glossary pane with two sections:

  • Frame concepts: definitions for RootQuestion, SubQuestion, Term, Interpretation, Checkpoint, LogicalGate, Conclusion, Authority, Premise. Each entry is one short paragraph.
  • Legal-mode concepts (visible only when mode is legal): jurisdiction, binding vs. persuasive authority, standard of review, burden of proof, holding vs. dictum.

Glossary entries are also reachable inline: hovering over a node-type icon or a legal-term label anywhere in the UI shows a small tooltip with the same one-paragraph definition. Hover-glossary keeps the explanation contextual without forcing a full help-pane open.

Onboarding for returning users on new features

When a new feature ships in a future version, the onboarding system surfaces a one-time notice on first encounter ("New: per-instance options_box override") with a "tell me more" link to a longer explanation. This is the same coachmark mechanism, applied to feature additions rather than first-use.

Article II commitments — compliance check

  • § 1 Frame/Argument Split. Honored. E2 (Frame Building) is structurally distinct from E3 (Argument Running) — different layouts, different available controls, different visual chrome (E6). Argument-layer overlays are explicitly hidden in Frame Building. The operating-mode toggle affordance reflects the data-layer enforcement, not creates it.
  • § 2 Determinism. Honored. The interview pane (E3) renders the open-items list as derived by D2's deterministic algorithm; no UI logic re-orders or filters in a way that breaks determinism. The output viewer renders the ConditionalOutput object verbatim. No build-time AI is introduced into rendering paths; LLM hooks remain Stream G's domain.
  • § 3 Per-Node Satisfaction Criteria. Honored. The options_box editor (E2) exposes the SatisfactionPolicy library, the override chain, and per-instance overrides where Stream A/B permit them. Visual surfacing of failed_conditions (E4) makes the per-node criteria transparent to the user.
  • § 4 Independent Versioning. Honored. E5 surfaces FrameVersion and ArgumentSessionVersion histories separately; in Argument Running, a second tab shows the underlying FrameVersion history read-only. No UI affordance couples the two version streams.
  • § 5 Human-Readable Structure. Honored. Graph layout (E1) is the canonical view; the outline tree provides a linear-readable equivalent. Status visualization (E4) makes the runtime output legible from the graph alone. The CheckpointOption virtual-edge rendering preserves the schema's invariants while keeping routing visually present.
  • § 6 Use-Case Commitment. Honored. Mode-aware visibility (D4/D5) governs every editor surface in E2 and E3. Personal flavor's "enable Source fields" opt-in is exposed in frame settings (E6). No UI surface privileges legal mode at the architecture layer; legal-only affordances appear conditionally.

Article III quality criteria — score

  • § 1 Correctness. Every action surfaced in Stream D maps to a UI affordance in Stream E. Every node type and every edge type has a defined visual representation in E1. Every status value has a defined badge in E4. Spot checks: D1 frame deletion → E2 cascade-with-confirmation dialog; D2 interview ordering → E3 interview pane with jurisdictional bubble; D3 mode-change blocking items → E6 inline-editor resolution UI. Coverage is complete.
  • § 2 Legal Practice Fit. Top-down hierarchical layout (E1) matches how lawyers read a case or a treatise: from question down through analytical structure to conclusion. The interview's "recommended next + free navigation" pattern (E3) matches how lawyers actually engage a brief — a sense of order, but skipping ahead and circling back routinely. Status badges (E4) communicate state without forcing the reader through a separate "results" screen, which would feel computationally alien. Authority chips (E1) and binding/persuasive sub-flag display (E4) speak the legal vocabulary directly. The version history (E5) preserves drafts the way a lawyer expects of a working document. Where idioms differ across modes (E5 milestones, E6 mode chip wording, E7 glossary), the legal idiom is the default in legal mode and academic/personal idioms surface only when the user has selected those flavors.
  • § 3 Determinism and Auditability. The output viewer renders ConditionalOutput from Stream C verbatim, with no client-side reordering or summarization. The interview pane lists items in the deterministic D2 order. Validation surfaces (E2) name the rule id and link to the implicated nodes. No UI element summarizes runtime state in a way that the user cannot audit by clicking through to the underlying node.
  • § 4 Clarity. Three coordinated validation surfaces (E2 decision #9), three coordinated output views (E3, A9), one consistent status palette across all surfaces (E4). The outline tree (E2) gives a keyboard-navigable linear view for users who prefer that mode. Glossary on hover (E7) keeps explanations contextual.
  • § 5 Reversibility. Version history (E5) preserves every save; restore creates a forked branch rather than overwriting. Mode change (E6) commits only after user confirmation and never deletes data. Cascade deletes (E2) require confirmation with explicit cascade summary. Coachmarks are dismissable and resettable (E7).
  • § 6 Interface Fit. Each subsection (E1–E7) is independently readable. A future session needing only the version-history flow can skip to E5; one needing only the status palette can jump to E4. Wireframe sketches use ASCII rather than embedded images so the document renders the same in any viewer. References to other streams use section ids, not quoted content.
