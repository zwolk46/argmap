stream_i_runtime_spec_v1.html

Per-module spec for src/runtime/  ·  Stream I session I.3  ·  May 10, 2026

This document is the implementation source of truth for src/runtime/. It commits per-file TypeScript signatures, the five-phase compute pipeline (validate → resolve foreclosure → compute reachable + active → compute status map → generate output), the LogicalGate evaluator with its named-slot semantics, the per-condition implementations of the 12-condition library, the C6 authority-binding determination, the F5 stub structure, the determinism implementation discipline, and Vitest acceptance tests for the module. The cross-module surface (what other modules import from @/runtime) is pinned in stream_i_contracts_v1.html Contract 2; this spec adds the per-file organization, internal types, and behavioral contracts the coding session implements against.

The runtime is the determinism boundary of the application (Article II § 2). src/runtime/ imports only @/schema and standard JavaScript built-ins; it imports nothing from @/persistence, @/ui, @/state, @/llm-hooks, @/modes, @/layout, React, Dexie, Zustand, ELK, or any third-party library that reads the clock or environment. Three tooling gates shipped in the I.1 skeleton enforce this: ESLint no-restricted-imports (the module boundary), the custom argmap-determinism/no-unsorted-iteration rule (sort-before-iterate), and vitest.config.ts's frozen-clock fake timers. The runtime spec below assumes all three are in force.

Conventions throughout: every function is pure (same inputs → same output, no side effects, no clock reads, no randomness, no environment branches); every iteration over a Set, Map, or plain object goes through one of the helpers in iteration-helpers.ts (the custom ESLint rule fails CI otherwise); arrays whose order is semantic (e.g., active_path, ConditionalOutput.branches, open_gates) are produced by walking already-sorted inputs so the output order is itself a deterministic function of the inputs; the orchestrator threads a single computed_at timestamp into every NodeStatus.evaluated_at and ConditionalOutput.computed_at field, never reading the wall clock.

Cross-document resolution notes (for self-review)

While compiling this spec from stream_c_runtime_v1.html against stream_i_contracts_v1.html Contract 2 and stream_i_schema_spec_v1.html, three precision questions arose where the contracts doc and the canonical schema spec diverged or were silent. In each case this spec adopts the schema spec's resolution (the schema spec is canonical for the persisted shapes the runtime produces; per stream_i_schema_spec_v1.html's self-review and the existing F-003 register flag, the contracts doc's paraphrased shapes are corrected at a future I.1 review session and downstream coding is not gated on the bump). The three items are itemized in this session's self-review (Article VIII § 4) and one of them — Contract 2's silence on how NodeStatus.evaluated_at is supplied — is logged as a register flag with the rationale that adopting Contract 2's silence as-is would force the runtime either to read the clock (forbidden by Article II § 2) or to make evaluated_at optional in the schema (a schema change). The constitutionally clean resolution is the one this spec adopts: the compute input gains a required computed_at: string field that the orchestrator threads into every NodeStatus and ConditionalOutput. The contracts doc is the doc that needs the additive correction, not the schema or the runtime.

// src/runtime/iteration-helpers.ts

The I.1 skeleton ships iteration-helpers.ts with working sortedIter, sortedKeys, and sortedEntries; this spec re-states the signatures as the contract and adds sortedBy, which the runtime needs for sorting nodes and edges by id when the source is a ReadonlyArray rather than a Set/Map. The helpers are the sole sanctioned iteration entry points inside src/runtime/; the custom argmap-determinism/no-unsorted-iteration ESLint rule fails CI on any direct iteration over a Set, Map, or plain object.

/**
 * Iterate over a Set in lexicographic id order.
 * `keyFn` extracts the sort key; defaults to the value itself (for Set<string>).
 */
export function sortedIter<T>(
  xs: ReadonlySet<T>,
  keyFn?: (x: T) => string,
): T[];

/**
 * Iterate over a Map's keys in lexicographic order. Returns a fresh array.
 */
export function sortedKeys<K extends string, V>(m: ReadonlyMap<K, V>): K[];
export function sortedKeys<V>(m: { [k: string]: V }): string[];

/**
 * Iterate over a Map's [key, value] entries in lexicographic-by-key order.
 */
export function sortedEntries<K extends string, V>(m: ReadonlyMap<K, V>): Array<[K, V]>;
export function sortedEntries<V>(m: { [k: string]: V }): Array<[string, V]>;

/**
 * Stable sort of an array by a string key. Returns a fresh array.
 * Used wherever the runtime iterates a ReadonlyArray whose source order is not
 * already deterministic (e.g., FrameVersion.nodes in input position).
 */
export function sortedBy<T>(xs: ReadonlyArray<T>, keyFn: (x: T) => string): T[];

Implementation notes for the coding session: each helper allocates one fresh array and sorts via String.prototype.localeCompare with explicit "en" locale and { sensitivity: "variant" } options to make ordering machine-independent. The helpers are tested in tests/runtime/iteration-helpers.test.ts already shipped in the skeleton; this spec adds a property test (Vitest acceptance tests below) asserting that two invocations of any helper on a structurally equal input produce structurally equal output regardless of original Set/Map insertion order.

// src/runtime/graph.ts

Reifies a FrameVersion into an in-memory Graph structure used by every downstream phase. The graph exposes both stored edges (from FrameVersion.edges) and synthetic edges that carry routing information stored on node fields rather than as Edge entities — specifically CheckpointOption.target_node_id (synthesized as OPTION_LEADS_TO) and LogicalGate.output_target (synthesized as GATES when not already present as an explicit GATES edge). Synthetic edges never persist; they exist so traversal code does not have to special-case Checkpoint and LogicalGate at every callsite (per Stream C "CheckpointOption target traversal model").

import type {
  FrameVersion, Node, Edge, NodeRef, EdgeType,
  Checkpoint, LogicalGate, RootQuestion,
} from "@/schema";

/**
 * The set of edge types that carry the structural resolution flow of a frame.
 * V-FR-6 (DAG check) operates on the structural sub-graph; reachability,
 * topological sort, foreclosure cascade, and active-set computation all
 * filter to these edge types unless otherwise noted.
 *
 * Note: includes the synthetic OPTION_LEADS_TO type produced by Checkpoint
 * options, which is not a member of the persisted EdgeType union.
 */
export const STRUCTURAL_EDGE_TYPES: ReadonlySet<TraversalEdgeType> = new Set([
  "DECOMPOSES_INTO", "TURNS_ON", "INTERPRETED_AS",
  "LEADS_TO", "GATES", "OPTION_LEADS_TO",
]);

/** Edge type union extended with the synthetic OPTION_LEADS_TO. */
export type TraversalEdgeType = EdgeType | "OPTION_LEADS_TO";

/**
 * Synthetic edges carry routing information not stored as Edge entities.
 * Discriminated by `synthetic: true`; otherwise structurally equivalent to a
 * subset of the persisted Edge fields.
 */
export interface SyntheticEdge {
  readonly synthetic: true;
  readonly type: "OPTION_LEADS_TO" | "GATES";
  readonly source: NodeRef;
  readonly target: NodeRef;
  /** Present only when `type === "OPTION_LEADS_TO"`. */
  readonly option_id?: string;
}

/** Either a persisted Edge or a SyntheticEdge produced by Graph.outTraversal. */
export type TraversalEdge = Edge | SyntheticEdge;

/**
 * Pure, opaque graph projection of a FrameVersion. Holds no references to
 * mutable state. Caller-side mutation of the input FrameVersion is undefined
 * behavior; the Graph captures a fresh view at construction time.
 */
export interface Graph {
  /** Lookup a node by id; throws if absent (callers should validate first). */
  readonly nodeById: (id: NodeRef) => Node;
  /** All node ids, sorted lexicographically. Cached. */
  readonly allNodeIds: ReadonlyArray<NodeRef>;
  /** All persisted edges (FrameVersion.edges) sorted by id. Cached. */
  readonly allEdges: ReadonlyArray<Edge>;
  /**
   * Yield every outgoing edge (persisted + synthetic) from `nodeId`,
   * in deterministic order: persisted edges by edge id, then synthetic
   * edges by (type, target).
   */
  readonly outTraversal: (nodeId: NodeRef) => ReadonlyArray<TraversalEdge>;
  /**
   * Yield every incoming edge (persisted only — synthetic edges are not
   * indexed by target). Used by status-determination ancestor walks.
   */
  readonly incoming: (nodeId: NodeRef) => ReadonlyArray<Edge>;
}

/** Build a Graph from a FrameVersion. Pure. */
export function buildGraph(frame: FrameVersion): Graph;

/**
 * Find the unique RootQuestion. V-FR-1 guarantees exactly one in a valid
 * frame; if zero or more than one is present, throws RuntimeStructuralError
 * (the orchestrator catches and short-circuits to incomplete output).
 */
export function findRoot(frame: FrameVersion): RootQuestion;

/**
 * Forward reachability from `start` along `STRUCTURAL_EDGE_TYPES`,
 * skipping nodes in `blocked`. Returns a sorted array (caller uses as Set
 * via `new Set(forwardReach(...))` when set semantics are needed).
 */
export function forwardReach(
  start: NodeRef,
  graph: Graph,
  blocked?: ReadonlySet<NodeRef>,
): ReadonlyArray<NodeRef>;

/**
 * Topological sort over the structural sub-graph. Returns a node-id ordering
 * such that for every structural edge u → v, `topo.indexOf(u) < topo.indexOf(v)`.
 * Throws RuntimeStructuralError if a cycle is detected (V-FR-6 should have
 * caught this, but the runtime is defensive per Stream C decision #9).
 *
 * Stability: ties are broken by lexicographic id, so the output is fully
 * deterministic given the input frame.
 */
export function topologicalSort(graph: Graph): ReadonlyArray<NodeRef>;

/**
 * Defensive error for runtime structural violations that V-FR validation
 * should have caught. Thrown by graph utilities; caught by the orchestrator
 * which converts to ValidationResult and short-circuits to incomplete output.
 */
export class RuntimeStructuralError extends Error {
  constructor(
    public readonly kind: "no_root" | "multiple_roots" | "cycle" | "missing_node",
    public readonly node_ids: ReadonlyArray<NodeRef>,
    message: string,
  ) {
    super(message);
    this.name = "RuntimeStructuralError";
  }
}

Implementation notes:

  • Synthetic edges from LogicalGate.output_target. Schema-spec LogicalGate variants (AndGate, OrGate, NotGate, IfThenGate, UnlessGate) all carry an optional output_target?: NodeRef. buildGraph emits a SyntheticEdge of type "GATES" for each gate whose output_target is set AND for which no persisted GATES edge from the gate to the same target already exists in frame.edges. The de-duplication keeps the traversal surface clean even when an author drew an explicit GATES edge in addition to the field.
  • Synthetic edges from CheckpointOption.target_node_id. For each Checkpoint, for each option in checkpoint.options sorted by option.id, emit a SyntheticEdge of type "OPTION_LEADS_TO" when option.target_node_id is set. The option_id field on the synthetic edge identifies which option produced it (consumed by the active-set walk in reachable-active.ts).
  • outTraversal ordering. Persisted edges first, sorted by edge.id; then synthetic edges, sorted lexicographically by (type, target) with type as the primary key. The whole projection is pre-computed at buildGraph time and cached per node id; outTraversal returns a stable ReadonlyArray reference for the same node across calls.
  • topologicalSort algorithm. Kahn's algorithm walking nodes in lexicographic id order at each step, with the structural sub-graph filtered via STRUCTURAL_EDGE_TYPES. Annotation/citation/argument-layer edges (FORECLOSES, BINDING_IN, CITES, DISTINGUISHED_BY, SUPPORTS, CONTRADICTS, ANSWERS) are excluded; this matches V-FR-6's DAG scope. The sort is total: every node in frame.nodes appears exactly once, including Premise and Authority nodes (which are leaves in the structural sub-graph and appear at the end of any valid ordering).
  • findRoot. Walks frame.nodes filtering for type === "RootQuestion"; throws RuntimeStructuralError("no_root") or "multiple_roots" if the count is not exactly one. The orchestrator catches and emits a synthetic ValidationResult with rule_id: "V-FR-1" + severity: "error" in addition to whatever validation phase 1 produced.
// src/runtime/foreclosure.ts

Phase 2 of the pipeline (Stream C2). Computes the foreclosed node-id set from two independent sources — explicit FORECLOSES edges sourced from selected Interpretations, and dispositive Term auto-foreclosure — then cascades forward along the structural sub-graph. Output is a ReadonlySet<NodeRef> consumed by phases 3, 4, and 5. The phase also exports foreclosureScopeFor, which the status determiner consults in phase 4 to distinguish "moot" foreclosure (status foreclosed) from "decided" foreclosure (status satisfied via structural_resolution).

import type {
  FrameVersion, ArgumentSession, NodeRef, Term, Interpretation,
  ForeclosesEdge, InterpretationSelection,
} from "@/schema";
import type { Graph } from "./graph";

/** "moot" — branch was made irrelevant; status becomes `foreclosed`.
 *  "decided" — branch was resolved by the foreclosing Interpretation; status
 *  becomes `satisfied` via `structural_resolution`. */
export type ForeclosureScope = "moot" | "decided";

/**
 * Phase 2 entry point. Composes explicit + dispositive sources, then cascades.
 * Pure. The returned set's iteration order is unspecified; consumers must use
 * `sortedIter` if they need order-sensitive output.
 */
export function resolveForeclosure(
  frame: FrameVersion,
  session: Pick<ArgumentSession, "interpretation_selections">,
  graph: Graph,
): ReadonlySet<NodeRef>;

/**
 * Determine the foreclosure scope for a node that is in the foreclosed set.
 * Walks `frame.edges` for FORECLOSES edges targeting `node_id` whose source
 * is a selected Interpretation: any such edge with `scope === "decided"`
 * makes the node decided; otherwise moot. Dispositive auto-foreclosure is
 * always moot (no FORECLOSES edge involved).
 *
 * Returns "moot" by default if the node is not actually in the foreclosed set;
 * callers must check membership first.
 */
export function foreclosureScopeFor(
  node_id: NodeRef,
  frame: FrameVersion,
  session: Pick<ArgumentSession, "interpretation_selections">,
): ForeclosureScope;

Algorithm — explicit FORECLOSES edges

For each selected interpretation id (collected from session.interpretation_selections via flat-map over selected_interpretation_ids), walk frame.edges sorted by edge.id: every FORECLOSES edge whose source is a selected interpretation id contributes its target to the direct foreclosed set. The scope field is read by foreclosureScopeFor in phase 4; phase 2 itself does not branch on scope.

Algorithm — dispositive Term auto-foreclosure

For each parent question (RootQuestion or SubQuestion) in lexicographic id order, collect its child Terms (via TURNS_ON edges, sorted by edge id). Sort the children by (term.order, term.id) — order is the primary key, lexicographic id is the deterministic tiebreaker for equal-order Terms. Walk the sorted children: the first Term that is both dispositive: true AND has at least one selected Interpretation in session.interpretation_selections is the winner. If a winner exists, every other sibling Term under the same parent — including later dispositive Terms — is added to the direct foreclosed set. If no winner exists, no auto-foreclosure happens at this parent.

Per Stream C2: explicit FORECLOSES and dispositive auto-foreclosure are independent sources; the direct foreclosed set is the union. Both run before cascade.

Algorithm — cascade

Cascade is a reachability subtraction. Compute fullReach = forwardReach(root, graph) with no blocking; compute liveReach = forwardReach(root, graph, blocked = direct). The cascade-foreclosed set is (fullReach − liveReach) ∪ direct. This guarantees every node strictly downstream of a directly-foreclosed node along the structural sub-graph is itself foreclosed unless reachable through some other live ancestor. The cascade respects the structural-edge filter: annotation edges (CITES, DISTINGUISHED_BY, BINDING_IN-as-edge) are excluded by forwardReach's use of STRUCTURAL_EDGE_TYPES.

Note on iteration determinism: forwardReach internally uses a queue/stack initialized with [start] and pushes targets in the order they appear in graph.outTraversal(cur) (which is itself sorted). The visited-set check prevents re-walks. The output array's order does not affect downstream consumers (only set membership is queried), but the array itself is produced deterministically.

Algorithm — foreclosureScopeFor

Walk frame.edges sorted by edge id; for each FORECLOSES edge whose target === node_id, check whether the source is in the set of selected interpretation ids AND the edge's scope === "decided". If any such edge exists, return "decided". Otherwise return "moot". Dispositive auto-foreclosure produces no FORECLOSES edge, so a node foreclosed only by dispositive auto-foreclosure correctly resolves to "moot" by default.

The function is called once per foreclosed node by the status determiner. It does not need to be O(1); a per-call linear scan over the edges is acceptable at v1 frame sizes (low thousands of edges). If profiling later shows hot-spotting, a precomputed map from node_id to scope can be added behind the same signature.

// src/runtime/reachable-active.ts

Phase 3 of the pipeline. Computes two derived sets the downstream phases depend on. Reachable: every node visitable from the RootQuestion along the structural sub-graph, honoring foreclosure but ignoring whether a branch is currently selected. This is the set of nodes the user could reach by changing answers. Active: every node visitable from the RootQuestion honoring both foreclosure AND following only branches the session's selections, responses, and gate evaluations actually route through. Nodes outside the active set get status not_applicable in phase 4. The two sets diverge at branching nodes (Term, Checkpoint, LogicalGate); both are needed.

import type {
  FrameVersion, ArgumentSession, NodeRef, Node,
  Term, Checkpoint, LogicalGate, CheckpointResponse, InterpretationSelection,
} from "@/schema";
import type { Graph, TraversalEdge } from "./graph";

/**
 * The set of nodes reachable from the RootQuestion along structural edges,
 * skipping foreclosed nodes. Pure.
 */
export function computeReachableSet(
  frame: FrameVersion,
  graph: Graph,
  foreclosed: ReadonlySet<NodeRef>,
): ReadonlySet<NodeRef>;

/**
 * The set of nodes the session's current selections, responses, and gate
 * evaluations actually visit. Pure. Skips foreclosed nodes; consults
 * selection records, checkpoint responses, and gate evaluation to decide
 * which branches to follow at each node.
 *
 * The active set may contain a node multiple times under different parent
 * paths in the conceptual sense, but the returned Set deduplicates by id;
 * `traceActivePath` in `output.ts` reconstructs the per-path information
 * separately when it walks for output generation.
 */
export function computeActiveSet(
  frame: FrameVersion,
  session: Pick<ArgumentSession,
    "interpretation_selections" | "checkpoint_responses" | "status_map">,
  graph: Graph,
  foreclosed: ReadonlySet<NodeRef>,
): ReadonlySet<NodeRef>;

Algorithm — computeReachableSet

Direct call to forwardReach(findRoot(frame).id, graph, blocked = foreclosed) from graph.ts; collect the result into a Set and return as ReadonlySet<NodeRef>. Pure and deterministic.

Algorithm — computeActiveSet

BFS from the RootQuestion. Initialize a queue with [rootId]. While the queue is non-empty: pop the current node id; if it is in foreclosed or already in active, skip. Add to active. For each outgoing traversal edge from the current node (in the order produced by graph.outTraversal, which is already deterministic): if the edge type is not in STRUCTURAL_EDGE_TYPES, skip; otherwise consult branchIsFollowed(node, edge, session) — if it returns true, push the edge target onto the queue.

The per-node-type branch-following logic is internal to this file (not exported); it is the single point where the runtime decides which structural edges out of a node should be considered "active" for the current session. Pseudocode:

function branchIsFollowed(node: Node, edge: TraversalEdge,
                          session: SessionLike): boolean {
  switch (node.type) {
    case "RootQuestion":
    case "SubQuestion":
      // All structural outgoing edges (DECOMPOSES_INTO, TURNS_ON, LEADS_TO,
      // GATES) are followed; questions don't branch — they decompose.
      return true;

    case "Term": {
      // Only follow INTERPRETED_AS edges to interpretations that are in
      // the session's selection record for this Term. Other edge types
      // out of a Term (none in v1) would be followed.
      if (edge.type !== "INTERPRETED_AS") return true;
      const selectedIds = selectedInterpretationIdsFor(node.id, session);
      return selectedIds.has(edge.target);
    }

    case "Interpretation":
      // Interpretations route via LEADS_TO to Checkpoint / SubQuestion /
      // Conclusion / LogicalGate. All such edges are followed; routing
      // is fixed by the frame author.
      return edge.type === "LEADS_TO";

    case "Checkpoint": {
      // Only follow OPTION_LEADS_TO edges whose option_id matches the
      // session's CheckpointResponse for this Checkpoint. Persisted
      // edges out of a Checkpoint are not part of the routing flow.
      if (edge.type !== "OPTION_LEADS_TO") return false;
      const response = responseFor(node.id, session);
      if (!response) return false;
      return (edge as SyntheticEdge).option_id === response.selected_option_id;
    }

    case "LogicalGate":
      // Only follow GATES edges (persisted or synthetic from output_target)
      // when the gate evaluates to "satisfied". Indeterminate or
      // not_satisfied gates do not route. The gate evaluator is invoked
      // here (it is pure and consults status_map only for input nodes,
      // which precede the gate in topological order).
      if (edge.type !== "GATES") return false;
      return gateRoutesTo(node, edge, session);

    default:
      // Premise / Authority / Conclusion: no outgoing structural edges
      // followed in active-set computation.
      return false;
  }
}

The accessor helpers (selectedInterpretationIdsFor, responseFor) are local to this file: each scans the relevant session array sorted by lexicographic id and returns either a Set<NodeRef> (for selections, since multiple per Term are permitted per the alternative-paths pattern) or a single CheckpointResponse | undefined. Both are pure and side-effect-free; both treat the absence of a record as "no selection / no response," which causes the active-set walk to stop at that node (and the status determiner to report open).

Gate-evaluation timing. gateRoutesTo requires the status of the gate's input nodes to be known. In active-set computation, a gate's inputs may be in the active set but their status has not yet been computed (status determination is phase 4, downstream of active-set). The runtime resolves this by passing session.status_map as an input to computeActiveSet — but status_map is itself the output of a later phase, creating an apparent dependency loop. The resolution: on the first compute pass, status_map is empty; gateRoutesTo falls back to treating any unresolved input as indeterminate, which means the gate does not route (per Stream C, only satisfied routes). The active set computed under an empty status map is therefore a lower bound. Phase 4 then computes status with the lower-bound active set, and phase 5's path tracer (output.ts) re-walks from the root using the current status map for accurate gate-routing decisions during output generation. This two-pass arrangement is the simplest way to keep the pipeline acyclic; an iterative fixpoint version is reserved for v2 if profiling shows the lower-bound active set materially under-estimating coverage in real frames.

The two-pass arrangement is invisible to compute's caller and does not affect determinism: every input combination produces the same active set, status map, and active path. It is documented here because the coding session will see computeActiveSet taking a status_map argument that is empty on first call and must understand why.

// src/runtime/gates.ts

Cross-cutting LogicalGate evaluation. Called by reachable-active.ts (for gate routing during active-set computation), status.ts (for the gate's own NodeStatus), and output.ts (for path tracing and indeterminate-gate branch generation). Honors the schema spec's named-slot encoding: AndGate/OrGate use inputs: NodeRef[]; NotGate uses input: NodeRef; IfThenGate uses antecedent and consequent; UnlessGate uses main and exception. The evaluator branches on gate.gate_type with TypeScript exhaustiveness. There is no uniform inputs[] on LogicalGate — gates know their slot names from their type.

import type {
  LogicalGate, AndGate, OrGate, NotGate, IfThenGate, UnlessGate,
  NodeRef, NodeStatus,
} from "@/schema";
import {
  isAndGate, isOrGate, isNotGate, isIfThenGate, isUnlessGate,
} from "@/schema";

/** A gate's evaluation result. Routing happens only on `"satisfied"`. */
export type GateResult = "satisfied" | "not_satisfied" | "indeterminate";

/**
 * Boolean projection of a NodeStatus for use as a gate input.
 * Per Stream C "Boolean evaluation of non-gate inputs":
 *   satisfied                            -> satisfied
 *   foreclosed (scope === "decided")     -> satisfied
 *   foreclosed (scope === "moot")        -> indeterminate
 *   not_applicable, open, contested      -> indeterminate
 *
 * Callers that pass `undefined` (input not yet evaluated) get `indeterminate`,
 * which is the conservative default and the value that prevents premature
 * routing during the first-pass active-set computation.
 */
export function booleanizeStatus(
  status: NodeStatus | undefined,
): "satisfied" | "indeterminate";

/**
 * Evaluate a LogicalGate against the current status_map. Pure.
 *
 * Inputs are looked up by NodeRef from `status_map`; for gates whose inputs
 * are themselves gates, status_map must already contain the input gates'
 * statuses (status determination walks reverse-topologically, so this
 * ordering is guaranteed in phase 4).
 */
export function evaluateGate(
  gate: LogicalGate,
  status_map: ReadonlyMap<NodeRef, NodeStatus>,
): GateResult;

/**
 * Convenience predicate used by `reachable-active.ts` to decide whether
 * a gate's GATES outgoing edge should be followed during active-set walk.
 * Equivalent to `evaluateGate(gate, status_map) === "satisfied"`.
 */

Per-gate-type evaluation

Pseudocode for evaluateGate's switch arms; each arm uses TypeScript narrowing through the schema's type guards (isAndGate, etc.) so the named slots are accessible by name without further casting.

// AND: every input satisfied. Indeterminate iff at least one input is
//      indeterminate AND no input is not_satisfied. Otherwise not_satisfied.
function evaluateAnd(g: AndGate, sm): GateResult {
  let hasIndeterminate = false;
  for (const inputId of sortedBy(g.inputs, x => x)) {  // NodeRef[] is string[]; sort lexicographically
    const b = booleanizeStatus(sm.get(inputId));
    if (b === "indeterminate") {
      // Check whether any input is definitively not_satisfied (which would
      // short-circuit to not_satisfied). Need to peek raw status, not just boolean.
      const raw = sm.get(inputId);
      if (raw && raw.status !== "satisfied"
                 && raw.status !== "open" && raw.status !== "contested"
                 && raw.status !== "not_applicable"
                 && !(raw.status === "foreclosed")) {
        // Reserved: no current status maps to "definitively not satisfied"
        // for AND; the indeterminate path is the conservative default.
      }
      hasIndeterminate = true;
    }
  }
  if (hasIndeterminate) return "indeterminate";
  // All inputs satisfied (booleanizeStatus returned "satisfied" for each).
  return "satisfied";
}

// OR: any input satisfied. Indeterminate iff every non-satisfied input is
//     indeterminate. Otherwise not_satisfied.
function evaluateOr(g: OrGate, sm): GateResult {
  let anySatisfied = false;
  let allNonSatisfiedAreIndeterminate = true;
  for (const inputId of sortedBy(g.inputs, x => x)) {
    const b = booleanizeStatus(sm.get(inputId));
    if (b === "satisfied") anySatisfied = true;
    // booleanizeStatus only returns "satisfied" or "indeterminate" in v1;
    // there is no "not_satisfied" projection because no NodeStatus value
    // maps to a definitive boolean false. Indeterminate covers "open",
    // "contested", "not_applicable", and "foreclosed (moot)"; "satisfied"
    // covers actual satisfaction and "foreclosed (decided)".
  }
  if (anySatisfied) return "satisfied";
  // No satisfied input found and only "satisfied"/"indeterminate" possible;
  // therefore every input is indeterminate.
  return "indeterminate";
}

// NOT: input not_satisfied. Indeterminate iff input indeterminate.
//      satisfied iff input not_satisfied; not_satisfied iff input satisfied.
//
// Caveat: because booleanizeStatus collapses many states into "indeterminate",
// NOT can only produce "satisfied" when the input has a hard not_satisfied
// signal. In v1 the only such signal is an upstream gate that itself
// evaluated to "not_satisfied" — propagated through status_map by phase 4
// recording the gate's NodeStatus.failed_conditions accordingly. For
// non-gate inputs whose boolean projection is "indeterminate", NOT is
// "indeterminate"; this is the conservative behavior and matches Stream C.
function evaluateNot(g: NotGate, sm): GateResult {
  const inputStatus = sm.get(g.input);
  if (!inputStatus) return "indeterminate";
  // Treat the upstream gate's own evaluated result, if recorded, as authoritative.
  // The status determiner stores gate evaluations as NodeStatus.failed_conditions
  // entries: ["gate_evaluated_not_satisfied"] for not_satisfied, none / "satisfied"
  // for satisfied, ["inputs_indeterminate"] for indeterminate.
  if (inputStatus.status === "satisfied") return "not_satisfied";
  if (inputStatus.status === "open"
      && (inputStatus.failed_conditions ?? []).includes("gate_evaluated_not_satisfied")) {
    return "satisfied";
  }
  return "indeterminate";
}

// IF_THEN: antecedent not_satisfied OR consequent satisfied.
//          Indeterminate iff antecedent indeterminate, OR (antecedent
//          satisfied AND consequent indeterminate).
//          Otherwise not_satisfied (antecedent satisfied AND consequent
//          not_satisfied).
function evaluateIfThen(g: IfThenGate, sm): GateResult {
  const a = sm.get(g.antecedent);
  const c = sm.get(g.consequent);
  if (!a) return "indeterminate";
  if (a.status !== "satisfied") {
    // Antecedent not satisfied (open / contested / not_applicable / foreclosed-moot
    // / explicitly not_satisfied via gate). For an antecedent that is itself
    // a gate evaluated not_satisfied, IF_THEN is satisfied (vacuously).
    if (a.status === "open"
        && (a.failed_conditions ?? []).includes("gate_evaluated_not_satisfied")) {
      return "satisfied";
    }
    return "indeterminate";
  }
  // Antecedent satisfied; consequent decides.
  if (!c) return "indeterminate";
  if (c.status === "satisfied") return "satisfied";
  if (c.status === "open"
      && (c.failed_conditions ?? []).includes("gate_evaluated_not_satisfied")) {
    return "not_satisfied";
  }
  return "indeterminate";
}

// UNLESS: main satisfied AND exception not_satisfied.
//         Indeterminate iff main indeterminate, OR main satisfied AND
//         exception indeterminate. Otherwise not_satisfied.
function evaluateUnless(g: UnlessGate, sm): GateResult {
  const m = sm.get(g.main);
  const x = sm.get(g.exception);
  if (!m || m.status !== "satisfied") {
    if (m && m.status === "open"
        && (m.failed_conditions ?? []).includes("gate_evaluated_not_satisfied")) {
      return "not_satisfied";
    }
    return "indeterminate";
  }
  if (!x) return "indeterminate";
  if (x.status === "open"
      && (x.failed_conditions ?? []).includes("gate_evaluated_not_satisfied")) {
    return "satisfied";
  }
  if (x.status === "satisfied") return "not_satisfied";
  return "indeterminate";
}

Implementation note on the "gate_evaluated_not_satisfied" failed-condition convention: when phase 4's status determiner evaluates a LogicalGate and gets not_satisfied, it stores the gate's NodeStatus as { status: "open", failed_conditions: ["gate_evaluated_not_satisfied"] }. This is the runtime's hard signal that propagates not-satisfaction up through chained gates. Without it, downstream NOT / IF_THEN / UNLESS gates would all collapse to indeterminate in the booleanization step, which is wrong for chained logic. This convention is invisible at the NodeStatus consumer surface (UI shows the gate as open either way; the failed condition appears in the inspector tooltip) and is consumed only inside gates.ts. The string is a constant exported as GATE_NOT_SATISFIED_MARKER from this file for the status determiner to reference symbolically.

The short-circuit-friendliness mentioned in Stream C ("AND can resolve to not_satisfied as soon as one input fails") is honored in implementation by early returns inside the for-loops; the truth table outcome is identical to a full evaluation, so determinism is preserved.

Cycle defense

V-FR-6 forbids cycles in the structural sub-graph, which includes GATES edges. The runtime is defensive: if evaluateGate is called for a gate whose input is itself the gate (transitively), the input lookup in status_map returns undefined (because status determination walks reverse-topologically and a cycle would have been caught by topologicalSort in graph.ts), and booleanizeStatus(undefined) === "indeterminate" propagates safely. The orchestrator emits a synthetic ValidationResult at the catch site and the cycle never silently routes.

Per-condition implementations

One arm per condition.kind, exhaustive on the schema's discriminated union. Each implementation is described in prose with the small implementation details a coding session needs; pseudocode is provided where the algorithm is non-obvious. Conditions whose semantics are pinned in Stream C (e.g., burden_met) are restated here in full because the spec is the surface the coding session implements.

premise_attached — applies to Checkpoint nodes only. Walk session.argument_edges sorted by edge id; pass iff at least one ANSWERS edge has target === node.id. via is empty.

interpretation_selected — applies to Term nodes (and, by inheritance through Term.linked_to, to linked Terms). Walk session.interpretation_selections sorted by (term_id, selected_at); pass iff at least one record has term_id === node.id AND selected_interpretation_ids.length >= 1. For a linked Term (node.linked_to non-null), follow the chain (resolved by frame.nodes lookup, walking until no further linked_to and detecting cycles defensively even though V-NODE-3 guarantees acyclicity), and consult the selection record on the chain endpoint. via is empty.

all_children_resolved — applies to RootQuestion / SubQuestion / Conclusion nodes. Children are the targets of outgoing edges of types {DECOMPOSES_INTO, TURNS_ON, LEADS_TO} (collected by walking frame.edges sorted by edge id, filtering by source === node.id and the type set). If the children list is empty, fail (a question with no children cannot be resolved). For each child, look up status_map.get(childId); treat satisfied as resolved (and accumulate the child's via array into this condition's via); treat foreclosed as resolved (do NOT contribute to via); anything else (open, contested, not_applicable, missing) fails the condition. The "foreclosed children do not block parent satisfaction" semantics implements the legal idiom "ruled on one ground, declined to reach the others" (Stream C). via is the union of satisfied children's via arrays, sorted and deduplicated.

path_complete — applies to Conclusion nodes. Pass iff there exists at least one path from the RootQuestion to node.id through the active set, every node on which has status satisfied or foreclosed with scope decided. Implementation: walk reverse-incoming from the Conclusion using graph.incoming, ascending until reaching the RootQuestion; if any node on the chosen ancestor path is missing from status_map or has a non-satisfied / non-decided-foreclosed status, fail. When multiple ancestor paths exist (the alternative-paths pattern), pass iff at least one path satisfies the condition. via reflects the via labels accumulated along the satisfied path.

not_contradicted — applies to Checkpoint, Interpretation, Conclusion. Walk session.argument_edges sorted by edge id; pass iff no CONTRADICTS edge has target === node.id. via is empty.

premise_kind_in — applies to Checkpoint, Interpretation, Conclusion. Pass iff at least one Premise from session.premises (sorted by premise id) is connected to this node via an ANSWERS, SUPPORTS, or CONTRADICTS edge in session.argument_edges AND the premise's kind is in the condition's kinds array. via is empty (premise kind alone does not contribute to via labels). The schema spec types kinds: string[] as an open string union; the policy editor (Stream E) filters offered kinds by frame mode/flavor.

burden_met — applies to Checkpoint nodes only. Categorical scoring per Stream C decision #6. The runtime imports the scoring tables and threshold map from this file (the F-001 wiring section below explains how G5-calibrated thresholds override the defaults). Pseudocode:

function condBurdenMet(node, frame, session, status_map, params): ConditionResult {
  if (node.type !== "Checkpoint") return { passed: false, via: [] };
  const level = params.level;                  // BurdenLevel; required by Condition shape
  const thresholds = frame.llm_settings_snapshot?.calibrated_thresholds
                   ?? DEFAULT_BURDEN_THRESHOLD;
  const threshold = thresholds[level];
  const weightTable = pickKindWeights(frame.mode, frame.flavor);
  let score = 0;
  const via = new Set<ConditionViaLabel>();
  for (const e of sortedBy(session.argument_edges, x => x.id)) {
    if (e.target !== node.id) continue;
    if (e.type !== "SUPPORTS" && e.type !== "ANSWERS" && e.type !== "CONTRADICTS") continue;
    const prem = session.premises.find(p => p.id === e.source);
    if (!prem) continue;
    const kw = weightTable[prem.kind] ?? 30;
    const ew = EDGE_WEIGHT_MULTIPLIER[(e as any).weight ?? "moderate"] ?? 0.66;
    if (e.type === "SUPPORTS" || e.type === "ANSWERS") {
      score += kw * ew;
      if (prem.kind === "stipulated") via.add("stipulation");
    } else {
      score -= kw * ew;
    }
  }
  return { passed: score >= threshold, via: sortedIter(via) };
}

The kind-weight tables are NOT subject to G5 override in v1 (the calibrated_thresholds field carries thresholds only); a future schema extension can broaden the projection without changing the runtime's interface.

function condAuthorityBinding(node, frame, session, status_map, params): ConditionResult {
  // Collect every Authority cited toward `node` from two sources:
  //   1. Frame-layer CITES edges to this node (Interpretation only via V-EDGE-1).
  //   2. Argument-layer Premises whose authority_ref is set, where the Premise
  //      ANSWERS or SUPPORTS this node.

  for (const e of sortedBy(frame.edges, x => x.id)) {
    if (e.type !== "CITES" || e.target !== node.id) continue;
    const auth = lookupAuthority(e.source, frame, session);
    if (auth) cited.push(auth);
  }
  for (const p of sortedBy(session.premises, x => x.id)) {
    if (!p.authority_ref) continue;
    const reaches = session.argument_edges.some(e =>
      (e.type === "ANSWERS" || e.type === "SUPPORTS")
      && e.source === p.id && e.target === node.id);
    if (!reaches) continue;
    const auth = lookupAuthority(p.authority_ref, frame, session);
    if (auth) cited.push(auth);
  }
  // Determine binding for each cited Authority against the frame's default
  // jurisdiction. via = "binding_authority" if any binds, "persuasive_authority"
  // if at least one Authority is cited but none binds, empty otherwise.
  const via = new Set<ConditionViaLabel>();
  let anyBinding = false;
  for (const auth of sortedBy(cited, x => x.id)) {
    if (!frame.jurisdiction_default) continue;
    const result = isBinding(auth, frame.jurisdiction_default);
    if (result.binding) {
      via.add("binding_authority");
      anyBinding = true;
    } else if (cited.length > 0) {
      via.add("persuasive_authority");
    }
  }
  return { passed: anyBinding, via: sortedIter(via) };
}

The condition passes iff at least one cited Authority is binding in the frame's default jurisdiction (or in the Interpretation's jurisdiction_override if applicable — to be added when Interpretation jurisdiction-override semantics are reviewed; v1 uses frame default). Per-Interpretation override is reserved for v2 and does not block coding.

not_distinguished — applies to Interpretation, Conclusion (legal mode). Walk frame.edges sorted by edge id; pass iff no DISTINGUISHED_BY edge has target === node.id. via is empty.

standard_of_review_applied — applies to RootQuestion, SubQuestion, Conclusion (legal mode). Pass iff node.standard_of_review is non-null AND non-empty. (For SubQuestion that inherits from a parent, the inherited value satisfies the condition; the runtime walks graph.incoming ascending until reaching a node with a populated standard_of_review or running out of ancestors.) Conclusion nodes do not carry standard_of_review on the schema; for Conclusions, ascend the incoming graph until the nearest ancestor with a populated value. via is empty.

not_foreclosed — universal. Pass iff node.id is NOT in the foreclosed set. Implementation note: the policy evaluator does not have direct access to the foreclosed set as a parameter; instead, the orchestrator passes foreclosure state into condition evaluation by reading the prior foreclosure check that already happened in phase 4 step 1 (see status.ts). For nodes that reach phase 4 step 4 (policy evaluation), the condition trivially passes (the foreclosure check is the very first step of the status determiner; foreclosed nodes never reach policy evaluation). This condition exists in the policy library for completeness — frames can list it explicitly to make the precondition visible — and as a future-proofing point if the precedence ladder is ever revised. In v1 it is effectively always-pass when reached; the implementation is a simple { passed: true, via: [] }. via is empty.

Policy evaluator

export function evaluatePolicy(
  policy: SatisfactionPolicy,
  node: Node,
  frame: FrameVersion,
  session: ArgumentSession,
  status_map: ReadonlyMap<NodeRef, NodeStatus>,
): PolicyResult {
  const via = new Set<ConditionViaLabel>();
  const failed: string[] = [];

  // all_of (AND-combined). Iterate in the order conditions appear in the
  // policy array — the policy itself is a deterministic ordered list, so
  // no extra sort is needed here.
  for (const cond of policy.all_of) {
    const r = evaluateCondition(cond, node, frame, session, status_map);
    for (const v of r.via) via.add(v);
    if (!r.passed) failed.push(cond.kind);
  }

  // any_of, if present. At most one AnyOf group; no nesting (per A3).
  if (policy.any_of && policy.any_of.length > 0) {
    let anyPassed = false;
    for (const cond of policy.any_of) {
      const r = evaluateCondition(cond, node, frame, session, status_map);
      if (r.passed) {
        anyPassed = true;
        for (const v of r.via) via.add(v);
        // Do NOT break: continue collecting via labels from other passing
        // conditions in the AnyOf group, so the via record is complete.
      }
    }
    if (!anyPassed) failed.push("any_of_group_unsatisfied");
  }

  return {
    passed: failed.length === 0,
    via: sortedIter(via),                       // lexicographic; deterministic
    failed: [...failed],
  };
}

The OR-group implementation continues iterating after finding a passing condition (it does not short-circuit) so that via labels from every passing condition are collected. This produces a richer audit trail at no behavioral cost; determinism is preserved since the iteration order is the policy's array order.

Phase 4 of the pipeline (Stream C3). Walks the frame in reverse topological order and assigns a NodeStatus to every non-source node (Premise and Authority are sources, not gates, and receive no status — per Article II § 3 and the schema spec's policy table). The walk applies the fixed precedence ladder from current_state.html "Status enum and precedence": foreclosed > not_applicable > contested > open > satisfied. foreclosed with scope decided resolves to satisfied with via: ["structural_resolution"]; this is recorded as a special branch within the foreclosure check rather than a separate precedence tier. The threading of computed_at into every NodeStatus.evaluated_at is the responsibility of this module — the orchestrator passes the timestamp; the determiner stamps it on every status it produces.

Algorithm

Compute the structural topological sort once at the top (via topologicalSort(graph)); reverse it; iterate in that order. For each node id in the reversed sort:

  1. Premise / Authority skip. If the node's type is "Premise" or "Authority", do not assign status; continue. (These nodes are still in the topological sort because they appear in frame.nodes; they simply don't receive status assignments. Map omission, not undefined, is the encoding.)
  2. Foreclosure check. If the node id is in foreclosed, consult foreclosureScopeFor(node_id, frame, session): if "decided", assign { status: "satisfied", via: ["structural_resolution"], evaluated_at: computed_at } with no failed_conditions. If "moot", assign { status: "foreclosed", evaluated_at: computed_at }. Continue.
  3. Active-set check. If the node id is NOT in active (and was not foreclosed in step 2), assign { status: "not_applicable", evaluated_at: computed_at }. Continue.
  4. LogicalGate evaluation. If the node is a LogicalGate, call evaluateGate(node, status_map) against the in-progress status map (which contains every input's status, since gates depend on structurally-prior nodes and we walk reverse-topologically). Map the gate result to NodeStatus as: "satisfied" → { status: "satisfied", evaluated_at }; "not_satisfied" → { status: "open", failed_conditions: [GATE_NOT_SATISFIED_MARKER], evaluated_at }; "indeterminate" → { status: "open", failed_conditions: ["inputs_indeterminate"], evaluated_at } (per Stream C decision #7). Continue.
  5. Contested check (Checkpoint-specific). If the node is a Checkpoint, evaluate the contested heuristic: a Checkpoint is contested when (a) at least one ANSWERS edge exists in session.argument_edges targeting this Checkpoint, AND (b) at least one CONTRADICTS edge exists from a Premise targeting this Checkpoint OR the chosen option's routes_to_status === "contested" (per the schema spec's CheckpointOption.routes_to_status). If contested, assign { status: "contested", failed_conditions: ["contradicting_premises"], evaluated_at }. Continue.
  6. Policy evaluation. Resolve the effective policy via resolveEffectivePolicy(node.type, node.options_box, frame.default_satisfaction_policies[node.type]) from @/schema. Pass it to evaluatePolicy(policy, node, frame, session, status_map); build the NodeStatus from the result: { status: result.passed ? "satisfied" : "open", via: result.passed ? result.via : [], failed_conditions: result.failed, evaluated_at }. The via array is omitted (rather than empty) when the status is open — Stream C and the schema spec both treat via as a satisfied-state badge.

Notes:

  • Order of checks. Foreclosure → active → gate-or-contested → policy. The first matching check wins; the precedence ladder is implemented by the order of these branches, not by post-hoc comparison. This makes status assignment a single deterministic pass per node.
  • via labels for satisfied nodes. The via array is the union of: any labels the policy evaluator collected from condition results, plus "binding_authority" / "persuasive_authority" for legal-mode satisfied Checkpoints / Interpretations / Conclusions whose Authority condition contributed (these come through the authority_binding condition's via output). The status determiner does not add any via labels of its own beyond the foreclosed-decided branch's "structural_resolution"; everything else flows through condition results.
  • Sub_flags display naming (current_state.html). The schema spec stores via values as "binding_authority" / "persuasive_authority" (no prefix); the UI layer renders them as the chips described in current_state.html ("⚖ B" / "⚖ P"); the contracts doc's sub_flags shape with the "via_" prefix is a paraphrase. The runtime emits the schema spec's bare names; the UI / Stream E does any prefixing for display.
  • Performance. Reverse topological iteration is O(V); each node's check is O(degree) for foreclosure/active membership and O(|policy.all_of| + |policy.any_of|) for policy evaluation; total is O(V + E). At v1 frame sizes (low hundreds of nodes), recompute on every session edit is fast enough that the incremental strategy in Stream C is deferred to v2 (per Stream C "Incremental computation strategy"). The runtime exposes only the full-recompute path; an incremental cache layer can be added behind compute's signature without changing it.

Output shape determination

Apply the precedence ladder from Stream C "Output shape determination":

  1. incomplete — phase 1 produced any error ValidationResult, OR no Conclusion is reachable from the active set. Output carries open_gates describing the structural gaps.
  2. contested — at least one Checkpoint on any path that reaches a Conclusion has status contested. Output carries contested_nodes (ids of contested Checkpoints on Conclusion-reaching paths, sorted lexicographically) and a best_inference Conclusion selected by the deterministic tiebreaker.
  3. conditional — at least one Term on the active path has no InterpretationSelection, or at least one Checkpoint on the active path has no CheckpointResponse, or at least one LogicalGate on the active path is indeterminate, AND alternative branches each lead to a different Conclusion. Output carries branches: ConditionalBranch[].
  4. determinate — exactly one Conclusion is reachable through fully-satisfied (or decided-foreclosed) nodes; no open gates remain on the path. Output carries conclusion and primary_path.

Implementation: compute the four shape predicates as booleans (hasErrors, hasContestedOnPath, hasUnresolvedBranchPoint, hasSinglePathToOneConclusion); apply the ladder. The first matching shape wins. Edge cases: a frame with no Conclusions reachable but no errors is still incomplete (the second clause of the rule). A frame with both contested and conditional features is contested (precedence). A frame with errors but also contested points is incomplete (errors win).

Path walking

The walker is a depth-first walk from the RootQuestion, branching at each node according to per-type rules. Multiple selected Interpretations under one Term produce a fork; the walk follows each fork and unions the resulting paths into all_segments. The per-node-type branching logic mirrors branchIsFollowed from reachable-active.ts but additionally records open gates and branches when a routing decision cannot be made.

// Pseudocode for traceActivePath. The actual implementation is iterative
// (explicit stack) to avoid recursion depth on deep frames.

function walk(node_id: NodeRef, prefix: NodeRef[],
              acc: WalkAcc): void {
  if (foreclosed.has(node_id)) return;            // foreclosed: cut here
  const status = status_map.get(node_id);
  if (status?.status === "not_applicable") return;
  const newPrefix = [...prefix, node_id];
  const node = graph.nodeById(node_id);

  if (node.type === "Conclusion") {
    acc.segments.push(newPrefix);
    return;
  }

  const nextTargets: NodeRef[] = [];

  switch (node.type) {
    case "RootQuestion":
    case "SubQuestion":
      // Follow every structural outgoing edge in deterministic order.
      for (const e of graph.outTraversal(node_id)) {
        if (e.type === "DECOMPOSES_INTO" || e.type === "TURNS_ON" || e.type === "LEADS_TO") {
          nextTargets.push(e.target);
        }
      }
      break;

    case "Term": {
      const sel = selectedInterpretationsFor(node.id, session);
      if (sel.size === 0) {
        // Open gate: user has not picked an interpretation.
        acc.open_gates.push({
          node_id, reason: "structural",
          prompt: `Pick an interpretation of '${node.name}'.`,
        });
        return;
      }
      for (const iid of sortedIter(sel)) {
        if (!foreclosed.has(iid)) nextTargets.push(iid);
      }
      break;
    }

    case "Interpretation":
      for (const e of graph.outTraversal(node_id)) {
        if (e.type === "LEADS_TO") nextTargets.push(e.target);
      }
      break;

    case "Checkpoint": {
      const response = responseFor(node_id, session);
      if (!response) {
        if (status?.status === "contested") {
          acc.branches.push(...branchesFromUnresolvedCheckpoint(
            node, newPrefix, frame, session, graph, foreclosed, status_map,
          ));
        } else {
          acc.open_gates.push({
            node_id,
            reason: hasContradictingPremise(node_id, session)
                  ? "premise_contradicted"
                  : "no_premise",
            prompt: node.question,
          });
        }
        return;
      }
      const opt = node.options.find(o => o.id === response.selected_option_id);
      if (opt?.target_node_id && !foreclosed.has(opt.target_node_id)) {
        nextTargets.push(opt.target_node_id);
      }
      break;
    }

    case "LogicalGate": {
      const result = evaluateGate(node, status_map);
      if (result === "satisfied") {
        // Walk via persisted GATES edge or synthetic from output_target,
        // whichever produced the routing path. graph.outTraversal yields
        // both; we follow each (typically there is only one).
        for (const e of graph.outTraversal(node_id)) {
          if (e.type === "GATES" && !foreclosed.has(e.target)) nextTargets.push(e.target);
        }
      } else if (result === "indeterminate") {
        acc.branches.push(...branchesFromIndeterminateGate(
          node, newPrefix, frame, session, graph, foreclosed, status_map,
        ));
        return;
      } else {
        // not_satisfied: gate does not route. The path stops here.
        return;
      }
      break;
    }

    default:
      // Premise / Authority should never appear on a path being walked.
      return;
  }

  // Walk targets in deterministic (sorted) order.
  for (const t of sortedBy(nextTargets, x => x)) walk(t, newPrefix, acc);
}

Primary-path selection

From all_segments, pick a single primary path by the fixed tiebreaker:

  1. Longest fully-satisfied path. Count the segments where every non-foreclosed node has status satisfied. Among these, the longest wins.
  2. Most binding-authority via labels. If two segments tie on length, count the number of nodes on each segment whose status's via array contains "binding_authority"; the higher count wins.
  3. Lexicographic Conclusion id. If still tied, the segment whose terminal Conclusion id sorts first wins.

The same tiebreaker is used for best_inference in the contested shape. Centralized in a helper pickPrimary(segments, status_map) within output.ts (not exported).

Branches from unresolved Checkpoints and indeterminate gates

When a Checkpoint with status contested is encountered with no response, OR a LogicalGate with result indeterminate is encountered, the walker emits ConditionalBranch entries. Each branch enumerates a small set of choices at the unresolved point and walks forward hypothetically from each:

function branchesFromUnresolvedCheckpoint(
  node: Checkpoint, prefix: NodeRef[],
  frame, session, graph, foreclosed, status_map,
): ConditionalBranch[] {
  const out: ConditionalBranch[] = [];
  for (const opt of sortedBy(node.options, x => x.id)) {
    if (!opt.target_node_id || foreclosed.has(opt.target_node_id)) continue;
    const subPath = walkAssuming(
      opt.target_node_id, frame, session, graph, foreclosed, status_map,
      { kind: "checkpoint", node_id: node.id, option_id: opt.id },
    );
    if (subPath.length === 0) continue;
    const terminal = subPath[subPath.length - 1];
    if (graph.nodeById(terminal).type !== "Conclusion") continue;
    out.push({
      id: branchIdFromConditions(node.id, opt.id),  // deterministic id derivation
      conditions: [{
        gate_node_id: node.id,
        required_value: opt.id,
        required_value_label: opt.label,
      }],
      resulting_conclusion: terminal,
      intermediate_path: [...prefix, ...subPath],
      prose: renderBranchProse(node, opt, subPath, frame),
    });
  }
  return mergeConvergentBranches(out);
}

The analogous branchesFromIndeterminateGate enumerates the gate's input-satisfaction states that would each produce a definite routing decision, and walks forward from each. For an IF_THEN gate whose antecedent is indeterminate, the two branches are "if antecedent satisfied" and "if antecedent not satisfied"; for an UNLESS gate whose exception is indeterminate but main is satisfied, the two branches are "if exception not satisfied" (gate satisfies, routes) and "if exception satisfied" (gate fails, no route). For AND/OR gates with multiple indeterminate inputs, the enumeration covers only the conjunction-of-decisions space that reaches a Conclusion — the implementation enumerates each indeterminate input's two boolean states and prunes to the satisfying combinations; for v1 this is bounded by a small input fan-in (in practice 2–4 indeterminate inputs at most). A larger fan-in degrades gracefully to a single open gate entry rather than an explosion of branches.

walkAssuming is a hypothetical walk: it temporarily treats the named assumption as if already in the session and walks forward. The walk is purely local — it does not re-run foreclosure or status determination for the rest of the frame. This is acceptable because the assumption is a single resolution at a single node, which only affects nodes downstream of that point. walkAssuming takes the same foreclosed and status_map from the outer compute call, plus an assumption object describing the local override; it returns a NodeRef[] that may or may not terminate at a Conclusion.

Convergent branches are merged by mergeConvergentBranches: when two branches end at the same Conclusion via different routes, their conditions arrays are combined into a disjunction (a single ConditionalBranch with both BranchCondition entries) and a single representative intermediate_path is chosen by the same primary-path tiebreaker. Merging keeps the conditional output compact for frames where multiple paths converge.

Branch ids: branchIdFromConditions derives a deterministic id from a stable hash of the sorted condition tuple (gate_node_id + required_value), so the same branch always has the same id across runs. The hash is SHA-256 of a canonical string form, truncated and prefixed with "br_"; SHA-256 is available in any modern JS runtime via crypto.subtle, but to keep the runtime free of async APIs, crypto.createHash from Node's built-in node:crypto module is used (the runtime is consumed in browser builds via Vite's polyfill). If the polyfill route proves brittle, the coding session may swap in a small synchronous SHA-256 implementation under iteration-helpers.ts rather than introduce a third-party dependency.

Open gates

Open gates are emitted by the walker when a structural opening is encountered (Term with no selection, Checkpoint with no response, etc.) AND when phase 4's status determiner produced an open NodeStatus on the active path with a meaningful failed_conditions entry. The output module collects both and merges by node_id, preferring the more specific reason when both sources fire for the same node. Reason mapping:

  • "no_premise" — Checkpoint with no ANSWERS edge.
  • "premise_contradicted" — Checkpoint with both ANSWERS and CONTRADICTS edges; phase 4 already labeled it contested rather than open, so this reason fires only when contested-checkpoint branch generation produced no usable branches.
  • "burden_unmet" — Checkpoint where failed_conditions includes "burden_met".
  • "authority_missing" — Checkpoint where failed_conditions includes "authority_required" or "authority_binding".
  • "structural" — everything else, including unresolved Terms, indeterminate gates, and dangling SubQuestions.

The open-gates list is sorted by node_id for output stability.

Prose summary generation

Per A9 and Stream C, the default view is a deterministic prose summary. The implementation is a fixed template walk over the active path or branch structure. No LLM, no clock reads, no randomness. The prose is plain text; the UI may mark it up further (links to nodes, etc.) but the canonical prose is what gets exported and what tests pin against.

Pseudocode mirrors Stream C's render_prose directly; the coding session implements per-shape formatters (renderDeterminateProse, renderConditionalProse, renderContestedProse, renderIncompleteProse) plus the shared confidence-breakdown footer (renderConfidenceBreakdown). Format conventions:

  • Sections are separated by single blank lines.
  • Lists use a leading " - " (two spaces, dash, space).
  • Direction is rendered via renderDirection(conclusion.direction, frame): legal-mode directions use the value verbatim ("affirm", "reverse", etc.) with "custom" using the Conclusion's custom_label; general-mode directions look up frame.positions[].label by position_id.
  • Node short forms come from renderNodeShort(nid, frame) which formats per-type: questions render as statement; Terms render as name; Interpretations render as statement; Checkpoints render as question; Gates render as "<gate_type> gate"; Conclusions render as statement. All are truncated at 80 characters with a trailing ellipsis when longer; truncation is at byte boundaries (no UTF-8 splitting).
  • The output never includes the date or the frame title; only the deterministic content. Determinism testing pins against the prose; including a date would defeat the snapshot test.

Confidence breakdown

Per A1, confidence is computed at output time and never stored on the Conclusion. The breakdown counts integer occurrences from the status map filtered to Checkpoints on the primary path; it is order-stable because it iterates the path in its given order. Mapping:

function computeConfidenceBreakdown(
  status_map: ReadonlyMap<NodeRef, NodeStatus>,
  primary_path: ReadonlyArray<NodeRef>,
  frame: FrameVersion,
): ConfidenceBreakdown {
  const checkpointsOnPath = primary_path.filter(
    nid => frame.nodes.find(n => n.id === nid)?.type === "Checkpoint"
  );
  const out: ConfidenceBreakdown = {
    total_checkpoints_on_path: checkpointsOnPath.length,
    satisfied_via_binding: 0,
    satisfied_via_persuasive: 0,
    satisfied_via_stipulation: 0,
    satisfied_via_structural: 0,
    contested: 0,
    open: 0,
  };
  for (const nid of checkpointsOnPath) {
    const st = status_map.get(nid);
    if (!st) continue;
    if (st.status === "satisfied") {
      const via = st.via ?? [];
      // Precedence among via labels for counting: binding > persuasive >
      // stipulation > structural > default. A satisfied Checkpoint counts in
      // the highest category for which it has a via label; if it has none,
      // it is counted as persuasive (the default for a satisfied node with
      // no via labels is "we got here on argument, not authority").
      if (via.includes("binding_authority"))           out.satisfied_via_binding++;
      else if (via.includes("persuasive_authority"))   out.satisfied_via_persuasive++;
      else if (via.includes("stipulation"))            out.satisfied_via_stipulation++;
      else if (via.includes("structural_resolution"))  out.satisfied_via_structural++;
      else                                              out.satisfied_via_persuasive++;
    } else if (st.status === "contested") out.contested++;
    else if (st.status === "open") out.open++;
  }
  return out;
}

The breakdown counts integers and is order-stable. The user-facing confidence display is rendered by Stream E; the breakdown itself is the canonical numeric record stored on ConditionalOutput.confidence_breakdown.

Resolution order

  1. Manual override. If authority.is_binding is set explicitly (boolean, non-null), return { binding: authority.is_binding, source: "manual_override", certainty: "exact" }. The user's explicit toggle wins over every algorithmic determination.
  2. F5 hierarchy database. If F5_AVAILABLE AND both authority.jurisdiction?.hierarchy_node_id AND target_jurisdiction.hierarchy_node_id are populated, dispatch to queryF5Binding(authority.jurisdiction.hierarchy_node_id, target_jurisdiction.hierarchy_node_id). If the call returns a result, return { binding: result.binding, source: "hierarchy_database", certainty: "exact", reasoning: result.reasoning, database_version: result.database_version }. In v1, F5_AVAILABLE is false and this branch is never taken; the F5 wire-in session sets it to true and populates the function body.
  3. String-match heuristic. If neither manual override nor F5 produced a result, fall back to:
    • U.S. Supreme Court special case. If authority.court matches "Supreme Court of the United States" or "United States Supreme Court" or "U.S. Supreme Court" (case-insensitive, exact-form list — the string match is fixed in code as a constant), return { binding: true, source: "string_match", certainty: "heuristic", reasoning: "U.S. Supreme Court is binding everywhere in the U.S. system" }. This special case exists because the F5 fallback would otherwise miss the rule; when F5 lands, the rule moves into the database and the special case is removed.
    • Same-circuit federal match. If authority.jurisdiction?.level === "federal" AND target_jurisdiction.level === "federal" AND both have non-null region AND the regions are identical (case-insensitive string equality), return { binding: true, source: "string_match", certainty: "heuristic" }.
    • Same level + same region. If authority.jurisdiction?.level === target_jurisdiction.level AND both have non-null region AND regions are identical (case-insensitive), return { binding: true, source: "string_match", certainty: "heuristic" }.
    • Anything else. Return { binding: false, source: "string_match", certainty: "heuristic" }.
  4. No-jurisdiction terminal. If authority.jurisdiction is absent OR target_jurisdiction's level/region are both absent, the string-match step has nothing to match on; return { binding: false, source: "no_jurisdiction", certainty: "exact" }. The "exact" certainty here reflects that the answer is unambiguously "we cannot determine binding without jurisdiction information" — it is not a heuristic gap.

The function is total: it always returns a BindingResult. It never throws (callers loop over many Authorities and would otherwise need to catch). Pure: identical inputs always produce identical outputs; no clock reads, no environment reads, no I/O.

  • The Stream F F6-v2 specification widens the result envelope additively with kind and reasoning and database_version fields. The v1 type already includes the optional reasoning and database_version fields; the F5 wire-in session does not need to change BindingResult's shape.
  • The Stream F "carry-forward determinism mechanism" (current_state.html open question) is honored at the runtime layer by reading the F5 database version from a snapshot field on the FrameVersion when one is present (the F5 wire-in session decides which exact field to populate). The runtime treats the snapshot field as opaque input; the choice between bundled-pinned and snapshot-on-session caching is invisible to isBinding's caller.
  • The via labels emitted by the authority_binding condition in conditions.ts ("binding_authority" / "persuasive_authority") are independent of BindingResult.source; the condition emits the via label based on BindingResult.binding alone. The source is recorded at the audit layer (HookInvocationRecord-style) and surfaced in tooltips per Stream E, but does not affect routing.

Pipeline composition

export function compute(...args: unknown[]): ComputeResult {
  const input: ComputeInput = normalizeArgs(args);
  const { frame_version_snapshot: frame, computed_at } = input;

  // Synthesize an ArgumentSession-shaped value for downstream phases
  // that take a session-like argument. The runtime does not require the
  // full ArgumentSession; it requires only the fields below.
  const session = {
    frame_version_snapshot: frame,
    checkpoint_responses: input.checkpoint_responses,
    interpretation_selections: input.interpretation_selections,
    premises: input.premises,
    argument_edges: input.argument_edges,
    session_authorities: input.session_authorities,
    status_map: {},                              // empty on first pass; see two-pass note below
  } as unknown as ArgumentSession;

  // Phase 1 — Validate.
  let validation_results: ValidationResult[];
  try {
    validation_results = [...runValidation(frame, session)];
  } catch (e) {
    return incompleteResult(frame, [{
      rule_id: "RUNTIME-INTERNAL", severity: "error",
      message: `Validation failed unexpectedly: ${(e as Error).message}`,
    }], computed_at);
  }
  if (validation_results.some(r => r.severity === "error")) {
    return incompleteResult(frame, validation_results, computed_at);
  }

  // Build the graph projection. Cycle detection here catches anything
  // V-FR-6 missed (defensive per Stream C decision #9).
  let graph;
  try {
    graph = buildGraph(frame);
  } catch (e) {
    if (e instanceof RuntimeStructuralError) {
      const synthetic: ValidationResult = {
        rule_id: e.kind === "cycle" ? "V-FR-6"
               : e.kind === "no_root" || e.kind === "multiple_roots" ? "V-FR-1"
               : "RUNTIME-STRUCTURAL",
        severity: "error",
        message: e.message,
      };
      return incompleteResult(frame, [...validation_results, synthetic], computed_at);
    }
    throw e;
  }

  // Phase 2 — Foreclosure.
  const foreclosed = resolveForeclosure(frame, session, graph);

  // Phase 3 — Reachable and active. Active is computed against an empty
  // status_map on this first pass; see the two-pass note in
  // reachable-active.ts. Phase 5 re-walks with the populated status_map.
  const reachable = computeReachableSet(frame, graph, foreclosed);
  const active_pass1 = computeActiveSet(frame, session, graph, foreclosed);

  // Phase 4 — Status. Walks reverse-topologically; threads computed_at
  // into every NodeStatus.evaluated_at.
  const status_map = computeStatusMap(
    frame, session, graph, foreclosed, active_pass1, computed_at,
  );

  // Phase 3 redux — recompute active set with the now-populated status_map
  // so gate-routing decisions during output generation see correct gate
  // results. Cheap because it's the same algorithm, different inputs.
  const active = computeActiveSet(
    frame, { ...session, status_map: mapToObject(status_map) }, graph, foreclosed,
  );

  // Phase 5 — Output.
  const { active_path, output, open_gates } = generateOutput(
    frame, session, graph, foreclosed, active, status_map, computed_at,
  );

  return {
    validation_results,
    foreclosed_set: foreclosed,
    reachable_set: reachable,
    active_set: active,
    status_map,
    active_path,
    output,
    open_gates,
  };
}

The orchestrator is the only place in the runtime where iteration over the input arrays for normalization happens; from that point onward, all internal iterations go through the iteration helpers. The orchestrator does not catch errors thrown by phases 2–5: those phases are required to be total over valid inputs, and any throw is a programmer error that should surface, not a contractual failure mode.

incompleteResult is a small helper that constructs a ComputeResult with empty foreclosed/reachable/active sets, an empty status map, an empty active path, and an output with shape: "incomplete" and the supplied validation results listed as open gates with reason: "structural". The prose summary is rendered through renderProse against the incomplete output for consistency. The computed_at stamp is honored.

The two-pass active-set computation is a deliberate choice over a single iterative-fixpoint version because (a) on most real inputs the two passes produce identical sets (gates are rare and resolve quickly when their inputs are settled), and (b) two-pass is straightforwardly deterministic without needing fixpoint-convergence proofs. If profiling shows a bottleneck, the iterative version can replace this without changing compute's signature.

Determinism implementation discipline

The runtime is the determinism boundary of the application. Article II § 2 fixes the contract; this section restates the implementation discipline at the file level so the coding session knows where each gate fires.

Module boundary (Stream H, ESLint no-restricted-imports)

Files under src/runtime/ may import only from @/schema and standard JavaScript built-ins. The skeleton's eslint.config.js wires the boundary; the CI typecheck + lint pass on every PR. Forbidden specifically: @/persistence, @/ui, @/state, @/llm-hooks, @/modes, @/layout, React, Dexie, Zustand, ELK, anything that reads the clock or DOM. The single permitted built-in that touches a non-deterministic surface is node:crypto's createHash for SHA-256 of canonical strings (used in output.ts's branchIdFromConditions); SHA-256 is a pure function.

Sort before iterate (custom ESLint rule argmap-determinism/no-unsorted-iteration)

Every for (const x of set) / for (const k of map.keys()) / for (const k in obj) / set.forEach(...) / map.entries() in src/runtime/ must go through sortedIter, sortedKeys, or sortedEntries from iteration-helpers.ts. The custom rule fires on the AST shapes the rule's documentation enumerates; the I.1 skeleton ships the rule and a self-test. Arrays whose order is already deterministic (e.g., frame.nodes sorted-by-id at construction time, policy.all_of as the policy's own ordered list) may be iterated directly.

Frozen clock in tests (vitest.config.ts)

Every test in tests/runtime/ runs under vi.useFakeTimers() with the clock pinned to 2026-05-10T00:00:00Z (the I.1 skeleton's default). Property tests (Vitest acceptance tests below) run each generated case twice with two different frozen times to surface incidental clock dependencies; both runs must produce byte-identical outputs.

No clock reads in the body

The runtime never calls Date.now(), new Date() with no arguments, performance.now(), or any equivalent. Timestamps come into the runtime via the computed_at input field; everywhere a NodeStatus, ConditionalOutput, or other shape requires an ISO-8601 string, the runtime threads computed_at. The ESLint no-restricted-syntax rule (added by the I.3 coding session as part of the runtime configuration block) flags any of these primitives in src/runtime/ source. The custom rule needs one entry; the rule signature is documented in the skeleton's eslint.config.js comment block.

No randomness, no environment branches

The runtime never calls Math.random(), crypto.getRandomValues, process.env (Node), import.meta.env (Vite), navigator.*, or any other source of non-determinism. Branch ids derived in output.ts use SHA-256 of canonical input strings, never a random component.

No LLM in the body

Article II § 2 forbids LLM calls in the routing path. The runtime's import boundary makes this structurally impossible — @/llm-hooks is not importable. The G5 calibrated thresholds reach compute through frame.llm_settings_snapshot.calibrated_thresholds on the FrameVersion, which is a static projection populated at build time by the bake-and-commit hook flow. The G6 prose-rewrite override lives on ArgumentSessionVersion.output_overrides and is consulted only by Stream E's prose tab; compute never reads or writes it.

F-001 wiring — calibrated thresholds in burden_met

Per the F-001 register flag (resolved/APPROVED): G5 calibrates the burden thresholds at build time; the calibrated values are projected onto each FrameVersion as llm_settings_snapshot.calibrated_thresholds: BurdenThresholdMap. The runtime reads this projection from the FrameVersion snapshot it receives and uses the calibrated map in place of DEFAULT_BURDEN_THRESHOLD when present. The data flow:

  1. Build time. The user invokes G5 in the Frame Builder editor; the hook calls the configured LlmProvider; the response (a BurdenThresholdMap with source: "g5_calibrated" and calibrated_from stamped) is reviewed by the user and committed via the three-button confirmation contract. The commit writes frame.llm_settings.calibrated_thresholds.
  2. Frame version save. When a new FrameVersion is saved (per Stream H autosave or milestone), the persistence layer projects frame.llm_settings.calibrated_thresholds onto the new FrameVersion's llm_settings_snapshot.calibrated_thresholds. This freezes the calibration for that version; later edits to frame.llm_settings do not retroactively change older versions.
  3. Compute. The runtime's burden_met condition reads frame.llm_settings_snapshot?.calibrated_thresholds and falls back to DEFAULT_BURDEN_THRESHOLD when absent. The choice is purely additive: legacy FrameVersions without the field resolve to default thresholds via the existing fallback.

The runtime never invokes G5 or any other hook directly; the projection is the only contact point. The G5 wire-in is invisible to compute's caller. Tests pin both branches: a frame fixture with llm_settings_snapshot populated produces a status map computed against the calibrated thresholds; the same fixture with the projection removed produces a status map computed against defaults; the difference is auditable in the snapshot test.

F-002 NOT in the runtime

Per the F-002 register flag (resolved/APPROVED with notes): G6's prose rewrite is a parallel field on ArgumentSessionVersion.output_overrides, never a replacement for ConditionalOutput.prose_summary. The runtime is unaware of output_overrides: compute does not read it; renderProse does not consult it; the canonical prose is the deterministic baseline. Stream E's prose tab reads both fields and presents the canonical prose as the primary surface with the rewritten prose clearly secondary and AI-attributed (per the F-002 NOTE in the original register flag). The runtime spec does not need to do anything for F-002; the boundary is upheld by the import direction (runtime never imports from @/llm-hooks or any module that touches output_overrides).
