This is the project's append-only register of flags raised, resolved, and escalated. It supports three flag categories under Article X. Entries are added but never deleted; superseded entries are marked superseded and cross-referenced.

Flag categories

Three categories, governed by Article X:

  • In-session flag (escalated to register). Article X § 1. In-session flags are normally raised and resolved within the same session and live in that session's self-review report. They are escalated to this register only when the resolution reverses a prior stream decision or is otherwise non-obvious.
  • Register flag. Article X § 2. Logged here for resolution by a review session or a subsequent work session. Used when (a) the flag's resolution requires a fresh perspective, (b) the flag affects multiple streams, or (c) the flag concerns a prior stream's decision.
  • User flag. Article X § 3. Logged here and surfaced as a halting escalation under Article VIII § 4(f). Used only when an escalation trigger under Article V § 3 fires. Proposed Protocol amendments are User flags under Article XIII § 2.

Numbering and identifiers

Each entry receives a unique id of the form F-NNN, monotonically increasing across all categories. Numbering does not reset by category. The register is the authoritative source for the next available number. Next available id: F-017.

Field schemas

In-session escalated flag

id: F-NNN raised_at: ISO-8601 date raised_by: session identifier (e.g., "2026-05-09 / consolidation") subject: short title (one line) description: what the flag is about and why it was escalated to the register streams_affected: comma-separated stream ids (e.g., "B, C") prior_decision_reversed: yes | no resolution: how it was resolved within the session status: resolved | superseded

Register flag

id: F-NNN raised_at: ISO-8601 date raised_by: session identifier subject: short title (one line) description: the flag content; the question to be resolved streams_affected: comma-separated stream ids deferred_reason: (a) needs fresh perspective | (b) multi-stream | (c) concerns prior stream decision | other (specify) status: open | resolved | superseded resolved_at: ISO-8601 date or "—" resolved_by: session identifier or "—" resolution: the answer reached, with constitutional reasoning if relevant review_outcome (when resolved by a review session): APPROVED | APPROVED WITH NOTES | DENIED, with constitutional citations

User flag

id: F-NNN raised_at: ISO-8601 date raised_by: session identifier subject: short title (one line) trigger: which Article V § 3 subsection fired ((a)–(f)); brief justification streams_affected: comma-separated stream ids question_presented: the bounded question delivered to the user under Article IV § 5 response_format: yes/no | short list | free response description: full context the user needs to decide status: pending | answered | withdrawn user_response: verbatim or summarized; date resolution: how the agent applied the user's response amendment_target (Protocol-amendment User flags only, per Article XIII § 2): proposed text + adoption_status (pending | adopted | rejected). Adopted only when the user updates project instructions.

Each is documented below. (1) V-NODE-8. Spec text: "Every CheckpointOption with satisfies: true has a non-null target_node_id, OR the option's target is a Conclusion (terminal answer)." On a literal reading the second clause is tautological — "the option's target is a Conclusion" requires target_node_id to be non-null. An alternative reading: the second clause refers to the Checkpoint's outgoing LEADS_TO targeting a Conclusion ("terminal-answer pattern"); under that reading, an option with satisfies: true and no explicit target_node_id is allowed when the parent Checkpoint structurally LEADS_TO a Conclusion. The implementation initially adopted the alternative reading but Checkpoint is not a valid LEADS_TO source under V-EDGE-1 (only Interpretation and LogicalGate are), so the alternative-reading branch is unreachable at the schema layer. V-NODE-8 was therefore simplified to: satisfies: true requires target_node_id. The simplified rule is functionally equivalent to the literal reading; the alternative reading was a misinterpretation that disagreed with V-EDGE-1. (2) V-FR-8. Spec text: "Every path from RootQuestion eventually terminates at a Conclusion (no dangling Term/Checkpoint that doesn't continue)." V-EDGE-1 forbids Checkpoint as a LEADS_TO source, so a Checkpoint structurally has no outgoing path. The spec's literal reading would make every reachable Checkpoint a V-FR-8 violation, which is plainly wrong. The intended reading: a Checkpoint's path "continues" through its CheckpointOption.target_node_ids. The implementation extends the path-termination DFS to treat each option's target_node_id as a virtual outgoing edge from the Checkpoint. With this extension, a Checkpoint whose options collectively reach a Conclusion is non-dangling; a Checkpoint with no satisfying option that reaches Conclusion is dangling. V-FR-7 ("legal-mode frame has at least one jurisdictional SubQuestion") and V-ARG-3 (Premise kind matches frame mode/flavor vocabulary) both require knowledge of the frame's mode. FrameVersion does not carry mode (it lives on Frame); the schema spec's "Mode-aware schema behavior summary" lists the legal-only fields without saying how a rule operating on FrameVersion alone resolves mode.
