Per-module spec for src/layout/  ·  Stream I session I.7  ·  May 11, 2026

This document is the implementation source of truth for src/layout/. It commits per-file TypeScript signatures and the concrete behavioral bodies of: the pure mapping from FrameVersion to an ELK layered-graph input (anchor handling, collapsed-subtree exclusion, per-node-type default dimensions, sorted iteration), the pure mapping from ELK's result back to the cross-module LayoutResult shape, the per-node-type dimensions table, the ELK algorithm-parameter constants (Sugiyama top-down with orthogonal routing), the Web Worker entry point (elk-worker.ts), the main-thread bridge that owns Worker lifecycle and exposes the public layout() function (elk-bridge.ts), and the test seam (runLayoutSync) that lets Vitest exercise the same logic without a Worker. The cross-module surface (what other modules import from @/layout) is pinned in stream_i_contracts_v1.html Contract 5; this spec refines that contract's layout() signature by adding an optional deps parameter for clock injection (logged as a single in-session-escalated flag in the F-003 family).

The layout module sits outside the runtime determinism boundary (Article II § 2 binds the runtime, not the layout module). src/layout/ may read the clock for the computed_at field on LayoutResult and may use elkjs's internal Promise scheduling. What it must not do is leak any of those non-determinism sources into the runtime: the layout module is never imported by @/runtime (ESLint no-restricted-imports rule from the I.1 skeleton already pins this). The layout module imports only @/schema from the application codebase and elkjs as its sole third-party dependency. The implementing agent gives the module a clean test seam (the synchronous-in-spirit runLayoutSync helper, which is itself async only because elkjs is) so Vitest can exercise the full mapping logic without instantiating a Web Worker.

Cross-document resolution notes (for self-review)

While compiling this spec from stream_i_contracts_v1.html Contract 5, the Stream E E1 design source, and the persistence spec's clock-injection pattern, one precision item arose. It is resolved by this spec in the constitutionally clean direction. It joins the F-003 family (the same family as the I.3, I.4, I.5, and I.6 pending items) and is logged as a single in-session-escalated flag for a future I.1 review session to merge into a contracts-doc bump. Downstream coding is not gated on the merge; this spec is canonical for the layout module per Stream I conventions.

  1. layout() accepts an optional deps parameter for clock injection. Contract 5's layout(frame: FrameVersion, opts?: Partial<LayoutOptions>): Promise<LayoutResult> signature does not provide a way to inject now(). The LayoutResult.computed_at field that the same contract pins requires a clock read at some point. Two constitutional facts make a third parameter the cleanest resolution. (a) Layout is off the determinism boundary (Article II § 2 binds the runtime); the wall-clock read for computed_at is fine in production. (b) But Vitest tests want deterministic outputs (the persistence spec, the runtime spec, the modes spec, and the state spec all follow the same pattern: optional deps with a frozen-clock default in tests). The constitutionally clean resolution is for layout() to accept an optional third parameter deps?: LayoutDeps where LayoutDeps = { now?: () => string } defaults to wall-clock time. The signature becomes layout(frame: FrameVersion, opts?: Partial<LayoutOptions>, deps?: LayoutDeps): Promise<LayoutResult>; production callers omit deps; tests pass a frozen-clock stub. Identical to the LayoutDeps shape runLayoutSync takes internally — the bridge constructs LayoutDeps from its own arguments and threads it through. The dependency graph in Contract 5 is unchanged (layout still imports only @/schema).

The Article V § 3 trigger-recheck for this refinement is in the self-review at the end of this document.

Determinism inside the layout module (practical, not constitutional)

Layout is off the Article II § 2 boundary, but the spec commits a practical determinism property anyway: given identical FrameVersion + identical LayoutOptions + identical deps.now(), two invocations of layout() produce identical LayoutResult. This is not a constitutional guarantee — elkjs may produce non-identical results across versions, and a layout-engine swap is permitted — but it is a useful production property and a hard requirement for the Vitest snapshot tests below. The four mechanisms that make it hold:

  1. Sorted input. frameToElkGraph sorts frame.nodes by id and frame.edges by id before building the ELK input. ELK's layered algorithm uses input order as a tiebreaker in two places (rank assignment ties and intra-rank ordering ties); sorted input pins these.
  2. Model-order tiebreaker. The org.eclipse.elk.layered.considerModelOrder.strategy option is set to NODES_AND_EDGES in ELK_LAYERED_OPTIONS, telling ELK to respect input order as the primary intra-rank ordering criterion when the crossing-minimization heuristic produces ties.
  3. Deterministic ELK algorithm. The layered algorithm in elkjs is deterministic given identical input and options. The random algorithm (which uses a seeded RNG) is not used.
  4. Injected clock. deps.now() is the only non-determinism source in the module; tests inject a frozen-clock stub.

If a future elkjs release breaks property 3, the spec's Vitest snapshot tests will fail in CI before the version reaches production; the implementing agent investigates and either pins the prior elkjs version or updates snapshots. The cross-module contract (LayoutResult shape) is unaffected.

import type { NodeRef, FrameVersion } from "@/schema";

/**
 * One per-node position from a layout computation.
 * `node_id` is a NodeRef; `x` and `y` are in layout units (1 layout unit == 1px
 * for the React Flow consumer in v1; the unit is unspecified at the contract
 * level so the layout module remains tech-stack-neutral).
 */
export interface LayoutPosition {
  node_id: NodeRef;
  x: number;
  y: number;
}

/**
 * Output of a layout computation. `positions` is sorted by `node_id`
 * (lexicographic) for deterministic snapshot equality. `width` and `height`
 * are the bounding box of the laid-out graph as reported by ELK; the UI uses
 * them to set the React Flow viewport's content extent.
 *
 * Off the Article II § 2 boundary: `computed_at` is a wall-clock ISO-8601
 * string in production and an injected frozen-clock value in tests.
 */
export interface LayoutResult {
  positions: LayoutPosition[];
  width: number;
  height: number;
  computed_at: string;
}

/**
 * Caller-tunable layout options. Each field has a sane default; `Partial`
 * is what `layout()` accepts at the call site.
 *   direction: which way layers grow. "DOWN" is the Stream E E1 default
 *     (Sugiyama top-down). "RIGHT" is reserved for v2 alternate views and
 *     functions but is not the visual default for any v1 screen.
 *   honor_user_anchors: when true, Node.x / Node.y (when both present)
 *     are passed to ELK as fixed positions and ELK lays out around them.
 *     Default true. The Stream E "auto-arrange" command runs layout() with
 *     this false (one-shot fresh layout, no anchor effect, no persistence).
 *   collapse_subquestions: when true, descendants of any SubQuestion with
 *     `collapsed: true` are excluded from the layout (per the exclusion
 *     rule in elk-mapping.ts). Default true.
 */
export interface LayoutOptions {
  direction: "DOWN" | "RIGHT";
  honor_user_anchors: boolean;
  collapse_subquestions: boolean;
}

export const DEFAULT_LAYOUT_OPTIONS: LayoutOptions = {
  direction: "DOWN",
  honor_user_anchors: true,
  collapse_subquestions: true,
};

/**
 * Injected dependencies. Production omits this object; tests supply
 * a frozen-clock stub for snapshot determinism.
 *   now: returns an ISO-8601 timestamp string. Defaults to
 *     () => new Date().toISOString().
 */
export interface LayoutDeps {
  now?: () => string;
}

/**
 * Internal helper: resolves `Partial<LayoutOptions>` against
 * DEFAULT_LAYOUT_OPTIONS. Pure; not re-exported from index.ts.
 */
export function resolveLayoutOptions(
  opts: Partial<LayoutOptions> | undefined
): LayoutOptions {
  return { ...DEFAULT_LAYOUT_OPTIONS, ...(opts ?? {}) };
}

/**
 * Internal helper: resolves `LayoutDeps` against wall-clock defaults.
 * Pure; not re-exported from index.ts.
 */
export function resolveLayoutDeps(deps: LayoutDeps | undefined): Required<LayoutDeps> {
  return { now: deps?.now ?? (() => new Date().toISOString()) };
}

Behavioral contract: resolveLayoutOptions(undefined) equals DEFAULT_LAYOUT_OPTIONS; passing a partial overrides only the named fields. resolveLayoutDeps(undefined).now is a wall-clock function; passing { now: () => "2026-05-11T00:00:00.000Z" } pins the result. Both helpers are pure and have unit tests in tests/layout/types.test.ts.

// src/layout/dimensions.ts

Per-NodeType default dimensions, in layout units. The implementing agent treats these as the single source of truth for layout-time node sizing; the UI's React Flow node components (I.9) consume the same constants by importing this file, so the canvas geometry and the ELK-computed positions stay consistent. Values are chosen to match the Stream E E1 shape table: questions are wider than terms (room for prose), Conclusion is widest and tallest (it has the most field content), gates are square (diamond), Premise / Authority are not in this table because they are argument-layer or annotation surfaces and are not laid out by the main canvas (Stream E line 115: argument-layer overlays are surfaced by the argument inspector panel, not the main layout). Authority is present in frame.nodes in legal mode (per Stream B B1.8) and is laid out; its dimension is the "document-shaped" Stream E entry.

import type { LayoutOptions } from "./types";

/**
 * Base ELK options for the layered (Sugiyama) algorithm. Stable across calls;
 * the only per-call variation is direction, supplied by elkDirectionOptions().
 *
 * Rationale per choice (each can be tuned in the visual-polish pass without
 * affecting the cross-module LayoutResult contract):
 *
 *   "elk.algorithm" = "layered"
 *     The Sugiyama-style hierarchical algorithm Stream E E1 commits.
 *
 *   "elk.layered.considerModelOrder.strategy" = "NODES_AND_EDGES"
 *     Asks ELK to break ties in intra-rank ordering and rank-assignment by
 *     input order. Combined with sorted input from frameToElkGraph, this is
 *     mechanism 2 of the four practical-determinism mechanisms above.
 *
 *   "elk.edgeRouting" = "ORTHOGONAL"
 *     Stream E E1 commits "orthogonal or smoothed-bezier paths"; orthogonal
 *     is the default. The UI's React Flow edge components may re-route to
 *     smoothed bezier at render time (a UI concern); the layout's job is to
 *     place nodes such that orthogonal routing reads well. The visual-polish
 *     pass may swap to "POLYLINE" or "SPLINES" without touching this module.
 *
 *   "elk.layered.spacing.nodeNodeBetweenLayers" = 80
 *   "elk.spacing.nodeNode" = 48
 *   "elk.spacing.edgeNode" = 32
 *   "elk.spacing.edgeEdge" = 16
 *     Agent-discretionary spacing defaults. Chosen so the dimensions in
 *     dimensions.ts produce a comfortable canvas at 1.0 zoom on the
 *     three-pane editor's center column. The visual-polish pass may revise.
 *
 *   "elk.padding" = "[top=24,left=24,bottom=24,right=24]"
 *     Outer padding around the graph. Aligns the laid-out content with the
 *     canvas viewport's content extent.
 *
 *   "elk.hierarchyHandling" = "INCLUDE_CHILDREN"
 *     The graph is flat in v1 (no ELK-level nested children — collapsed
 *     subtrees are excluded at the mapping stage, not nested as ELK groups).
 *     This setting is a no-op for flat graphs but documents the intent.
 *
 *   "elk.layered.thoroughness" = "7"
 *     ELK's effort knob for crossing minimization; 7 is the elkjs default
 *     for the layered algorithm. Stream E E1's frames (legal analysis,
 *     typical sub-hundred-node frames) lay out in well under 200 ms at this

// src/layout/elk-mapping.ts

The two pure transforms that bracket the elkjs call. Both are synchronous, pure, sort their inputs, and produce JSON-serializable values (so the Worker boundary's postMessage structured-clone marshals them without custom serializers).

frameToElkGraph(frame, opts)

Maps a FrameVersion to the ELK input graph. Five steps in order:

  1. Compute the visible-node set. If opts.collapse_subquestions is false, every node in frame.nodes is visible. If true, walk from RootQuestion via the BFS-traversal edge types (DECOMPOSES_INTO, TURNS_ON, INTERPRETED_AS, LEADS_TO, GATES, FORECLOSES) treating any SubQuestion with collapsed: true as a frontier: the collapsed SubQuestion itself is visible, but its outgoing structural edges are not traversed. Any node not reached by this BFS is hidden, with two exceptions added unconditionally: Authority nodes are always visible (they may be referenced by multiple SubQuestions, some collapsed, some not — keeping them visible matches the user expectation that citations remain reachable); Premise nodes, if present (defensive — they live on ArgumentSessionVersion, not FrameVersion), are also marked visible so they reach step 3 where the missing-dimension warning fires.
  2. Compute the visible-edge set. An edge is visible iff both its source and target are visible nodes and its type is in LAID_OUT_EDGE_TYPES: the six BFS-traversal types plus the annotation types CITES, BINDING_IN, DISTINGUISHED_BY. Argument-layer edges (SUPPORTS, CONTRADICTS, ANSWERS) do not exist in frame.edges — they live on ArgumentSessionVersion.argument_edges; the filter is defensive against schema drift.
  3. Build ELK nodes. For each visible node, in sorted-id order, construct an ELK child with id (the NodeRef as-is), width and height (from dimensionsFor(node.type); nodes whose type yields undefined are skipped with a console.warn and an entry added to the test-only warnings array), and — when opts.honor_user_anchors is true and the node has both x and y set — an x and y field plus layoutOptions: { "elk.position": "(x,y)", "elk.layered.fixed": "true" }. ELK respects fixed positions during layered layout.
  4. Build ELK edges. For each visible edge, in sorted-id order, construct an ELK edge with id (the EdgeRef as-is), sources: [edge.source], targets: [edge.target]. No edge labels are passed to ELK (label rendering is a UI concern; ELK does not need labels for routing).
  5. Assemble. Return { id: "root", children: nodes, edges: edges, layoutOptions: { ...ELK_LAYERED_OPTIONS, ...elkDirectionOptions(opts.direction) } }. The graph is rooted at the literal string "root" (a constant; not derived from frame.id) so two frames with identical visible-node sets produce identical ELK input.
import type { FrameVersion, Node, Edge, EdgeType, NodeRef } from "@/schema";
import { dimensionsFor } from "./dimensions";
import { ELK_LAYERED_OPTIONS, elkDirectionOptions } from "./elk-options";
import { resolveLayoutOptions, type LayoutOptions } from "./types";

/**
 * Edge types that propagate visibility in the collapse BFS. Reaching a node
 * via one of these from a visible non-collapsed predecessor marks that node
 * visible. FORECLOSES is included so Conclusions reachable only as
 * foreclosure targets remain visible.
 */
const BFS_TRAVERSAL_EDGES: ReadonlySet<EdgeType> = new Set([
  "DECOMPOSES_INTO",
  "TURNS_ON",
  "INTERPRETED_AS",
  "LEADS_TO",
  "GATES",
  "FORECLOSES",
]);

/**
 * Edge types that are passed to ELK as part of the laid-out graph. The
 * BFS-traversal set plus the annotation edges that connect frame-layer
 * nodes (CITES from a SubQuestion / Conclusion / Term to an Authority,
 * BINDING_IN from an Authority to a jurisdictional surface,
 * DISTINGUISHED_BY between Authorities). Annotation edges do not propagate
 * BFS visibility — they are followed by ELK for routing-quality purposes
 * only, after visibility is decided.
 *
 * Notably NOT included: ANSWERS, SUPPORTS, CONTRADICTS. These are
 * argument-layer edges living on ArgumentSessionVersion.argument_edges,
 * not on FrameVersion.edges; they reach the canvas through the UI's
 * argument-overlay rendering layer (Stream E E1 "Argument-layer overlay
 * rendering"), not through ELK. If schema drift ever placed them on
 * FrameVersion.edges, the filter below would drop them defensively.
 */
const LAID_OUT_EDGE_TYPES: ReadonlySet<EdgeType> = new Set([
  ...BFS_TRAVERSAL_EDGES,
  "CITES",
  "BINDING_IN",
  "DISTINGUISHED_BY",
]);

/** JSON-serializable shape elkjs accepts for `layout(graph)`. */
export interface ElkGraph {
  id: string;
  children: ElkNode[];
  edges: ElkEdge[];
  layoutOptions: Record<string, string>;
}

export interface ElkNode {
  id: string;
  width: number;
  height: number;
  x?: number;
  y?: number;
  layoutOptions?: Record<string, string>;
}

export interface ElkEdge {
  id: string;
  sources: string[];
  targets: string[];
}

/**
 * Maps a FrameVersion to the ELK input graph. Pure; sorts inputs; ignores
 * Premise nodes defensively (they should not appear in frame.nodes).
 *
 * `warnings` is a side-channel for the test harness: in tests, the caller
 * passes a fresh array and asserts on its contents after the call. In
 * production, the caller passes undefined and warnings are emitted via
 * console.warn instead.
 */
export function frameToElkGraph(
  frame: FrameVersion,
  opts?: Partial<LayoutOptions>,
  warnings?: string[]
): ElkGraph {
  const resolved = resolveLayoutOptions(opts);

  // Step 0: sort node and edge collections by id once.
  const sortedNodes = [...frame.nodes].sort((a, b) => a.id.localeCompare(b.id));
  const sortedEdges = [...frame.edges].sort((a, b) => a.id.localeCompare(b.id));
  const nodeById = new Map<NodeRef, Node>(sortedNodes.map((n) => [n.id, n]));

  // Step 1: visible-node set.
  const visible = computeVisibleNodes(sortedNodes, sortedEdges, resolved.collapse_subquestions);

  // Step 2: visible-edge set (filter sortedEdges).
  const visibleEdges = sortedEdges.filter(
    (e) => visible.has(e.source) && visible.has(e.target) && LAID_OUT_EDGE_TYPES.has(e.type)
  );

  // Step 3: build ELK nodes.
  const elkNodes: ElkNode[] = [];
  for (const n of sortedNodes) {
    if (!visible.has(n.id)) continue;
    const dim = dimensionsFor(n.type);
    if (!dim) {
      pushWarning(warnings, `frameToElkGraph: skipping node with unhandled type "${n.type}" (id=${n.id})`);
      continue;
    }
    const elkNode: ElkNode = { id: n.id, width: dim.width, height: dim.height };
    if (resolved.honor_user_anchors && typeof n.x === "number" && typeof n.y === "number") {
      elkNode.x = n.x;
      elkNode.y = n.y;
      elkNode.layoutOptions = {
        "elk.position": `(${n.x},${n.y})`,
        "elk.layered.fixed": "true",
      };
    }
    elkNodes.push(elkNode);
  }

  // Step 4: build ELK edges.
  const elkEdges: ElkEdge[] = visibleEdges.map((e) => ({
    id: e.id,
    sources: [e.source],
    targets: [e.target],
  }));

  // Step 5: assemble.
  return {
    id: "root",
    children: elkNodes,
    edges: elkEdges,
    layoutOptions: { ...ELK_LAYERED_OPTIONS, ...elkDirectionOptions(resolved.direction) },
  };
}

/**
 * BFS from RootQuestion; treats collapsed SubQuestions as frontier nodes
 * (visible themselves, but their outgoing structural edges are not
 * traversed). When collapse_subquestions is false, every node is visible.
 *
 * Traversal uses STRUCTURAL_EDGE_TYPES only — annotation edges (CITES /
 * BINDING_IN / DISTINGUISHED_BY / FORECLOSES) do not propagate visibility
 * because they are cross-cutting and would defeat the collapse semantic
 * (e.g., a CITES edge from an Authority into a collapsed subtree's Term
 * should not make the Term visible — the user expected the subtree
 * hidden).
 */
function computeVisibleNodes(
  sortedNodes: Node[],
  sortedEdges: Edge[],
  collapse: boolean
): Set<NodeRef> {
  const all = new Set<NodeRef>(sortedNodes.map((n) => n.id));
  if (!collapse) return all;

  // Authority and Premise nodes (Premise should not be here, but defensively):
  // these are not reachable from RootQuestion via structural edges. Include
  // them as visible iff they exist; the collapse semantic does not hide them.
  const reachableViaStructure = new Set<NodeRef>();
  const root = sortedNodes.find((n) => n.type === "RootQuestion");
  if (root) {
    const queue: NodeRef[] = [root.id];
    reachableViaStructure.add(root.id);
    const nodeById = new Map<NodeRef, Node>(sortedNodes.map((n) => [n.id, n]));
    const outgoingByNode = new Map<NodeRef, Edge[]>();
    for (const e of sortedEdges) {
      if (!BFS_TRAVERSAL_EDGES.has(e.type)) continue;
      const list = outgoingByNode.get(e.source) ?? [];
      list.push(e);
      outgoingByNode.set(e.source, list);
    }
    while (queue.length > 0) {
      const cur = queue.shift()!;
      const curNode = nodeById.get(cur);
      // Collapsed SubQuestion: visible but a frontier. Do not enqueue children.
      if (curNode?.type === "SubQuestion" && (curNode as { collapsed?: boolean }).collapsed) {
        continue;
      }
      const outs = outgoingByNode.get(cur) ?? [];
      // Sort outs by edge id for stable BFS order (already sorted at the
      // sortedEdges level; the partition into outgoingByNode preserves
      // that order because the map is built by iterating sortedEdges).
      for (const e of outs) {
        if (!reachableViaStructure.has(e.target)) {
          reachableViaStructure.add(e.target);
          queue.push(e.target);
        }
      }
    }
  }

  // Authority and Premise: not reachable via structural edges. Include them
  // unconditionally if they appear in frame.nodes; Premise should not appear
  // but the union is harmless if it does (it will be skipped at step 3 with
  // a warning anyway).
  const visible = new Set<NodeRef>(reachableViaStructure);
  for (const n of sortedNodes) {
    if (n.type === "Authority" || n.type === "Premise") visible.add(n.id);
  }
  return visible;
}

function pushWarning(sink: string[] | undefined, message: string): void {
  if (sink) sink.push(message);
  else if (typeof console !== "undefined") console.warn(message);
}

elkResultToLayoutResult(elk_result, computed_at)

Maps elkjs's result object back to the cross-module LayoutResult shape. ELK's layout() resolves with a graph whose children carry x, y, width, height and whose top-level node carries an overall width and height. The mapping extracts per-node positions, sorts by node_id for stable snapshot equality, and emits the bounding box.

import type { LayoutPosition, LayoutResult } from "./types";

/** JSON-serializable shape elkjs produces from `layout(graph)`. */
export interface ElkResult {
  id: string;
  width: number;
  height: number;
  children: { id: string; x: number; y: number; width: number; height: number }[];
}

export function elkResultToLayoutResult(elk_result: ElkResult, computed_at: string): LayoutResult {
  const positions: LayoutPosition[] = elk_result.children
    .map((c) => ({ node_id: c.id, x: c.x, y: c.y }))
    .sort((a, b) => a.node_id.localeCompare(b.node_id));
  return {
    positions,
    width: elk_result.width,
    height: elk_result.height,
    computed_at,
  };
}

Behavioral contracts for elk-mapping.ts:

  • Stable input → stable output. Two invocations of frameToElkGraph(frame, opts) with identical inputs produce structurally identical ElkGraph values (deep equality). The same property holds for elkResultToLayoutResult.
  • Insensitive to input order. Reshuffling frame.nodes or frame.edges in the input has no effect on the output — the function sorts before building the ELK graph.
  • Collapse semantics. A SubQuestion with collapsed: true appears in children; its descendants reachable only through it do not. Shared descendants (reachable from a non-collapsed ancestor) still appear. collapse_subquestions: false includes everything regardless of collapsed hints.
  • Anchor semantics. With honor_user_anchors: true, every node with both x and y set carries the ELK fixed-position options; nodes with one of the two unset are not anchored (defensive: treat partial hints as no hint). With honor_user_anchors: false, no x / y / elk.position / elk.layered.fixed survives into the ELK input.
  • Edge filtering. Only LAID_OUT_EDGE_TYPES reach ELK; argument-layer edges (SUPPORTS, CONTRADICTS) are skipped — they do not exist in frame.edges at all, but the filter is defensive against schema drift. Edges incident to a hidden node are dropped.
  • Defensive against missing RootQuestion. A frame without a RootQuestion (mid-edit, transient V-FR-1 violation) returns an empty children array under collapse_subquestions: true (the BFS has no seed) plus Authority nodes. The implementing agent does not throw; the caller handles the degenerate result. Tests cover this case.
// src/layout/run.ts

The core executor. Wires frameToElkGraph → new ELK().layout(graph, opts) → elkResultToLayoutResult. Async because elkjs returns a Promise; otherwise synchronous-in-spirit. The Worker entry point and the bridge fallback both call this function; tests call it directly without instantiating a Worker.

import ELK from "elkjs/lib/elk.bundled.js";
import type { FrameVersion } from "@/schema";
import { frameToElkGraph } from "./elk-mapping";
import { elkResultToLayoutResult, type ElkResult } from "./elk-mapping";
import { resolveLayoutDeps, type LayoutDeps, type LayoutOptions, type LayoutResult } from "./types";

/**
 * Returns a LayoutResult for the given frame. Reads deps.now() exactly
 * once at the start so the computed_at field is stable for tests.
 *
 * Used by:
 *   - elk-worker.ts (in-Worker call path)
 *   - elk-bridge.ts (fallback when Worker unavailable)
 *   - tests/layout/*  (direct call without Worker indirection)
 *
 * The `warnings` parameter is the test-only side channel passed to
 * frameToElkGraph; production callers omit it.
 */
export async function runLayoutSync(
  frame: FrameVersion,
  opts?: Partial<LayoutOptions>,
  deps?: LayoutDeps,
  warnings?: string[]
): Promise<LayoutResult> {
  const resolvedDeps = resolveLayoutDeps(deps);
  const computed_at = resolvedDeps.now();
  const elkGraph = frameToElkGraph(frame, opts, warnings);
  const elk = new ELK();
  const elkResult = (await elk.layout(elkGraph)) as ElkResult;
  return elkResultToLayoutResult(elkResult, computed_at);
}

Behavioral contract: runLayoutSync is referentially transparent given identical inputs and identical deps.now; the elkjs layered algorithm is deterministic per the practical-determinism property above. One ELK instance is constructed per call; elkjs's instance is cheap to construct (it sets up internal worker-like scheduling in a way that does not start OS threads). Reusing a single instance across calls is not a correctness requirement; the implementing agent may add memoization later as a performance refinement, with care to preserve byte-identity of LayoutResult across calls.

Important: elkjs/lib/elk.bundled.js is the bundled distribution that includes the layered algorithm and ELK's worker-pool implementation (a JavaScript-level pool, not a Web Worker; orthogonal to our Worker). The bundled distribution works identically in node (Vitest) and the browser (production), which is what makes the test seam clean. The non-bundled elkjs/lib/elk-api.js + an external elk-worker.min.js URL is the alternative path; it has been considered and rejected for v1 — the bundled path keeps dependencies under one npm version, simplifies the Vite bundling story, and removes one production failure mode (worker-script fetch error).

// src/layout/elk-worker.ts

The Web Worker entry point. Imports runLayoutSync; receives layout requests via self.onmessage; posts back the result or a serialized error. Owns no state across requests; one Worker can service many sequential requests safely.

/// <reference lib="webworker" />
import { runLayoutSync } from "./run";
import type { FrameVersion } from "@/schema";
import type { LayoutOptions, LayoutResult } from "./types";

/**
 * Request envelope sent main-thread → worker. `request_id` is the bridge's
 * correlation handle; the worker echoes it back in the response so the
 * bridge can route results to the right Promise resolver.
 */
export interface LayoutWorkerRequest {
  kind: "layout_request";
  request_id: string;
  frame: FrameVersion;
  opts?: Partial<LayoutOptions>;
  computed_at: string;   // bridge-supplied so the Worker does not read the clock
}

/**
 * Response envelope worker → main-thread. Exactly one of `result` or
 * `error` is populated; never both.
 */
export interface LayoutWorkerResponse {
  kind: "layout_response";
  request_id: string;
  result?: LayoutResult;
  error?: { message: string; name: string };
}

const ctx: DedicatedWorkerGlobalScope = self as unknown as DedicatedWorkerGlobalScope;

ctx.onmessage = async (event: MessageEvent<LayoutWorkerRequest>) => {
  const req = event.data;
  if (!req || req.kind !== "layout_request") return;
  try {
    const result = await runLayoutSync(req.frame, req.opts, { now: () => req.computed_at });
    const response: LayoutWorkerResponse = {
      kind: "layout_response",
      request_id: req.request_id,
      result,
    };
    ctx.postMessage(response);
  } catch (err) {
    const response: LayoutWorkerResponse = {
      kind: "layout_response",
      request_id: req.request_id,
      error: {
        message: err instanceof Error ? err.message : String(err),
        name: err instanceof Error ? err.name : "UnknownError",
      },
    };
    ctx.postMessage(response);
  }
};

Behavioral contract: the Worker reads no clock — computed_at is supplied by the bridge so the value is consistent regardless of whether the layout ran in the Worker or via the synchronous fallback. The Worker swallows no exceptions; the response always arrives. The Worker imports only ./run and the schema types it needs for the request envelope. tsconfig.json declares the WebWorker library at the file level via the triple-slash reference so self resolves as DedicatedWorkerGlobalScope.

The Vite bundler picks up this file by URL at the bridge's new Worker(new URL("./elk-worker.ts", import.meta.url), { type: "module" }) call site. No worker-specific Vite configuration is needed; the default { type: "module" } path uses native ES module workers in modern browsers.

// src/layout/elk-bridge.ts

The main-thread bridge. Single Worker instance, lazily constructed on first call, reused for subsequent calls. Each layout() invocation:

  1. Resolves deps against the wall-clock default and computes computed_at on the main thread (so the value is identical whether the request runs in the Worker or the fallback).
  2. Generates a unique request_id (a monotonic counter; sufficient because request lifetime is per-bridge-instance).
  3. Registers a Promise resolver in the in-flight map keyed by request_id.
  4. Posts the request envelope to the Worker (or, in the no-Worker fallback, calls runLayoutSync directly).
  5. Returns the Promise.

When the Worker's response arrives, the bridge looks up the resolver by request_id, removes the in-flight entry, and resolves (or rejects, on Worker-reported errors) the Promise.

import type { FrameVersion } from "@/schema";
import { runLayoutSync } from "./run";
import { resolveLayoutDeps, type LayoutDeps, type LayoutOptions, type LayoutResult } from "./types";
import type { LayoutWorkerRequest, LayoutWorkerResponse } from "./elk-worker";

let workerInstance: Worker | null = null;
let nextRequestId = 1;
const inFlight = new Map<string, { resolve: (r: LayoutResult) => void; reject: (e: Error) => void }>();

function ensureWorker(): Worker | null {
  if (typeof Worker === "undefined") return null;     // Vitest in node, no Worker fallback
  if (workerInstance) return workerInstance;
  const w = new Worker(new URL("./elk-worker.ts", import.meta.url), { type: "module" });
  w.onmessage = (event: MessageEvent<LayoutWorkerResponse>) => {
    const res = event.data;
    if (!res || res.kind !== "layout_response") return;
    const handler = inFlight.get(res.request_id);
    if (!handler) return;                              // stale; bridge.terminate() may have cleared
    inFlight.delete(res.request_id);
    if (res.error) handler.reject(Object.assign(new Error(res.error.message), { name: res.error.name }));
    else if (res.result) handler.resolve(res.result);
    else handler.reject(new Error("layout_response missing both result and error"));
  };
  w.onerror = (event: ErrorEvent) => {
    // Catastrophic Worker error (e.g., script load failure). Reject every
    // pending request; clear the in-flight map; null out the instance so
    // the next call reconstructs.
    const err = new Error(event.message || "layout worker error");
    for (const { reject } of inFlight.values()) reject(err);
    inFlight.clear();
    workerInstance = null;
  };
  workerInstance = w;
  return w;
}

/**
 * Public entry point (Contract 5, refined per the F-003-family flag for
 * the optional `deps` parameter). Returns a Promise that resolves with the
 * LayoutResult for `frame` under `opts`. Reads `deps.now()` exactly once
 * on the main thread.
 */
export function layout(
  frame: FrameVersion,
  opts?: Partial<LayoutOptions>,
  deps?: LayoutDeps
): Promise<LayoutResult> {
  const resolvedDeps = resolveLayoutDeps(deps);
  const computed_at = resolvedDeps.now();

  const w = ensureWorker();
  if (!w) {
    // Fallback: run on the main thread. We pass the computed_at-as-frozen-clock
    // into runLayoutSync so the result is identical to the Worker path.
    return runLayoutSync(frame, opts, { now: () => computed_at });
  }

  const request_id = String(nextRequestId++);
  const promise = new Promise<LayoutResult>((resolve, reject) => {
    inFlight.set(request_id, { resolve, reject });
  });
  const req: LayoutWorkerRequest = {
    kind: "layout_request",
    request_id,
    frame,
    opts,
    computed_at,
  };
  w.postMessage(req);
  return promise;
}

/**
 * Test-only utility: terminates the Worker (if any), clears the in-flight
 * map. Not part of the public surface in production code, but exported so
 * test teardown can release resources. Subsequent layout() calls will
 * reconstruct the Worker.
 */
export function terminate(): void {
  if (workerInstance) {
    workerInstance.terminate();
    workerInstance = null;
  }
  for (const { reject } of inFlight.values()) {
    reject(new Error("layout worker terminated"));
  }
  inFlight.clear();
  nextRequestId = 1;
}

Behavioral contracts:

  • Lazy Worker construction. No Worker is created until the first layout() call. Apps that never lay out (unlikely, but defensible) pay no startup cost.
  • Single Worker reuse. One Worker services all calls. Sequential layout() calls queue at the Worker (postMessage is buffered); the Worker processes them in order. Concurrent calls are not rejected — each gets its own Promise correlated by request_id.
  • Identical results across paths. The Worker path and the fallback path produce byte-identical LayoutResult for the same input, because both call runLayoutSync with the same computed_at-as-frozen-clock and the same opts.
  • Catastrophic-error recovery. If the Worker errors out (e.g., script load failure), all pending Promises reject with a single shared error message, and the next layout() call reconstructs the Worker. The bridge does not retry automatically; the caller decides whether to retry.
  • Test-only termination. terminate() is exposed for Vitest teardown; tests that exercise the Worker path call it in afterEach to avoid hanging open Workers. Production code does not call it; the Worker lives until the page unloads.
  • No request cancellation in v1. Cancellation would require a Worker-side per-request abort signal; not worth the v1 complexity. The UI debounces layout calls (only one in-flight at a time per FrameVersion); stale results that arrive after a newer call started are discarded at the UI layer by checking the FrameVersion id, not at the bridge. Logged as a v2 refinement candidate below.

Web Worker lifecycle and message protocol

The Worker is a thin executor — it owns no state across requests, and the message protocol is a single request/response pair per layout.

main thread (bridge)                    worker thread
─────────────────────                   ─────────────
layout(frame, opts, deps)
  │
  │ computed_at = deps.now()
  │ request_id  = String(nextRequestId++)
  │ inFlight.set(request_id, { resolve, reject })
  │ worker.postMessage({
  │   kind: "layout_request",
  │   request_id, frame, opts, computed_at
  │ })
  │                                       onmessage receives request
  │                                       runLayoutSync(frame, opts,
  │                                         { now: () => computed_at })
  │                                       postMessage({
  │                                         kind: "layout_response",
  │                                         request_id,
  │                                         result | error
  │                                       })
  │ onmessage receives response          ◄────────────────────────
  │ handler = inFlight.get(request_id)
  │ inFlight.delete(request_id)
  │ handler.resolve(result) or handler.reject(error)
  │
  └─► Promise resolved to LayoutResult

Lifecycle invariants:

  • Worker outlives individual requests. The Worker is constructed once (lazy) and reused for all subsequent layout calls in the page session.
  • Message order is preserved. Browsers preserve postMessage ordering per channel; multiple concurrent requests posted before any response arrives are processed in send order by the Worker. This is not relied on for correctness — request_id correlation is the canonical mechanism — but it produces predictable timing for tests.
  • No shared memory. The structured-clone serialization at postMessage ensures the Worker sees a fresh copy of frame; any in-Worker mutation would not affect the main thread (and no in-Worker mutation occurs anyway — frameToElkGraph is pure).
  • No DOM or window access from the Worker. The Worker has access only to elkjs and the schema-type imports (which are erased at compile time). No DOM, no localStorage, no fetch, no navigator is used.
  • Worker teardown. The Worker is not terminated automatically on idle in v1; it lives until the page unloads. The terminate() export is test-only. A v2 refinement candidate is idle-timeout termination — logged below.
