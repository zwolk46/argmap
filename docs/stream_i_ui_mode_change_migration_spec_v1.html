stream_i_ui_mode_change_migration_spec_v1.html

Per-module spec for src/ui/mode-change/ and src/ui/session-migration/  ·  Stream I session I.9d2  ·  May 11, 2026

This document is the implementation source of truth for two related but page-scoped UI surfaces delivered in the same session: src/ui/mode-change/ (the architectural-mode-change dialog and its lighter sibling, the flavor-change dialog — both reached from FrameSettingsPanel inside FrameBuildingPage) and src/ui/session-migration/ (the session-migration dialog reached from FrameVersionDriftIndicator inside ArgumentRunningPage). It commits per-file TypeScript signatures and behavioral contracts for: the architectural-mode-change dialog with its target picker, scan-result body, per-Conclusion ConclusionDirectionEditor inline editors, optional inline positions editor for the legal→general-with-no-positions case, and advisory list; the flavor-change dialog as a lighter ConfirmDialog wrapper that commits immediately per E6; the session-migration dialog with its grouped OrphanCandidate rows, per-row resolution pickers pre-seeded with suggested_kind, and the new SessionStore.previewMigration wiring; and the small page-level refinements that lift the dialog open-state into FrameBuildingPage and ArgumentRunningPage rather than the App (the dialogs are page-local; F-006's App-level mount pattern does not apply here). The I.9a substrate (Dialog, ConfirmDialog, DialogHeader/Body/Footer) and the I.9b / I.9c page-prop callbacks (FrameSettingsPanel.on_open_mode_change_dialog reserved by F-004; FrameVersionDriftIndicator.on_open_migration_dialog reserved by F-005) are the foundation this spec consumes.

Two cross-spec refinements are committed and held in the consolidated F-007 register flag in the same F-003 / F-004 / F-005 / F-006 family pattern: (1) a new FramePatch kind architectural_mode_changed in the state spec that atomically applies mode (and flavor, when relevant) + per-Conclusion direction resolutions + optional new positions in one FrameVersion — parallel to I.9b's presentation_hints_reset_all precedent in F-004; and (2) a new SessionStore.previewMigration(target_frame_version_id): Promise<OrphanCandidate[]> method that wraps Repository.loadFrameVersion + enumerateOrphanCandidates from @/modes and re-exports OrphanCandidate through @/state so the UI consumes through the selector boundary. Three minor refinements join the same flag: a new dispatch-table entry in the modes spec for architectural_mode_changed; extension of metadata_edited.partial's Pick tuple to include flavor (joining the F-004 note on archived); and removal of the now-superseded onOpenModeChangeDialog? and on_open_migration_dialog? page-props from FrameBuildingPageProps and ArgumentRunningPageProps respectively (the dialogs are page-local, so the page provides its own handler internally — the I.9b and I.9c "I.9d wires this through the App" reservations are honored by this concrete wiring rather than by an App-level mount). Downstream coding sessions are not gated on a contracts-doc bump; the per-module specs are canonical per Stream I conventions.

What this spec does not commit

This spec deliberately omits:

  • The session-settings panel (I.9d3 scope). The placeholder disabled IconButton slot the I.9c buildTopBarSlots renders is unchanged by this session.
  • The onboarding wizard and coachmark machinery (I.9d4 scope).
  • App-level dialog mount or app-routes refinement. Both I.9d2 dialogs are page-local (architectural-mode-change is reached only from FrameBuildingPage's settings panel; session-migration is reached only from ArgumentRunningPage's drift indicator). The F-006 App-level pattern (a single mount serving multiple pages via useRoute()) does not apply. app-routes.tsx is not touched by this session.
  • Tech-stack decisions. The dialog primitives, focus-trap mechanics, and React Flow integration come from the I.9a substrate; this spec composes them without restating their internals.
  • Migration-target selection beyond the current FrameVersion. Per D2 and the I.9c FrameVersionDriftIndicator, migration always targets the parent Frame's current_version_id. Migrating a session to an older specific version is reachable only through E5's restore flow on the frame side first; the I.9d2 dialog does not offer arbitrary-version-target selection.

Cross-document resolution notes (for self-review)

Five small refinements relative to prior I.9 sibling specs. Each is technical, additive (or wiring-promotion of placeholders the prior specs explicitly reserved), and within agent-autonomous scope per Article V § 2(c). All five are held in the consolidated F-007 register flag below in the same F-003 / F-004 / F-005 / F-006 family pattern.

  1. New FramePatch kind: architectural_mode_changed. The architectural-mode-change dialog needs to atomically commit (a) the new Frame.mode value, (b) the new Frame.flavor value (when target is general), (c) optionally new Frame.positions entries (when target is general and current positions are empty/absent and the frame has Conclusions), and (d) per-Conclusion direction updates resolving the blocking items the D3 scan surfaced. The existing metadata_edited patch operates on Frame-metadata fields only; the existing node_edited patch operates on one node at a time. Multiple sequential dispatches would produce N FrameVersions and N autosave debounces — awkward in the version-history surface and not atomic. The state spec's FramePatch union gains | { kind: "architectural_mode_changed"; target_mode: Mode; target_flavor?: Flavor; positions_added?: Position[]; conclusion_direction_resolutions: Array<{ node_id: NodeRef; direction: ConclusionDirection }> } and the modes spec's FrameActionDispatchTable gains a corresponding entry that, in one pass, produces a next_version in which frame.mode = target_mode, frame.flavor = target_flavor (when defined; otherwise undefined for legal target), frame.positions = [...(frame.positions ?? []), ...(positions_added ?? [])], and each Conclusion in the resolutions array has its direction field replaced with the resolved value. The dispatch entry follows the single-clock-read property (one deps.now()) and the sorted-iteration discipline (the resolutions array is walked in lexicographic node_id order even if the caller supplied it in arbitrary order, ensuring byte-identical output). The patch's change_summary is constructed by the caller (the dialog) per E6's "mode changed: legal → general" template; the dispatch entry itself does not synthesize the change summary. Purely additive; no existing patch shape is changed. Parallel in spirit to F-004's presentation_hints_reset_all.
  2. New SessionStore.previewMigration method. The session-migration dialog needs the OrphanCandidate[] list before the user picks resolutions. Two constitutional facts shape the surface. (a) The state layer is the application's source of truth for entities passed to @/modes functions per the F-003 family discipline (modes does not import Repository for pure helpers; state does and threads through). (b) The UI consumes selector-shape outputs from @/state rather than reaching into @/modes directly per the I.9c precedent (selectInterviewItems wraps computeInterviewOrder and re-exports InterviewItem through @/state). The constitutionally clean resolution is for the SessionStore to expose a previewMigration(target_frame_version_id: FrameVersionId): Promise<OrphanCandidate[]> method that wraps this.deps.repository.loadFrameVersion(target_id) + enumerateOrphanCandidates(this.snapshot.session, target_frame_version), returning the canonical-order candidates array. The state spec re-exports OrphanCandidate from @/state's barrel so the UI imports it via import type { OrphanCandidate } from "@/state". The method is read-only (no patch dispatched, no FrameVersion created, no autosave touched); the actual migration commit happens through the existing session_store.migrateToFrameVersion(target_id, resolutions) method.
  3. New dispatch-table entry in the modes spec. The modes spec's FrameActionDispatchTable gains architectural_mode_changed as a synchronous pure transformation: (args: { current_frame: Frame; current_version: FrameVersion; target_mode: Mode; target_flavor?: Flavor; positions_added?: Position[]; conclusion_direction_resolutions: Array<...>; deps: FrameActionDeps }) => FrameTransformResult. The implementation walks current_version.nodes sorted by id, replacing each Conclusion's direction when its id appears in conclusion_direction_resolutions (a Map built from the resolutions array for O(1) lookup); updates current_frame's mode, flavor, and positions fields; produces the new FrameVersion with created_at = deps.now() and a fresh id from deps.generateId(). Behavioral contracts: pure synchronous; deterministic (single deps.now(), sorted iteration, sorted-by-id resolutions lookup); over-supplied resolutions (entries whose node_id does not match an existing Conclusion) are silently dropped; under-supplied resolutions (Conclusions whose direction.kind would mismatch the new mode but no resolution provided) are caught by an invariant check that throws — the dialog enforces completeness before dispatch, so this is a dev-time wiring guard. The Vitest acceptance tests in the modes module cover all three paths.
  4. Extension of metadata_edited.partial Pick tuple to include flavor. The flavor-change dialog dispatches frame_store.applyEdit({ kind: "metadata_edited", partial: { flavor: target } }). The current Pick tuple in the state spec is "title" | "description" | "tags" | "jurisdiction_default" | "positions"; F-004 already noted that archived joins this tuple. This refinement adds flavor in parallel — both are simple Frame-metadata fields and both are absorbed by the existing metadata_edited dispatch-table entry without further changes. The schema spec already exposes Frame.flavor as the source field; no schema change. Mode is NOT added to the Pick tuple because every Frame.mode change flows through the new architectural_mode_changed patch above.
  5. Removal of onOpenModeChangeDialog? and on_open_migration_dialog? page-props. F-004 reserved FrameBuildingPageProps.onOpenModeChangeDialog?: (target: "mode" | "flavor") => void for I.9d wiring, and F-005 reserved ArgumentRunningPageProps.on_open_migration_dialog?: () => void in parallel. Both reservations anticipated an App-level dialog mount in the F-006 pattern. I.9d2's discretionary call under Article V § 2(c): both dialogs are page-local (architectural-mode-change is reached only from FrameBuildingPage's FrameSettingsPanel; session-migration is reached only from ArgumentRunningPage's FrameVersionDriftIndicator), so a page-internal mount is simpler and avoids the cross-page coordination the App-level pattern was designed for in the version-history pane. The two page-props are removed from their respective Props interfaces; in their place, FrameBuildingPage and ArgumentRunningPage each own a small piece of dialog open-state (specified under "Page-level refinements" below) and pass an internally-defined handler down to FrameSettingsPanel.on_open_mode_change_dialog / FrameVersionDriftIndicator.on_open_migration_dialog. The component-level callback props on FrameSettingsPanel and FrameVersionDriftIndicator are unchanged — only the upstream page-prop reservation is dissolved. This honors F-004's and F-005's intent (the placeholder behavior is replaced by real wiring) without requiring an app-routes.tsx refinement parallel to F-006.

Per-file specs — src/ui/mode-change/

// src/ui/mode-change/architectural-mode-change-dialog.tsx

The top-level architectural-mode-change dialog. Composes the I.9a Dialog primitive with a header, scrollable body, and footer. Owns the staged-resolution state until commit. Reads the current Frame and current FrameVersion from useFrameStore(); runs the D3 architectural scan via useModeChangeScan; tracks per-Conclusion resolutions and per-staged-position additions in local component state; gates the Commit button on "every blocking item has a resolved direction AND positions are present-or-staged when target=general-with-Conclusions." On commit, dispatches the new architectural_mode_changed FramePatch through frame_store.applyEdit and closes.

import * as React from "react";
import type {
  NodeRef, Mode, Flavor, Position, ConclusionDirection,
} from "@/schema";
import type { ConclusionDirectionEditor } from "@/modes";

export interface ArchitecturalModeChangeDialogProps {
  open: boolean;
  onClose: () => void;
  /** Which target the dialog opens against. "mode" means the user clicked
   *  "Change mode" (flips legal ↔ general). "flavor" means the user clicked
   *  "Change flavor" within general mode — but flavor changes do NOT use this
   *  dialog; the page routes flavor target to FlavorChangeDialog instead.
   *  This prop is included for forward-compatibility and asserts at runtime
   *  that target === "mode" when the dialog mounts; the page guards this. */
  target: "mode" | "flavor";
  /** Optional focus-on-mount target for the implicated Conclusion node when
   *  the dialog opens from a deep-link or coachmark; v1 unused. */
  focus_node_id?: NodeRef;
}

export function ArchitecturalModeChangeDialog(
  props: ArchitecturalModeChangeDialogProps,
): React.ReactElement | null;

/** Local component state shape (not exported). Held in a useReducer so the
 *  state transitions are testable in isolation. */
type DialogReducerState =
  | { phase: "idle" }
  | {
      phase: "staging";
      target_mode: Mode;
      target_flavor: Flavor | undefined;          // required when target_mode = "general"
      staged_positions: Position[];                // empty initially; filled when positions-inline-editor stages
      direction_resolutions: Map<NodeRef, ConclusionDirection>; // per-Conclusion resolutions
      scan: ReturnType<typeof useModeChangeScan>["result"] | null;
    }
  | { phase: "committing" };

type DialogReducerAction =
  | { type: "open"; current_mode: Mode; current_flavor: Flavor | undefined }
  | { type: "set_target_flavor"; flavor: Flavor }
  | { type: "scan_complete"; scan: ReturnType<typeof useModeChangeScan>["result"] }
  | { type: "stage_position"; position: Position }
  | { type: "remove_staged_position"; position_id: string }
  | { type: "set_direction_resolution"; node_id: NodeRef; direction: ConclusionDirection }
  | { type: "begin_commit" }
  | { type: "reset" };

Behavioral contract. Mount and unmount: the dialog component receives open as a prop; when open is false the component returns null (and the inner Dialog is not mounted at all — the focus-trap reset is the I.9a Dialog primitive's responsibility on its own close handler). On open, the reducer transitions to "staging" with default target_mode = the opposite of current_mode and default target_flavor = "personal" if target_mode = "general" else undefined. Body composition: <TargetModePicker /> at the top, followed by <PositionsInlineEditor /> (conditionally — only when target_mode = "general", current frame.positions is empty/absent, and frame has at least one Conclusion), followed by <ScanResultBody />. The scan runs via useModeChangeScan({ current_version, current_mode, target_mode, staged_positions }); the hook rescans when staged_positions changes (because new positions change the options[] for direction editors). Footer: "Cancel" (secondary) and "Commit mode change" (primary, destructive=false; the button is destructive=false because per D3 no data is destroyed — fields become inactive metadata). Commit is enabled iff can_commit selector returns true: (a) target_mode is defined; (b) when target_mode === "general", target_flavor is defined; (c) when target_mode === "general" and frame has Conclusions, frame.positions has at least one entry OR staged_positions has at least one entry; (d) every ConclusionDirectionEditor in scan.inline_editors has an entry in direction_resolutions. Commit dispatch: builds the architectural_mode_changed patch payload with change_summary = the canonical template from buildModeChangeSummary(current_mode, current_flavor, target_mode, target_flavor) (defined in change-summary.ts below; produces e.g. "mode changed: legal → general (personal)"); calls frame_store.applyEdit(patch); on resolve transitions reducer to "idle" and calls props.onClose(). Cancel: transitions to "idle" and calls props.onClose(); the in-flight reducer state is discarded (no draft is preserved across dialog re-opens; the user starts fresh each time, which is consistent with E5's reasoning that draft state should not be silently restored). Escape and outside-click: I.9a Dialog's default handlers fire onClose, which the component routes through the cancel path. Determinism: the dialog reads no clock; the eventual deps.now() read is inside the modes dispatch-table entry, single-clock-read-per-dispatch per F-003 family discipline.

Two visual-polish notes from stream_e_visual_polish_v1.html § "Per-component token assignments": dialog surface uses --color-surface-elevated, --shadow-lg, --radius-lg; blocking-item rows in the scan-result body carry a --color-severity-error left-border; advisory rows use --color-text-tertiary for body text. The dialog size is size="lg" per the I.9a primitive (architectural-mode change is consequential and benefits from wider scan-result inspection).

The body section that renders the TransitionResult from attemptTransition('architectural', ...). Two grouped sections: blocking items (each with an inline ConclusionDirectionEditorRow) and advisory items (read-only).    *  When target = legal, undefined. The ConclusionDirectionEditor's options
   *  array already encodes the available choices from the modes-layer scan,

export function ScanResultBody(props: ScanResultBodyProps): React.ReactElement;

Behavioral contract. Section ordering: blocking section first (when non-empty), advisory section second (when non-empty). Determinism: inline_editors and advisory are consumed verbatim and rendered in their canonical order; no client-side sorting. The direction_resolutions map is read but never mutated.

UI-layer determinism discipline (restated for I.9d2's surfaces)

Per the I.9a / I.9b / I.9c / I.9d1 disciplines, the UI layer's contribution to the application's determinism guarantee (Article II § 2) is four obligations: D1 canonical output verbatim, D2 single-clock-read-per-dispatch, D3 deterministic order (no client-side sorting of canonical lists), D4 stable cross-render identity for memoized projections. Each surface in this spec honors them:

  • Architectural-mode-change dialog: D1 — TransitionResult.blocking / advisory messages and the per-Conclusion ConclusionDirectionEditor rows are rendered verbatim from the modes-layer scan. D2 — the dialog reads no clock; the eventual deps.now() read is inside the modes dispatch-table entry's single application of architectural_mode_changed, per F-003 family discipline. D3 — inline_editors, blocking, advisory arrays consumed in their canonical id-sorted order; the AdvisoryList's rule_id grouping preserves first-occurrence order; positions added through the inline editor are appended in insertion order (also deterministic for the user's session). D4 — the dialog reducer's state machine has stable transitions; the scan hook memoizes on a stable input tuple.
  • Flavor-change dialog: D1 — scanFlavorChange's advisory[] rendered verbatim. D2 — no clock read in the dialog; metadata_edited dispatch reads deps.now() once in its existing dispatch-table entry. D3 — advisory[] consumed in canonical order. D4 — applicable to the memoized scan result.
  • Session-migration dialog: D1 — OrphanCandidate.display_summary rendered verbatim; the per-row resolution picker's labels and ordering are constants. D2 — no clock read in the dialog; the eventual writes (Repository.migrateSession) thread their own atomicity contract. D3 — candidates consumed in canonical (carrier_kind, carrier_id) order; partitioning into groups preserves order; resolutions emitted in candidates' order at commit time. D4 — the resolutions Map keyed by carrier_id has stable identity across render; the candidates array reference is stable until a re-load.
