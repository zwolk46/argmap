Conceptual questions resolved autonomously (Article V § 1 / § 2(c))

The prior current_state.html I.9d4 reading-list paragraph flagged three technical questions for this session to verify or resolve, with the instruction that each is within agent-autonomous scope unless substantial hesitation under Article IV § 4 fires. None fires; all three resolve below.

Q1 — Does dismissed_coachmarks already exist on AppState?

Yes, as coachmark_dismissals. The canonical AppState definition lives in stream_i_persistence_spec_v1.html at line 129; line 137 reads coachmark_dismissals: { [coachmark_id: string]: boolean };. The state spec's AppStateStore.dismissCoachmark(coachmark_id, dismissed?) (line 377) and selectCoachmarkDismissed(state, coachmark_id) (line 740) are already pinned. The reading-list paragraph used the name dismissed_coachmarks informally; the canonical name is coachmark_dismissals and is in place from I.4 onward (it predates F-004, which added the parallel-but-distinct dismissed_warnings array). The useCoachmark hook in this spec consumes the existing surfaces unchanged; no schema work is required for coachmark dismissal itself.

Q2 — Does the one-time new-feature notice mechanism need schema work?

Yes, minimally: a new seen_new_feature_notices field on AppState. Two defensible designs.

  • Option Reuse. Use the existing coachmark_dismissals map with an id-prefix convention (e.g., "new_feature:options_box_override_v1"). Pro: zero schema change. Con: conflates reset semantics. E7's "reset coachmarks" affordance is designed to "restore them for testing or for users who want a refresher" — re-surfacing tutorial coachmarks. A user wanting a refresher of the coachmarks does not want to re-see new-feature notices announcing features that have been live for months; those should remain one-time per user per feature.
  • Option Separate. Add seen_new_feature_notices: { [notice_id: string]: boolean } to AppState alongside coachmark_dismissals. Pro: clean reset semantics — resetCoachmarks() clears coachmark_dismissals only, leaving notices untouched. Con: one additional additive field.

Decision: Option Separate. The semantic distinction E7 implicitly carries (coachmarks are first-encounter contextual hints; new-feature notices are one-time-per-user feature announcements) compiles into different reset rules; preserving the distinction in the schema lets the reset affordance behave correctly without prefix-string heuristics. Article III § 4 (Clarity) and § 5 (Reversibility) both favor the separate field. The substantial-hesitation test does not fire: both options are reasonable, the choice is reversible, and the difference is purely technical. Recorded as F-009 refinement #4 below.

Q3 — Where does the reset-coachmarks affordance live?

Inside the I.9a HelpGlossaryPane, as a composed section at the bottom. Four candidates considered.

  • FrameSettingsPanel (I.9b). Rejected. The panel is per-frame; coachmark dismissals are per-user (AppState-resident, per I.9b line 142: "Cross-page state (recents, pinned, coachmark dismissals, default output view) lives in app_state"). Placing a per-user reset inside a per-frame panel mixes scopes.
  • SessionSettingsPanel (I.9d3). Rejected by I.9d3 itself (stream_i_ui_session_settings_spec_v1.html line 38: "The session-settings panel does not host a 'reset coachmarks for this session' affordance; coachmark dismissals are AppState-scoped (per F-007's read of selectCoachmarkDismissed), so resetting is a frame-or-app-level concern that I.9d4 places elsewhere"). The deferral is honored here.
  • New user-preferences surface. Rejected as scope creep. v1 has no other user-preferences-scoped UI fields beyond default_output_view (already exposed elsewhere) and app_provider_default (a hidden field not surfaced in v1 UI per Stream G). Building a new top-level surface for a single toggle is not justified; if a real user-preferences surface emerges later, the affordance migrates there.
  • HelpGlossaryPane (I.9a). Chosen. The pane is the only existing user-level surface (opened from the chrome's HelpButton, not scoped to any Frame or Session). E7's onboarding ladder — welcome screen → coachmarks → help/glossary → returning-user new-feature notices — places the help/glossary surface in the same semantic neighborhood as the onboarding system; locating the reset toggle there matches user expectations ("I want to see the help and tutorials again"). Implementation is small: OnboardingPreferencesSection (this module) renders as a composed child below the existing two glossary sections.

Article III § 6 (Interface Fit) and § 5 (Reversibility) both favor the HelpGlossaryPane placement. Substantial-hesitation test does not fire: reversible, reachable from constitutional principles, not user-personal. Recorded as F-009 refinement #11 below.

// src/ui/onboarding/onboarding-wizard.tsx

The top-level component. Replaces the I.9a placeholder. Returns null when open is false. When open is true, renders a non-dismissible I.9a Dialog; Escape and click-outside route through the same dismiss path as the explicit "Skip introduction" button so the AppState dismissal is recorded uniformly. The wizard does not own AppState or store dispatches; it receives onSkip and onSubmit as props from app-routes.tsx, which wires them to app_state_store.dismissCoachmark and app_state_store.createFrame per refinement #10.

import * as React from "react";
import type { ReactElement } from "react";
import { Dialog } from "@/ui";
import type { CreateBlankFrameArgs, CreateFromTemplateArgs } from "@/state";
import { WelcomeScreen } from "./welcome-screen";
import { NewFrameWizard } from "./new-frame-wizard";

export interface OnboardingWizardProps {
  /** When true, the wizard is mounted and visible. Driven by selectFirstLaunchDismissed. */
  open: boolean;
  /** Fires when the user chooses "Skip introduction" on the welcome screen, or
      Escape / click-outside the dialog. The wrapper in app-routes.tsx dispatches
      app_state_store.dismissCoachmark(FIRST_LAUNCH_COACHMARK_ID, true). */
  onSkip: () => void;
  /** Fires from the wizard's final step. The wrapper in app-routes.tsx awaits
      app_state_store.createFrame(args), then dismisses first-launch, then
      navigates to the new frame. */
  onSubmit: (args: CreateBlankFrameArgs | CreateFromTemplateArgs) => Promise<void>;
}

export function OnboardingWizard(props: OnboardingWizardProps): ReactElement | null;

Behavioral contract. open false → returns null. open true → mounts Dialog, seeds local stage to "welcome". WelcomeScreen.onStartGuided transitions stage to "wizard". NewFrameWizard.onSubmit calls props.onSubmit(args) and awaits; while pending, the wizard's internal submitting flag is true (managed by the NewFrameWizard's reducer and gated at its submit button). On resolve, the parent's dismissal flow flips props.open to false on the next render — the wizard unmounts naturally; no internal stage reset is required. The Dialog's dismiss_on_outside_click and dismiss_on_escape are both true; both route to a shared dismiss handler that calls props.onSkip (so dismissal records AppState consistently regardless of which gesture the user uses). Route-change does not dismiss. If the user navigates away mid-wizard (e.g., by hand-editing the URL hash), the wizard remains mounted because open is route-independent.

UI-layer determinism discipline (I.9a four obligations)

The onboarding module is not on the Article II § 2 determinism boundary. The four UI-layer obligations restated for I.9d4's surfaces:

  1. D1 — Canonical surface verbatim. The welcome screen renders WELCOME_SCREEN_COPY verbatim from a static constant; each coachmark renders COACHMARK_MESSAGES[id] verbatim; new-feature notices render NEW_FEATURE_NOTICE_DEFINITIONS[id].short_message verbatim. The wizard's step captions (MODE_CAPTIONS, FLAVOR_CAPTIONS) are static. No client-side string interpolation derives content from runtime data.
  2. D2 — Single-clock-read per dispatch. resetCoachmarks and markNewFeatureNoticeSeen route through the existing set(patch) wrapper (one autosave-schedule). createFrame awaits modes.createBlankFrame or modes.createFrameFromTemplate, each reading deps.now() exactly once for the new-version timestamp per F-003 family discipline. The wizard reads no clock; the wizard's reducer has no time-derived fields.
  3. D3 — Deterministic order. The wizard's step sequence is fixed by reducer rules. The coachmark anchor positioning is deterministic for a given (anchor-rect, viewport-rect, preference) triple. The OnboardingPreferencesSection renders one button — no ordered list. The wizard's template-step list of frame summaries is rendered in the canonical order returned by Repository.listFrameSummaries (already deterministic per Stream H H1); the UI does not re-sort.
  4. D4 — Stable identity across renders. The useCoachmark hook returns the same ref instance across renders. WIZARD_STEP_TITLES and the caption tables are frozen constants. The wizard's reducer produces new state objects (no in-place mutation). The useAppStateStore selector subscriptions use the existing useSyncExternalStoreWithSelector identity discipline per the state spec.

The implementing session adds these obligations to src/ui/onboarding/README.md in the I.9a "what we do not do" pattern.
