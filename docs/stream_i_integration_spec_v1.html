§ 1 — Home page module: src/ui/home/

The Home page is the App's landing surface for users who have completed (or skipped) the first-launch onboarding wizard. It is the v1 realization of { kind: "home" } in src/ui/chrome/routing.ts, replacing the one-line placeholder the I.9a substrate spec reserves. The page renders a pinned-frames row, a recents grid, a search affordance, two new-frame buttons that mount NewFrameWizard in a Dialog, and a help-button surface lifted from the I.9a chrome.

Stream E E2 line 207 names this surface: "From Home, the user enters Frame Building by: opening an existing frame, clicking 'New frame' (which runs the new-frame wizard from E7), or clicking 'New frame from template.'" E7's empty-state copy is verbatim ("No frames yet. Create your first frame, or start from a template."). The page does not duplicate the welcome-screen or coachmark machinery; the wizard component is the same one I.9d4 ships, mounted in a different parent.

Layout

The TopBar renders in a "Home" variant: the operating-mode toggle is disabled with the explanatory tooltip "Open a frame or session to switch modes"; the version-history toggle is disabled with "Available inside a frame or session"; the HelpButton is enabled. Per-component behavioral contracts

// src/ui/home/home-page.tsx
import * as React from "react";
import { useFrameStore, useAppStateStore, selectPinnedFrames } from "@/state";
import type { FrameSummary, FrameId } from "@/state";
import { TopBar } from "@/ui/chrome/top-bar";
import { RecentsSection } from "./recents-section";
import { PinnedSection } from "./pinned-section";
import { SearchResultsSection } from "./search-results-section";
import { HomeSearchRow } from "./home-search-row";
import { NewFrameButtonRow } from "./new-frame-button-row";
import { HomeEmptyState } from "./home-empty-state";
import { useHomeSearch } from "./use-home-search";

export interface HomePageProps {
  /** Optional initial search query (deep-link support; v1 reserves but does not surface from URL). */
  initial_query?: string;
}

export function HomePage(props: HomePageProps): React.ReactElement {
  const frame_store = useFrameStore();
  const recents: FrameSummary[] = frame_store.recents();
  const pinned_ids = useAppStateStore(s => selectPinnedFrames(s.current()));

  const pinned_summaries = recents.filter(s => pinned_ids.includes(s.id));
  const recents_minus_pinned = recents.filter(s => !pinned_ids.includes(s.id) && !s.archived);
  const search = useHomeSearch(props.initial_query);
  const is_empty = pinned_summaries.length === 0 && recents_minus_pinned.length === 0 && !search.is_searching;

  return (
    <>
      <TopBar variant="home" />
      <main className="argmap-home">
        <HomeSearchRow query={search.query} onChange={search.set_query} />
        {pinned_summaries.length > 0 && <PinnedSection summaries={pinned_summaries} />}
        {search.is_searching
          ? <SearchResultsSection results={search.results} />
          : <RecentsSection summaries={recents_minus_pinned} />}
        {is_empty ? <HomeEmptyState /> : <NewFrameButtonRow />}
      </main>
    </>
  );
}

Behavioral contract. HomePage renders synchronously from snapshot data; FrameStore.recents() is populated by the I.9a App-mount sequence (which calls frame_store.refreshRecents() immediately after seeding). On first render with an unseeded store, recents is an empty array and the empty-state renders until the seed completes — the visual transition is one frame and is not part of the test contract. The page reads no clock and dispatches no patches directly; all dispatches go through the section components. Search: while a query is active, RecentsSection is replaced in-place by SearchResultsSection; the pinned row is preserved (pinned remains visible across search to support "pin then search-discover other entries"). Empty state: the HomeEmptyState CTAs mirror NewFrameButtonRow behaviorally so empty-state users have the same affordances.

// src/ui/home/frame-summary-card.tsx
import * as React from "react";
import { useNavigate } from "@/ui/chrome/routing";
import { useAppStateStore } from "@/state";
import { PinnedCapReached } from "@/state";
import type { FrameSummary } from "@/state";
import { ModeFlavorChip, PinToggle, TagList } from "@/ui/primitives";
import { formatRelative } from "@/ui/primitives/format-relative";
import { useToast } from "@/ui/chrome/toast";

export interface FrameSummaryCardProps {
  summary: FrameSummary;
  variant?: "recents" | "pinned" | "search_result";
}

export function FrameSummaryCard(props: FrameSummaryCardProps): React.ReactElement {
  const navigate = useNavigate();
  const app_state_store = useAppStateStore();
  const toast = useToast();
  const { summary } = props;

  const handleOpen = React.useCallback(() => {
    // Deterministic order: recordRecent fire-and-forget (debounced), then navigate.
    void app_state_store.recordRecent(summary.id);
    navigate({ kind: "frame_building", frame_id: summary.id });
  }, [app_state_store, navigate, summary.id]);

  const handleTogglePin = React.useCallback(async (e: React.MouseEvent) => {
    e.stopPropagation();  // do not also trigger the card-open
    try { await app_state_store.togglePinned(summary.id); }
    catch (err) {
      if (err instanceof PinnedCapReached) {
        toast({ kind: "error", message: err.message });
      } else { throw err; }
    }
  }, [app_state_store, summary.id, toast]);

  return (
    <article
      className={`frame-summary-card frame-summary-card--${props.variant ?? "recents"}`}
      tabIndex={0} role="button"
      onKeyDown={e => { if (e.key === "Enter" || e.key === " ") { e.preventDefault(); handleOpen(); } }}
      onClick={handleOpen}
      aria-label={`Open frame: ${summary.title}`}
    >
      <header>
        <h3>{summary.title}</h3>
        <ModeFlavorChip mode={summary.mode} flavor={summary.flavor} />
        <PinToggle pinned={summary.pinned} onToggle={handleTogglePin} />
      </header>
      {summary.description && <p className="description">{summary.description}</p>}
      <footer>
        <time dateTime={summary.updated_at}>{formatRelative(summary.updated_at)}</time>
        {summary.tags && summary.tags.length > 0 && <TagList tags={summary.tags} />}
      </footer>
    </article>
  );
}

Behavioral contract. The card is the only place in the Home module where dispatches fire. Open: synchronous; recordRecent is fire-and-forget (the navigation does not wait), since the dispatch is debounced through the 1-second AppState autosave and the user's perception of "open" is the navigation. Toggle pin: awaited; on PinnedCapReached surfaces a toast and leaves state unchanged. Keyboard: Enter and Space both dispatch open (matches the I.9a IconButton convention); the pin toggle is reachable via Tab and has its own activation. The card emits no other interactions; right-click context menus are reserved for v1.5.

// src/ui/home/use-home-search.ts
import * as React from "react";
import { useRepository } from "@/state";
import type { FrameSearchHit } from "@/state";

export interface HomeSearch {
  query: string;
  set_query: (q: string) => void;
  results: ReadonlyArray<FrameSearchHit>;
  is_searching: boolean;
}

export const HOME_SEARCH_DEBOUNCE_MS = 300;

export function useHomeSearch(initial_query?: string): HomeSearch {
  const repo = useRepository();
  const [query, set_query] = React.useState(initial_query ?? "");
  const [results, set_results] = React.useState<ReadonlyArray<FrameSearchHit>>([]);
  const [is_searching, set_is_searching] = React.useState(false);

  React.useEffect(() => {
    if (query.trim() === "") { set_results([]); set_is_searching(false); return; }
    set_is_searching(true);
    const t = setTimeout(async () => {
      const hits = await repo.searchFrames(query);
      set_results(hits);
    }, HOME_SEARCH_DEBOUNCE_MS);
    return () => clearTimeout(t);
  }, [query, repo]);

  return { query, set_query, results, is_searching };
}

Behavioral contract. The hook calls repository.searchFrames with a 300ms trailing-edge debounce. Empty query resets results to empty and is_searching to false. The hook does not own loading-state UX (which is the section component's concern); it owns the data-fetching contract.

// src/ui/home/new-frame-button-row.tsx
import * as React from "react";
import { Dialog, Button } from "@/ui/primitives";
import { NewFrameWizard } from "@/ui/onboarding";    // F-009 reusable component
import { useNavigate } from "@/ui/chrome/routing";
import { useAppStateStore } from "@/state";
import type { CreateFrameArgs } from "@/state";

export function NewFrameButtonRow(): React.ReactElement {
  const navigate = useNavigate();
  const app_state_store = useAppStateStore();
  const [open_kind, set_open_kind] = React.useState<"blank" | "template" | null>(null);

  const handleSubmit = React.useCallback(async (args: CreateFrameArgs) => {
    const frame_id = await app_state_store.createFrame(args);
    set_open_kind(null);
    navigate({ kind: "frame_building", frame_id });
  }, [app_state_store, navigate]);

  return (
    <>
      <div className="new-frame-button-row">
        <Button onClick={() => set_open_kind("blank")}>+ New frame</Button>
        <Button variant="secondary" onClick={() => set_open_kind("template")}>+ New frame from template</Button>
      </div>
      <Dialog open={open_kind !== null} onClose={() => set_open_kind(null)}>
        <NewFrameWizard
          start_at_template_step={open_kind === "template"}
          onSubmit={handleSubmit}
          onSkip={() => set_open_kind(null)}
        />
      </Dialog>
    </>
  );
}

Behavioral contract. Two buttons, one Dialog reused for both. The "template" variant passes start_at_template_step: true — an additive F-009 prop on NewFrameWizard that pre-positions the reducer state at step 5 (template-selection) while still allowing back-navigation to earlier steps. The wizard's existing onSubmit / onSkip contracts are unchanged; the Home page does not duplicate creation logic. Note for the implementing session: if F-009's NewFrameWizard does not already expose start_at_template_step, the I.10 implementation adds it as a one-line additive prop in the wizard's reducer-initial-state computation (additive; default false; legacy mountings unaffected); this is a small refinement recorded as I.10 #4 in the cross-document resolution notes below.

UI-layer determinism discipline (the four obligations)

The Home module is not on the Article II § 2 runtime determinism boundary. The four UI-layer obligations restated for this module:

  1. D1 — Canonical surface verbatim. The empty-state copy is the verbatim E7 string, exported as HOME_EMPTY_STATE_COPY. The two button labels are static constants. Mode/flavor chip labels are read from FrameSummary.mode and FrameSummary.flavor; no UI-side string interpolation derives content from runtime data beyond the standard relative-time pattern that formatRelative handles deterministically (pure function of timestamp + clock anchor).
  2. D2 — Single-clock-read per dispatch. The Home module dispatches three actions: recordRecent, togglePinned, and createFrame. Each goes through the existing AppStateStore wrappers that read deps.now() exactly once. The card's open-action does not read the clock; the navigation is a Route update, not a data mutation.
  3. D3 — Deterministic order. Recents are rendered in the canonical order returned by FrameStore.recents() (the order maintained by AppStateStore.recordRecent, which dedupes and prepends). Pinned frames are rendered in the canonical order of state.pinned (the order in which the user pinned them). Search results are rendered in the order repository.searchFrames returns. The Home page never re-sorts.
  4. D4 — Stable identity across renders. The useHomeSearch hook returns the same set_query callback identity across renders. The button labels and empty-state copy are frozen constants. The FrameSummaryCard is React-memoized on summary.id + summary.updated_at + summary.pinned so re-renders are O(changed-cards).

The implementing session adds these obligations to src/ui/home/README.md.

§ 2 — Scenario-grouped Vitest integration tests: tests/integration/

Eight scenario files mirroring the buildkit's eight high-level acceptance criteria. Each file mounts the full App with the integration-test harness (per stream_i_contracts_v2.html Cross-cutting: integration-test harness expectations) and exercises the user flow at the React-Testing-Library level. The tests assert behavior, ARIA roles, text content, and dispatched patch shapes — not pixel positions, not snapshot diffs of rendered HTML. Each scenario produces a discrete, named test plan; coding sessions implement until the named cases pass.

The eight scenarios

Scenario 1 — determinism-gate.test.tsx

Maps to buildkit criterion 1. Verifies the runtime's byte-identity property survives the action-runner + Repository + AutosaveController round trip.

  • Two runs with byte-identical input produce byte-identical persisted blobs (modulo timestamp fields created_at, evaluated_at, computed_at, updated_at).
  • The validation_results array is identical across runs.
  • The status_map is identical across runs.
  • The active_path is identical across runs.
  • The decision_tree node order is identical across runs.
  • Iteration-order audit: re-runs with a different frozen-clock anchor produce identical outputs everywhere except the explicit timestamp fields.

Scenario 2 — frame-building.test.tsx

Maps to buildkit criterion 2. Exercises the full Frame Building loop in both legal and general modes.

  • Create a blank legal frame via the wizard from Home; lands in Frame Building with one stub RootQuestion and its coachmark visible.
  • Add every NodeType from the palette; the canvas reflects each addition with the correct status badge.
  • For each NodeType combination valid per V-EDGE-1, drag from the connector handle to create an edge; the inspector shows the edge type and the satisfaction-policy filter offered matches Mode + Flavor (legal-only conditions hidden in general; authority_required opt-in for academic).
  • Edit per-instance options_box on Checkpoint, Question, and Interpretation nodes; the inspector's three-tier display (effective / source / edit) shows the override correctly.
  • Save a milestone via the TopBar affordance; the version-history pane lists the milestone with the supplied change_summary.
  • Open the version-history pane, preview the prior version (read-only canvas), restore it; the canvas reflects the restored version, the version-tree shows the new fork.
  • Attempt Frame → Argument transition with a failing V-FRAME-* rule; the transition is blocked with the validation drawer pre-expanded.
  • Fix the violation, re-attempt; the transition succeeds, lands in Argument Running with the open-items interview populated.
  • Parallel happy paths: general / personal flavor with observation/value/assumption/stipulated Premise kinds; general / academic flavor with empirical/definitional/normative/stipulated kinds. Each verified against the inspector's Premise-kind picker.

Scenario 3 — argument-running.test.tsx

Maps to buildkit criterion 3. Exercises the open-items interview, the three output views, and cross-pane status propagation.

  • Create a session against a seeded legal frame; the interview pane renders open items in the D2 order with jurisdictional items grouped first.
  • The recommended-next row is decorated.
  • Answer a Checkpoint with each of the four canonical answer-types (boolean true, boolean false, multiple-choice each option, graded met/not_met/unclear); the status-map updates and the recommended decoration moves.
  • Select an Interpretation; the dependent Term's status updates.
  • Attach a Premise to a Checkpoint with an optional Authority reference (legal mode); the Premise pool and the Authority section both reflect the addition.
  • The three output views (path overlay default, decision tree, prose summary) each render the same conclusion shape from the same compute result.
  • Toggle the G6 prose rewrite once it has been cached by the MockLlmProvider — the AIAttributionChip renders verbatim from HookInvocationRecord.
  • The status palette colors are consistent across the canvas, the outline tree, and the interview pane (via selectStatusBadge).
  • The output_view_tab_choice_by_frame persists across reload.

Scenario 4 — persistence.test.tsx

Maps to buildkit criterion 4. Exercises the H2 debounce, the H1 atomicity contracts, template deep-copy, session migration, version restore, the cross-tab BroadcastChannel, and the search index.

  • Make N edits within the 5-second debounce window; only one save persists at the trailing edge.
  • Make an edit at T0, idle until T+5s; the save persists.
  • Make continuous edits for 30 seconds; the 30-second cap fires a save mid-stream.
  • createFrameFromTemplate deep-copies every node and edge with new UUIDs; the original template is unchanged; both end in a single Dexie transaction.
  • migrateSessionToFrameVersion + orphan resolutions land in one transaction (verified by attempting a mid-transaction read which sees the prior version).
  • restoreFrameVersion from an ancestor creates a fork; the resulting FrameVersion has parent_version_id equal to the ancestor's parent.
  • Cross-tab edit conflict surfaces via peer_save on the BroadcastChannel; the second tab's save-attempt creates a fork (no data loss).
  • Search across title, description, node text, tags, and Conclusion statements all return the expected FrameSearchHit[] with the correct match_kind.

§ 3 — Playwright E2E plan: tests/e2e/

Playwright complements the Vitest integration tests by exercising the same scenarios against a real Vite build with a real browser. The trade-off: Playwright catches real-browser regressions (CSS, real React Flow rendering, IndexedDB-in-browser quirks, hashchange routing in a real window) that jsdom misses, at the cost of slower runtime. Playwright runs in a separate CI job after the build step. This dual surface honors both the buildkit's I.10 framing ("end-to-end Playwright scenarios") and current_state.html's framing ("scenario-grouped Vitest integration tests"); the two surfaces serve different roles.

The buildkit skeleton already ships tests/e2e/smoke.spec.ts verifying the placeholder page renders. I.10 extends with five additional scenario specs covering the canonical happy paths. Each Playwright scenario is a lightweight version of its Vitest sibling — fewer test cases (3–5 vs. 8–12), focused on the user-visible happy path rather than internal patch-shape assertions.

What this spec does not commit

  • Visual polish. Specific color values, typography sizing, animation curves, drop-shadows, the precise card visual treatment, the breakpoint pixel values for the responsive grid — visual-polish-doc territory. The spec pins behavior and structure.
  • Drag-and-drop reordering of pinned frames. v1 pins in insertion order. A future polish pass may add explicit reordering; not in v1 because state.pinned is a FrameId[] array with no per-frame ordering metadata beyond array position, and a reordering surface would require either a wrap-on-AppState dispatch or a per-pin ordering field that v1 schemas do not carry.
  • Tag-based filtering or grouping on Home. Tags are already searchable via the search row's full-text query (which the persistence search index covers). A dedicated tag-filter chip set is reserved for v1.5 if usage data shows demand.
  • Archived frames surface. v1 hides archived frames from Recents and from the default Pinned view (a pinned archived frame is hidden by the !s.archived filter). Surfacing archived frames is reserved for v1.5 (a "Show archived" toggle in the filter row).
  • Sessions list on Home. The Home page lists frames only. Per-frame session lists land inside Frame Building (E2 entry surface) and inside Argument Running. The buildkit's E2 line 305 framing — "from Home's session list" — is honored by not introducing a separate top-level sessions surface on Home; sessions are reached through their parent frame.
  • Detailed Playwright fixture scaffolding beyond the six specs above. Each Playwright spec pre-seeds via window.__argmap_test_seed; the implementing session writes the page-level helper. Fixture sharing (e.g., a beforeAll-seeded frame used by multiple specs) is a polish concern.
  • Cross-browser Playwright coverage. v1 ships Chromium-only Playwright. Firefox and WebKit specs land in v1.5 if cross-browser regressions surface.
  • Performance benchmarks. The buildkit's H2 performance budget is not formalized as a CI gate in v1 because the budget is itself agent-discretionary (Stream H deferred concrete numbers). v1.x polish may add benchmark assertions to specific scenarios.
  • The F-006 v1.5 Repository batch loader. Recorded as a deferral note in stream_i_contracts_v2.html Contract 3. Not built in v1. If a future polish pass surfaces a perceptible Compare-view flash, the batch loader is the clean additive surface.
  • Concrete UUID values for integration-test assertions. The harness emits a counter-backed UUID sequence; tests that snapshot output assert against the sequence by relative position, not against absolute UUID strings. This insulates the test plan from UUID-generator implementation choice; should the implementing session pick a different counter shape (e.g., RFC 4122 v7), the assertion pattern adapts without re-specifying scenarios.
