Stream E — Visual Polish (Design Tokens)

Companion to stream_e_uiux_v1.html  ·  Session date: May 10, 2026  ·  Version 1

This document specifies the visual polish layer that stream_e_uiux_v1.html deliberately deferred under "Items deferred → Specific color values, typography, and animation curves." The structure that Stream E committed — pairing rules, layout topology, interaction patterns, every surface-to-surface consistency contract — is not modified. This document fills in concrete defaults for color, typography, spacing, radii, elevation, and motion as a CSS-custom-property token set, compileable against the React Flow CSS variable surface committed in Stream H H4 and against the AI-attribution affordance specified in Stream G. The token set is the canonical artifact; the prose around it explains rationale, coverage, and constraint compliance.

How to read this

The tokens in §§ Color / Typography / Spacing / Borders / Elevation / Motion are the deliverable. The §§ Surface coverage / React Flow integration / F-002 carry / Reduce motion sections show the tokens applied to specific Stream E surfaces, so an implementer can tell, for each Stream E surface, which tokens are involved without having to reverse-engineer the choices. The end-of-document self-review verifies the work against the Constitution.

Conventions used here:

  • All token names are CSS custom properties at :root scope; no namespace prefix. The implementing agent may add a project prefix (e.g., --ag-) at scaffolding time without changing the token system.
  • Color values are given in HSL plus hex for clarity. HSL is the canonical form because adjacent-hue manipulations (chip backgrounds at higher lightness, hover states at lower lightness) read more naturally from HSL.
  • Where Stream E or Stream G fixed an icon or shape, this document specifies only the color / size / motion treatment, not the icon itself.

Decisions made in this session

The decisions below are agent-discretionary under Article V § 2(c). Each is sanctioned by Stream E's "agent-discretionary defaults" framing and by Article IV § 1 (the user delegates technical decisions, and these are visual-technical defaults). None reverses a prior stream decision; none weakens an architectural commitment; none forecloses a use case.

  1. Aesthetic register: serious, neutral, lawyerly; closer to a working editor than to a marketing surface. Article III § 2 (Legal Practice Fit) is the load-bearing constraint here. The product is for appellate analysis, statutory work, and trial prep; the chrome should feel like something a federal judge would tolerate. That rules out high-saturation accents, soft "playful" curves, celebratory motion, and consumer-SaaS gradients. The tokens prefer warm-tinted neutrals as the workhorse, restrained accents, and a single muted accent per operating mode.
  2. Token system over per-component CSS. The deliverable is design tokens, not implemented components. Component implementation is the implementing coding agent's job under Article V § 2(c) and is best done at Stream I. Tokens give the implementer a fixed palette to compose against.
  3. Font selection: Inter (sans), Source Serif 4 (serif, prose surfaces only), JetBrains Mono (mono). All three are SIL OFL or Apache-2.0 licensed and free for commercial use, removing licensing risk for an open-source-friendly v1. Inter is validated for information-dense interfaces (Linear, Notion, Stripe). Source Serif 4 is reserved for the prose-summary tab where lawyerly idiom benefits from a serif. JetBrains Mono carries ids and machine-readable fields without competing with the body type.
  4. Status palette: hue choices that are accessibly distinct AND tonally restrained. Per Stream E E4, the pairing rule (color + icon) is structural and fixed; this document picks the actual hues. The picks aim for WCAG AA contrast against the canvas background (target ≥ 4.5:1 for the icon glyphs at their target sizes) and for distinguishability under common color-vision deficiencies. The icon does the work for users who can't differentiate the hues; the color does the work for everyone else.
  5. Operating-mode accent shift: slate-blue (Frame Building) ↔ burnt sienna (Argument Running), used minimally. Per Stream E E6 the shift is "minimal — enough to confirm the mode at a glance, not enough to feel like two different applications." The two accents are equally restrained and complementary; they share lightness and chroma, differing primarily in hue. They appear only in: top-bar accent line, active operating-mode pill background, focus rings, and primary-button hover states.
  6. AI-attribution accent: muted plum, distinct from every structural color. Per Stream G the "✦" chip identifies AI-touched fields. The plum hue (around HSL 280) is intentionally outside the palette of status colors, mode accents, edge colors, and severity colors, so a plum chip cannot be mistaken for any other signal. Muted, not vibrant — the chip is informational, not promotional.
  7. Motion: 150 ms standard, single easing function, reduce-motion-aware. Stream E E4 fixed ~150 ms for the status badge transition; this document adopts that as the base duration and supplies a small motion scale (instant / fast / base / medium / slow) plus a single dominant easing curve. The "soft pulse" on the recommended-next item (Stream E E3) gets its own gentler curve and longer duration. All animations collapse to instant under prefers-reduced-motion: reduce.
  8. Single light theme in v1; tokens structured for theme swapping. Dark theme is not in Stream E's specification and not in any other stream's. Token names are theme-neutral (--color-surface-canvas rather than --color-white) so a dark theme can be added by overriding the same token names under [data-theme="dark"] without touching components. This is consistent with Article III § 5 (Reversibility) and with Stream E's "Items deferred → Accessibility detailed audit," which explicitly defers polish work that touches contrast modes.

Color tokens

Surfaces and text

The base substrate. Warm-tinted neutrals so the chrome reads as paper-and-ink rather than as a clinical white-on-pure-white.

:root { /* Surfaces (warm-neutral grays) */ --color-surface-canvas: hsl(36 16% 97%); /* #F8F6F2 — main background, graph canvas */ --color-surface-pane: hsl(36 12% 95%); /* #F2EFEA — left and right panes, side panes */ --color-surface-elevated: hsl(0 0% 100%); /* #FFFFFF — inspector cards, popovers, dialogs */ --color-surface-overlay: hsla(30 8% 12% / 0.42); /* dialog backdrop scrim */ --color-surface-hover: hsl(36 14% 92%); /* node and row hover wash */ --color-surface-selected: hsl(36 18% 89%); /* node and row selected wash */ /* Text */ --color-text-primary: hsl(30 8% 12%); /* #1F1C18 — body, node statements */ --color-text-secondary: hsl(30 6% 38%); /* #65605B — labels, captions, secondary UI */ --color-text-tertiary: hsl(30 5% 56%); /* #918C87 — placeholders, hints, disabled */ --color-text-on-accent: hsl(0 0% 100%); /* white on filled accent surfaces */ --color-text-on-status: hsl(30 8% 12%); /* dark on pale status backgrounds (chips) */ /* Borders */ --color-border-subtle: hsl(30 8% 88%); /* pane separators, table dividers */ --color-border-default: hsl(30 8% 80%); /* card outlines, input borders */ --color-border-strong: hsl(30 8% 65%); /* selected/focused outlines, RootQuestion thicker border */ }

Status palette (Stream E E4)

The five statuses each get a distinct hue, paired with an icon at the component layer (per E4's structural pairing rule). Backgrounds (-bg variants) are used for chip fills behind the icon at low lightness so the badge reads at small sizes.

:root { /* satisfied — restrained green, not festive */ --color-status-satisfied: hsl(145 36% 38%); /* #3F8155 */ --color-status-satisfied-bg: hsl(145 32% 92%); /* #DEEEDF */ /* open — neutral mid-gray; "no signal yet" reads as neutral */ --color-status-open: hsl(30 6% 45%); /* #797470 */ --color-status-open-bg: hsl(30 8% 92%); /* #ECE9E5 */ /* contested — warm amber, attention without alarm */ --color-status-contested: hsl(34 75% 44%); /* #C47A1C */ --color-status-contested-bg: hsl(34 75% 93%); /* #FAEBD8 */ /* foreclosed — dark gray glyph + muted brick-red strikethrough */ --color-status-foreclosed: hsl(30 8% 32%); /* #57524C — the X glyph */ --color-status-foreclosed-accent: hsl(355 38% 42%); /* #94434A — the strikethrough overlay */ --color-status-foreclosed-bg: hsl(30 8% 88%); /* #DFDBD6 */ /* not_applicable — light desaturated gray; visibly faded vs. open */ --color-status-not-applicable: hsl(30 5% 70%); /* #B8B3AE */ --color-status-not-applicable-bg: hsl(30 6% 95%); /* #F3F1ED */ }

Sub-flag chips — legal mode (Stream E E4)

The "⚖ B" / "⚖ P" chips appear on satisfied badges in legal mode only. Binding reads weighty (deep institutional navy); persuasive reads lighter (medium slate). Both chips sit on a pale wash so they remain legible at small sizes.

:root { --color-subflag-binding: hsl(218 42% 30%); /* #2C3F6E — "⚖ B" glyph and label */ --color-subflag-binding-bg: hsl(218 38% 94%); /* #E4ECF6 */ --color-subflag-persuasive: hsl(215 22% 48%); /* #607692 — "⚖ P" glyph and label */ --color-subflag-persuasive-bg: hsl(215 22% 95%); /* #EAEEF3 */ }

Edge color families (Stream E E1)

Edge styling per type encodes relation visually. The structural family is the workhorse neutral; foreclosure carries muted red weight; argument-layer overlays use distinct positive/negative hues that are deliberately offset from the satisfied/contested status hues so an edge cannot be misread as a node-status echo.

:root { --color-edge-structural: hsl(30 6% 32%); /* #5A5550 — DECOMPOSES_INTO, TURNS_ON, INTERPRETED_AS, LEADS_TO, GATES, CheckpointOption (virtual) */ --color-edge-foreclosure: hsl(355 36% 44%); /* #99494F — FORECLOSES, dashed with stop-cap */ --color-edge-supports: hsl(182 40% 36%); /* #378187 — SUPPORTS, dotted, weight = strength tier */ --color-edge-contradicts: hsl(12 62% 47%); /* #BE5530 — CONTRADICTS, dotted, weight = strength tier */ --color-edge-answers: hsl(268 16% 46%); /* #6F647F — ANSWERS, dotted, neutral argument-layer */ --color-edge-annotation: hsl(30 6% 62%); /* #A29D97 — CITES, DISTINGUISHED_BY, thin gray with glyph */ }

Validation severity (Stream E E2)

The three severities cover the validation indicator (top bar), validation drawer entries, per-node chips, and inspector validation block. Pass and warning intentionally reuse the satisfied and contested hues so the visual vocabulary is consistent across the runtime status surface and the validation surface. Error uses a more saturated red than the foreclosure red because errors must be unmistakable.

:root { --color-severity-pass: var(--color-status-satisfied); --color-severity-pass-bg: var(--color-status-satisfied-bg); --color-severity-warning: var(--color-status-contested); --color-severity-warning-bg: var(--color-status-contested-bg); --color-severity-error: hsl(0 68% 46%); /* #C32B2B — distinct from foreclosure red; for V-FR-* errors and FrameVersion blocking */ --color-severity-error-bg: hsl(0 68% 95%); /* #F9E2E2 */ }

Operating-mode accents (Stream E E6)

The "subtle chrome accent shift" between Frame Building and Argument Running. Two restrained tones at matched lightness/chroma. The implementing component reads --color-mode-current-accent (set on <html> or a top-level wrapper) and uses that single token everywhere chrome adapts to the operating mode. This keeps every chrome surface in lockstep without branching component logic.

:root { --color-mode-frame-accent: hsl(216 28% 42%); /* #4D627F — slate-blue, Frame Building */ --color-mode-frame-accent-bg: hsl(216 28% 94%); /* #E5EAF1 */ --color-mode-argument-accent: hsl(18 38% 44%); /* #9B5839 — burnt sienna, Argument Running */ --color-mode-argument-accent-bg: hsl(18 38% 94%); /* #F5E5DD */ } [data-mode="frame-building"] { --color-mode-current-accent: var(--color-mode-frame-accent); --color-mode-current-accent-bg: var(--color-mode-frame-accent-bg); } [data-mode="argument-running"] { --color-mode-current-accent: var(--color-mode-argument-accent); --color-mode-current-accent-bg: var(--color-mode-argument-accent-bg); }

AI-attribution accent (Stream G "✦")

Per Stream G's UI confirmation contract and AI-attribution section, fields whose current value was committed via an LLM hook show a "✦" chip. The plum hue is intentionally outside the palette of status, severity, mode, and edge colors so a plum chip cannot be mistaken for any other signal. The accent appears at three intensities depending on context (chip glyph, chip background fill, surface left-border for the rewritten_prose subordinate surface from F-002).

:root { --color-ai-accent: hsl(282 30% 45%); /* #7C5292 — "✦" glyph and chip border */ --color-ai-accent-bg: hsl(282 30% 95%); /* #EFE8F3 — chip fill, rewritten_prose surface wash */ --color-ai-accent-strong: hsl(282 32% 36%); /* #654276 — chip on focus / hover, AI-settings link in inspector */ }

Milestone marker (Stream E E5)

The "★" star marking is_milestone: true versions in the version-history side pane. A warm gold reads as the canonical "saved/marked" signifier without competing with status colors.

:root { --color-milestone-star: hsl(38 60% 48%); /* #C49321 — milestone star glyph */ }

Focus ring

Keyboard-focus indication. Reads from the current operating-mode accent so focus feels native to whichever screen the user is on, with a fixed offset and width so the ring is consistent.

:root { --focus-ring: 2px solid var(--color-mode-current-accent); --focus-ring-offset: 2px; }

Typography tokens

Font families

Three families. Sans is the default for nearly everything. Serif appears in exactly one place (the prose-summary tab) where the legal/academic idiom benefits from it. Mono carries ids, hashes, and code-style fields.

:root { --font-sans: 'Inter', system-ui, -apple-system, BlinkMacSystemFont, 'Segoe UI', Roboto, sans-serif; --font-serif: 'Source Serif 4', Georgia, 'Times New Roman', serif; --font-mono: 'JetBrains Mono', 'Fira Code', Menlo, Consolas, monospace; }

Type scale

A tight scale optimized for an information-dense editor. base is the workhorse (13 px); the surrounding sizes step in modest increments to keep the interface from feeling stratified.

:root { --font-size-2xs: 10px; /* sub-flag chip labels ("B"/"P"), edge mid-labels ("yes"/"no"), Premise count chips */ --font-size-xs: 11px; /* chip labels, table headers (with letter-spacing), captions, validation drawer secondary text */ --font-size-sm: 12px; /* inspector field labels, outline tree rows, table cells, validation drawer entries, tooltips */ --font-size-base: 13px; /* body text, default node statements, interview pane primary text, coachmark text */ --font-size-md: 15px; /* primary node statements (RootQuestion, SubQuestion, Conclusion), side pane titles, prose-summary body */ --font-size-lg: 17px; /* top-bar titles (frame title, session title) */ --font-size-xl: 22px; /* screen-level headings (Home, settings) */ }

Font weights

Three weights only. Heavier than 600 reads as shouting in this kind of dense interface; lighter than 400 loses legibility at small sizes.

:root { --font-weight-regular: 400; /* body, list rows */ --font-weight-medium: 500; /* labels, headers, button labels, table header text */ --font-weight-semibold: 600; /* primary node statements, top-bar titles, recommended-next list item */ }

Line height and letter spacing

:root { --line-height-tight: 1.30; /* large titles, top-bar text */ --line-height-snug: 1.45; /* multiline node statements */ --line-height-normal: 1.60; /* body, validation messages, tooltips */ --line-height-relaxed: 1.75; /* prose-summary tab (legal prose feel) */ --letter-spacing-tight: -0.01em; /* large titles 17px+ */ --letter-spacing-normal: 0; --letter-spacing-wide: 0.04em; /* uppercase: table headers, category labels */ }

Spacing, borders, radii

Spacing scale

Four-pixel base, six steps. Used for padding, gap, margin, and inter-pane spacing so the interface aligns to a single rhythm.

:root { --space-1: 4px; --space-2: 8px; --space-3: 12px; --space-4: 16px; --space-5: 24px; --space-6: 32px; --space-8: 48px; }

Border weights

:root { --border-hairline: 0.5px; /* pane separators, table dividers */ --border-thin: 1px; /* card outlines, input borders, default canvas edges */ --border-medium: 1.5px; /* selected node outlines, focus rings, RootQuestion border */ --border-thick: 2px; /* Conclusion's double-border outer ring */ }

Border radii

:root { --radius-sm: 3px; /* chips, small badges */ --radius-md: 6px; /* cards, inspector sections, popovers, drawers */ --radius-lg: 10px; /* modals, dialogs */ --radius-pill: 9999px; /* status badges, tag chips, sub-flag chips, segmented toggle */ }

Elevation

Restrained shadow set. Three levels: cards / popovers / modals. Shadows tinted with the warm-neutral text color so they read as ink, not as bluish web-default drop-shadows.

:root { --shadow-sm: 0 1px 2px hsla(30 8% 12% / 0.05); /* inspector cards, hover lift */ --shadow-md: 0 2px 6px hsla(30 8% 12% / 0.08), 0 1px 2px hsla(30 8% 12% / 0.05); /* popovers, drawers, side panes */ --shadow-lg: 0 8px 24px hsla(30 8% 12% / 0.12), 0 2px 6px hsla(30 8% 12% / 0.06); /* modals, dialogs */ }

Motion

Duration scale

Anchored on Stream E E4's ~150 ms status-transition spec; --duration-base is that exact value. Other durations supplied for consistency across the surfaces that animate.

:root { --duration-instant: 50ms; /* button-press feedback */ --duration-fast: 100ms; /* tooltip appear, hover state changes */ --duration-base: 150ms; /* status badge transitions (Stream E E4 spec), tab switches */ --duration-medium: 250ms; /* drawer expand/collapse, side pane slide-in */ --duration-slow: 350ms; /* mode-accent chrome shift on operating-mode toggle */ --duration-pulse: 1800ms; /* recommended-next item soft pulse, full cycle */ }

Easing curves

:root { --ease-standard: cubic-bezier(0.20, 0.00, 0.20, 1.00); /* general motion: status fade-and-scale, tab switches, hover */ --ease-emphasized: cubic-bezier(0.20, 0.00, 0.00, 1.00); /* drawer/pane appear; more decisive entry */ --ease-soft: cubic-bezier(0.25, 0.10, 0.25, 1.00); /* recommended-next pulse; gentle */ }

Composed animation primitives

These are not tokens themselves; they are the canonical applications of the tokens above for animations Stream E specifies by name.

  • Status badge transition (Stream E E4): transition: opacity var(--duration-base) var(--ease-standard), transform var(--duration-base) var(--ease-standard); with a soft fade-and-scale (0.92 → 1.0) on status change. No celebration animation; intentionally minimal per E4 ("the application is for serious analysis").
  • Recommended-next pulse (Stream E E3): animation: pulse var(--duration-pulse) var(--ease-soft) infinite; oscillating opacity 1.0 → 0.85 → 1.0. Background tint rather than scale change so the surrounding layout doesn't reflow.
  • Mode-accent chrome shift (Stream E E6): transition: background-color var(--duration-slow) var(--ease-standard), border-color var(--duration-slow) var(--ease-standard); applied to top-bar accent line and operating-mode toggle pill.
  • Recompute pulse indicator (Stream E E3): a small dot near the interview pane that fades in for --duration-base, holds for 200 ms, fades out for --duration-base, on every recompute.

Reduce-motion accommodation

Stream E E4 specifies a "reduce motion" accessibility preference. The token-level enforcement is a single media query that collapses all durations and the pulse to instant. Components do not need to branch on the preference; the tokens flip globally.

@media (prefers-reduced-motion: reduce) { :root { --duration-instant: 0.01ms; --duration-fast: 0.01ms; --duration-base: 0.01ms; --duration-medium: 0.01ms; --duration-slow: 0.01ms; --duration-pulse: 0.01ms; } /* The pulse animation, by being keyed off --duration-pulse via animation-duration, also collapses. */ }

Stream G AI-attribution chip — explicit treatment

Per Stream G's "AI attribution" section, fields whose current value was committed via an LLM hook show a "✦" chip in the inspector when the field is selected, with a tooltip listing hook id, prompt version, and timestamp. The chip uses the AI-accent tokens; the chip is informational only (does not constrain editing).

  • Chip dimensions: --space-4 tall, --space-1 horizontal padding around the "✦" glyph + label, --radius-pill.
  • Chip fill: --color-ai-accent-bg.
  • Glyph and label color: --color-ai-accent.
  • Label text: hook short name (e.g., "summary"), --font-size-2xs --font-weight-medium.
  • Hover: chip border in --color-ai-accent-strong, --duration-fast --ease-standard.
  • Tooltip on hover: standard tooltip treatment (--color-surface-elevated, --shadow-md, --radius-md), with chip-attribution lines in --font-size-sm --color-text-secondary.

Dark theme — deferral note

A dark theme is not in Stream E's specification and not requested by any other stream. v1 ships with the single light theme above.
