Stream C — Runtime Behavior and Algorithms

Companion to handoff_document_v2.html, master_plan_v1.html, stream_a_resolutions_v1.html, and stream_b_schema_v1.html  ·  Session date: May 4, 2026  ·  Version 1

This document specifies the runtime behavior of the app: how the structures defined in Stream B are evaluated to produce a result for the user. It covers Stream C end-to-end: path tracing (C1), foreclosure resolution (C2), status determination (C3), conditional output generation (C4), the frame completeness checker (C5), and authority binding determination (C6). Algorithms are written in pseudocode at a level that a coding agent can implement directly. Determinism is treated as a load-bearing requirement throughout — a consequence of the A2 AI-strategy decision that no LLM sits in the routing path.

How to read this

This document is algorithm-first rather than data-first. The Stream B spec is the source of truth for type definitions; this document references those types by section id (B1.5 = Checkpoint, B5 = ConditionalOutput, etc.) rather than redefining them. Algorithms are written in a Python-like pseudocode that elides import statements, error wrapping, and obvious null handling so the structure is visible. Where a decision had multiple defensible options, the chosen option is documented with rationale in the "Decisions made in this session" block; that block is the place to look first if any algorithm seems surprising.

Conventions:

  • Pure functions. Every named function is pure — same inputs, same output, no side effects, no clock reads, no randomness. State changes happen in one place: the session writer at the orchestration layer.
  • Stable iteration. Wherever the algorithm iterates over a set or map, sort by id (lexicographically) before iterating. This is the only thing that keeps "set" semantics deterministic across runtimes and machines.
  • Reverse topological order. Status computation walks children before parents. The frame DAG (V-FR-6) makes this well-defined.
  • "Active path" vs. "reachable set." Reachable = could be visited if all branches were taken. Active = visited under the current selections and responses. The two diverge at branching nodes (Term, Checkpoint, LogicalGate). Both are needed; the algorithms compute them separately.
  • NodeRef resolution. Every algorithm operates on a single FrameVersion snapshot plus an ArgumentSession's argument-layer entities. Cross-version references are not legal here; a session that drifts across frame versions is migrated explicitly before runtime evaluation (Stream D concern).

Decisions made in this session

The following technical refinements were resolved while writing this spec. Items marked stream B amendment are minimal additions to the schema; they are minor and follow directly from conditions and language already locked in A3 and B. They are flagged for inclusion when the handoff doc and stream_b doc are next updated. Each decision is locatable in the body of this document.

  1. stream B amendment InterpretationSelection added to ArgumentSession. The A3 condition library already names interpretation_selected ("this Term has at least one Interpretation marked selected in the session"), but Stream B did not specify the data structure that records the selection. A new entity, InterpretationSelection, lives on ArgumentSession alongside checkpoint_responses. See "Schema amendment" below.
  2. Dispositive Term semantics fully specified. A dispositive Term, once any of its Interpretations is selected, automatically forecloses all sibling Terms under the same parent Question. The Term order field decides which dispositive Term takes effect first when several siblings are dispositive. This matches the handoff doc's plain-language description and A7's "order is purely a foreclosure mechanism." Explicit FORECLOSES edges still exist and are processed independently; the two foreclosure sources merge into one foreclosed set.
  3. CheckpointOption target traversal model. CheckpointOption.target_node_id is treated as a logical edge for graph algorithms (reachability, topological sort, path walking) even though it is not promoted to an Edge entity in the B schema. ANSWERS edges live in the argument layer and represent "this Premise is the response to this Checkpoint"; option targets live in the frame layer and represent "if this option is chosen, route to this node." Different things; both needed.
  4. Five-phase computation pipeline. Validate → resolve foreclosure → compute reachable + active sets → compute status map → generate output. The phases are ordered by dependency; each is a pure function whose inputs are the outputs of prior phases. This is the load-bearing structural commitment of this spec.
  5. Status precedence ordering. When a node's status could plausibly be assigned from multiple sources (e.g., it's both foreclosed and contested), precedence is fixed: foreclosed > not_applicable > contested > open > satisfied. The first applicable wins. This makes status assignment a single deterministic pass.
  6. Burden_met evaluation: categorical, not stochastic. The burden_met condition (B's condition library) is evaluated by a fixed scoring rule: each supporting Premise contributes a weight by kind and edge weight, and contradicting Premises subtract; the total is compared against a threshold per burden level. The scoring table is fixed in this spec, and an LLM_HOOK is reserved for a future calibration override (G5). No probabilistic reasoning runs in v1.
  7. LogicalGate output for unresolved inputs is "indeterminate," not "open." A gate whose inputs are partly unresolved produces an indeterminate result that propagates conditional branches into ConditionalOutput. The gate's own NodeStatus is open with failed_conditions: ["inputs_indeterminate"]. This separates "the gate is broken" from "the gate is computing a conditional answer," which is the right distinction for the conditional output view.
  8. Best-inference Conclusion in contested cases is selected by a fixed tiebreaker. When the output shape is contested, the best-inference Conclusion is chosen by: (a) most fully-resolved path, (b) most binding-authority support, (c) lexicographic Conclusion id. No probabilistic ranking, no LLM. The user always sees this Conclusion flagged as tentative.
  9. The frame-completeness checker (C5) runs in two contexts with two strictness levels. "Pre-mode-transition" mode treats every error rule as a hard block. "Editing" mode reports both errors and warnings as advisory and never blocks. Same rule set, different consumer policy.
  10. Authority binding (C6) has a v1 implementation that doesn't depend on F5. Until the court hierarchy database lands, binding is determined by: (a) explicit is_binding override from the user, (b) string match between Authority's jurisdiction and Frame's jurisdiction, (c) "soft heuristic" flag in the result so UI can mark it as preliminary.
  11. Cycle detection is defensive, not load-bearing. V-FR-6 guarantees no cycles in the resolution flow. Runtime algorithms still check, and abort with a structured error if a cycle is observed. This catches bugs in the validation layer without slowing the common path materially.

Schema amendment (amendment to B4)

The Stream B spec needs a small addition to record explicit Interpretation selections. Without it, the A3 condition interpretation_selected has no underlying data to evaluate. The amendment is minimal and additive — no changes to existing fields.

ArgumentSession { // ... existing fields ... interpretation_selections: InterpretationSelection[] // new in this amendment // ... existing fields ... } InterpretationSelection { term_id: NodeRef // the Term being resolved selected_interpretation_ids: NodeRef[] // usually one; multiple permitted for "alternative paths" supporting_premise_ids: NodeRef[]? // optional; cross-references SUPPORTS edges supporting_authority_ids: NodeRef[]? // optional; cross-references CITES edges notes?: string selected_at: timestamp }

An InterpretationSelection is the argument-layer record of how a Term was resolved. Multiple selections under one Term are permitted and represent the "alternative paths to the same Conclusion" pattern: e.g., the holding may rest on either of two readings of "willful," and the user wants to record that the conclusion follows under either. Frame validators do not require a single selection per Term; the runtime simply walks each selected branch.

A new validation rule is added: V-ARG-6 (error) — every selected_interpretation_id in an InterpretationSelection points to an Interpretation that is a child (via INTERPRETED_AS) of the named term_id. A new validation rule is also added: V-ARG-7 (warning) — at session-time, every Term reachable from the active path has at least one InterpretationSelection record. This is a warning because an unselected Term simply produces an open gate in the output.

Linked Terms (A10): when Term.linked_to is non-null, the linked Term inherits the selection of its target. No InterpretationSelection record is required for the linked Term itself; if one is provided, V-ARG-8 (warning) is raised reminding the user that the linked target's selection takes precedence.

Determinism guarantee

The runtime computation pipeline is deterministic in the strict sense: given identical inputs, it produces byte-identical outputs.

Formally, the function

compute( frame_version_snapshot: FrameVersion, checkpoint_responses: CheckpointResponse[], interpretation_selections: InterpretationSelection[], premises: Premise[], argument_edges: Edge[], session_authorities: Authority[] ) -> { status_map: Map<NodeRef, NodeStatus>, active_path: NodeRef[], output: ConditionalOutput, open_gates: OpenGate[], validation_results: ValidationResult[] }

satisfies: for any two invocations with the same six inputs, the returned five-tuple is identical. This holds across machines, processes, and library upgrades — provided the implementation respects the conventions in this document (stable iteration, no clock reads in the body, no environment-dependent branches, no LLM in the body).

This guarantee is the practical meaning of A2's "Same frame + same premises = same path, every time." It is also what allows a session to be replayed against an older frame version, exported, audited, or reproduced from a textual log months later. Every test case in the runtime test suite is a tuple of (input fixture, expected output snapshot); a determinism regression is detectable by any diff.

The runtime computation pipeline

Five phases, each pure, each consuming only the outputs of prior phases:

The orchestrating runtime function is roughly:

def compute(frame, session): validation = run_validators(frame) if any(r.severity == "error" for r in validation): # Frame is structurally broken; produce an "incomplete" output and stop. return incomplete_result(validation) foreclosed = resolve_foreclosure(frame, session) # C2 reachable = compute_reachable_set(frame, foreclosed) active = compute_active_set(frame, session, foreclosed) # used by C1 status_map = compute_status_map(frame, session, foreclosed, active) # C3 active_path, output, open_gates = generate_output( # C1 + C4 frame, session, status_map, foreclosed, active ) return ComputeResult( validation_results=validation, foreclosed_set=foreclosed, reachable_set=reachable, active_set=active, status_map=status_map, active_path=active_path, output=output, open_gates=open_gates, )

C6 (authority binding) does not appear in the orchestrator because it is consumed inline by the authority_binding condition during phase 4, and again by the prose-summary generator in phase 5. It is a leaf-level utility, not a pipeline phase.

Pre-pipeline: the frame as a graph

Before any phase runs, the FrameVersion is reified into a graph data structure. This is mechanical but worth specifying because several algorithms below assume the conversion has happened.

def build_graph(frame_version): nodes_by_id = { n.id: n for n in frame_version.nodes } outgoing = { n.id: [] for n in frame_version.nodes } incoming = { n.id: [] for n in frame_version.nodes } for e in frame_version.edges: outgoing[e.source].append(e) incoming[e.target].append(e) # Logical edges from CheckpointOption.target_node_id and # LogicalGate.output_target are stored on the source node, not as Edge entities. # Algorithms iterate over both real edges and these logical edges via: def out_traversal(node_id): for e in outgoing[node_id]: yield e node = nodes_by_id[node_id] if node.type == "Checkpoint": for opt in node.options: if opt.target_node_id: yield SyntheticEdge( source=node_id, target=opt.target_node_id, type="OPTION_LEADS_TO", option_id=opt.id ) if node.type == "LogicalGate" and node.output_target: yield SyntheticEdge( source=node_id, target=node.output_target, type="GATES" ) return Graph(nodes_by_id, outgoing, incoming, out_traversal)

SyntheticEdge is a transient object; it never lives on disk. It exists so that traversal code does not have to special-case Checkpoint and LogicalGate at every callsite.

From the rest of this document, "edge" without qualification means either a stored Edge or a SyntheticEdge produced by out_traversal. Where a distinction matters, it is called out.

Phase 1 — Frame validation (C5)

The completeness checker runs every rule in B6 and returns a list of results. Two consumer policies use the same output:

  • Editing (default). All results are surfaced in the UI as advisories. Errors are highlighted; warnings are shown but do not block any user action. The user can always save and continue editing.
  • Pre-mode-transition (strict). Errors block the transition (Frame → Argument cannot be entered if any error is present). Warnings are surfaced for acknowledgment but do not block.

Rule evaluator

Each rule is a function with a uniform signature so the checker can iterate over them:

Rule = (frame: FrameVersion, graph: Graph) -> ValidationResult[] ValidationResult { rule_id: string // e.g. "V-FR-6" severity: "error" | "warning" node_ids: NodeRef[] // nodes implicated edge_ids?: EdgeRef[] // edges implicated; optional message: string // pre-rendered for UI suggested_fix?: string // optional; for advisory display }

The full evaluator just composes rules:

def run_validators(frame): graph = build_graph(frame) results = [] for rule in ALL_RULES: # sorted by rule_id for stable order results.extend(rule(frame, graph)) return results

Rule implementations (selected)

Most rules are local checks on a single node or edge. A few require global graph properties; those are spelled out below. The full rule list lives in B6; this section gives implementation sketches for the non-obvious ones.

V-FR-4 — every Checkpoint reachable from RootQuestion

def v_fr_4(frame, graph): root = find_root(frame) if not root: return [] # V-FR-1 already caught this visited = set() stack = [root.id] while stack: cur = stack.pop() if cur in visited: continue visited.add(cur) for e in graph.out_traversal(cur): # For reachability, follow every branch (not just selected ones) if e.type in {"DECOMPOSES_INTO", "TURNS_ON", "INTERPRETED_AS", "LEADS_TO", "GATES", "OPTION_LEADS_TO"}: stack.append(e.target) unreachable_checkpoints = [ n for n in frame.nodes if n.type == "Checkpoint" and n.id not in visited ] return [ ValidationResult("V-FR-4", "error", [c.id], [], f"Checkpoint '{c.question[:40]}' is unreachable from the root question.", "Connect this Checkpoint to the rest of the frame, or delete it.") for c in sorted(unreachable_checkpoints, key=lambda n: n.id) ]

V-FR-6 — frame is a DAG

The frame graph, restricted to "structural" edges (everything except FORECLOSES, BINDING_IN, CITES, DISTINGUISHED_BY, SUPPORTS, CONTRADICTS), must be acyclic. Argument-layer edges and annotation edges do not count toward the DAG check.

STRUCTURAL_EDGE_TYPES = { "DECOMPOSES_INTO", "TURNS_ON", "INTERPRETED_AS", "LEADS_TO", "GATES", "OPTION_LEADS_TO" } def v_fr_6(frame, graph): # Tarjan or 3-color DFS; same outcome. WHITE, GRAY, BLACK = 0, 1, 2 color = { n.id: WHITE for n in frame.nodes } cycles = [] def dfs(node_id, path): if color[node_id] == GRAY: cycles.append(path[path.index(node_id):] + [node_id]) return if color[node_id] == BLACK: return color[node_id] = GRAY for e in graph.out_traversal(node_id): if e.type in STRUCTURAL_EDGE_TYPES: dfs(e.target, path + [node_id]) color[node_id] = BLACK for n in sorted(frame.nodes, key=lambda x: x.id): if color[n.id] == WHITE: dfs(n.id, []) return [ ValidationResult("V-FR-6", "error", cycle, [], "Cycle in resolution flow: " + " -> ".join(cycle), "Remove an edge from the cycle.") for cycle in cycles ]

V-FR-8 — every path terminates at a Conclusion

A path from RootQuestion that does not eventually reach a Conclusion is a dangling branch. Detection is by reverse reachability: compute the set of nodes from which a Conclusion is reachable; flag any reachable-from-root node that is not in this set.

def v_fr_8(frame, graph): conclusions = { n.id for n in frame.nodes if n.type == "Conclusion" } # Reverse-reach: nodes from which any Conclusion is reachable. reach_concl = set(conclusions) changed = True while changed: changed = False for e in iter_all_traversal_edges(frame, graph): if e.target in reach_concl and e.source not in reach_concl: reach_concl.add(e.source) changed = True # Forward-reach: from root, ignoring annotation edges. root = find_root(frame).id fwd = forward_reach(root, graph) dangling = sorted((fwd - reach_concl) - conclusions) return [ ValidationResult("V-FR-8", "error", [n], [], f"Node '{n}' is reachable from the root but no Conclusion is reachable from it.", "Add an outgoing branch that eventually terminates at a Conclusion.") for n in dangling ]

V-FR-9 — compound Checkpoint detection (heuristic)

This rule is intentionally fuzzy: detecting whether a Checkpoint asks more than one question is a language judgment, not a structural one. The v1 heuristic flags Checkpoints whose question field contains a conjunction at top level (" and ", " plus ", semicolons followed by question phrases) or matches a small set of compound patterns. False positives are kept down by emitting warnings only; an LLM_HOOK is reserved for a future precision upgrade (G4).

The heuristic is fixed-string and case-insensitive. The patterns are documented in code as COMPOUND_HEURISTICS; updates to the patterns require a schema_version bump because they affect what gets flagged.

Phase 2 — Foreclosure resolution (C2)

Foreclosure is the mechanism by which selecting one branch makes others moot. The phase produces a single set of foreclosed node ids that downstream phases honor as "do not visit, do not evaluate, status is foreclosed." There are two sources of foreclosure; both are processed and their outputs unioned.

Source 1 — explicit FORECLOSES edges

An Interpretation node may carry FORECLOSES edges to specific Term, SubQuestion, or Checkpoint nodes (per B2). When the Interpretation is selected (per the InterpretationSelection record), every target of its FORECLOSES edges is added to the foreclosed set.

def explicit_foreclosure(frame, session): foreclosed = set() selected_interp_ids = { iid for sel in session.interpretation_selections for iid in sel.selected_interpretation_ids } for e in sorted(frame.edges, key=lambda x: x.id): if e.type != "FORECLOSES": continue if e.source in selected_interp_ids: foreclosed.add(e.target) # scope handled at status-time: # scope=="moot" -> status foreclosed # scope=="decided" -> status satisfied via structural_resolution return foreclosed

Source 2 — dispositive Term auto-foreclosure

Per the handoff doc and A7: a dispositive Term, when any of its Interpretations is selected, forecloses all sibling Terms under the same parent Question. The Term order field decides priority when multiple sibling Terms are dispositive: the lowest-ordered dispositive Term whose Interpretation is selected wins; later dispositive Terms in the same Question are themselves foreclosed by the winner.

def dispositive_foreclosure(frame, session): foreclosed = set() selected_term_ids = { sel.term_id for sel in session.interpretation_selections if sel.selected_interpretation_ids } # Group Terms by their parent Question (RootQuestion or SubQuestion). parent_of_term = {} for e in frame.edges: if e.type == "TURNS_ON": parent_of_term[e.target] = e.source siblings_by_parent = group_terms_by_parent(frame, parent_of_term) for parent_id, terms in siblings_by_parent.items(): sorted_terms = sorted(terms, key=lambda t: (t.order, t.id)) winner = None for t in sorted_terms: if t.dispositive and t.id in selected_term_ids: winner = t break if winner: for t in sorted_terms: if t.id != winner.id: foreclosed.add(t.id) return foreclosed

Cascade

Once direct foreclosure is computed, foreclosure cascades: every node strictly downstream of a foreclosed node (along structural edges) is itself foreclosed. The cascade stops at convergent nodes that remain reachable through other (non-foreclosed) ancestors. This is a standard reachability subtraction.

def cascade_foreclosure(frame, graph, direct_foreclosed): # A node is "live" if reachable from RootQuestion through structural edges # whose source is not foreclosed. Cascade-foreclosed = reachable-from-root # minus reachable-from-root-without-foreclosed. root = find_root(frame).id full_reach = forward_reach(root, graph, blocked=set()) live_reach = forward_reach(root, graph, blocked=direct_foreclosed) cascaded = (full_reach - live_reach) | direct_foreclosed return cascaded def resolve_foreclosure(frame, session): graph = build_graph(frame) direct = explicit_foreclosure(frame, session) | dispositive_foreclosure(frame, session) return cascade_foreclosure(frame, graph, direct)

The cascade pass guarantees: if a Term is foreclosed, all its Interpretations and their downstreams are foreclosed; if an Interpretation is foreclosed, its downstream Checkpoint, SubQuestion, or Conclusion is foreclosed unless reachable through another live branch.

Foreclosure scope: "moot" vs. "decided"

FORECLOSES edges carry an optional scope field (B2): "moot" (default) means the foreclosed branch never gets evaluated; "decided" means the foreclosed branch is treated as if the foreclosing Interpretation's outcome resolved it.

Phase 2 outputs the foreclosed set without distinguishing scope. The scope is consulted in Phase 4 by the status determiner: a node foreclosed by a "moot" edge gets status foreclosed; a node foreclosed by a "decided" edge gets status satisfied with via: ["structural_resolution"]. Dispositive auto-foreclosure is always "moot."

def foreclosure_scope_for(node_id, frame, session): # Returns "moot" or "decided" for a foreclosed node. selected_interps = collect_selected_interpretations(session) decided_via = [] for e in frame.edges: if (e.type == "FORECLOSES" and e.target == node_id and e.source in selected_interps and e.scope == "decided"): decided_via.append(e.source) if decided_via: return "decided" return "moot"

Phase 3 — Reachable and active sets

Two derived sets are needed for downstream phases.

Reachable set: every node visitable from RootQuestion along structural edges, ignoring foreclosure and ignoring whether a branch is currently selected. This is the set of nodes the user could possibly visit by changing answers.

Active set: every node visitable from RootQuestion along structural edges, honoring foreclosure and following only branches currently selected by InterpretationSelection / CheckpointResponse / gate evaluation. Nodes outside the active set get status not_applicable.

def compute_reachable_set(frame, foreclosed): return forward_reach(find_root(frame).id, build_graph(frame), blocked=foreclosed) def compute_active_set(frame, session, foreclosed): graph = build_graph(frame) active = set() queue = [find_root(frame).id] while queue: cur = queue.pop() if cur in foreclosed or cur in active: continue active.add(cur) node = graph.nodes_by_id[cur] for e in graph.out_traversal(cur): if e.type not in STRUCTURAL_EDGE_TYPES: continue if not branch_is_followed(node, e, session, foreclosed): continue queue.append(e.target) return active def branch_is_followed(node, edge, session, foreclosed): if node.type == "Term": if edge.type == "INTERPRETED_AS": return edge.target in selected_interpretation_ids_for(node.id, session) return True if node.type == "Checkpoint": if edge.type == "OPTION_LEADS_TO": response = response_for(node.id, session) return response and edge.option_id == response.selected_option_id return False # Checkpoints have no other outgoing edges if node.type == "LogicalGate": return gate_routes_to(node, edge, session, foreclosed) return True # Question, Interpretation: follow all structural outgoing

gate_routes_to is defined in the LogicalGate evaluation section below; it returns true if the gate's evaluation routes to the given outgoing target.

For the active set, the case where a Term has multiple selected Interpretations (the alternative-paths pattern) produces multiple parallel active branches. This is intended; the output generator collapses convergent paths in C4.

LogicalGate evaluation

LogicalGates are referenced by status determination, active-set computation, and path tracing, so their evaluation is specified in one place and called from each consumer.

A gate's evaluation produces one of three results:

  • satisfied — the gate's logical condition holds.
  • not_satisfied — the gate's logical condition does not hold.
  • indeterminate — at least one input has not been resolved (status is open or contested); the gate cannot decide.

For routing purposes, a gate "routes to" its output_target only when its result is satisfied. The not_satisfied and indeterminate outcomes do not route. (Frames that need an explicit "if-not" branch use an UNLESS gate or a separate IF_THEN gate with a NOT-prefixed antecedent.)

Per-gate-type semantics

Gate typeInputssatisfied iffindeterminate iffelse
ANDinputs[]every input satisfiedany input indeterminate AND no input not_satisfiednot_satisfied
ORinputs[]any input satisfiedevery non-satisfied input is indeterminatenot_satisfied
NOTinputinput not_satisfiedinput indeterminatenot_satisfied (when input satisfied)
IF_THENantecedent, consequentantecedent not_satisfied OR consequent satisfiedantecedent indeterminate, OR antecedent satisfied AND consequent indeterminatenot_satisfied (antecedent satisfied AND consequent not_satisfied)
UNLESSmain, exceptionmain satisfied AND exception not_satisfiedmain indeterminate, OR main satisfied AND exception indeterminatenot_satisfied

The truth tables are short-circuit-friendly: AND can resolve to not_satisfied as soon as one input fails; OR to satisfied as soon as one input succeeds; IF_THEN to satisfied as soon as antecedent fails. The implementation honors short-circuiting for performance but produces the same results as the full evaluation.

Boolean evaluation of non-gate inputs

Gates take Checkpoints, Interpretations, SubQuestions, RootQuestions, and other gates as inputs (per V-GATE-5/6). Each input is evaluated as boolean by reading its NodeStatus:

Input NodeStatusBoolean value
satisfiedsatisfied
foreclosed (scope=decided)satisfied
foreclosed (scope=moot)indeterminate (the gate cannot rely on it)
not_applicableindeterminate
openindeterminate
contestedindeterminate

The mapping for "moot foreclosure" → indeterminate is a deliberate choice: a gate whose input branch was made moot by an unrelated dispositive Term has nothing to compute against, and its output should not silently flip based on a side channel.

Cycle in gate dependency

V-FR-6 forbids cycles in the structural graph, but defensive runtime checking still applies. If a gate's input depends transitively on the gate itself (impossible if validation passed but possible during a partial-state computation), the gate is reported as indeterminate with a structured error attached, and a non-fatal log entry is emitted. The orchestrator surfaces this as a runtime validation failure.

Phase 4 — Status determination (C3)

The status determiner walks the frame in reverse topological order (children before parents) and assigns a NodeStatus to every node in the active set, plus foreclosed or not_applicable to nodes outside it.

Status precedence

Decision (#5 above): when more than one status applies, precedence is fixed.

  1. foreclosed — node is in the foreclosed set with scope=moot.
  2. satisfied via structural_resolution — node is foreclosed with scope=decided.
  3. not_applicable — reachable but not in the active set.
  4. contested — active, but Premises support both sides of a binary or graded Checkpoint.
  5. open — active, but at least one condition in the satisfaction policy fails.
  6. satisfied — active and every condition passes.

Premise and Authority nodes do not get NodeStatus assignments (per A3). They are sources, not gates.

Algorithm

def compute_status_map(frame, session, foreclosed, active): graph = build_graph(frame) status_map = {} # Reverse topological sort over structural edges. topo = topological_sort(frame, graph, edge_filter=STRUCTURAL_EDGE_TYPES) rev_topo = list(reversed(topo)) for node_id in rev_topo: node = graph.nodes_by_id[node_id] if node.type in {"Premise", "Authority"}: continue # sources, not gates # Step 1: foreclosure if node_id in foreclosed: scope = foreclosure_scope_for(node_id, frame, session) if scope == "decided": status_map[node_id] = NodeStatus( status="satisfied", via=["structural_resolution"], evaluated_at=session.computed_at_clock_pure, failed_conditions=[], ) else: status_map[node_id] = NodeStatus( status="foreclosed", via=[], evaluated_at=..., failed_conditions=[], ) continue # Step 2: out-of-active-set if node_id not in active: status_map[node_id] = NodeStatus( status="not_applicable", via=[], evaluated_at=..., failed_conditions=[], ) continue # Step 3: contested check (Checkpoint-specific shortcut) if node.type == "Checkpoint" and is_contested(node, session, status_map): status_map[node_id] = NodeStatus( status="contested", via=[], evaluated_at=..., failed_conditions=["contradicting_premises"], ) continue # Step 4: evaluate satisfaction policy policy = node.options_box or frame.default_satisfaction_policies[node.type] result = evaluate_policy(policy, node, frame, session, status_map) status_map[node_id] = NodeStatus( status="satisfied" if result.passed else "open", via=result.via, evaluated_at=..., failed_conditions=result.failed, ) return status_map

The evaluated_at field is filled with the orchestrator's "logical computation time" (a single timestamp passed in from the orchestrator), not a per-call clock read. This keeps the function pure for fixed inputs and lets test fixtures pin timestamps.

Condition evaluation

Each condition in the A3/B condition library is implemented as a pure function with a uniform signature:

Condition = ( node: Node, frame: FrameVersion, session: ArgumentSession, status_map: Map<NodeRef, NodeStatus>, # partial; children resolved, parents not params: Dict<string, any>, ) -> { passed: bool, via: string[] } def evaluate_policy(policy, node, frame, session, status_map): via = [] failed = [] # all_of (AND) for cond in policy.all_of: result = evaluate_condition(cond, node, frame, session, status_map) via.extend(result.via) if not result.passed: failed.append(condition_name(cond)) # any_of (OR), if present — at most one group, no nesting any_of_passed = True if policy.any_of: any_of_passed = False for cond in policy.any_of: result = evaluate_condition(cond, node, frame, session, status_map) if result.passed: any_of_passed = True via.extend(result.via) break if not any_of_passed: failed.append("any_of_group_unsatisfied") return PolicyResult( passed=(not failed), via=sorted(set(via)), failed=failed, )

burden_met

Decision (#6 above): categorical scoring, fixed table, no probabilistic reasoning. A burden level resolves to a numeric threshold; supporting Premises contribute by kind and edge weight; contradicting Premises subtract; the comparison is total ≥ threshold.

PREMISE_KIND_WEIGHT = { # legal mode "stipulated": 100, # effectively certain "found": 70, "disputed": 30, "procedural": 50, } ACADEMIC_KIND_WEIGHT = { "stipulated": 100, "empirical": 70, "definitional": 50, "normative": 30, } PERSONAL_KIND_WEIGHT = { "stipulated": 100, "observation": 60, "value": 40, "assumption": 20, } EDGE_WEIGHT_MULTIPLIER = { "strong": 1.0, "moderate": 0.66, "weak": 0.33 } BURDEN_THRESHOLD = { "scintilla": 25, "preponderance": 50, "substantial_evidence": 60, "clear_and_convincing": 80, "beyond_reasonable_doubt": 95, } def cond_burden_met(node, frame, session, status_map, params): level = params.get("level", "preponderance") threshold = BURDEN_THRESHOLD[level] weight_table = pick_kind_weights(frame.mode, frame.flavor) score = 0.0 via_set = set() for e in session.argument_edges: if e.target != node.id: continue prem = find_premise(e.source, session) if not prem: continue kw = weight_table.get(prem.kind, 30) ew = EDGE_WEIGHT_MULTIPLIER.get(e.weight or "moderate", 0.66) if e.type == "SUPPORTS" or e.type == "ANSWERS": score += kw * ew if prem.kind == "stipulated": via_set.add("stipulation") elif e.type == "CONTRADICTS": score -= kw * ew # LLM_HOOK G5: future calibration override may replace this scoring. return Result(passed=(score >= threshold), via=sorted(via_set))

The scoring table is fixed for v1, exposed in code as a single module so it is easy to find. The threshold values are documented in the spec but treated as configurable per-frame in v2 (deferred). The score is a simple linear sum; nothing about it is probabilistic, and the user-visible "confidence" reported on a Conclusion (per A1) is computed separately from these numbers.

def cond_authority_binding(node, frame, session, status_map, params): cited_authorities = [ e for e in frame.edges if e.type == "CITES" and e.target == node.id ] cited_authorities += [ ("via_premise", p) for p in session.premises if p.authority_ref and any(e.type == "ANSWERS" and e.source == p.id and e.target == node.id for e in session.argument_edges) ] via_set = set() for entry in cited_authorities: if isinstance(entry, tuple): authority = resolve_authority(entry[1].authority_ref, frame, session) else: authority = resolve_authority(entry.source, frame, session) if authority and is_binding(authority, frame.jurisdiction_default): # C6 via_set.add("binding_authority") return Result(passed=("binding_authority" in via_set), via=sorted(via_set))

Contested-state heuristic for Checkpoints

A Checkpoint is contested (status precedence step 3) when both of the following hold:

  1. An ANSWERS edge exists, AND
  2. At least one CONTRADICTS edge from a Premise targeting this Checkpoint exists, OR the answer routes to a graded option whose routes_to_status is "contested".

The contested status is independent of burden_met: a Checkpoint can be contested even when one side has a higher burden score, because the user's frame author wants to expose the dispute to the reader. The output layer (C4) decides whether to treat contested as a partial answer or a fork.

SubQuestion / RootQuestion status

SubQuestion and RootQuestion use the default policy { all_of: ["all_children_resolved", "not_foreclosed"] }. The all_children_resolved condition is implemented as:

def cond_all_children_resolved(node, frame, session, status_map, params): children = [ e.target for e in frame.edges if e.source == node.id and e.type in {"DECOMPOSES_INTO", "TURNS_ON", "LEADS_TO"} ] if not children: return Result(False, []) via_set = set() for c in children: st = status_map.get(c) if st is None: return Result(False, []) # shouldn't happen in reverse-topo order if st.status == "satisfied": via_set.update(st.via) continue if st.status == "foreclosed": # Foreclosed children do not block parent satisfaction; # they are explicitly skipped per dispositive-Term semantics. continue return Result(False, []) return Result(True, sorted(via_set))

This is the implementation of "ruled on one ground, declined to reach the others": foreclosed children do not block; satisfied children pass; anything else fails.

Phase 5 — Path tracing and output generation (C1 + C4)

This phase produces three artifacts: the active_path (an ordered NodeRef list), the output (a ConditionalOutput per B5), and the open_gates list. They share computation; combining them in one phase avoids walking the graph twice.

Output shape determination

The shape field of ConditionalOutput is derived from the status map and the active set:

  • determinate — the active set contains exactly one Conclusion, every node on its path is satisfied (or foreclosed with scope=decided), and no open gates remain. One Conclusion, one path.
  • conditional — the active set is well-formed but at least one Term or Checkpoint on the path is open in a way that produces multiple branches each leading to a different Conclusion. The output enumerates branches.
  • contested — at least one Checkpoint on the path is contested. The output names the contested nodes and supplies a best-inference Conclusion as tentative.
  • incomplete — the frame is structurally broken (errors from Phase 1) or no Conclusion is reachable from the active set. The output names the structural gaps.

Precedence among shapes when several conditions seem to apply: incomplete > contested > conditional > determinate.

Path walking

The active path is constructed by a depth-first walk from RootQuestion, following branches in the direction the session's selections and responses dictate. Multiple selected Interpretations under one Term produce a fork; the walk follows each fork and unions the resulting paths.

def trace_active_path(frame, session, status_map, foreclosed, active): graph = build_graph(frame) path_segments = [] # list of NodeRef-list paths from root open_gates = [] branches = [] # for "conditional" shape def walk(node_id, prefix): if node_id in foreclosed: return # foreclosed; cut here st = status_map.get(node_id) if st is None or st.status == "not_applicable": return new_prefix = prefix + [node_id] node = graph.nodes_by_id[node_id] if node.type == "Conclusion": path_segments.append(new_prefix) return next_targets = [] if node.type in {"RootQuestion", "SubQuestion"}: for e in graph.out_traversal(node_id): if e.type in {"DECOMPOSES_INTO", "TURNS_ON", "LEADS_TO"}: next_targets.append(e.target) elif node.type == "Term": sel = selected_interpretations_for(node_id, session) if not sel: open_gates.append(OpenGate(node_id, "no_selection", f"Pick an interpretation of '{node.name}'.")) return for interp_id in sel: if interp_id in foreclosed: continue next_targets.append(interp_id) elif node.type == "Interpretation": for e in graph.out_traversal(node_id): if e.type == "LEADS_TO": next_targets.append(e.target) elif node.type == "Checkpoint": response = response_for(node_id, session) if not response: if st.status == "contested": branches.extend(branches_from_unresolved_checkpoint( node, new_prefix, frame, session, status_map, foreclosed )) else: open_gates.append(OpenGate(node_id, "no_premise", node.question)) return opt = next((o for o in node.options if o.id == response.selected_option_id), None) if opt and opt.target_node_id: next_targets.append(opt.target_node_id) elif node.type == "LogicalGate": result = evaluate_gate(node, status_map, frame, session) if result == "satisfied" and node.output_target: next_targets.append(node.output_target) elif result == "indeterminate": branches.extend(branches_from_indeterminate_gate( node, new_prefix, frame, session, status_map, foreclosed )) return else: return # gate not_satisfied; no routing for t in sorted(next_targets): walk(t, new_prefix) walk(find_root(frame).id, []) # Collapse convergent paths and pick a primary path for display. primary_path = pick_primary_path(path_segments, status_map) return primary_path, path_segments, open_gates, branches

pick_primary_path selects one path for default rendering by: (a) longest fully-satisfied path, (b) most binding-authority status sub-flags, (c) lexicographic ordering of node ids. This is the same fixed tiebreaker used for best-inference Conclusion selection (decision #8 above), applied to paths.

Branches in conditional output

When a Term has no selection, or a Checkpoint has no response, or a gate is indeterminate, the output may still be useful: the structure tells the reader "if X then A; if Y then B." This is generated by enumerating the small number of choices at the unresolved point and walking forward from each.

def branches_from_unresolved_checkpoint(node, prefix, frame, session, status_map, foreclosed): branches = [] for opt in sorted(node.options, key=lambda o: o.id): if not opt.target_node_id or opt.target_node_id in foreclosed: continue sub_path = walk_assuming( opt.target_node_id, frame, session, status_map, foreclosed, assumption=("checkpoint_response", node.id, opt.id), ) if not sub_path: continue terminal = sub_path[-1] if graph.nodes_by_id[terminal].type != "Conclusion": continue branches.append(ConditionalBranch( id=fresh_id(), conditions=[BranchCondition( gate_node_id=node.id, required_value=opt.id, required_value_label=opt.label, )], resulting_conclusion=terminal, intermediate_path=sub_path, prose=render_branch_prose(opt, sub_path, frame), )) return branches

walk_assuming is a hypothetical walk: it temporarily treats the named assumption as if it were already in the session, and walks forward from the assumed-resolved node. The walk is purely local — it does not re-run foreclosure or status determination for the rest of the frame. This is acceptable because the assumption is a single resolution at a single node, which only affects nodes downstream of that point.

Branches are emitted in alphabetical-by-condition order so the user sees the same enumeration each time. When two branches converge on the same Conclusion (e.g., both options of a Checkpoint route to the same downstream Conclusion via different paths), the branches are merged by combining their conditions arrays into a disjunction and recording one ConditionalBranch.

Prose summary generation

Per A9, the default view is a deterministic prose summary. The summary is generated by a fixed template walk over the active path or branch structure. No LLM in the routing path; an LLM_HOOK is reserved for a future "rewrite for fluency" pass that runs only at the user's request and is auditably separate from the deterministic baseline.

def render_prose(output, frame, session): parts = [] parts.append(f"On the question: {find_root(frame).statement}") parts.append("") if output.shape == "determinate": c = node_for(output.conclusion, frame) parts.append(f"The frame resolves to: {c.statement}.") parts.append(f"Direction: {render_direction(c.direction, frame)}.") parts.append("") parts.append("Path:") for nid in output.primary_path: parts.append(f" - {render_node_short(nid, frame)}") elif output.shape == "conditional": parts.append(f"The frame produces {len(output.branches)} alternative outcomes " f"depending on unresolved questions.") parts.append("") for b in output.branches: cond_text = " and ".join( f"the answer to '{label_for(c.gate_node_id, frame)}' is '{c.required_value_label}'" for c in b.conditions ) concl = node_for(b.resulting_conclusion, frame) parts.append(f" - If {cond_text}, then: {concl.statement}.") elif output.shape == "contested": parts.append("The frame contains contested points that prevent a determinate answer.") for nid in output.contested_nodes: parts.append(f" - Contested: {label_for(nid, frame)}") if output.best_inference: c = node_for(output.best_inference, frame) parts.append("") parts.append(f"Tentative best inference: {c.statement}. (Subject to resolving the contested points.)") elif output.shape == "incomplete": parts.append("The frame is incomplete and cannot be fully resolved.") for og in output.open_gates: parts.append(f" - Open: {og.prompt}") parts.append("") parts.append(render_confidence_breakdown(output.confidence_breakdown)) return "\n".join(parts)

The output of render_prose is plain text, deterministic byte-for-byte for the same inputs. The UI layer can mark it up further (links to nodes, etc.) but the canonical prose is what gets exported and what tests pin against.

Confidence breakdown

Per A1, confidence is computed at output time and never stored on the Conclusion. The breakdown is populated during status computation; output generation just reads it.

def compute_confidence_breakdown(status_map, primary_path, frame): cps_on_path = [ nid for nid in primary_path if frame.node_by_id(nid).type == "Checkpoint" ] breakdown = ConfidenceBreakdown( total_checkpoints_on_path=len(cps_on_path), satisfied_via_binding=0, satisfied_via_persuasive=0, satisfied_via_stipulation=0, satisfied_via_structural=0, contested=0, open=0, ) for nid in cps_on_path: st = status_map[nid] if st.status == "satisfied": if "binding_authority" in st.via: breakdown.satisfied_via_binding += 1 elif "persuasive_authority" in st.via: breakdown.satisfied_via_persuasive += 1 elif "stipulation" in st.via: breakdown.satisfied_via_stipulation += 1 elif "structural_resolution" in st.via: breakdown.satisfied_via_structural += 1 else: breakdown.satisfied_via_persuasive += 1 elif st.status == "contested": breakdown.contested += 1 elif st.status == "open": breakdown.open += 1 return breakdown

The breakdown is integer-counted and order-stable. The user-facing "confidence display" is rendered from the breakdown by the UI layer (Stream E); the breakdown itself is the canonical numeric record.

C6 — Authority binding determination

Determining whether a cited Authority is binding in a frame's jurisdiction is the hinge for the authority_binding condition and for the legal-mode "satisfied via binding authority" status sub-flag. The full implementation depends on F5 (court hierarchy database), which is its own sub-project. For v1, a fallback is specified that works without F5 and is clearly marked as preliminary in the result.

Inputs and outputs

def is_binding(authority: Authority, target_jurisdiction: Jurisdiction) -> { binding: bool, source: BindingSource, certainty: "exact" | "heuristic" } BindingSource = "manual_override" | "hierarchy_database" | "string_match" | "no_jurisdiction"

Resolution order

  1. Manual override. If authority.is_binding is set explicitly (boolean, non-null), the result uses that value with source: "manual_override" and certainty: "exact". The user has spoken; algorithms defer.
  2. Hierarchy database (F5). If authority.jurisdiction.hierarchy_node_id and target_jurisdiction.hierarchy_node_id are both populated, the binding query is dispatched to F5 and the result returned with source: "hierarchy_database", certainty: "exact". F5 must answer "is the authority's court binding on cases heard in the target jurisdiction's court hierarchy" using the actual court relationships.
  3. String-match heuristic (v1 fallback). If F5 is not yet wired or hierarchy ids are missing, the algorithm compares level and region strings:
    • Same level and same region: binding.
    • authority.level == "federal" and target.level == "federal", same circuit: binding.
    • authority is U.S. Supreme Court: binding everywhere.
    • Anything else: not binding (or persuasive only, but the binding determination itself is "not binding").
    Result carries source: "string_match", certainty: "heuristic". The UI surfaces the heuristic flag so the reader knows the binding claim is provisional until F5 is wired.
  4. No jurisdiction information. If the authority has no jurisdiction or the target frame has no jurisdiction_default, return not binding with source: "no_jurisdiction", certainty: "exact" (because the answer is unambiguously "we cannot say binding").

The U.S. Supreme Court special-case is wired in by name in v1 because the rule is unambiguous and the F5 fallback would otherwise miss it. When F5 lands, the fallback path is removed and the special case becomes part of the database.

Incremental computation strategy

The pipeline as written recomputes the entire status map and output from scratch on every change. For a small frame this is cheap; for large frames, the user's edit-to-feedback latency might suffer. This section specifies an incremental strategy that an implementation can layer on without changing the algorithmic spec.

Edit dispatch table

Each kind of session edit affects a known subset of the pipeline:

Frame edits invalidate everything below them, but frame edits only happen in Frame Building mode, where output computation is not running.

The implementation rule is: "ancestors" means the transitive set of nodes whose status policies depend on the changed node, computed by walking up the structural edges. The set is bounded by the path from the change to the root.

When to skip the cache

Three cases force a full recompute:

  1. Schema migration. Any change to schema_version.
  2. Validation change. Any frame edit that changes the validation results (a fresh frame load, a structural edit).
  3. Foreclosure flip. Any change that adds or removes nodes from the foreclosed set forces recompute of the cascade and downstream phases.

Determinism is preserved either way: the cached and full-recompute paths must produce byte-identical outputs for the same inputs. Test fixtures verify this by running both paths on every test case.

Determinism testing strategy

The reproducibility guarantee is testable at multiple levels.

  • Snapshot tests. A fixture is (frame_version_snapshot.json, session.json) → (status_map.json, active_path.json, output.json). The test serializes the result and compares against the recorded snapshot. New runs produce identical bytes.
  • Property tests. Random inputs (well-formed frames, random sessions): assert that two runs of compute produce equal outputs.
  • Cache equality. For every snapshot test case, run both the full-recompute and incremental paths and assert byte-equality.
  • Iteration order audit. A unit test asserts that every for loop over a Set or dict in the algorithm modules sorts before iterating. Detected by AST inspection of the algorithm modules; CI gate.
  • Clock isolation. The compute pipeline is tested with a frozen clock; outputs must be identical at any wall-clock time.

These tests live with Stream H's persistence and tooling, but the contracts they enforce are specified here.

Reserved / deferred design space

  • Probabilistic burden evaluation. The current burden_met uses a fixed scoring table. A future v2 may replace it with a Bayesian or LLM-calibrated model. The condition's signature is stable; the implementation is replaceable. v2
  • Authority weighting beyond binding/persuasive. v1 distinguishes binding vs. persuasive. v2 may add concepts like "vacated," "overruled," "questioned" — these would surface in C6 and the via array. v2
  • Constraint-solver path tracing. The current walk is a simple DFS. For very large frames, a constraint solver could compute conditional outputs more efficiently. The interface (input frame + session, output ConditionalOutput) is solver-agnostic. v2 if needed
  • Differential output. Showing the user "what changed since the last save" requires diffing two ConditionalOutputs. The data structures support this, but the diff algorithm and UI are deferred. v2
  • Cross-session best-inference. When the user has multiple sessions against the same frame, a "what does the body of sessions say" view could aggregate. Out of scope for v1; the schema does not preclude it. v2
  • Re-run on Authority update. If F5 ships and updates a court's binding scope, sessions whose results depend on that authority would benefit from re-evaluation. The hooks for "stale because external data changed" exist (the frame_version_drift_warning pattern), but the F5-driven variant is deferred. v2
