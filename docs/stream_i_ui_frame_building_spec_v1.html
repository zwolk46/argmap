Stream I per-module spec — Frame Building UI (page + left pane + right pane + validation drawer + cascade dialog + frame settings + auto-arrange)  ·  Stream I session I.9b  ·  May 11, 2026

This document is the implementation source of truth for src/ui/frame-building/. It commits per-file TypeScript signatures and behavioral contracts for: the FrameBuildingPage top-level composition (which replaces the I.9a placeholder export of the same name); the three-pane layout shell with bottom-drawer mount; the left pane (node palette + hierarchical outline tree); the right pane (context-sensitive inspector dispatching to per-node-type and per-edge-type editors, the options_box editor for SatisfactionPolicy per Article II § 3 / A3, the per-field AI-attribution decoration, and the inspector validation block); the bottom validation drawer rendering frame_store.validation with severity grouping, per-rule jump-to-node, and warning dismissal; the cascade-delete dialog consuming useCascadeConfirmation from I.9a; the frame settings panel (title / description / jurisdiction / positions / mode / flavor / default policies / pin / archive / delete) with the routing into I.9d's architectural-mode-change dialog; and the auto-arrange flow (confirmation + new FramePatch kind dispatched to the state layer). The I.9a substrate (chrome + canvas + primitives + suggestion drawer + cross-cutting hooks) is the foundation this spec consumes; cross-module integration surfaces in stream_i_contracts_v1.html (Contracts 5, 7, 8), stream_i_state_spec_v1.html, and stream_i_modes_spec_v1.html are cited by name and not restated.

All imports from @/runtime (value), @/persistence, @/modes (value), and value-level imports from @/llm-hooks are forbidden per Contract 8 and the I.9a substrate's no-restricted-imports wiring. The frame-building module's reads come from @/state selectors and snapshots; its writes go through frame_store.applyEdit(patch) with FramePatch from @/state; its type-only imports of HookInvocationRecord / SuggestionResult / ConfirmationDecision / HookId from @/llm-hooks are limited to the per-field AI-attribution decoration helpers and to the suggestion drawer's own type surface (which lives in I.9a; this spec only consumes it).

Cross-document resolution notes (for self-review)

The I.9b spec carries small refinements relative to stream_i_ui_chrome_canvas_spec_v1.html (I.9a), stream_i_state_spec_v1.html (I.5), stream_i_modes_spec_v1.html (I.6), and Contracts 7 and 8 in stream_i_contracts_v1.html. None is constitutionally novel; each is consolidated into a single in-session-escalated register flag (same family as F-003 and the I.3 / I.4 / I.5 / I.6 / I.7 / I.8 / I.9a pending flags). Held in flag_pending_from_i9b.html for merge.

  1. State-spec selector named: selectCascadeSummary. The I.9a register flag left "the useCascadeSummary selector is named in the state spec for I.9b's useCascadeConfirmation hook to consume" as an open item. This spec resolves it: the state spec's selectors.ts gains selectCascadeSummary(snapshot: FrameStoreSnapshot, node_id: NodeRef): CascadeReport | null. The function delegates to computeDeletionCascade (re-exported from @/modes's cascade.ts) when the snapshot has a current_version and the node_id exists in current_version.nodes; otherwise returns null. CascadeReport is re-exported from @/state's barrel so I.9b imports it through the state surface per Contract 8 boundary (the UI does not value-import from @/modes). The useCascadeConfirmation hook in I.9a is implemented against this selector.
  2. New FramePatch kind: presentation_hints_reset_all. The auto-arrange flow needs to clear x / y on every node atomically (single FrameVersion, single change_summary line, single autosave debounce). The existing presentation_hint_edited patch operates per-node and would require N dispatches with N FrameVersions; this is awkward in the version-history surface and the autosave path. The state spec's FramePatch union gains | { kind: "presentation_hints_reset_all" } and the modes spec's FrameActionDispatchTable gains a corresponding entry that, in one pass, produces a next_version in which every node has x and y set to undefined while collapsed is preserved. The dispatch entry follows the single-clock-read property (one deps.now()) and the sorted-iteration discipline. This is purely additive; no existing patch shape is changed.
  3. AppState extension: dismissed_warnings. The validation drawer supports per-warning dismissal (errors are never dismissable). The dismissal state is persisted in app_state per Stream H H2 (1-second debounced; never versioned). The AppState interface in @/persistence gains dismissed_warnings: ReadonlyArray<{ rule_id: string; node_ids: ReadonlyArray<NodeRef> }> (deduplicated by rule_id). The AppStateStore (Contract 7) is extended with dismissWarning(rule_id, node_ids): Promise<void> and restoreDismissedWarning(rule_id, node_ids): Promise<void> methods that wrap set; the state spec's app-state-store.ts gains the implementation. Pure-additive; no existing field is changed.
  4. Frame schema field: archived: boolean. The pin/archive/delete section exposes archive as a non-destructive alternative to delete. The schema spec's Frame interface gains archived: boolean (default false); the home-screen recents filter excludes archived frames. This is an additive metadata field; FramePatch.metadata_edited's partial shape covers it via Pick<Frame, ... | "archived">. The schema-spec's existing metadata_edited partial-target tuple is extended additively. The B7 forward-compatible envelope absorbs the addition; no migration is required (the Dexie schema's frames store is JSON-shaped per H1, and the I.4 persistence spec already round-trips arbitrary additional fields).
  5. I.9d integration callback: onOpenModeChangeDialog. The architectural-mode-change dialog is owned by I.9d (per I.9a "What this spec does not commit"). I.9b's FrameSettingsPanel exposes an onOpenModeChangeDialog callback that I.9d wires at integration time. Until I.9d lands, the "Change mode" button is rendered but disabled with the tooltip "Mode change is available in I.9d." The App-boots-end-to-end property is preserved by this disabled state; no placeholder dialog is added.

These five items consolidate into one in-session-escalated register flag for I.9b. Downstream coding is not gated on a contracts-doc bump; the per-module specs (this one, plus the state/modes/schema specs once amended) are canonical for their respective modules per Stream I conventions. The I.9a / I.9b / I.9c / I.9d coding sessions read the merged refinements via this spec's "Cross-document resolution notes" section plus the relevant per-module spec's amendment.

Behavioral contract. Mount lifecycle: on mount, the page calls frame_store.open(frame_id) if the store's current frame id is null or differs from the route's frame_id; while the store reports loading: true, the page renders a centered loading affordance (using the substrate's tokens). On unmount, the page calls frame_store.close() only when the route is changing to a non-frame-building route (the App's router signals this via a sentinel; for navigation between two frame-building routes the page re-mounts naturally per React keying on frame_id).

Selection: stored in a single useState<InspectorSelection>. The InspectorSelection union is { kind: "empty" } | { kind: "node"; node_id: NodeRef } | { kind: "edge"; edge_id: EdgeRef } | { kind: "multi"; node_ids: NodeRef[]; edge_ids: EdgeRef[] }. The page passes selection and selection-setter to both the FrameCanvas (via selected + on_selected_change) and the OutlineTree. The canvas works in NodeRef / EdgeRef arrays; the page translates between the canvas's array shape and the inspector's tagged-union shape (an empty array → empty; one node → single; one edge → single edge; otherwise → multi).

Top-bar wiring: the page constructs the slot fills via buildTopBarSlots(deps) and passes them to the substrate's TopBar with operating_mode="frame_building". The OperatingModeToggle's onSwitchToArgument callback opens a session-choice prompt the page hosts: it queries app_state.recents for sessions tied to this frame's current_version_id; for each session it offers an "Open existing" option, plus an "Open existing on older version" option, plus a "Create new session" option. The chosen session triggers navigate({ kind: "argument_running", session_id }). onValidationBlocked(errors) opens the bottom validation drawer and seeds the displayed list to the blocking errors; the user resolves and clicks the toggle again. onSwitchWithWarnings proceeds to the session-choice prompt directly.

Cascade-delete: the page mounts useCascadeConfirmation at top level and renders the CascadeDeleteDialog always (the dialog renders null when state is idle). Components that originate a delete (the canvas's keyboard delete, the inspector's delete button, the cascade context-menu entry) call cascadeConfirmation.request(node_id); the dialog opens; on confirm, cascadeConfirmation.confirm() dispatches frame_store.applyEdit({ kind: "node_deleted", node_id, cascade_confirmed: true }). The selection state is set to { kind: "empty" } on confirm.

Auto-arrange: the page mounts AutoArrangeFlow always; the canvas-toolbar's on_auto_arrange callback opens the flow's confirm dialog; on confirm, the flow dispatches frame_store.applyEdit({ kind: "presentation_hints_reset_all" }). The page also clears the selection on auto-arrange because node identities are preserved but positions change.

Edge creation: the canvas's on_edge_created emits an EdgeCreationCandidate. The page routes by candidate kind: "edge" → frame_store.applyEdit({ kind: "edge_added", edge: { id: generateId(), type, source, target } }); "logical_gate_slot" → frame_store.applyEdit({ kind: "node_edited", node_id: gate_id, partial: { ...slot-specific assignment } }); "checkpoint_option_routing" → frame_store.applyEdit({ kind: "node_edited", node_id: checkpoint_id, partial: { options: nextOptions } }) where nextOptions is the existing options array with the targeted option's target_node_id updated. The generateId available to the page comes through the same RepositoryProvider the store consumes; the I.9a substrate exposes it via a useGenerateId() hook that context.tsx in the state layer adds as a parallel to useRepository (this is one of the small naming items the state spec's context.tsx already exposes per its FrameStoreOptions.generateId).

Drawer / panel toggles: the four right-side overlays (validation drawer, settings panel, help pane, suggestion drawer) follow a "topmost overlay handles Escape" pattern. The substrate's Drawer primitive does not focus-trap; only the cascade-delete Dialog does. When two right-side overlays are simultaneously open, the most-recently-opened wins keyboard focus. The page tracks open order in a small stack and routes Escape to the top.

Keyboard shortcuts: the page calls useKeyboardShortcuts with bindings derived from DEFAULT_SHORTCUTS, customizing each handler to act against the page's local state: zoom_in / zoom_out / zoom_fit / zoom_100 call frameCanvasHandle.current?.zoom(...); focus_search focuses the canvas-toolbar's search input via a ref the page threads through; open_help toggles the help pane; close_overlay closes the topmost overlay. Right pane — options_box editor (SatisfactionPolicy / A3)

The options_box editor is the Article II § 3 / A3 surface for editing a node's SatisfactionPolicy. A3 commits four mechanics, all of which this editor implements: (i) per-instance options_box is allowed on Checkpoint, RootQuestion, SubQuestion, and Interpretation; other types use the frame default for that type, edited via the frame-settings panel's default-policies section; (ii) policies are an AND list of Condition entries plus at most one AnyOf group (the disjunctive escape hatch); (iii) the offered conditions vary by mode/flavor per D4 + D5 and per the schema's OFFERED_CONDITIONS_BY_MODE_FLAVOR table; (iv) the effective policy at any time is computed by resolveEffectivePolicy(frame, node) from @/schema (per-instance > frame-default-for-type > library-default), and this resolved policy is rendered read-only above the editor.

The panel is structured as labeled sections; per Stream E6, sections that are not relevant to the current mode/flavor are hidden (not just disabled), so the panel's visual density tracks the frame's operating mode. Editable fields route through frame_store.applyEdit with metadata_edited or default_policy_edited patches as appropriate; non-edit actions (pin, archive, delete) route through app_state_store directly (these are at the frame-record level, not the FrameVersion level, so they are not in the FramePatch union).

Behavioral contract. Renders Drawer anchored to the right edge, fixed width per the I.9a layout tokens. Body is a scroll container with the seven sections, in canonical order: Metadata, Mode & Flavor, Jurisdiction (legal only), Positions (general only), Default Policies, Pin / Archive / Delete, and the closing footer with "Done" (just closes the panel). The panel does not own form state — each section reads directly from useFrameStore and dispatches patches on change, so closing and re-opening the panel never loses edits.

Auto-arrange flow

Auto-arrange is the affordance that resets per-node x/y presentation hints so that the canvas's React Flow layout re-runs from a clean slate. Per the cross-document resolution notes refinement #2, the patch is a new FramePatch kind: { kind: "presentation_hints_reset_all" }, applied atomically by the state layer (a single FrameVersion delta clearing x and y for every node while preserving collapsed). The entry point is the onAutoArrange slot on the I.9a FrameCanvas; the page wires this slot to the flow described here.

// src/ui/frame-building/auto-arrange/auto-arrange-flow.tsx
import * as React from "react";

export interface AutoArrangeFlowProps {
  open: boolean;
  on_close: () => void;
}

export function AutoArrangeFlow(props: AutoArrangeFlowProps): React.ReactElement;

Behavioral contract. Wraps the I.9a ConfirmDialog primitive. Title: "Auto-arrange nodes?". Body: a short prose explanation that all manual node positions in the current frame version will be cleared so the canvas can re-layout, and a note that this only affects the current frame version (older versions retain their positions). Primary action "Auto-arrange" dispatches the presentation_hints_reset_all FramePatch through frame_store.applyEdit; secondary action "Cancel" closes. The dialog also exposes the H2 1-second debounced undo: after confirmation, a toast appears with "Auto-arranged. Undo" and a 1-second window; clicking undo dispatches the inverse patch (the state layer maintains the pre-patch positions in the version delta, so undo is a direct revert of one FrameVersion entry — no special inverse logic is required at the UI layer). The flow is mounted once per page and toggled by the canvas's onAutoArrange slot.

Determinism — UI-layer discipline

The four UI-layer determinism obligations established in stream_i_ui_chrome_canvas_spec_v1.html apply unchanged here, with the following I.9b-specific surfaces called out:

  1. Canonical validation order. Both the InspectorValidationBlock (per-node) and the ValidationDrawer (frame-wide) render results in the order returned by selectValidationDrawer / selectValidationByNode, which themselves sort by (severity, VALIDATION_RULE_PRIORITY[rule_id], node_id, edge_id) per the I.5 state spec. No I.9b component re-sorts.
  2. AI-attribution chips on every hook-touched field. The FieldAttributionDecoration wraps every editable field in every per-node-type editor and every ConditionRow. The decoration's lookup is keyed on the canonical fieldAttributionPath (e.g., "statement", "options_box.and[0]", "checkpoint.options[1].label"); the path scheme is committed in use-field-attribution.ts and is the same scheme the LLM hooks layer writes against when emitting HookInvocationRecord.target_field_path.
  3. Canonical cascade-report ordering. CascadeSummaryTree never re-sorts the CascadeReport; it consumes the already-sorted deleted_node_ids and deleted_edge_ids arrays from computeDeletionCascade. The outline-tree projection within the dialog uses buildOutlineShape, which is itself deterministic per the I.9a substrate's determinism commitments (it walks edges in OUTLINE_EDGE_PRIORITY order).
  4. Deterministic options_box rendering. ConditionList renders conditions in the canonical order from CONDITION_KIND_PRIORITY (defined in @/schema); the AnyOf group, when present, always renders below the AND list. The ConditionPicker menu lists kinds in the same canonical order, regardless of what is or isn't currently in the policy. This guarantees that two users with the same frame state and the same mode/flavor see the same UI, byte-for-byte, in the inspector — a precondition for collaborative work and for git-diff stability of saved frames.

Article II §§ 1–6 commitments

§ 1 (Logical-frame / premise separation). The frame-building module deals exclusively with the logical frame — nodes, edges, options_box policies, frame metadata, mode/flavor. Premises are not edited here; the inspector's per-node-type editors do not surface premise-side fields. The Argument Running surface (I.9c) will be the premise-editing locus.

§ 2 (Implementation-level commitments). Every file in the file map has a per-file behavioral contract and a TypeScript signature block. Cross-file boundaries (component prop shapes, hook return types, FramePatch routing paths) are explicit. The inspector's selection-dispatch logic, the options_box editor's three-tier layout, the validation drawer's grouping + dismissal flow, the cascade-delete dialog's CascadeReport consumption, the frame-settings panel's section structure with mode/flavor visibility, and the auto-arrange flow's new FramePatch kind are all committed at implementation granularity.

§ 3 (Per-node satisfaction criteria / A3). The options_box editor implements all four A3 mechanics: per-instance allowed on Checkpoint/RootQuestion/SubQuestion/Interpretation; AND list + at-most-one AnyOf group; offered conditions vary by mode/flavor; effective policy is the resolveEffectivePolicy precedence chain (per-instance > frame-default-for-type > library-default), rendered read-only above the editor.

§ 4 (Mode/flavor surface visibility). The consolidated visibility table from current_state.html is honored across every editor (per-node-type editors, edge editor, options_box editor's condition picker, frame-settings panel's Jurisdiction and Positions sections, and the palette's NODE_TYPES_BY_MODE_FLAVOR filter). No surface in this spec leaks a legal-mode field into general mode or vice versa.

§ 5 (Deterministic ordering). The "Determinism — UI-layer discipline" section commits four invariants and points to the canonical-order sources for each. No I.9b component re-sorts data; all components consume already-sorted selectors or pure shape utilities.

§ 6 (AI-attribution surfacing). The FieldAttributionDecoration wraps every editable field in every editor and condition row. The fieldAttributionPath scheme is committed in use-field-attribution.ts and is the same scheme the LLM hooks layer writes against.
