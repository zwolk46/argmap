stream_i_ui_session_settings_spec_v1.html

Per-module spec for src/ui/session-settings/  ·  Stream I session I.9d3  ·  May 11, 2026

This document is the implementation source of truth for src/ui/session-settings/ — the session-settings panel the I.9c buildTopBarSlots reserves a placeholder slot for and the I.9c ModeFlavorChip.onOpenSettings wires its callback into. The panel is the argument-running parallel to I.9b's FrameSettingsPanel: a right-edge Drawer with labeled sections that compose section-local form state read directly from useSessionStore, dispatching SessionPatch on change. The panel is page-local — reached only from ArgumentRunningPage — so its open-state is lifted into the page, following the F-007 page-local-dialog precedent rather than the F-006 App-level pane precedent. This spec commits per-file TypeScript signatures and behavioral contracts for: the panel shell (session-settings-panel.tsx); the metadata section (title + description); the G6 remove-rewrite section that dispatches the new output_overrides_cleared SessionPatch kind; the G12 cross-implication advisory section that displays the frame-scope llm_settings read-only with a "change in frame settings" affordance; the archive-and-delete section parallel to I.9b's pin-archive-delete (with Pin omitted — see "What this spec does not commit"); the page-level refinements lifting the dialog open-state into ArgumentRunningPage; and the small buildTopBarSlots deps extension that replaces the I.9c-disabled placeholder IconButton with a real callback.

Six cross-spec refinements are committed and held in the consolidated F-008 register flag in the same F-003 / F-004 / F-005 / F-006 / F-007 family pattern: (1) a new SessionPatch kind output_overrides_cleared in the state spec that atomically clears ArgumentSessionVersion.output_overrides in one new ArgumentSessionVersion; (2) a new SessionActionDispatchTable.output_overrides_cleared entry in the modes spec as a synchronous pure transformation; (3) an additive ArgumentSession.archived: boolean field in the schema spec (parallel to F-004's Frame.archived); (4) extension of session_metadata_edited.partial's Pick tuple to include archived (parallel to the F-004 + F-007 pattern on the frame side); (5) a new AppStateStore.deleteSession(session_id: SessionId): Promise<void> orchestration wrapper in the state spec, parallel to the I.9b-implied app_state_store.deleteFrame; (6) extension of ArgumentRunningTopBarDeps in the argument-running spec to include on_open_session_settings: () => void, replacing the I.9c "disabled placeholder" notation with a real callback that the page wires from its own session-settings-panel open-state. Downstream coding sessions are not gated on a contracts-doc bump; the per-module specs are canonical per Stream I conventions.

What this spec does not commit

This spec deliberately omits:

  • The onboarding wizard and coachmark machinery (I.9d4 scope). The session-settings panel does not host a "reset coachmarks for this session" affordance; coachmark dismissals are AppState-scoped (per F-007's read of selectCoachmarkDismissed), so resetting is a frame-or-app-level concern that I.9d4 places elsewhere.
  • Pin (in the v1 panel). The current_state.html description says "Pin / Archive / Delete (parallel to I.9b's pin-archive-delete section)," but sessions have no schema-level pinning surface analogous to AppState.pinned: FrameId[]: there is no AppState.pinned_sessions, no ArgumentSession.pinned: boolean, no v1 home-screen surface that surfaces pinned sessions, and no session-switching UI in I.9b's "Open existing session" flow (lines 183 of stream_i_ui_frame_building_spec_v1.html) that consumes a session-pin state. Adding session-pinning as part of I.9d3 would require either inventing a new pinning semantic ("mark as the user's primary session for this frame" — a domain concept the existing design does not commit) or adding a new AppState.bookmarked_session_by_frame map (additive but consumed by zero v1 surfaces, so the affordance does nothing visible). Both options are speculative; Article III § 5 (Reversibility) and § 6 (Interface Fit) favor omitting Pin until a consuming surface emerges. The decision is within Article V § 2(c) and § 2(e). When a consuming surface lands (likely a frame-side "default session" affordance or a session-list in the home screen), Pin can be added additively. The section is therefore named Archive and Delete in this v1 spec; if a future session reinstates Pin, the section is renamed to match without breaking the file or the field shapes.
  • Cross-session output-override "permanent suppression." The phrase "Remove rewrite permanently for this session" in current_state.html is interpreted at the per-version semantic: dispatching output_overrides_cleared creates a new ArgumentSessionVersion with output_overrides set to undefined, which is the canonical "no rewrite present for this session" state until the user manually re-invokes G6. There is no parallel session-scope flag preventing G6 from ever being invoked again on this session — invocation is always user-initiated per Stream G's "G6 manual trigger" contract, so "permanent suppression" is achieved by the user simply not re-invoking. If a future session adds an opt-out flag (ArgumentSession.g6_disabled: boolean), it joins additively.
  • A session-scope G12 override. Per the conceptual question's resolution at session-start (recorded under "Conceptual question resolution" below), the G12 toggle in the session-settings panel reads the frame-scope llm_settings.runtime_hooks_enabled and llm_settings.per_hook_enabled?.cross_implications read-only and renders a "change in frame settings" link rather than introducing a session-scope override field. No new schema is added for G12; the panel surfaces existing state.
  • Tech-stack decisions. The drawer primitive, focus mechanics, segmented-toggle composition, confirmation-dialog wrapping, and React Flow integration come from the I.9a substrate; this spec composes them without restating their internals. The visual-polish tokens (color, typography, spacing, motion) come from stream_e_visual_polish_v1.html via the I.9a token surface.
  • App-level mount. The session-settings panel is reached only from ArgumentRunningPage. The F-006 App-level pattern (a single mount serving multiple pages via useRoute()) does not apply. app-routes.tsx is not touched by this session. The F-007 page-local-dialog precedent governs.

Conceptual question resolution (recorded at session start)

The I.9d2 self-review surfaced one conceptual question for I.9d3: does the G12 toggle in the session-settings panel display the frame-scope llm_settings.runtime_hooks_enabled read-only with a "change in frame settings" link (no new schema), or add a session-scope override field (additive schema extension)? Applying Article IV § 4 (substantial-hesitation test) at session start:

  • (a) Multiple defensible options with material downstream consequences not clearly addressed by the Constitution or prior user direction? Two defensible options. The downstream consequences, however, are clearly addressed by Stream G: stream_g_llm_hooks_v1.html line 729 states "Runtime sidecar hooks (G11, G12): disabled by default in v1. The user opts in per-frame in Frame settings; G11 also has a one-time per-frame confirmation dialog…" The per-frame opt-in is a Stream G design commitment. Under Article VI § 1, Stream G as a working document binds agent judgment unless the Constitution overrides; the Constitution does not. Option A directly honors the Stream G commitment; Option B would contradict it.
  • (b) Reasoning depends on inferences about user preferences not directly supported? No. The choice is reachable from constitutional principles (Article III § 5 Reversibility, § 6 Interface Fit) and the prior-stream commitment alone.
  • (c) Decision commits the project to a path costly to retreat from? Option A is more reversible. If a future session decides session-scope override is needed, adding the field is trivially additive (no migration; new field starts undefined and the resolution rule defaults to frame-scope when absent). Option B locks in a schema commitment that may never be exercised in v1 — G12 ships a no-op fallback per stream_i_llm_hooks_spec_v1.html line 1445, so the override would have no v1 user-visible effect.

Decision: Option A. The G12 toggle row in the session-settings panel displays the current frame-scope state (an aggregate of frame.llm_settings?.runtime_hooks_enabled and the per-hook override, if any) read-only, with a link that — when clicked — closes the session-settings panel and opens the frame-settings panel via a navigation-and-callback pattern specified below. No new schema is added for G12 in this session. The decision is recorded in this spec; the substantial-hesitation test did not fire; no escalation to Zach is warranted. If, after v1 ships, user feedback indicates session-scope granularity is needed, a future session adds ArgumentSession.llm_overrides?: { runtime_hooks_enabled?: boolean; per_hook_enabled?: { [hook_name: string]: boolean } } with the resolution rule "session-scope wins over frame-scope when defined; otherwise frame-scope" — a pattern that compiles cleanly into the existing Stream G hook-activation discipline.

Cross-document resolution notes (for self-review)

Six small refinements relative to prior I.9 sibling specs. Each is technical, additive (or wiring-promotion of placeholders the I.9c spec explicitly reserved), and within agent-autonomous scope per Article V § 2(c). All six are held in the consolidated F-008 register flag below in the same F-003 / F-004 / F-005 / F-006 / F-007 family pattern.

  1. New SessionPatch kind: output_overrides_cleared. The G6 remove-rewrite affordance needs to atomically clear ArgumentSessionVersion.output_overrides in one new ArgumentSessionVersion. The existing session_metadata_edited patch operates on session-metadata fields only; G6's own commit writes into output_overrides.rewritten_prose (per stream_i_llm_hooks_spec_v1.html line 1197) — there is no patch shape that clears the override. The state spec's SessionPatch union gains | { kind: "output_overrides_cleared" } (a zero-payload patch — the action is fully determined by the kind). Parallel in spirit to F-004's presentation_hints_reset_all (zero-payload Frame-side atomic reset) and F-007's architectural_mode_changed (atomic-FramePatch family). The patch is versioned: true by virtue of being a SessionPatch — every session-side patch creates a new ArgumentSessionVersion in the normal autosave flow per stream_i_state_spec_v1.html behavioral contracts. Purely additive; no existing patch shape is changed.
  2. New dispatch-table entry in the modes spec. The modes spec's SessionActionDispatchTable gains output_overrides_cleared as a synchronous pure transformation: (args: { current_session: ArgumentSession; current_version: ArgumentSessionVersion; deps: SessionActionDeps }) => SessionTransformResult. The implementation produces a new ArgumentSessionVersion that copies every field from current_version except output_overrides, which is set to undefined; created_at = deps.now(); the version id is fresh from deps.generateId(); parent_version_id is current_version.id; is_milestone: false; change_summary is the constant string "Removed rewritten prose." The resulting ArgumentSession.current_version_id is updated to the new version id; updated_at = deps.now() (same value read once per dispatch per F-003 family discipline). Pure synchronous; deterministic (single deps.now(), no iteration that could depend on map ordering); no input validation is needed because the patch has no payload. The Vitest acceptance tests in the modes module cover the field-clearing semantics and the determinism property.
  3. Additive schema field: ArgumentSession.archived: boolean. Parallel to F-004's Frame.archived: boolean. The schema spec's ArgumentSession interface gains archived: boolean (default false; the schema-level default is enforced by Stream B B7's forward-compatible envelope — legacy sessions absent the field are read as false). This is a session-record-level flag (not on per-version snapshot), so it survives version restores and lives parallel to title, description, etc. The home-screen and session-switching surfaces filter archived sessions out of default lists; the consuming UI surfaces are out of scope for this spec (the I.9b "Open existing session" flow is a frame-building-side surface; surfacing of archived sessions in that flow is an I.9b-or-later refinement). The field is mutated through session_metadata_edited per refinement #4 below.
  4. Extension of session_metadata_edited.partial Pick tuple to include archived. The current Pick tuple in the state spec (line 276 of stream_i_state_spec_v1.html) is Partial<Pick<ArgumentSession, "title" | "description">>. The session-settings panel dispatches session_store.applyEdit({ kind: "session_metadata_edited", partial: { archived: true } }) for the archive toggle and { archived: false } for unarchive. This refinement extends the tuple to Partial<Pick<ArgumentSession, "title" | "description" | "archived">> in parallel to the F-004 + F-007 frame-side pattern (where archived joined the frame's metadata_edited Pick tuple and flavor joined it via F-007). The modes spec's SessionActionDispatchTable.session_metadata_edited entry already iterates the partial object's keys generically (per its existing behavioral contract) and applies each key to the matching field on ArgumentSession; no dispatch-table-body change is needed beyond widening the type.
  5. New AppStateStore.deleteSession method. The session-delete flow requires an orchestration wrapper that handles the close-then-delete sequence cleanly, parallel to the I.9b-implied app_state_store.deleteFrame (referenced in stream_i_ui_frame_building_spec_v1.html line 999 but not yet pinned on the AppStateStore interface in stream_i_state_spec_v1.html — a pre-existing gap that the consolidated F-003/F-004 contracts-doc bump is expected to close). The new method signature is deleteSession(session_id: SessionId): Promise<void>. Behavioral contract: the method delegates to this.deps.repository.deleteSession(session_id) (existing at stream_i_persistence_spec_v1.html line 801) inside a try/catch; on resolve, the method completes; on reject, the method re-throws with a wrapped RepositoryError. The page is responsible for calling session_store.close() before calling app_state_store.deleteSession (the call order matters because session_store.close() stops further autosave scheduling against the about-to-be-deleted session; reversing the order risks a write-after-delete race). The method does not remove the deleted session from any AppState-level cache (sessions are not enumerated in AppState; only Frames are, via AppState.recents and AppState.pinned). The session-versions are bulk-deleted by Repository.deleteSession's own transaction. The method's parallel on the frame side — app_state_store.deleteFrame — is referenced by I.9b without being formally pinned; this refinement notes that both methods belong on the AppStateStore interface and that the future contracts-doc bump should formalize both together. The I.9b reference to app_state_store.deleteFrame remains the canonical surface for the frame side; this spec does not redefine it.
  6. Extension of ArgumentRunningTopBarDeps to include on_open_session_settings. The I.9c spec (lines 251 and 253 of stream_i_ui_argument_running_spec_v1.html) renders <ModeFlavorChip onOpenSettings={openSessionSettings} /> in the chips slot and a disabled <IconButton ... onClick={openSessionSettings} disabled /> in the buttons slot, where openSessionSettings is a forward reference that I.9c did not pin in the ArgumentRunningTopBarDeps interface (lines 234–243). I.9c reserved the name as a placeholder, with the IconButton rendered with disabled and a Tooltip "Available in I.9d." This refinement adds on_open_session_settings: () => void as a required field on ArgumentRunningTopBarDeps, and replaces the disabled attribute on the buttons-slot IconButton with the live callback. The ModeFlavorChip's onOpenSettings remains required (it always was per the I.9a chrome contract); it now resolves to the live callback instead of the inert forward reference. The Tooltip is removed from the IconButton. The buildTopBarSlots function signature widens by exactly one deps field; the page passes its openSessionSettingsPanel handler as that field. No new component surfaces are introduced; this is wiring promotion.

// src/ui/session-settings/session-settings-panel.tsx

The panel. Renders as an I.9a Drawer (right-edge, fixed width per the I.9a layout tokens — --space-drawer-default); composes the four labeled sections in canonical order: Metadata, Output rewrite, Advisory notes, Archive and Delete. Open state is owned by ArgumentRunningPage (page-local mount; F-007 precedent); opened by the chrome's ModeFlavorChip.onOpenSettings callback and by the buttons-slot IconButton's onClick. Closes on Escape (the Drawer primitive's dismissOnEscape default), click-outside (Drawer default), or the panel's explicit close button (in the Drawer footer).

import * as React from "react";

export interface SessionSettingsPanelProps {
  open: boolean;
  on_close: () => void;
  /** Called when the user clicks the "change in frame settings" link in the
      G12 section. The page closes the session-settings panel first, then
      navigates to the frame-building route for the session's parent frame and
      opens the frame-settings panel. */
  on_open_frame_settings: () => void;
  /** Called when the user confirms session deletion. The page closes the
      session_store, calls app_state_store.deleteSession, and navigates back
      to the parent frame. */
  on_delete_session: () => void;
}

export function SessionSettingsPanel(props: SessionSettingsPanelProps): React.ReactElement;

Behavioral contract. The panel renders the I.9a Drawer with open={props.open}, onClose={props.on_close}, title="Session settings", and the body as a vertical scroll container holding the four sections in canonical order. The Drawer's side="right" is the default. The footer holds a single "Done" button (just calls on_close); destructive actions are inside Archive and Delete and have their own confirmation dialogs. The panel does not own any form state — each section reads from useSessionStore (and the G12 section additionally from useFrameStore) and dispatches patches on change. Closing the panel mid-edit and re-opening it preserves all in-flight values because the source-of-truth is the store, not the panel-local state. The panel passes on_open_frame_settings to the G12 section and on_delete_session to the Archive-and-Delete section. The Metadata and Output-rewrite sections take no extra props beyond their store reads.

Section render order: Metadata first (the most common edit), then Output rewrite (G6 affordance — visible when an override exists; otherwise still rendered as a read-only "no rewrite present" line), then Advisory notes (G12), then Archive and Delete (destructive section, visually separated by a band per the I.9b precedent).

Visual hierarchy: Each section has a small heading (--text-sm-medium), a body, and a thin divider between sections (a single --color-border-tertiary hairline, no extra heading-decoration). The Archive-and-Delete section is preceded by an extra spacing band (--space-6) and the destructive button uses --color-severity-error per the ConfirmDialog's destructive: true prop. The G12 section's read-only display uses the Pill primitive with a neutral surface — no toggle interaction is presented because the row is read-only by the conceptual-question resolution.

Coexistence with the prose-tab "Remove" affordance: The I.9c prose tab also exposes a "Remove" affordance for G6 rewrites (line 605 of stream_i_ui_argument_running_spec_v1.html, currently disabled with a Tooltip "Available in I.9d"). Both entry points dispatch the same output_overrides_cleared SessionPatch and produce the same result. The prose-tab "Remove" affordance is wired by I.9d3 as part of refinement #1 above: the implementing session enables the affordance (removes the disabled attribute and Tooltip) and wires its onClick to dispatch the new SessionPatch. The prose-tab affordance and the session-settings panel affordance use distinct visual treatments (the prose tab has a small "Remove" button beside the rewritten paragraph; the session-settings panel has a destructive button inside its G6 section), but both dispatch identically. The implementing session writes a small integration test asserting that dispatching from either surface produces the same patch payload and the same resulting ArgumentSessionVersion shape.

// src/ui/session-settings/g6-remove-rewrite-section.tsx

The G6 prose-rewrite affordance in the session-settings panel. Conditionally rendered based on the presence of an override.

import * as React from "react";

export function G6RemoveRewriteSection(): React.ReactElement;

Behavioral contract. The section reads current_version.output_overrides from useSessionStore. Two render paths:

No override present (current_version.output_overrides === undefined OR current_version.output_overrides.rewritten_prose === undefined): renders a single read-only paragraph in secondary text: "No rewritten prose for this session. Use the 'Rewrite' button in the prose tab of the output viewer to generate one." No interactive controls in this render path.

Override present (current_version.output_overrides?.rewritten_prose is a non-empty string): renders a small block with three pieces.

  • The AiAttributionChip labeled "G6 · rewritten {time-ago}" where the time-ago is derived from current_version.output_overrides.rewritten_at via the I.9a time-ago helper (the same helper the I.9c prose tab uses).
  • A preview paragraph showing the first ~120 characters of rewritten_prose followed by an ellipsis when the full text is longer. Full text is available in the prose tab; the panel deliberately shows a preview only, to avoid duplicating the prose tab's primary surface.
  • A destructive "Remove rewrite" button (icon-button-ish, label visible) that opens a ConfirmDialog with:
    • title: "Remove the rewritten prose?"
    • body: "This creates a new session version (v{N+1}). The canonical deterministic output is unchanged."
    • confirmLabel: "Remove rewrite"
    • cancelLabel: "Cancel"
    • destructive: true
    Confirm dispatches session_store.applyEdit({ kind: "output_overrides_cleared" }) and closes the dialog. The session-settings panel itself remains open after the dispatch; the section re-renders into the "no override present" path on the next snapshot tick.

{N+1} interpolation in the body string uses current_version.version_number + 1 read at render time. The implementing session writes a test that asserts the correct version number is interpolated for both v1 → v2 and v17 → v18 cases.

export function g12EffectiveEnabled(
  llm_settings: LlmSettings | undefined,
): boolean {
  if (!llm_settings) return false;                                  // schema default

Close path: any close (Escape, click-outside, the Drawer's close button via the Done footer button) calls closeSessionSettingsPanel which sets the local state to false; the Drawer animates out and returns null on the next snapshot tick.

G12 link path: the user clicks the "change in frame settings" link in the G12 section. The section calls props.on_open_frame_settings, which on the page side is onOpenFrameSettingsFromG12Link. UI-layer determinism discipline (restated for I.9d3's surfaces)

Per the I.9a / I.9b / I.9c / I.9d1 / I.9d2 disciplines, the UI layer's contribution to the application's determinism guarantee (Article II § 2) is four obligations: D1 canonical output verbatim, D2 single-clock-read-per-dispatch, D3 deterministic order (no client-side sorting of canonical lists), D4 stable cross-render identity for memoized projections. Each surface in this spec honors them:

  • Session-settings panel shell: D1 — none of the panel surfaces render output canonically derived from compute() directly (G6's rewritten prose is read verbatim from session_version.output_overrides.rewritten_prose as a session-versioned field, not from ConditionalOutput; G12's enabled state is derived from frame.llm_settings through a pure helper). D2 — the panel reads no clock; the eventual deps.now() reads are inside the SessionStore's session_metadata_edited and output_overrides_cleared dispatch-table entries, each consuming a single clock read per dispatch per F-003 family discipline. D3 — the four sections are rendered in canonical order (Metadata, Output rewrite, Advisory notes, Archive and Delete); no list is consumed from upstream that could be re-sorted. D4 — section-local memoizations (the G12 effective-enabled derivation; the G6 preview-string truncation) are pure functions of their inputs and stable across renders.
  • G6 remove-rewrite section: D1 — the rewritten prose preview is rendered verbatim from output_overrides.rewritten_prose, character-truncated only; the AI-attribution chip's label is rendered from output_overrides.rewritten_at via the I.9a time-ago helper which is deterministic given its input. D2 — the ConfirmDialog's body interpolates current_version.version_number + 1 read at render time, not at click time; the clock read happens once inside the SessionStore's dispatch path when Confirm is clicked. D3 — no list rendering. D4 — the preview-string truncation is a pure function of the input string.
  • G12 advisory-toggle section: D1 — the effective-enabled derivation is a pure function of frame.llm_settings; the Pill's label is one of two constants ("Enabled" / "Disabled") chosen by the derivation. D2 — the section reads no clock; the eventual deps.now() reads only fire if the user follows the link to frame-settings and changes the per-frame value there (Frame-side dispatch). D3 — no list rendering. D4 — g12EffectiveEnabled is pure-total over its input type.
  • Archive-and-Delete section: D1 — the destructive-confirm body interpolates session.title verbatim into the type-to-confirm field's matcher; no derived strings. D2 — the section reads no clock; the eventual deps.now() reads are in the SessionStore's session_metadata_edited({ archived: ... }) dispatch (for the archive toggle) and in the Repository transaction (for the delete). D3 — no list rendering. D4 — the type-match check is a pure equality comparison.
