stream_i_contracts_v1.html

Cross-module integration contracts  ·  Stream I session I.1  ·  May 10, 2026  ·  Sibling of stream_i_buildkit_index_v1.html

This document is the canonical TypeScript boundary between modules. Per-module Stream I specs (I.2–I.10) cite specific interfaces from this document by name; coding sessions implement to these signatures. The contracts are shape-only — they fix the type surface that crosses module boundaries, not the implementation. Stream-specific internal types live in the per-module specs and need not appear here.

Where a contract refers to a Stream-B-defined entity (Frame, FrameVersion, ArgumentSession, etc.), the canonical shape is in stream_b_schema_v1.html; this document re-states only the cross-module-relevant surface and points to Stream B for the rest.

Module dependency graph

Allowed import directions (read "→" as "may import from"):

schema       ← runtime  ← persistence ← state ← ui
                   ↑                       ↑      ↑
                   └─────── modes ─────────┘      │
                                                  │
schema ← llm-hooks ← state ← ui                   │
schema ← layout ← state ← ui ─────────────────────┘
                                                  │
schema ← modes ← state ← ui                       │

Hard rules:

  • schema imports nothing from this codebase. Pure types and validators.
  • runtime imports only schema (Article II § 2; Stream H part 1; ESLint no-restricted-imports). It does not import persistence, ui, state, llm-hooks, modes, or layout. It does not import any third-party library that reads the clock or environment.
  • llm-hooks imports schema (and may import the Repository interface from persistence for hook contexts; see HookContext below). It is never imported by runtime.
  • persistence imports schema and runtime (the latter only to call compute on session restore; the runtime is itself pure).
  • modes imports schema, runtime, persistence, and llm-hooks. Modes orchestrates user-initiated actions; it is the layer where action surfaces (D1, D2) live.
  • layout imports schema only. Pure layout against frame structure; main thread speaks to the layout Web Worker via the bridge.
  • state imports everything except ui. Stores hold references to Repository, dispatch into modes, subscribe to runtime outputs.
  • ui imports state, schema, and primitives from layout and llm-hooks. No direct calls to persistence or runtime; routing through stores.

Notation conventions

  • Types written in TypeScript. Where Stream B uses a TypeScript-like notation (with ? for optional), this document follows TypeScript exactly.
  • NodeRef and EdgeRef are aliases for string (entity ids). They live in schema/identifiers.ts.
  • timestamp is string (ISO-8601). Stream B fixes this; reading it as Date happens in UI/state; runtime treats it as opaque.
  • Imports are written as @/<module>/<file>; the @ alias is wired in tsconfig.json and vite.config.ts.
  • Where a type is defined in another contract section of this document, it is referenced by name without re-stating its shape.

Contract 2 — runtime → schema

The runtime exports a single deterministic top-level function. Internal phase functions (resolve_foreclosure, compute_status_map, etc.) are not part of the cross-module surface.

// Iteration helpers (re-exported for tests; not for consumption outside src/runtime/).
export { sortedIter, sortedKeys, sortedEntries } from "./iteration-helpers";

The runtime never exposes mutable references. Callers receive ReadonlySet/ReadonlyMap/ReadonlyArray projections; mutating them is a programmer error and not contractually supported.

Contract 8 — ui → state, schema, layout, llm-hooks (primitives)

The UI does not talk to runtime, persistence, or modes directly. Every action goes through a state store; every read comes from a store selector or its computed projection. UI components export React component types; the cross-module surface is the prop shapes below. Per-component design lives in stream_i_ui_spec_v1.html (I.9).

// Top-level shell. Mounts stores, repository, autosave, providers.

// Each subdirectory exports its own page-level component:
//   @/ui/frame-building → FrameBuildingPage
//   @/ui/argument-running → ArgumentRunningPage
//   @/ui/version-history → VersionHistoryPane
//   @/ui/onboarding → OnboardingWizard, Coachmark
//   @/ui/primitives → StatusBadge, TypeIcon, GlossaryTooltip, etc.

Cross-cutting: validation result attachment

Three modules produce ValidationResult[]: schema (rule definitions), runtime (phase 1 evaluation), and llm-hooks (G4 gap-detection advisories with severity "warning"). All three flow into the state layer's validation projection, which the UI surfaces via the per-node chips, bottom drawer, and inspector block (E2). The contracts above pin ValidationResult in @/schema/validation-rules as the canonical shape.

Cross-cutting: HookInvocationRecord audit trail

Hook commits write into FrameVersion fields and append to Frame.llm_settings.invocations. The append happens through Repository.saveFrame in the same transaction as the FrameVersion write that the commit produces. The state layer renders the audit projection per Stream E; the UI never invokes hooks directly without going through llm-hooks contracts.

Cross-cutting: F5 (court hierarchy database) deferred contract

The F5 module under src/runtime/f5/ is stubbed in v1. Its eventual interface is fixed by Stream F's F5Database API surface. v1 ships F6-v1 only; the F5 wire-in session adds the F5-database-lookup branch to authority-binding.ts per F6-v2's specification, which widens the result envelope additively (kind, reasoning, database_version) without changing the function signature. What I.2–I.10 add to this document

Each per-module spec extends this contracts doc with internal types and per-file signatures, but does not change the cross-module shapes here without flagging the change as an in-session-escalated flag (Article X § 1). Cross-module shape changes require the I.1 author (or a review session) to update this doc and bump its version (e.g., stream_i_contracts_v2.html). Per-module specs reference contracts here by file-section name; coding sessions follow the chain from the Stream I spec to this document and stop.
