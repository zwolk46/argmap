Resolutions

A1Conclusion node contentlocked

A Conclusion stores the answer plus a structural direction; confidence is computed from the path, not stored on the node; remedy is a v1 field on the Conclusion (legal mode only) with the schema designed so it can be promoted to a separate node type later if remedy sub-frames are needed.

Conclusion {
  statement: text  // required
  direction: enum  // required, mode-aware
  reasoning_summary: text?  // optional
  tags: string[]?
  remedy: text?  // legal mode only, v1
  notes: text?  // from A12
}

Direction enum, legal mode: favors_plaintiff, favors_defendant, affirm, reverse, remand, dismiss, custom. General mode: position_a, position_b, ... (mapped to user-named positions on the frame), custom.

Confidence is displayed on the Conclusion (computed from the session's status map — proportions of Checkpoints satisfied by binding authority vs. persuasive vs. contested vs. open) but never stored on the node. This prevents the user from asserting certainty the structure does not support.

A2Checkpoint interview UXlocked

Checkpoints have one of three answer types, all defined at frame-build time. Default for a new Checkpoint: multiple_choice pre-filled with two options ("yes"/"no").

  • boolean — yes/no. Each answer maps to a target branch.
  • multiple_choice — author-defined options, each mapped to a target branch or outcome.
  • graded — three-state for burden/threshold questions (met / not met / unclear). The middle state routes to contested.

Every answer requires the user to attach a Premise (created on the fly: text statement + optional source/Authority). Optional notes field on every answer.

In Legal Mode, Checkpoints can be flagged requires_authority: true, meaning the attached Premise must reference an Authority node, not just assert a fact.

Checkpoints are atomic. Multi-element tests (e.g., negligence: duty/breach/causation/damages) are modeled as multiple Checkpoints, not one Checkpoint with multiple parts.

Checkpoint {
  question: text
  answer_type: enum {boolean, multiple_choice, graded}
  options: { label, target_node_id, satisfies: bool }[]
  requires_premise: bool  // default true
  requires_authority: bool  // default false; legal mode only
  options_box: SatisfactionPolicy  // from A3
  notes: text?
}
A2 (AI strategy)Build-time AI / deterministic runtimelocked

The user originally proposed runtime LLM mapping of Checkpoint answers to branches. After discussion, this was rejected in favor of a build-time / runtime split that preserves determinism and auditability.

  • Build-time AI (suggestion only, user always confirms): Checkpoint question wording, answer option suggestions, routing suggestions, Interpretation suggestions with authorities. All suggestions are baked into the frame after user confirmation.
  • Runtime: zero LLM in the routing path. Argument Running uses only the pre-baked structure. Same frame + same premises = same path, every time.
  • Optional sidecar AI at runtime, never on routing: drafting Premise text from a fact pattern the user pastes; flagging cross-implications as advisory notes.

Each LLM hook gets a versioned .md prompt file with a defined input/output shape. This fits cleanly into Stream G and is a non-trivial change to G's scope: G now must specify build-time vs. runtime classification per hook, and all build-time hooks must produce structured output the user can edit before commit.

A fourth, non-deterministic freeform answer type was discussed and deferred to v2 unless a real use case forces it sooner. Frames using it would require an explicit "non-deterministic" warning in their audit trail.

A3Per-node satisfaction criteria — the "options box"locked

Each node type has a default satisfaction policy at the frame level (editable per-frame), and Checkpoints, Questions, and Interpretations can override the policy per-instance. Other node types use the default always.

Condition library (≈12 named conditions, each knows how to evaluate itself against the ArgumentSession):

  • Structural: premise_attached, interpretation_selected, all_children_resolved, path_complete
  • Quality: not_contradicted, premise_kind_in[list], burden_met(level)
  • Authority (legal/academic): authority_required, authority_binding, not_distinguished
  • Procedural (legal): standard_of_review_applied (LLM hook surface)
  • Universal: not_foreclosed

Default policy per node type:

NodeDefault conditions (all must hold)
Checkpointpremise_attached ∧ not_contradicted ∧ burden_met(preponderance) ∧ not_foreclosed
Interpretationselected ∧ not_foreclosed
Terminterpretation_selected ∧ not_foreclosed
SubQuestion / RootQuestionall_children_resolved ∧ not_foreclosed
Conclusionpath_complete ∧ not_foreclosed
Premise / Authority(no satisfaction policy — these are inputs/sources)

Composition: AND by default, plus one explicit AnyOf group per policy. A policy is AND(c1, c2, ..., AnyOf(c5, c6, c7)). No nested groups, no NOT (negative conditions are explicit, e.g. not_contradicted).

Status enum (refining the handoff doc's 5-state):

  • open — at least one condition unmet
  • satisfied — all conditions met; sub-flags via_binding_authority / via_persuasive_authority exposed for legal-mode display
  • contested — premises on both sides, neither dispositive
  • foreclosed — not_foreclosed failed
  • not_applicable — node lives on a path never reached

This refinement collapses the handoff doc's satisfied_binding and satisfied_persuasive into sub-flags rather than peer top-level states. The handoff doc's status section needs updating.

A4Mode differentiation: Personal / Academic / Legallocked

Two architectural modes, three presentation modes.

  • Architectural modes: legal and general. Differ in schema (Legal adds Authority binding, jurisdiction, court hierarchy hooks, standard-of-review fields).
  • Within general, a flavor: personal or academic. Flavor is metadata, not schema. It changes Authority node behavior, Premise kind vocabulary, and UI labels.

In academic flavor, Authority is fully active (citation, author, year, venue, position, URL). In personal, Authority is hidden by default but available — turned on per-frame if needed. Authority is labeled "Source" in personal flavor.

Premise kind vocabularies:

  • Legal: stipulated, found, disputed, procedural
  • Academic: empirical, definitional, normative, stipulated
  • Personal: observation, value, assumption, stipulated

"Scholarly consensus" is not a distinct concept in v1 — it's an Authority with a position field describing the consensus view.

A6Cross-frame referencesdeferred to v2

Live cross-frame references are deferred to v2. The v1 substitute is frame templates: at frame creation, the user can pick "start from existing frame" and get a deep copy. No live link, no version coupling, no circular reference detection needed. Schema's id design must be template-friendly (random IDs, no positional dependencies).

A7Resolution order semantics outside foreclosurelocked

Order is purely a foreclosure mechanism. Outside foreclosure, resolution order has no effect on what Interpretations are available for later Terms. Frame authors who want "Term X interpretation A constrains Term Y" must express that structurally — by routing X's interpretation through a SubQuestion that contains Y with the constrained Interpretation set.

Reasoning: silent coupling is invisible. The frame's logic should be readable from the graph, not from a hidden ordering variable.

The order field on Term is retained, used only by the foreclosure resolver and for default UI presentation order.

A8Jurisdiction structure (Legal Mode)locked

Jurisdictional pre-checks are ordinary SubQuestions with an is_jurisdictional: true flag. They use the same Term/Interpretation/Checkpoint machinery as everything else.

Behavior:

  • Rendered first in the UI ordering.
  • Visually distinguished (treatment decided in Stream E).
  • Frame completeness checker warns — does not block — when a Legal-mode frame has no jurisdictional SubQuestions.

This unblocks F2.

A9Conditional output presentationlocked

Three coordinated views, all rendered from the same ConditionalOutput object (B5 will spec):

  1. Default: prose summary. Deterministic textual rendering, no LLM. Exportable.
  2. Toggle: decision tree view. Inline tree showing only open gates and the Conclusions each branch leads to.
  3. Toggle: path overlay on the frame. Active path highlighted in the main frame view, with open gates marked.

Default view is a per-user setting; per-frame override available.

A10Term scope and reuselocked

Term has a nullable linked_to: term_id field. When the user creates a Term whose name matches an existing Term in the same frame, the UI offers a checkbox: "use the same interpretation as the existing Term." Checked → linked Term shares Interpretations with its target. Unchecked → independent Term with its own Interpretation branches.

Linking is one-way (a Term points at the Term it follows) and cannot create cycles. The linked Term inherits Interpretations and their selection state at runtime; it can have its own options_box override.

A11Premise typinglocked

Premise gets a kind enum, vocabulary determined by Frame mode/flavor (see A4). The kind feeds the contested status calculation and the display layer (e.g., disputed vs. stipulated premises render differently).

Premise {
  id, statement: text
  kind: enum  // vocabulary per Frame mode/flavor
  source: text?  // optional citation/source
  authority_ref: authority_id?  // when authority required
  notes: text?
}
A12Per-node noteslocked

Every node type has an optional notes: text field for the user's working thoughts. Not part of the formal logical structure; not used by path tracing or status determination. Persists with the frame.
