Stream B — Schema and Data Model

Companion to handoff_document_v2.html, master_plan_v1.html, and stream_a_resolutions_v1.html  ·  Session date: May 2, 2026  ·  Version 1

This document is the field-level data specification for the app. It covers Stream B end-to-end: node schemas (B1), edge schemas (B2), frame metadata (B3), argument session (B4), conditional output (B5), validation rules (B6), and the identifier and migration plan (B7). Everything here applies the Stream A resolutions and the contradiction fixes already adopted in the master plan. Stream C (runtime algorithms) and Stream D (mode definitions) build directly from this spec.

How to read this

Field definitions are written in a TypeScript-like notation chosen for compactness and readability. The actual on-disk format is JSON; types here describe its shape. Conventions:

  • field: type — required field.
  • field?: type — optional; may be absent or null. The two are treated identically (absent ≡ null) at the schema layer; tooling normalizes to absent.
  • type[] — an ordered array. Order is significant where called out (e.g., LogicalGate slot semantics); otherwise treated as a set.
  • "a" | "b" — string-literal union (an enum).
  • NodeRef — a node id; see "Identifiers and references" below.
  • Fields marked // legal mode only exist on the schema in all modes but are ignored by validators and renderers when the frame is not in legal mode. The same applies to // flavor: academic etc. Unknown-field tolerance: absence is fine; a field set inappropriately for the mode is a soft-warning, not a hard error, so frames don't break when their mode is changed.
  • All string fields are UTF-8 with no trimming. All timestamps are ISO 8601 with timezone (2026-05-02T14:23:00-05:00). All numeric ids/versions are monotonically increasing integers starting at 1.

Decisions made in this session

The following technical refinements were resolved while writing this spec. They preserve the conceptual decisions in the Stream A resolutions while giving them concrete shape. Each is flagged so it is easy to find and revisit:

  1. Identifier model refined. Each entity carries a stable opaque id (UUID v4 from day one) plus an optional human-readable slug. References use id; the slug is for logs, debug UIs, and exported documents. This eliminates the slug-to-UUID migration step from B7 and replaces it with a config flag (whether to auto-generate slugs). Rationale: a single migration is fragile across long-lived data, and slugs collide when frames are templated. See B7 below.
  2. FORECLOSES is modeled as a directed edge with a typed target set. One FORECLOSES edge per (Interpretation → foreclosed-target) pair, where the target is a Term, SubQuestion, or Checkpoint. Multi-target foreclosure is expressed by multiple edges. This keeps the "one source, one target" edge invariant.
  3. LogicalGate uses a discriminated union with named slots, not a uniform inputs[]. This makes IF_THEN's antecedent/consequent and UNLESS's main/exception roles structurally explicit and validatable, which the contradiction fix called for.
  4. Frame versions are stored as full snapshots, not deltas. Snapshots make replay and session-binding trivial; deltas would require a reconstruction algorithm at every read. Storage size is a Stream H concern; deduplication can be layered in later without a schema change.
  5. Argument-layer authorities are first-class. An Authority node may exist on the Frame (cited at build-time supporting an Interpretation) or on the ArgumentSession (added at runtime by a Premise's authority_ref). Both go in the same Authority shape; the locus is implicit from where the node lives.
  6. Conclusion direction in general mode references a frame-level Position table. Frames in general mode declare named positions (e.g., "Pro X", "Anti X"); Conclusions reference those by id. Legal mode uses the fixed enum from A1.
  7. NodeStatus carries a via array, not the original handoff-doc peer states satisfied_binding / satisfied_persuasive. The A3 resolution collapsed those into sub-flags; this spec implements that collapse as { status: "satisfied", via: ["binding_authority"] }.

Identifiers and references (B7)

Identifier model

Each persisted entity (Frame, FrameVersion, Node, Edge, ArgumentSession, Authority, Premise, etc.) carries:

  • id: string — a UUID v4. Always present, immutable for the lifetime of the entity. References use this.
  • slug?: string — a human-readable label, auto-generated in development environments and optional in production. Format: <type-prefix>_<humanized-noun>_<4-char-suffix>, e.g. term_willful_a3f9, checkpoint_contact-occurred_b71c. The 4-char suffix is base-36 random; the noun is a slugified excerpt of the user-visible label, truncated to 32 chars. Slugs are not unique by themselves; uniqueness is guaranteed only by the id.

This refinement removes the migration step the original handoff doc anticipated: there is nothing to migrate, because production already uses UUIDs. Turning slugs off in production is a single-line config change. The // TODO: swap slug for UUID comment in the original plan becomes // CONFIG: enable_slugs = false in prod.

NodeRef and EdgeRef

Wherever a field references another entity, it uses the bare id as a string. There are no embedded objects in references. NodeRef and EdgeRef are documentation aliases for string with the corresponding constraint that the value must resolve to an existing entity within the same FrameVersion (frame-layer references) or ArgumentSession (argument-layer references). Cross-layer references are allowed in one direction only: argument-layer entities may reference frame-layer entities; the reverse is forbidden.

Template-friendly id design

The id is opaque, randomly generated, and carries no positional information. Templating a frame (deep copy under a new Frame id) regenerates every node and edge id; the templating routine maintains an old-id → new-id map to rewrite all internal references in one pass. Slug regeneration is independent and doesn't affect references.

Common base fields

NodeBase

Every node type extends this base. The type field is the discriminator for the union.

NodeBase { id: string // UUID v4 slug?: string // human-readable, dev-friendly type: NodeType // discriminator layer: "frame" | "argument" // derivable from type; stored for fast filtering notes?: string // A12: per-node working thoughts; not used by logic presentation?: { // optional UI-layer hints, ignored by logic x?: number y?: number color_override?: string // CSS color or theme token collapsed?: boolean // for visual collapse in graph view } created_at: timestamp updated_at: timestamp created_by?: string // user id; multi-user is v2, ignored in v1 }

EdgeBase

EdgeBase { id: string // UUID v4 slug?: string type: EdgeType // discriminator layer: "frame" | "argument" source: NodeRef target: NodeRef label?: string // optional UI label, e.g. on a LEADS_TO branch notes?: string created_at: timestamp updated_at: timestamp }

Edges are always directed. A reverse-direction edge is a separate entity with its own id; the schema never implies bidirectionality.

Mode and flavor

Mode and flavor enums

Mode = "legal" | "general" Flavor = "personal" | "academic" // only meaningful when mode = "general"

Mode is architectural; flavor is presentational. Per A4, schema differences are governed by mode; field labels, vocabularies, and visibility defaults are governed by flavor. Switching mode after frame creation is allowed but emits warnings about fields that may become inert (e.g., jurisdiction_default on a frame switched from legal to general).

Jurisdiction (Legal Mode)

A jurisdiction is a structured reference into the court hierarchy database (Stream F5). It is a value object — never an entity with its own id — embedded into Frame, Authority, and Interpretation where applicable.

Jurisdiction { level: "federal" | "state" | "tribal" | "territory" | "international" region?: string // e.g., "9th Circuit", "California", "S.D.N.Y." court?: string // e.g., "Supreme Court of California" hierarchy_node_id?: string // reference into F5 court hierarchy when wired }

In v1, only level and region are required to be populated. The hierarchy_node_id is reserved for when F5 lands and the binding-determination algorithm becomes live. Until then, binding determination falls back to a soft heuristic (string match on region) and is marked as such on display.

Default policies per node type

Every Frame stores a default_satisfaction_policies object mapping NodeType → SatisfactionPolicy. New frames are seeded with the defaults below. Per A3, only Checkpoint, Question (Root and Sub), and Interpretation may carry per-instance overrides; other types use the frame default unconditionally.

default_satisfaction_policies = { Checkpoint: { all_of: ["premise_attached", "not_contradicted", { name: "burden_met", level: "preponderance" }, "not_foreclosed"] }, Interpretation: { all_of: ["interpretation_selected", "not_foreclosed"] }, Term: { all_of: ["interpretation_selected", "not_foreclosed"] }, SubQuestion: { all_of: ["all_children_resolved", "not_foreclosed"] }, RootQuestion: { all_of: ["all_children_resolved", "not_foreclosed"] }, Conclusion: { all_of: ["path_complete", "not_foreclosed"] }, LogicalGate: { all_of: ["not_foreclosed"] }, // gate_type semantics drive the rest // Premise and Authority do not have satisfaction policies; they are inputs/sources }

Override resolution

For Checkpoint, Question, and Interpretation: the per-instance options_box field, if present, replaces the frame-default policy entirely. Merging is intentionally not supported — partial overrides are confusing and the whole-replacement model keeps the policy each node uses inspectable in one place.

NodeStatus

NodeStatus { status: "open" | "satisfied" | "contested" | "foreclosed" | "not_applicable" via?: ("binding_authority" | "persuasive_authority" | "stipulation" | "structural_resolution" | "default")[] evaluated_at: timestamp failed_conditions?: string[] // list of condition names that did not hold; populated when status = "open" or "contested" }

Sub-flags in via are populated by the runtime status determiner (Stream C). Multiple may apply: a Checkpoint may be { status: "satisfied", via: ["binding_authority", "stipulation"] } if it is both stipulated and supported by binding precedent. The failed_conditions array makes the path-trace UI's "what's missing" panel trivial to render.

Node schemas (B1)

NodeType discriminator values: "RootQuestion", "SubQuestion", "Term", "Interpretation", "Checkpoint", "LogicalGate", "Conclusion", "Authority", "Premise".

B1.1RootQuestion
RootQuestion extends NodeBase { type: "RootQuestion" layer: "frame" statement: string // the question, e.g. "Does the contract bind D?" standard_of_review?: string // legal mode; e.g. "de novo", "abuse of discretion" options_box?: SatisfactionPolicy // per-instance override of frame default }

A frame has exactly one RootQuestion (validation rule V-FR-1). It has no incoming edges. Its outgoing edges are DECOMPOSES_INTO (to SubQuestions), TURNS_ON (directly to Terms when the question is atomic), or LEADS_TO (to a Conclusion when the question is trivially answered by a single fact, rare).

B1.2SubQuestion
SubQuestion extends NodeBase { type: "SubQuestion" layer: "frame" statement: string is_jurisdictional: boolean // A8; default false; legal mode only standard_of_review?: string // legal mode; inherits from parent if unset options_box?: SatisfactionPolicy }

SubQuestions decompose larger questions and may themselves decompose further. is_jurisdictional SubQuestions render first and are visually distinguished (Stream E). Frame completeness check warns (does not block) when a legal-mode frame has zero jurisdictional SubQuestions (V-FR-7).

B1.3Term
Term extends NodeBase { type: "Term" layer: "frame" name: string // the word or phrase, e.g. "willful" order: integer // 0-indexed within parent Question; controls foreclosure resolution and default UI ordering (A7) dispositive: boolean // per the handoff control concept (A); when true, this Term's resolution can foreclose siblings linked_to?: NodeRef // A10; another Term whose Interpretations this Term shares options_box?: SatisfactionPolicy }

A Term is the branching point of an interpretive choice. Per A7, order is used solely by the foreclosure resolver and as the default presentation order; it does not silently constrain the Interpretation set on later Terms. Per A10, linked_to creates a one-way reference: this Term shares the Interpretations of the target Term. Linked Terms cannot form cycles (V-NODE-3) and may carry their own options_box override.

Validation: every Term must have at least two outgoing INTERPRETED_AS edges, OR a non-null linked_to (V-FR-3). A Term with one Interpretation is a frame error: there is no choice to make.

B1.4Interpretation
Interpretation extends NodeBase { type: "Interpretation" layer: "frame" statement: string // the specific reading, e.g. "intentional with knowledge of consequences" jurisdiction_override?: Jurisdiction // legal mode; overrides frame default for this branch options_box?: SatisfactionPolicy }

An Interpretation represents one branch of how a Term is read. Outgoing edges: LEADS_TO (to a Checkpoint, SubQuestion, or Conclusion), and optionally FORECLOSES (marking sibling branches moot). Incoming edges: one INTERPRETED_AS from the parent Term, plus optional CITES from supporting Authorities and SUPPORTS/CONTRADICTS from session-time Premises.

B1.5Checkpoint (A2)
Checkpoint extends NodeBase { type: "Checkpoint" layer: "frame" question: string // e.g. "Did contact occur?" answer_type: "boolean" | "multiple_choice" | "graded" options: CheckpointOption[] // shape and length depends on answer_type requires_premise: boolean // default true requires_authority: boolean // default false; legal mode only burden_level?: "preponderance" | "clear_and_convincing" | "beyond_reasonable_doubt" | "scintilla" | "substantial_evidence" // legal mode; feeds burden_met condition options_box?: SatisfactionPolicy } CheckpointOption { id: string // stable within the Checkpoint; references it from session responses label: string // e.g. "yes", "no", "unclear", or author-defined target_node_id?: NodeRef // the node this option routes to; null only for "graded" middle state satisfies: boolean // whether selecting this option counts as satisfying the Checkpoint routes_to_status?: "contested" // optional override for "graded" middle state }

Per A2:

  • A new Checkpoint is created with answer_type: "boolean" and a default options pair: [{label: "yes", satisfies: true}, {label: "no", satisfies: false}]. The author wires target_node_id after.
  • multiple_choice Checkpoints have author-defined options with at least 2 entries; each routes to a target node. The satisfies flag determines whether the option counts as resolving the Checkpoint or as a routing-only choice (e.g., a routing branch that itself contains further unresolved structure).
  • graded Checkpoints carry exactly three options whose labels default to "met", "not_met", "unclear". The middle option carries routes_to_status: "contested" by default, which sends the node to contested status in the session (per A3 status enum). Authors may relabel the three options but the three-state structure is fixed.

Checkpoints are atomic. Multi-element tests (e.g., negligence's four elements) are modeled as multiple Checkpoints joined by an AND LogicalGate (V-FR-9 forbids "compound" Checkpoints).

B1.6LogicalGate

LogicalGate is a discriminated union over gate_type. Per the contradiction fix, inputs and outputs are formally typed; FORECLOSES is removed from the gate-type list (it is an edge only).

AND / OR gate

AndGate extends NodeBase { type: "LogicalGate" layer: "frame" gate_type: "AND" | "OR" inputs: NodeRef[] // minimum 2; each must resolve to a boolean-evaluable node output_target?: NodeRef // where the gate's resolution flows; rendered as a GATES edge }

NOT gate

NotGate extends NodeBase { type: "LogicalGate" gate_type: "NOT" input: NodeRef // exactly one; must be boolean-evaluable output_target?: NodeRef }

IF_THEN gate

IfThenGate extends NodeBase { type: "LogicalGate" gate_type: "IF_THEN" antecedent: NodeRef // if this is satisfied... consequent: NodeRef // ...this becomes active/required output_target?: NodeRef }

UNLESS gate

UnlessGate extends NodeBase { type: "LogicalGate" gate_type: "UNLESS" main: NodeRef // the rule that normally applies exception: NodeRef // suspends the rule when satisfied output_target?: NodeRef }

Boolean-evaluable nodes are: Checkpoint, Interpretation, SubQuestion, RootQuestion, and other LogicalGates. A LogicalGate that takes a Term as input is a frame error (Terms multi-resolve into Interpretations, not into a binary state); use one of the Term's Interpretations as the input instead.

Validation V-GATE-1 through V-GATE-5 enforce the arities above. The runtime status of a gate is computed in Stream C from its inputs' statuses per its gate_type semantics; it does not have its own options_box beyond not_foreclosed.

B1.7Conclusion (A1)
Conclusion extends NodeBase { type: "Conclusion" layer: "frame" statement: string // required; the answer in plain language direction: ConclusionDirection // required; mode-aware, see below reasoning_summary?: string // optional author summary; LLM_HOOK candidate (G3) tags?: string[] remedy?: string // legal mode v1; promotable to a Remedy node in v2 options_box?: SatisfactionPolicy } ConclusionDirection = | { kind: "legal", value: "favors_plaintiff" | "favors_defendant" | "affirm" | "reverse" | "remand" | "dismiss" | "custom" custom_label?: string } // required when value = "custom" | { kind: "general", position_id: string, custom_label?: string } // reference into Frame.positions

Confidence is not a stored field. It is computed at session render time from the status map (proportions of Checkpoints satisfied via binding authority vs. persuasive vs. stipulation vs. open vs. contested) and displayed alongside the Conclusion. Storing it on the node would let the user assert certainty the structure does not support; this was the explicit reasoning in A1.

The remedy field is a legal-mode v1 affordance. The schema is designed so the field can be deprecated and a separate Remedy node type promoted into its place without affecting existing Conclusions: a future migration would copy the string into a new node and link it via a HAS_REMEDY edge, leaving the field as a deprecated read-through.

B1.8Authority
Authority extends NodeBase { type: "Authority" layer: "frame" // may also be added at session-time; same shape citation: string // e.g. "Erie R.R. v. Tompkins, 304 U.S. 64 (1938)" // Legal mode fields: court?: string year?: integer holding_summary?: string is_holding: boolean // vs dicta; default true is_binding?: boolean // computed by F6 when live; manually settable v1 jurisdiction?: Jurisdiction // where this authority issues from // Academic flavor fields: author?: string venue?: string // journal, conference, publisher position?: string // what the author argues; or "represents scholarly consensus" // Universal: url?: string short_label?: string // for compact display, e.g. "Erie" }

Per the resolution-session decision (#5 above), the same Authority shape is used whether the node is created at frame-build time (citing an Interpretation) or at session-runtime (referenced by a Premise's authority_ref). The locus is implicit from where the node is stored. In personal flavor, the type is labeled "Source" in the UI; the schema name remains Authority for code consistency.

"Scholarly consensus" is not a separate concept (per A4) — it is an Authority node with position describing the consensus view, typically with a citation pointing to the survey paper or review article that establishes the consensus.

B1.9Premise (A11)
Premise extends NodeBase { type: "Premise" layer: "argument" // argument-layer only statement: string kind: PremiseKind // vocabulary per Frame.mode/flavor; see A11/A4 source?: string // free-text citation/source label authority_ref?: NodeRef // id of an Authority node (frame-layer or session-layer) } PremiseKind = (mode/flavor-dependent enum)

A Premise is the runtime answer to a Checkpoint (via ANSWERS), or evidence for/against an Interpretation (via SUPPORTS/CONTRADICTS). Per A2, every Checkpoint answer in the session creates a Premise; the Premise carries the user-supplied factual statement plus optional source. authority_ref is mandatory when the answered Checkpoint has requires_authority: true (V-ARG-2).

Edge schemas (B2)

EdgeType discriminator values: "DECOMPOSES_INTO", "TURNS_ON", "INTERPRETED_AS", "LEADS_TO", "FORECLOSES", "GATES", "ANSWERS", "SUPPORTS", "CONTRADICTS", "CITES", "BINDING_IN", "DISTINGUISHED_BY".

Per the contradiction fixes already adopted: CITED_BY is renamed CITES with the same direction (Authority → Interpretation); ALTERNATIVE_PATH_TO is removed (two convergent paths are expressed by two LEADS_TO edges with the same target).

Notes on individual edge types

  • BINDING_IN targets a Jurisdiction value, not a node. Since the EdgeBase requires a NodeRef target, this edge is implemented as an embedded relation: the Authority node carries an optional binding_in: Jurisdiction[] field directly. The BINDING_IN edge type is retained in documentation as a conceptual relation but does not exist as a row in the edges table. This is the cleanest way to keep the "edge has one node target" invariant and avoid a phantom Jurisdiction-as-node type.
  • FORECLOSES's scope field distinguishes "moot" (the foreclosed branch is no longer relevant; status flips to foreclosed) from "decided" (the foreclosed branch is taken to be answered the same way the foreclosing Interpretation answers; status flips to satisfied with via: ["structural_resolution"]). Default is "moot".
  • LEADS_TO's optional condition field lets authors annotate why the path leads where it leads; rendered on the edge in the graph view but not used by routing logic.
  • SUPPORTS and CONTRADICTS targeting a Conclusion are unusual but legitimate (e.g., a Premise establishing a fact that directly bears on the holding); v1 supports both. The status determiner treats Conclusion-targeted CONTRADICTS as flipping the Conclusion to contested.

Frame schema (B3)

Frame { id: string slug?: string title: string description?: string mode: Mode // "legal" | "general" flavor?: Flavor // required when mode = "general"; forbidden otherwise jurisdiction_default?: Jurisdiction // legal only positions?: Position[] // general only; required when frame has Conclusions default_satisfaction_policies: { [nodeType: string]: SatisfactionPolicy } tags: string[] pinned: boolean // default false; max 8 pinned per user enforced by app layer created_at: timestamp updated_at: timestamp last_opened_at?: timestamp current_version_id: string // → FrameVersion.id llm_settings?: LlmSettings // reserved for Stream G; opaque object in v1 } Position { id: string // stable within the Frame; e.g. "pos_pro_x_8b21" label: string // e.g. "Pro X" description?: string } LlmSettings { build_time_hooks_enabled: boolean // default true runtime_hooks_enabled: boolean // default false in v1; runtime is deterministic prompt_versions?: { [hook_name: string]: string } }

The positions table is the source of truth for general-mode Conclusion directions. Changing a Position's label updates all Conclusions that reference it (because they reference by id). Removing a referenced Position is blocked unless the user reassigns the dependent Conclusions first (V-FR-10).

FrameVersion schema

FrameVersion { id: string frame_id: string // → Frame.id version_number: integer // monotonic, 1-indexed parent_version_id?: string // previous version, for the version graph created_at: timestamp created_by?: string change_summary?: string // optional human-written summary; LLM_HOOK candidate nodes: Node[] // full snapshot edges: Edge[] // full snapshot is_milestone: boolean // for the version-history UI; true at named saves }

Decision #4 above: snapshots, not deltas. Each FrameVersion is a complete, self-contained graph. The current_version_id on Frame points to the latest version; older versions are kept indefinitely (per the handoff doc's "every version stored"). Storage compaction is a Stream H concern and is invisible at the schema layer.

Auto-save creates a new FrameVersion at most once per significant edit (debounced); the rate is a Stream H tuning parameter. Manual saves and mode transitions force a new version with is_milestone: true.

ArgumentSession schema (B4)

ArgumentSession { id: string slug?: string frame_id: string // → Frame.id frame_version_id: string // → FrameVersion.id; the version this session was created against frame_version_snapshot: FrameVersion // embedded copy; insulates session from later frame edits (per master plan fix #4) title: string description?: string premises: Premise[] // argument-layer nodes created in this session argument_edges: Edge[] // ANSWERS, SUPPORTS, CONTRADICTS edges; argument layer only session_authorities?: Authority[] // Authorities introduced at runtime by Premise.authority_ref; not on frame checkpoint_responses: CheckpointResponse[] // dense record of which option was chosen at each Checkpoint status_map: { [node_id: string]: NodeStatus } // computed; updated on each premise change output?: ConditionalOutput // computed; latest result of running the session active_path?: NodeRef[] // computed; primary determinate path through the frame frame_version_drift_warning?: string // populated when underlying Frame has advanced past frame_version_id created_at: timestamp updated_at: timestamp current_version_id: string // → ArgumentSessionVersion.id } CheckpointResponse { checkpoint_id: NodeRef selected_option_id: string // references CheckpointOption.id within the Checkpoint premise_id: NodeRef // the Premise that supplies this answer answered_at: timestamp notes?: string // optional per-answer note (per A2) } ArgumentSessionVersion { id: string session_id: string version_number: integer parent_version_id?: string created_at: timestamp created_by?: string change_summary?: string premises: Premise[] argument_edges: Edge[] session_authorities?: Authority[] checkpoint_responses: CheckpointResponse[] is_milestone: boolean }

The session embeds a full FrameVersion snapshot at creation. This is the master plan's fix #4 made concrete: the session never silently breaks if the underlying Frame is edited. If the user wants the session to track newer frame versions, they explicitly migrate it (a Stream D operation), which copies premises and responses forward where they still attach to existing nodes and lists the orphans.

Per the handoff doc, Frame and ArgumentSession are versioned independently. ArgumentSessionVersion mirrors FrameVersion's shape but for the argument layer. Sessions and Frames have no shared version-numbering.

frame_version_drift_warning populates lazily: when the user opens a session whose frame_version_id is older than the parent Frame's current_version_id, the field is set to a human-readable summary of what changed since. It does not block running the session; it is informational only.

ConditionalOutput schema (B5)

ConditionalOutput { shape: "determinate" | "conditional" | "contested" | "incomplete" // Populated when shape = "determinate": conclusion?: NodeRef // reference into the FrameVersion snapshot primary_path?: NodeRef[] // ordered nodes from RootQuestion to Conclusion // Populated when shape = "conditional": branches?: ConditionalBranch[] // Populated when shape = "contested": contested_nodes?: NodeRef[] best_inference?: NodeRef // optional Conclusion the system best guesses, with a "tentative" flag in display // Populated when shape = "incomplete": open_gates?: OpenGate[] // Always populated: prose_summary: string // deterministic textual rendering; the default view (A9) computed_at: timestamp confidence_breakdown: ConfidenceBreakdown // the proportions A1 calls for at display time } ConditionalBranch { id: string conditions: BranchCondition[] // the if-clauses that select this branch resulting_conclusion: NodeRef // the Conclusion this branch produces intermediate_path: NodeRef[] // the path from the divergence point to the Conclusion prose: string // deterministic prose for this branch } BranchCondition { gate_node_id: NodeRef // the Checkpoint or Term whose answer determines the branch required_value: string // for Checkpoint: a CheckpointOption.id; for Term: an Interpretation.id required_value_label: string // pre-rendered for display } OpenGate { node_id: NodeRef reason: "no_premise" | "premise_contradicted" | "burden_unmet" | "authority_missing" | "structural" prompt: string // the exact question to surface to the user } ConfidenceBreakdown { total_checkpoints_on_path: integer satisfied_via_binding: integer satisfied_via_persuasive: integer satisfied_via_stipulation: integer satisfied_via_structural: integer contested: integer open: integer }

Per A9, a single ConditionalOutput object drives all three views: the prose summary view (default), the decision tree view, and the path-overlay view. Renderers consume the same object; differences are presentational. The prose_summary field is generated deterministically (no LLM in the routing path, per A2's AI strategy) by walking the structure and templating standard sentences. An LLM_HOOK exists for prose-rewriting but is build-time / opt-in only.

The four shape values map to user-visible states:

  • determinate: every Checkpoint on the path is satisfied; there is one Conclusion. The view shows it directly.
  • conditional: at least one Checkpoint is unanswered, but the structure short-circuits to a small number of named branches. The view shows "if X then A; if Y then B."
  • contested: Premises exist on both sides of one or more Checkpoints; no determinate path exists. The view shows the contested points and any best-inference Conclusion as tentative.
  • incomplete: the frame itself is incomplete (paths don't terminate, gates have no inputs); the user is shown the open structural gaps and prevented from publishing the session.

Validation rules (B6)

Validation rules are the frame-completeness check that runs before mode transitions and on demand. Each rule has a stable id (e.g., V-FR-1) so failures can be referenced from UI and logs. Rules are tagged as error (blocks mode transition) or warning (advises but does not block).

Frame-level rules

IdSeverityRule
V-FR-1errorThe frame has exactly one RootQuestion.
V-FR-2errorEvery non-Root node has at least one incoming edge (no orphans).
V-FR-3errorEvery Term has at least two outgoing INTERPRETED_AS edges, OR a non-null linked_to.
V-FR-4errorEvery Checkpoint is reachable from RootQuestion through frame-layer edges.
V-FR-5errorEvery Conclusion is reachable from RootQuestion.
V-FR-6errorThe frame graph (excluding FORECLOSES, BINDING_IN, CITES, DISTINGUISHED_BY) is a DAG. No structural cycles in the resolution flow.
V-FR-7warningLegal-mode frame has at least one SubQuestion with is_jurisdictional: true. Per A8, this is a warning, not a blocker.
V-FR-8errorEvery path from RootQuestion eventually terminates at a Conclusion (no dangling Term/Checkpoint that doesn't continue).
V-FR-9errorNo "compound" Checkpoints: a Checkpoint's question field, by author convention, names a single test. Detection is best-effort and primarily a warning unless heuristic confidence is high.
V-FR-10error(general mode) Every Conclusion's direction.position_id resolves to an entry in Frame.positions. A Position cannot be deleted while referenced.
V-FR-11error(legal mode) Every Authority with is_binding: true has a non-null jurisdiction.
V-FR-12warningEvery Interpretation with no incoming CITES has at least one explanatory notes field, encouraging the author to defend the reading. Warning only.

V-NODE-2errorEvery node's type matches the structural shape of its fields (no Conclusion missing direction, no Term missing order, etc.).
V-NODE-3errorTerm linked_to chains terminate (no cycles).
V-NODE-4errorTerm linked_to targets a Term within the same FrameVersion.

Mode-aware schema behavior

This is a consolidated reference for how mode and flavor affect the schema, drawn from the per-section notes above.

Legal mode adds

  • Frame.jurisdiction_default
  • SubQuestion.is_jurisdictional (active)
  • RootQuestion.standard_of_review, SubQuestion.standard_of_review
  • Interpretation.jurisdiction_override
  • Checkpoint.requires_authority (active), Checkpoint.burden_level
  • Authority.court, .year, .holding_summary, .is_holding, .is_binding, .jurisdiction, .binding_in
  • Conclusion.direction uses the legal enum; Conclusion.remedy active
  • Premise kind vocabulary: legal set
  • BINDING_IN, DISTINGUISHED_BY edges active

General mode adds

  • Frame.flavor required
  • Frame.positions populated; Conclusion.direction uses the general variant referencing positions

Academic flavor

  • Authority fully visible; author, venue, position active
  • Premise kind vocabulary: academic set

Personal flavor

  • Authority hidden by default; usable per-frame opt-in (UI-level)
  • Authority labeled "Source" in UI; schema name unchanged
  • Premise kind vocabulary: personal set

JSON envelope and storage

An on-disk frame export consists of a Frame envelope embedding the current FrameVersion. Sessions export similarly with their snapshot and current ArgumentSessionVersion.

FrameExport { schema_version: integer // the schema spec version this file was written against app_version: string exported_at: timestamp frame: Frame current_version: FrameVersion history?: FrameVersion[] // optional; omitted by default for size } ArgumentSessionExport { schema_version: integer app_version: string exported_at: timestamp session: ArgumentSession current_version: ArgumentSessionVersion history?: ArgumentSessionVersion[] embedded_frame_export?: FrameExport // when sharing a session standalone }

The schema_version integer increments whenever a non-backward-compatible change to this spec ships. Reader code uses it to dispatch into the correct migration before validating the rest of the document.

Reserved / deferred design space

The following are intentionally not specified in v1 and are flagged here so future agents do not assume they must be designed now:

  • Cross-frame references (A6). Deferred to v2. Frame templates are the v1 substitute (deep copy at creation; no live link). The id design is template-friendly (random UUIDs) so v2 can layer cross-frame edges in without restructuring.
  • Confidence as a stored value. Always computed, never stored (per A1). If a future use case needs cached confidence, add a derived field to ArgumentSession.output; do not put it on Conclusion.
  • Multi-user collaboration / concurrent editing. The schema includes a created_by field on Node and Edge bases as a hook, but no concurrency model is specified. v1 is single-user.
  • Compound LogicalGates / nested AnyOf. A3 explicitly disallowed nesting in SatisfactionPolicy. If a future use case truly requires nested boolean combinators, model it with multiple LogicalGate nodes wired together, not by complicating the policy schema.
  • Promotion of remedy to a separate Remedy node type. Reserved for v2. The Conclusion field is sufficient for v1; the schema migration path is documented at B1.7.
  • Freeform Checkpoint answers. A2 deferred this. If introduced, a fourth answer_type: "freeform" would be added with a non-deterministic-warning flag on its session output.
  • Storage backend. Stream H concern. Schema is storage-agnostic.
