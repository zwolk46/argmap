stream_i_state_spec_v1.html

Per-module spec for src/state/  ·  Stream I session I.5  ·  May 10, 2026

This document is the implementation source of truth for src/state/. It commits per-file TypeScript signatures, the Zustand store shapes for Frame, Session, and AppState, the RepositoryProvider React Context that mounts Repository + AutosaveController + CrossTabBus at the application root, the subscription wiring between compute() results and the session store, the selector projections the UI consumes, and the action-runner pattern by which state composes @/modes actions with compute() and AutosaveController.schedule* into a single coherent dispatch surface. The cross-module surface (what other modules import from @/state) is pinned in stream_i_contracts_v1.html Contract 7; this spec narrows the placeholder types FramePatch and FrameStoreSnapshot that Contract 7 marked "I.5 narrows," adds the per-file organization, and pins the behavioral contracts the coding session implements against.

The state layer sits at the integration center of the application. It imports from @/schema, @/runtime, @/persistence, @/modes, and @/llm-hooks; only @/ui may import from it. State is the layer where the runtime's determinism boundary becomes operational for the rest of the app: state captures a single wall-clock read per user action and threads it into both the new ArgumentSessionVersion.created_at field and compute()'s required computed_at input, so replaying the same session edit always produces a byte-identical recomputation (Article II § 2; stream_i_runtime_spec_v1.html "Caller responsibility: supply computed_at from a deterministic source"). State itself is not inside the determinism boundary — it may read the clock, generate UUIDs, and hold mutable working state — but it is the layer that defends the boundary from the rest of the application by capturing each clock read at one well-defined point per action and never elsewhere.

Conventions throughout: every factory function is pure with respect to its options argument — given the same Repository, AutosaveController, CrossTabBus, and time/id injections, two factory invocations produce stores with byte-identical initial state and structurally identical subscription wiring. Store snapshots are immutable: each Zustand set() call replaces the snapshot rather than mutating it; the UI's React-reconciler-driven memoization works on reference equality. Selectors are pure functions of (snapshot, args) with no closure over external state; this is what makes them safe to call inside React render and reusable in tests. Action methods are async only because they await autosave.flush* at milestone time or Repository-composite operations (restore, migrate); the in-memory state transition (modes-action + compute) is synchronous, and the asynchrony exists only at the persistence boundary.

Cross-document resolution notes (for self-review)

While compiling this spec from stream_i_contracts_v1.html Contract 7, the runtime spec, the persistence spec, and the consolidated state in current_state.html, three precision items arose. In each case this spec adopts the constitutionally clean resolution; the contracts doc is the doc that needs the additive correction, not the per-module specs. The items are logged as a single in-session-escalated flag for a future I.1 review session to merge into a contracts-doc bump — same family as F-003 (I.2), the I.3 pending flag (flag_pending_from_i3.html), and the I.4 pending flag (flag_pending_from_i4.html). Downstream coding is not gated on the merge; this spec is canonical for the state module per Stream I conventions.

  1. Modes action signatures refined: synchronous pure transformations (current_entity, args, deps) => new_entity. Contract 4's FrameBuildingActions.createNode(args): Promise<NodeRef> (and parallel ArgumentRunningActions) implies modes mutates through Repository internally and returns identifiers. Two facts make that signature unworkable: (a) the persistence spec's resolution #2 committed that the state layer is the source of truth for PendingFrameSave / PendingSessionSave payloads (the autosave controller does not reach into @/state for the latest entity), so the state layer must know the new entity in full after every action, not just an id; and (b) the persistence spec's resolution #1 committed that persistence does not import runtime (recompute happens "at the state-layer subscription site after the restore returns"), so the state layer must drive compute() from the new entity, not learn the new entity by re-reading. The constitutionally clean resolution is for modes-actions to be synchronous pure transformations of (current_entity, args, deps) => new_entity; the state layer composes these with compute and autosave. Composite operations that genuinely span entities and need Repository (createFrameFromTemplate, migrateSession, restoreFrameVersion, restoreSessionVersion) remain on Repository already (per Contract 3) and are wrapped by modes orchestration functions that the state layer awaits directly; those wrappers are async because they go through Repository. The dependency graph in Contract 4 is unchanged (modes still imports Repository for the orchestration wrappers); only the per-action signatures change. This spec pins the dispatch table I.6 fulfills under FrameActionDispatchTable and SessionActionDispatchTable in action-runner.ts below.
  2. RepositoryProvider data-type renamed to RepositoryProviderValue; RepositoryProvider reserved for the React component. Contract 7 declares interface RepositoryProvider { repository, autosave, llm_settings_default } with the comment "React Context bundle the UI mounts at the root." The name RepositoryProvider is conventionally the React component (the JSX element the UI mounts), not the value it supplies. Per Contract 7's own "I.5 narrows" comment, this spec renames the data interface to RepositoryProviderValue and reserves RepositoryProvider for the component. The data fields the contracts doc names are preserved verbatim and extended additively (crosstab, the three stores, the time/id injections); no field is removed.
  3. Test-only devDependency added: happy-dom. The state layer's tests are mostly framework-free (factories, action runner, compute driver, selectors), but context.tsx exposes a React Context provider whose mounting and disposal behavior is meaningful only inside a DOM-like environment. happy-dom is added as a devDependency and vitest.config.ts uses it for tests under tests/state/context.test.tsx via a per-file environment override (// @vitest-environment happy-dom). No @testing-library/react is added in v1 — the context test exercises the provider with hand-rolled React (act() + ref-callback inspection) so the dep set stays minimal. Vite's tree-shaking excludes happy-dom from production; the buildkit index's "no additional dependency" rule, like the I.4 fake-indexeddb addition, applies to runtime/production dependencies. Logged for visibility.
// src/state/context.tsx

The React Context and provider component the UI mounts at the application root. The provider creates the three stores via their factory functions, injects the Repository, AutosaveController, CrossTabBus, time, and id dependencies, loads the singleton AppState from Repository at mount, wires the autosave-event and cross-tab-event subscriptions to the appropriate stores, and disposes everything on unmount. Components reach the value through the named hooks; direct useContext is not part of the public surface (the hooks throw a typed error if called outside the provider).

const RepositoryContext = React.createContext<RepositoryProviderValue | null>(null);

/**
 * The provider component. Behavior:
 *   1. useMemo over the long-lived deps: create the three stores via factory.
 *   2. useEffect on mount: load AppState from Repository; seed AppStateStore.
 *   3. useEffect on mount: subscribe to autosave save_succeeded / save_failed;
 *      route SaveEvents into frame_store.onSaveEvent / session_store.onSaveEvent /
 *      app_state_store.onSaveEvent by inspecting SaveEvent.kind.

 *      subscriptions. The Repository, AutosaveController, and CrossTabBus are
 *      owned by the caller and not disposed here.
 */
export function RepositoryProvider(props: RepositoryProviderProps): ReactElement;

/** Throws a typed `RepositoryProviderMissingError` outside a RepositoryProvider. */

Implementation notes for the coding session. The three stores are created inside a useMemo keyed on the deps (repository, autosave, crosstab, now, generateId, modes_actions). The AppState load uses a one-shot useEffect with a cleanup that aborts the seed if the provider unmounts before the load resolves. Autosave-event routing is one function that inspects SaveEvent.kind. Cross-tab routing mirrors it. The hooks use Zustand's useSyncExternalStoreWithSelector primitive internally so React 18's concurrent rendering does not tear stores.

// src/state/frame-store.ts

The Zustand store for the current Frame and FrameVersion. Holds the working frame, the working version, the recents list as FrameSummary[] for the index pages, the most recent compute-pipeline validation results, and a handful of UI-significant flags. Exposes imperative action methods that dispatch through the action runner.

/**
 * Discriminated union of frame edit operations. Each kind maps 1:1 to an entry
 * in FrameActionDispatchTable that @/modes (I.6) fulfills. The kinds cover

 * sub-action arguments flow through the relevant kind with a richer args shape.

  | { kind: "node_edited"; node_id: NodeRef; partial: Partial<Node> }

  | { kind: "edge_added"; edge: Edge }
  | { kind: "edge_edited"; edge_id: EdgeRef; partial: Partial<Edge> }
  | { kind: "edge_deleted"; edge_id: EdgeRef }
  | { kind: "metadata_edited"; partial: Partial<Pick<Frame,

export interface FrameStoreSnapshot {
  frame: Frame | null;
  current_version: FrameVersion | null;
  recents: FrameSummary[];
  /** validation_results from the latest compute() against (frame, empty session).
      Empty array when no frame is open or compute has not yet run after open. */
  validation: ReadonlyArray<ValidationResult>;
  loading: boolean;
  /** True when at least one FramePatch has been applied since the most recent
      save_succeeded for this frame. Drives onPeerSave's reload-vs-conflict choice. */
  has_pending_save: boolean;
  /** Populated by onPeerSave when a peer save arrives while has_pending_save is
      true. The UI surfaces "remote changes detected; next save will fork" until
      the user flushes or the next autosave fires (whichever first). */

  /** Populated when the most recent autosave attempt emitted save_failed.
      Cleared on the next save_succeeded. In-memory state is preserved. */
  save_error: { error_kind: "quota_exceeded" | "repository_error"; message: string } | null;
}

Behavioral contracts. open(frame_id): sets loading: true and emits; calls repository.loadFrame, then repository.loadFrameVersion(frame.current_version_id); runs validation through validateOnly(frame_version, compute_driver); replaces the snapshot with the loaded entities, loading: false, the new validation, has_pending_save: false, peer_save_conflict: null, save_error: null. Independently schedules refreshRecents(). applyEdit(patch): calls runFrameAction with the current snapshot's frame + current_version + patch + the action-runner deps; receives { next_frame, next_version, validation, pending_save }; replaces the snapshot accordingly; calls autosave.scheduleFrameSave(pending_save); sets has_pending_save: true. saveMilestone(change_summary): builds the latest PendingFrameSave from the in-memory state with new_version.is_milestone = true and the supplied change_summary; calls autosave.saveFrameMilestone(payload) and awaits; on resolve clears has_pending_save. restoreVersion(ancestor_id, change_summary): calls the modes orchestration wrapper for restore (which goes through Repository.restoreFrameVersion per persistence atomicity contract #3); on resolve sets the working version to the new fork, refreshes recents, re-runs validation. onSaveEvent: on save_succeeded with matching frame id and matching version_id, clear has_pending_save; if the version_id mismatches the in-memory current_version.id, leave it true (newer in-memory version is still pending). onPeerSave: if frame_id mismatches, no-op; if matches and !has_pending_save, silently reload (internal re-open without loading flicker); if matches and has_pending_save, set peer_save_conflict and emit — the user's next save creates a fork.

Behavioral contracts. open(session_id): loading: true; loads ArgumentSession; loads the corresponding ArgumentSessionVersion via session.current_version_id; computes a fresh ComputeResult through compute_driver.runFor(session); replaces the snapshot. applyEdit(patch): dispatches into runSessionAction(snapshot.session, snapshot.current_version, patch, deps); receives { next_session, next_version, compute_result, pending_save }; replaces the snapshot accordingly; calls autosave.scheduleSessionSave(pending_save); sets has_pending_save: true. The persisted ArgumentSession carries status_map, output, and active_path from compute_result; the persisted ArgumentSessionVersion does NOT carry those fields (the schema spec keeps them on the live session only, not on per-version snapshots — version snapshots carry raw user-data fields). saveMilestone / restoreVersion: parallel to the FrameStore. migrateToFrameVersion: calls the modes orchestration wrapper for session migration (Repository.migrateSession + atomicity contract #2); on resolve replaces session/version, recomputes against the new frame_version_snapshot, clears frame_version_drift. onSaveEvent / onPeerSave / onParentFrameSaved: parallel patterns to the FrameStore, with onParentFrameSaved setting/clearing the drift projection by comparing the supplied frame current_version_id against session.frame_version_id.

The load-bearing determinism point. runSessionAction reads now() exactly once and uses the returned ISO-8601 string as both ArgumentSessionVersion.created_at+ArgumentSession.updated_at AND compute()'s computed_at. This is what makes the runtime determinism guarantee operational across the application: the new version's stored timestamp and the runtime's reading of "now" match exactly. Replaying the same session edit with a different clock produces a new version with a different created_at, but the structural part of the ComputeResult is byte-identical because the runtime's only clock input — computed_at — flows through NodeStatus.evaluated_at and ConditionalOutput.computed_at only, both stamped values rather than gating values. Tests assert this.

// src/state/app-state-store.ts

The Zustand store for the singleton AppState entity. AppState is per-user, not versioned (Stream H H2), saved with a 1-second debounce. The store loads from Repository at mount via the provider, exposes per-field convenience methods, and routes every mutation through AutosaveController.scheduleAppStateSave.

// src/state/compute-driver.ts

A thin wrapper around compute() that pins the rule for computed_at sourcing. Every compute() invocation in the state layer goes through this driver, so the policy lives in one place: read now() exactly once per action and pass the value as computed_at; prefer the caller-supplied at when the caller has already pinned the clock for a multi-step action. buildComputeInput is exposed so callers (action-runner's validate-only path; tests) can construct the input shape without invoking compute.

   * once and threads the value through.

  /**
   * Build a ComputeInput from a session and a fixed computed_at. Exposed for
   * callers that need to inspect or transform the input before invoking
   * compute() (e.g., the validate-only path in the FrameStore.open flow,
   * which passes an empty argument-layer slice).

  now: () => string;

  session: ArgumentSession,

The driver operationalizes the runtime spec's preference order for computed_at sources: (1) ArgumentSessionVersion.created_at when computing as part of saving a new session version (the natural anchor used by runSessionAction); (2) ArgumentSession.updated_at when computing on session edit; (3) a caller-pinned constant in tests. A multi-step action that creates a new ArgumentSessionVersion first reads now() once into a local at, uses at as the new version's created_at, and passes at to compute_driver.runFor(session, at). The runtime gets exactly the timestamp that's already going to be stamped onto the persisted version. The no-argument overload is the fallback for callers that are not creating a new version (currently only the open/load path, where the result is immediately consumed into the snapshot and not persisted).

Subscription wiring — how the pieces connect at runtime

The provider mounts the three stores and wires three classes of subscription. This section pins the wiring contract; context.tsx implements it.

Compute → store

The state layer never subscribes to a "compute event stream" — compute() is a synchronous pure function, not an event source. The subscription, in the sense the user's prompt named, is the wiring by which a store-side change to session inputs reliably triggers a fresh compute() pass whose result lands in the store snapshot:

  1. UI calls session_store.applyEdit(patch).
  2. The store enters runSessionAction, which (a) reads now() once into at; (b) calls the modes-action to produce next_session_raw and next_version_raw with at stamped on both; (c) calls compute_driver.runFor(next_session_raw, at); (d) writes the computed projections back onto next_session_raw; (e) returns the four-tuple.
  3. The store sets the snapshot to { session: next_session, current_version: next_version, compute_result, ... } and emits to subscribers.
  4. Components reading compute_result, statusOf, or any selector that touches the compute result re-render with the new values.

The subscription contract is therefore: every snapshot transition that changes any compute input also produces a fresh compute_result in the same snapshot transition. The store never publishes a snapshot whose session and compute_result disagree (after the first computed snapshot). This is the durable equivalent of "subscribing to compute": the store transition itself is the subscription unit.

The frame parallel: every snapshot transition that changes the frame version also produces a fresh validation projection in the same snapshot. The FrameStore never publishes a snapshot whose current_version and validation disagree.

Autosave → store (save event routing)

The autosave controller emits save_succeeded and save_failed with SaveEvent payloads. The provider subscribes once via autosave.on("save_succeeded", listener) and once via autosave.on("save_failed", listener); the listener inspects SaveEvent.kind and routes to frame_store.onSaveEvent(event), session_store.onSaveEvent(event), or app_state_store.onSaveEvent({ kind: "app_state", error }). The store-side handlers:

  • save_succeeded for a frame/session: if the in-memory current_version.id matches event.version_id, clear has_pending_save. If they differ (a later applyEdit produced a newer in-memory version while the prior save was in flight), leave has_pending_save: true — the next autosave flush picks up the latest payload (the "leave the latest payload in slot" path from the persistence spec's flushFrame).
  • save_failed for a frame/session: set save_error in the snapshot with the error's name classified as "quota_exceeded" or "repository_error"; in-memory state is preserved. The UI surfaces a non-blocking banner via a selector; the user's next edit re-arms the timers, and the next successful save clears the error.
  • save_succeeded / save_failed for app_state: as documented in the AppStateStore section.

CrossTabBus → store (peer-save routing)

The provider subscribes to crosstab.subscribe("frame_saved", listener) and crosstab.subscribe("session_saved", listener); the listener routes to frame_store.onPeerSave(event) and session_store.onPeerSave(event). The handlers implement Stream H's decision tree:

  • If the event is for a frame/session the store is not currently holding (id mismatch): no-op.
  • If the event is for the held entity and !has_pending_save: reload the entity from Repository silently and re-run validation (frame) or compute (session). The user's view updates with the peer's changes.
  • If the event is for the held entity and has_pending_save: set peer_save_conflict but do NOT reload. The user's next save creates a fork — Stream H's "forked-branch as the worst case, which the version history already handles."

The FrameStore's onPeerSave handler additionally calls session_store.onParentFrameSaved(new_current_version_id) if a session is open against the same frame; this refreshes the SessionStore's frame_version_drift projection so the Argument Running top-bar updates without a session reload.

Wiring lifecycle

All subscriptions are attached in RepositoryProvider's mount effect and detached in the cleanup. The three stores own no subscriptions outside their listener Sets (which the dispose methods clear). Unmounting the provider calls dispose on every store and detaches every subscription; the Repository, AutosaveController, and CrossTabBus are owned by the caller (the application boot harness or the test) and are disposed there.

Determinism contract — clock and id boundaries at the state layer

The runtime is the determinism boundary (Article II § 2). The state layer is outside it, but is the layer that defends it from the rest of the application by capturing each clock and id read at well-defined points and threading the values consistently. The contract:

  1. One clock read per session-action. runSessionAction reads deps.now() exactly once per call and uses the returned ISO-8601 string as both ArgumentSessionVersion.created_at + ArgumentSession.updated_at AND compute()'s computed_at. runFrameAction reads deps.now() at most once per call (validate-only path inside validateOnly; the result is not persisted so this is informational, not constraining). No other code path inside src/state/ reads the clock.
  2. Id generation through the injection point. deps.generateId() is the only way src/state/ code creates new entity ids. Tests inject a deterministic counter; production injects () => crypto.randomUUID().
  3. No clock or id read in selectors. Selectors are pure functions of (snapshot, args). The acceptance test (selectors.test.ts) includes a smoke check asserting selectors produce structurally equal output across two invocations on the same snapshot.
  4. No clock or id read in store getters. The Zustand get path is pure-of-the-snapshot. The set path is where clock/id reads land; getters never read them.
  5. The validate-only path is allowed to drift on the clock. Its result is not persisted (the schema spec does not store validation on the FrameVersion), so the clock value affects no byte-identity of any persisted entity. The frame-store test asserts: two applyEdit calls with the same patch and the same id-injection produce byte-identical persisted FrameVersions even when the injected clock differs.

These five points operationalize the runtime spec's "Caller responsibility: supply computed_at from a deterministic source." The state layer is the single caller of compute() in the application; this discipline is what makes the runtime's determinism guarantee hold for the application as a whole.

How the state layer composes Repository + AutosaveController + CrossTabBus into the actions surface

This section names the composition so I.6 and I.9 can refer to it without reconstructing the picture.

The store, as the read surface

UI components and modes orchestration wrappers read snapshots via the named hooks (useFrameStore, useSessionStore, useAppStateStore) or via getSnapshot(). Selectors shape the read into a UI-specific projection. The store never returns Repository handles, runtime functions, or autosave handles into the UI; every read is a pure snapshot projection.

The store, as the write surface

UI components and orchestration wrappers write through the store's imperative action methods. Each action follows the same pipeline shape:

  1. Read the current snapshot.
  2. Dispatch into the action-runner, which: reads now() once, calls the modes-action to transform the entity, runs compute (sessions only) or validate-only (frames), assembles the new entity + version + (for sessions) compute_result, assembles the PendingFrameSave / PendingSessionSave payload.
  3. Replace the snapshot with the new entity + version + projections.
  4. Call autosave.scheduleFrameSave(payload) / autosave.scheduleSessionSave(payload) (or the milestone variant for saveMilestone). For Repository-composite operations (restoreVersion, migrateToFrameVersion), call the modes orchestration wrapper that goes through Repository directly inside a single transaction; on resolve, refresh the in-memory state from the result.
  5. Set has_pending_save appropriately.

The provider, as the wiring chassis

RepositoryProvider is the single place the four background subscriptions live: autosave save_succeeded, autosave save_failed, crosstab frame_saved, crosstab session_saved. The provider also hosts the cross-store linkage (FrameStore.onPeerSave calling SessionStore.onParentFrameSaved). Every subscription is attached on mount and detached on unmount; nothing in src/state/ creates a long-lived subscription outside the provider's lifecycle.

The clean way to think about the composition: actions are state-layer methods that bundle modes (transformation), runtime (recompute), and autosave (persistence-with-debounce) into a single dispatch the UI calls; reads are pure projections of the latest store snapshot; the provider keeps everything wired and disposed in lockstep with the React tree. Application code outside src/state/ never has to know whether a particular operation is going to autosave-debounce, immediately-flush, or atomically-restore — it calls a store method by name and the state layer handles the composition.

I.5 self-review (Article VIII § 4)

Architectural commitments (Article II)

  • § 1 Frame/Argument Split. Honored. FrameStore holds Frame/FrameVersion; SessionStore holds ArgumentSession/ArgumentSessionVersion with its embedded frame_version_snapshot. The two stores are independent; the bridge points (onPeerSave for the parent frame; the frame_version_drift projection) preserve insulation rather than blur it.
  • § 2 Determinism. Honored, and operationalized at the state-layer boundary by the five-point determinism contract. The state layer reads the clock exactly once per session action, threads the value into both created_at and computed_at, and never reads the clock from selectors or store getters. The action-runner test asserts the single-read property; the session-store test asserts byte-identity of the persisted ArgumentSessionVersion across clock variations (everything except created_at and downstream evaluated_at/computed_at stamps is byte-identical). This is the durable mechanism by which the runtime's required computed_at input is supplied deterministically from application code.
  • § 3 Per-Node Satisfaction Criteria. Honored. The state layer surfaces validation results and status badges built from NodeStatus + SatisfactionPolicy-evaluation outputs; failed_conditions flows directly into StatusBadge.failed_conditions for inspector display. The state layer does not redefine or constrain the satisfaction-policy framework.
  • § 4 Independent Versioning. Honored. FrameStore and SessionStore each track their own current_version; saveMilestone on one does not touch the other; restoreVersion on one does not touch the other; only the explicit onParentFrameSaved bridge crosses the boundary, and only in the read direction.
  • § 5 Human-Readable Structure. Honored. FramePatch and SessionPatch are explicit discriminated unions; reading a patch tells you exactly which action it triggers and which entity field it touches. No hidden ordering or coupling variables; each action is a pure transformation that can be replayed in any order against any prior snapshot, subject to the per-kind preconditions modes enforces.
  • § 6 Use-Case Commitment. Honored. Both legal and general modes (with personal/academic flavors) preserved end-to-end; FramePatch.metadata_edited carries jurisdiction_default and positions; the selectors expose legal-mode-only chips behind UI primitive filtering.

Article II § 2 — implementation-level confirmation

Restated for the record. The runtime is the determinism boundary; the state layer's job is to defend it. This spec defends it by:

  • Routing every compute() call through ComputeDriver, which threads computed_at from a single clock read per action.
  • Routing every persisted entity through runFrameAction / runSessionAction, which capture now() once and use the captured value as both the version's created_at and the runtime's computed_at.
  • Forbidding clock reads in selectors and store getters (asserted by the selectors test's spy).
  • Asserting via property test that two action runs with the same patch and same id-injection produce byte-identical persisted entities (modulo the version's own created_at stamp).

These four points together make the runtime spec's "Caller responsibility: supply computed_at from a deterministic source" hold in production. The state layer is the single application caller of compute(); this is the layer at which the discipline lives.
