This document is the implementation source of truth for src/ui/version-history/. It replaces the I.9a placeholder export of VersionHistoryPane with the real implementation, wires the onToggleVersionHistory page-prop callback reserved by I.9b on FrameBuildingPageProps and the on_toggle_version_history callback reserved by I.9c on ArgumentRunningPageProps, and commits per-file TypeScript signatures and behavioral contracts for: the App-level integration in app-routes.tsx (open-state lifting, preview-state provider mount); the VersionHistoryPane top-level composition rendered as an I.9a Drawer; the two-tab variant for Argument Running (Sessions / Frames-read-only); the version tree with branch indentation derived from parent_version_id per Stream H H3's buildVersionTree shape (UI-side equivalent operating over FrameVersionSummary[] / ArgumentSessionVersionSummary[] for lazy summary loading); the milestone-only filter; the selection footer (Preview / Restore / Compare buttons); the preview surfaces (FramePreviewView for read-only frame-canvas overlay; SessionPreviewView for read-only argument-canvas overlay) with the sticky preview banner; the restore confirmation dialog wrapping I.9a's ConfirmDialog; the structural-diff Compare view consuming diffFrameVersions / diffSessionVersions with the layout-only bucketing per Stream H H3; and the version-history preview React Context the pages consume to conditionally render the preview surfaces in place of their normal content.

All imports from @/runtime (value), @/modes (value), and value-level imports from @/llm-hooks are forbidden per Contract 8 and the I.9a substrate's no-restricted-imports wiring. The version-history module's reads come from @/state selectors and snapshots (for the current version ids and the parent-frame title) and from @/persistence's pure diff functions and useRepository() for async listing and loading; its writes go through frame_store.restoreVersion / session_store.restoreVersion for restore (which route through the modes orchestration wrappers and Repository atomicity contracts per the state spec). The module never calls compute(), never produces NodeStatus or ConditionalOutput, never invokes LLM hooks at the value level; preview surfaces render frame and session data without recomputation, so status badges and output payloads are absent during preview by design (status display reactivates on Exit Preview because the working state is unchanged).

Cross-document resolution notes (for self-review)

The I.9d1 spec carries five small refinements relative to the I.9a UI substrate spec, the I.9b frame-building UI spec, the I.9c argument-running UI spec, and the I.4 persistence spec. None is constitutionally novel; all are additive or callback-wiring promotions of placeholders to real implementations. Per the F-003 / F-004 / F-005 family pattern, the five items are consolidated into a single register flag (F-006) held in flag_pending_from_i9d1.html for merge into the live flags.html.

  1. Add read_only?: boolean to FrameCanvasProps in the I.9a UI substrate spec. The version-history preview surfaces render FrameCanvas in read-only mode (no node drag, no connector-handle gesture, no auto-arrange, no edge-creation popup). The I.9a spec's existing enable_editing?: boolean prop disables only the toolbar's auto-arrange button; it does not suppress the per-node connector handle or the node-drag affordance, which are needed for the read-only display. Adding read_only?: boolean as an optional additive prop (default false) preserves backward compatibility with I.9b's and I.9c's existing call sites (both omit the prop and get the existing editing behavior). The new prop's contract: when true, the canvas (a) hides connector handles on per-node renderers, (b) sets React Flow's nodesDraggable and edgesUpdatable to false, (c) disables the toolbar's auto-arrange button regardless of enable_editing, (d) ignores the on_node_moved and on_edge_created callbacks (they are not called); selection and search and zoom remain functional. The implementation is local to frame-canvas.tsx and to NodeFrame in the per-node-type renderer suite; no cross-module change.
  2. Promote onToggleVersionHistory in FrameBuildingPageProps from optional to required, and add version_history_open: boolean as a required prop. The I.9b spec marked onToggleVersionHistory?: () => void as optional with the comment "I.9d wires this through the App once the version-history pane lands. Until then, the chrome button toggles an inert state." I.9d1 is the wiring step. The refined shape: onToggleVersionHistory: () => void (no ?) and version_history_open: boolean (new). The page passes both into buildTopBarSlots at the existing on_toggle_version_history and version_history_open dep keys; no other I.9b file is touched. Refer to the page-composition skeleton in I.9b's frame-building-page.tsx section — the existing skeleton already references onToggleVersionHistory in the slot-builder call (line annotated "I.9d wires this through"); after I.9d1 the optionality is dropped and version_history_open is added.
  3. Promote on_toggle_version_history in ArgumentRunningPageProps from optional to required, and add version_history_open: boolean as a required prop. Symmetric to refinement #2 but for I.9c's argument-running spec. The I.9c skeleton already hard-codes version_history_open: false /* I.9d */ in buildTopBarSlots; after I.9d1 the literal false is replaced with the page-prop value, and on_toggle_version_history's optional fallback ?? noop is removed from the slot-builder call.
  4. Refine app-routes.tsx (I.9a-owned) to lift version_history_open state, mount the VersionHistoryPreviewProvider, mount a single VersionHistoryPane instance, and thread the callbacks through. I.9a's app-routes.tsx was described as "a tiny component that switches on the current Route and renders the relevant page." I.9b's and I.9c's specs noted that I.9b/c/d each "replace its imports of the placeholder page components with their real implementations." I.9d1 makes the corresponding change: replaces the placeholder import of VersionHistoryPane with the real one; introduces useState<boolean>(false) for version_history_open; wraps the RouterProvider's child tree in <VersionHistoryPreviewProvider>; renders <VersionHistoryPane open={version_history_open} onClose={() => setVersionHistoryOpen(false)} /> as a sibling of the active page component; passes onToggleVersionHistory={() => setVersionHistoryOpen((v) => !v)} and version_history_open={version_history_open} to whichever page is active. The pane reads useRoute() internally to scope to the current Frame or Session; the App-level mount avoids the alternative of duplicate-mounting the pane in both pages. The refinement is local to app-routes.tsx; no other I.9a file is touched. The I.9a barrel re-export VersionHistoryPane in src/ui/index.ts remains in place; only its source changes from the placeholder file (deleted) to the version-history module's barrel (added).
  5. Optional Repository extension (deferred to v1.5 by this spec): batch loader for diff source pairs. The Compare view loads two FrameVersions back-to-back via loadFrameVersion(a_id) then loadFrameVersion(b_id). For v1 the two awaits run in parallel via Promise.all at the call site, which is sufficient (each load is one IndexedDB read on the primary key); the persistence-layer Repository surface does not need a batch loader. If a future polish pass surfaces that the two parallel reads cause a perceptible Compare-view flash, a Repository.loadFrameVersionPair(a_id, b_id) method that batches the two reads into a single transaction would be a clean additive surface; this is reserved for v1.5 and not committed in I.9d1. The same reservation applies to loadSessionVersionPair. This item is documented as part of F-006 to keep the cross-spec audit consolidated, but no spec edits are required by I.9d1 to land it.

F-006 entries 1–4 are technical, additive (or required-from-optional in the page-prop case), and within agent-autonomous scope per Article V § 2(c). Entry 5 is a deferral note for future record-keeping. The five items are held as a single in-session-escalated register flag in the same family pattern as F-003 / F-004 / F-005; downstream coding sessions are not gated on a contracts-doc bump.

export function AppRoutes(): React.ReactElement {
  const route = useRoute();
  const [version_history_open, set_version_history_open] = React.useState(false);
  const on_toggle = React.useCallback(() => set_version_history_open((v) => !v), []);
  const on_close = React.useCallback(() => set_version_history_open(false), []);

  return (
    <VersionHistoryPreviewProvider>
      <PageWithPreview
        route={route}
        version_history_open={version_history_open}
        on_toggle={on_toggle}
      />
      <VersionHistoryPane open={version_history_open} onClose={on_close} />
    </VersionHistoryPreviewProvider>
  );
}

function PageWithPreview(props: {
  route: ReturnType<typeof useRoute>;
  version_history_open: boolean;
  on_toggle: () => void;
}): React.ReactElement {
  const preview = useVersionHistoryPreview();

  if (preview.state.kind === "frame") {
    return (
      <FramePreviewView
        frame_id={preview.state.frame_id}
        version_id={preview.state.version_id}
        version_number={preview.state.version_number}
      />
    );
  }
  if (preview.state.kind === "session") {
    return (
      <SessionPreviewView
        session_id={preview.state.session_id}
        version_id={preview.state.version_id}
        version_number={preview.state.version_number}
      />
    );
  }

  switch (props.route.kind) {
    case "home":         return <HomePage />;
    case "frame_building":
      return (
        <FrameBuildingPage
          frame_id={props.route.frame_id}
          onToggleVersionHistory={props.on_toggle}
          version_history_open={props.version_history_open}
        />
      );
    case "argument_running":
      return (
        <ArgumentRunningPage
          session_id={props.route.session_id}
          on_toggle_version_history={props.on_toggle}
          version_history_open={props.version_history_open}
        />
      );
  }
}

Behavioral contract. Three invariants: preview state lives outside the page (so the pane and the page agree on whether preview is active); the pane is mounted exactly once (the pane internally scopes to the current Route via useRoute() rather than being remounted per page, which would lose its selection state on every navigation); preview replaces the page (not overlaid; the page's chrome and side panels are hidden during preview, matching E5's "the current canvas dims and a banner reads 'Previewing version N'" framing — a true mode swap, not a translucent overlay). The "page mounted alongside" approach is rejected here in favor of the swap because preview is rare and intentional, so the user's working state needs no preservation behind the preview surface; on Exit Preview, the page remounts and re-loads from the (unchanged) stores via the existing open(frame_id) / open(session_id) lifecycle.

Behavioral contract. The pane reads useRoute() to determine scope; for the home route the pane renders null (version history has no entity scope to operate on); for frame_building the pane renders the single-Frame-versions variant; for argument_running the pane renders the two-tab variant.

Internal state. Five pieces of local state: (1) active_tab: PaneTabValue — only used in argument-running variant; defaults to "sessions"; reset to "sessions" on Route change; (2) milestone_filter: MilestoneFilterValue — defaults to "all"; (3) selected_version_id — the currently selected row; defaults to null; reset on tab change and on Route change; (4) compare_state: { from_id; to_id } | null — when non-null, the pane renders CompareView in place of the version list; (5) restore_dialog_open: boolean.

Selection footer wiring. On Preview click: read the preview-context controls, dispatch enterFramePreview or enterSessionPreview with the selected version's metadata; the pane stays open during preview (the user picks another version or clicks Exit Preview from the banner). On Restore click: set restore_dialog_open to true. On Compare click: read the current version id from the relevant store, set compare_state to { from_id: selected_version_id, to_id: current_version_id }; the pane re-renders with CompareView. The back-arrow from Compare sets compare_state to null.

Frame Building variant layout: a Drawer with DrawerHeader showing "Version history · {frame.title}"; DrawerBody rendering either the MilestoneFilter + VersionTree pair (when compare_state === null) or the CompareView (when active); DrawerFooter rendering the SelectionFooter with allow_restore=true. Determinism — UI-layer discipline

The I.9a chrome+canvas substrate established four UI-layer determinism obligations (canonical output verbatim; AI-attribution chips on AI-touched fields; deterministic order with no client-side re-sort; canonical validation-finding order). I.9d1's surfaces touch the first and third; the second is unaffected (no AI in version-history); the fourth is unaffected (no validation surface). This section enumerates the obligations for I.9d1's surfaces.

D1 — Canonical output is rendered verbatim

The Compare view's diff entries render the fields_changed arrays from the persistence-layer diffFrameVersions / diffSessionVersions verbatim. The UI does not rewrite field names, abbreviate, or stylistically transform the diff output. Statement previews render the node's statement field verbatim (truncated for display via CSS text-overflow only, never via JS-side truncation that would alter the underlying string). The preview banner renders the version number from FrameVersion.version_number verbatim. Verified by the Vitest case compare-view.test.tsx asserting that a fixture diff with fields_changed: ["statement", "notes"] renders the comma-separated string "statement, notes" unchanged.

D3 — Deterministic order is preserved; no client-side re-sort

Three lists in this module are sorted by upstream sources and must not be re-ordered:

  • The version tree (VersionTree) — order is supplied by buildVersionTreeShape, a deterministic DFS over the summary list (version_number ascending among children of a parent). The filterByMilestone pre-pass preserves the order; the component renders the resulting display-entry array in-order. The pane never re-sorts.
  • The Compare view's per-section entry lists — order is the order received from diffFrameVersions / diffSessionVersions, which the persistence-layer spec pins as canonical (sorted by node-id / edge-id lex ascending). CompareEntryList renders in-order; never re-sorts.
  • The Repository's listFrameVersionSummaries / listSessionVersionSummaries return summaries sorted by version_number ascending per the persistence-layer spec. useVersionSummaries passes the array through unchanged; VersionTree consumes it via buildVersionTreeShape's deterministic DFS.

D2 / D4 — Inapplicable

I.9d1's surfaces do not render AI-touched content (no G6 / G7 / G11 surfaces in version history); the AI-attribution chip obligation does not apply. I.9d1's surfaces do not render validation findings; the validation-canonical-order obligation does not apply.

What this spec does not commit

  • Storage compaction. Stream H H3 defers compaction (deduplicating identical nodes/edges across versions, or representing later versions as deltas) to v2 as-needed. I.9d1 does not surface compaction in the UI; if a future polish pass introduces it, the pane's API surface is unaffected because the Repository's interface is unchanged.
  • Pixel-level visual diff (E5 deferred). Side-by-side rendering of "version A on the left, version B on the right" is reserved for v2 per E5's v1 scope. I.9d1 implements only structural diff per StructuralDiff / SessionStructuralDiff.
  • Branch collapse affordances. The VersionTreeDisplayEntry.has_branch_children flag is reserved for a polish pass; v1 renders all branches expanded. The flag is computed in the pure helper but unused by the v1 row renderer.
  • Session-preview recomputation. Session preview does not invoke compute() against the historical session version; status badges are absent during preview. A v1.5 polish pass may add a "Restore to inspect status" affordance in the preview banner; the spec does not commit to this. Adding it requires no schema changes — only an affordance in PreviewBanner and wiring to the session store's restoreVersion.
  • Orphan-item badge in session preview. When the historical session version's argument_edges reference NodeRefs that don't resolve in the current frame_version_snapshot, v1 silently suppresses those edges per the FrameCanvas's existing tolerance. A future polish pass may render an "orphaned items: N" badge in the preview banner; v1 does not commit to this.
  • Search-within-version-history. A search box that filters versions by change_summary text is a natural extension but not in v1 scope. The pane's structural framing supports this addition cleanly; v1 ships without it.
  • Per-version export. The Repository's exportFrame / exportSession methods accept an include_history flag; exporting a single version through the pane is a natural extension but reserved. v1 ships export from the home / settings surfaces only.
  • The exact pixel width of the pane. E5 specifies "the right ~30% of the screen" as the target; the spec pins a CSS custom property --space-pane-version-history (default value left to the visual-polish pass; suggested 520px) that Drawer's width prop consumes. Fine-tuning is design's call.
  • Cross-version session-preview alignment for migrated sessions. When a session has been migrated, its historical versions reference an older frame_version_snapshot that may differ structurally from the current one. v1 renders historical session versions against the session's CURRENT frame_version_snapshot with orphan-tolerance; a more faithful alternative (loading the historical frame_version_snapshot from a hypothetical embedded copy) is out of v1 scope and would require a schema change. The v1 design suffices because (a) migration is rare, (b) preview is for "did the user have this premise at version 7?"-style inspection, which the current rendering supports.

I.9d1 self-review (Article VIII § 4)

The Constitution requires that every stream commit a self-review against the relevant Article II commitments and Article III quality criteria before persistence. This section records that review.

Article II — Architectural commitments

  • § 1 — Frame/Argument Split. Honored. The pane scopes by Route; Frame Building shows FrameVersions only; Argument Running shows ArgumentSessionVersions as the primary tab and FrameVersions as a read-only secondary tab. The argument-side Frames tab cannot dispatch frame-restore (Restore is disabled with a tooltip). The Frame Building variant cannot dispatch session-restore (variant never reaches the Sessions tab). Restore calls always go through the right store. No cross-store coupling.
  • § 2 — Determinism. Honored. Version-history reads from canonical sources whose order the persistence-layer spec pins; the UI never re-sorts. The diff functions are pure and deterministic; the UI consumes them as-received. The version-tree shape helper is a deterministic DFS. Preview surfaces render without invoking compute(); status badges and output payloads are absent during preview by design. Restore goes through the store's existing restoreVersion method, which routes through the modes orchestration wrapper and Repository atomicity contract 3. The Determinism section above enumerates the three applicable obligations and their verification surfaces.
  • § 3 — Per-node satisfaction. Unaffected. Version history does not surface or modify satisfaction policies; the diff view's "edited" entries list fields_changed verbatim, including options_box when changed, without interpretation.
  • § 4 — Independent versioning. Strongly honored — this stream is the user surface for independent versioning. The pane shows FrameVersions and ArgumentSessionVersions in separate views. Restore on one side does not touch the other. Preview on one side does not affect the other store. The "current" markers come from each store's own snapshot. The "session was authored against this version" annotation in the Argument Running Frames tab is a one-way read (session → frame), not a coupling — it surfaces drift transparently without coupling the streams' lifecycles. The pane never deletes a version (E5's "no deletion affordance" rule per Article II § 4); the only mutation is restore, which creates a new version off an ancestor, preserving the ancestor.
  • § 5 — Human-Readable Structure. Honored. The version tree's branch indentation makes the fork structure readable from the display itself; no hidden ordering or coupling variables. The diff entries display field-name → fields-changed pairs verbatim; no hidden interpretation.
  • § 6 — Use-Case Commitment. Honored. Both legal and general modes (personal/academic flavors) are first-class; the pane's chrome uses neutral vocabulary per Stream E E5's rule. The two-tab variant exists in both modes because session-vs-frame versioning applies to both. Preview surfaces render the canvas in the operating-mode appropriate to the kind being previewed; legal-mode-specific surfaces (Authority binding chips, jurisdiction visibility, BINDING_IN / DISTINGUISHED_BY edges) flow through unchanged via the FrameCanvas's existing mode-aware rendering.

Article II § 2 — Implementation-level confirmation

The single architectural risk in this stream is the temptation for the UI to recompute status during preview (because seeing status would feel useful in some preview workflows) and thereby invoke compute() via the compute driver against the previewed version. The spec resolves this by holding the line on "no recompute during preview" (preview shows raw data; status badges are absent) and by routing all mutation paths through the store's existing restoreVersion method. The preview surfaces' FrameCanvas mounts pass status_map={undefined} deliberately, and the no-op on_node_moved / on_edge_created callbacks ensure that even if an interaction inadvertently produces a drag event, no state changes. The page-swap-on-preview pattern ensures the user is unambiguously in a read-only state (banner present, no edit affordances, no live session overlay) for the duration of the preview.
