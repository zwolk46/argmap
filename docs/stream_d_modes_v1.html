Stream D — Mode Definitions

Companion to stream_a_resolutions_v1.html, stream_b_schema_v1.html, and stream_c_runtime_v1.html  ·  Session date: May 9, 2026  ·  Version 1

This document specifies the behavioral surfaces of the application: what a user can do in each mode, in what order, and under what constraints. It covers Stream D end-to-end: the Frame Building action inventory (D1), the Argument Running action inventory and interview flow (D2), mode transition rules (D3), Legal Mode toggle behavior (D4), and non-legal mode behavior (D5). It is the bridge between the static schema (Stream B) and the runtime (Stream C) on one side and the UI design (Stream E) on the other. UI affordances — visual layout, control placement, animations — are Stream E's concern; this document specifies the functional surface that any UI must expose.

How to read this

This document is action-first. The Stream B spec is the source of truth for type definitions; this document references those types by section id (B1.5 = Checkpoint, B5 = ConditionalOutput). The Stream C spec is the source of truth for runtime algorithms; this document references those by phase number (Phase 4 = status determination). Where a behavior turns on a runtime computation, this document specifies what happens and points at Stream C for how.

Conventions:

  • Action = a user-initiated operation that mutates state. Each action lists its preconditions, the mutation, and the post-mutation effects (validation pass, recompute, version bump).
  • Operating mode = Frame Building or Argument Running. These are architecturally enforced by data-layer scoping: frame-layer entities can only be created or edited in Frame Building; argument-layer entities only in Argument Running. The "mode toggle" is a UI affordance for switching context; the enforcement is structural (B1's layer field).
  • Architectural mode = legal or general (per A4). Distinct from operating mode. A frame is in exactly one architectural mode at a time.
  • Flavor = personal or academic, applicable only when architectural mode is general (per A4). Flavor changes vocabulary and visibility, not schema.
  • Active vs. visible vs. present: a field is present on every entity (the schema carries it), active when the runtime evaluates it (governed by mode/flavor), and visible when the UI surfaces it (governed by mode/flavor and per-frame opt-ins). A field can be present and not active and not visible.

Decisions made in this session

The following decisions were resolved while writing this spec. Each is justified against the Constitution's Article III quality criteria and is locatable in the body of this document. Items marked stream B extension add detail to existing Stream B specifications without modifying any existing decision; they extend the spec rather than amending it, and so do not require an Article XI amendment document.

  1. Frame creation auto-stubs a RootQuestion. Because every frame must have exactly one RootQuestion (V-FR-1), the create-frame action creates a stub RootQuestion with empty statement. The user fills it in. This avoids an ungrammatical empty-frame state and keeps validation passing trivially at the start.
  2. Deletion is cascade-with-confirmation, not block. Deleting a non-leaf node deletes its descendants and incident edges, with a confirmation dialog summarizing what will be removed. The alternative (block deletion until manual cleanup) penalizes the common case (changing one's mind partway through frame construction) and is harder to recover from. The version system makes accidents recoverable.
  3. CheckpointOption routing is rendered as virtual edges in the graph view. The frame schema (B2) does not promote CheckpointOption.target_node_id to an Edge entity, but the graph view displays a per-option arrow from the Checkpoint to its target. This keeps the graph readable while preserving the schema's "edges have one source, one target" invariant.
  4. Premise reuse is supported and suggested, not enforced. A Premise can have multiple ANSWERS / SUPPORTS / CONTRADICTS edges; when the user begins authoring a Premise text that overlaps with existing premises, the UI offers to reuse one. The user can always decline and create a new one.
  5. The interview ordering algorithm is fully deterministic. Open work items are sorted by: jurisdictional priority, then DFS pre-order from RootQuestion through the active path, then Term.order at unselected branch points, then lexicographic node id as final tiebreaker. The system highlights the topmost item as "recommended" but the user is free to address any item at any time.
  6. Recompute is automatic on every persisted change. Per Stream C's incremental-computation strategy, recomputation cost is bounded; v1 does not need a manual "compute" button. If performance degrades for very large frames, debouncing layers in transparently.
  7. Switching architectural mode is allowed but treated as significant. The mode-change action runs an immediate scan for fields and direction-values that become inconsistent under the new mode and presents the user with a list of items to resolve. The frame stays in its old mode until each item is resolved or the user cancels. Data is never lost; values that become inactive metadata stay on disk and reactivate on switching back.
  8. Switching flavor within general mode is lighter. Flavor changes the Premise kind vocabulary (A4/A11) and Authority visibility/labeling. Existing Premises with kinds outside the new vocabulary are flagged at session-time as advisories; they don't block the flavor change because they're argument-layer artifacts and the frame switch is a frame-level operation.
  9. Frame Building → Argument Running gate is the C5 strict pre-mode-transition check. No additional readiness check at the Stream D layer. If the C5 strict check passes, the user proceeds; if any error fires, the transition is blocked with a list of errors.
  10. Argument Running → Frame Building is always allowed at the navigation level. The user can leave an open session and edit the Frame at any time. The session is preserved; on return, a drift warning fires if the underlying Frame has advanced.
  11. "Source" vs. "Authority" labeling is governed by mode and flavor. Legal mode → "Authority." Academic flavor → "Authority" (the existing schema name fits academic citation practice). Personal flavor → "Source" (matching A4's resolution). The schema name Authority is unchanged everywhere; only the user-visible label varies.
  12. Frame templates are the v1 cross-frame mechanism (per A6). Creating a frame from a template deep-copies the source frame, regenerating all node and edge ids and optionally regenerating slugs. There is no live link to the source. This is exposed as an action in the Frame Building entry flow.

D1 — Frame Building actions

Frame Building is the operating mode in which the user constructs the logical frame: nodes, edges, satisfaction policies, frame metadata. Argument-layer entities (Premise, ArgumentSession, etc.) cannot be created or edited here. Frame Building's actions divide into: frame-level actions, per-node-type actions, edge actions, and version actions.

Per-node-type actions

Every node type supports the universal node actions edit notes, edit presentation hints (x, y, color_override, collapsed), and delete (subject to the constraints below). Type-specific actions are listed per type. All actions on frame-layer nodes occur within Frame Building.

RootQuestion (B1.1)

  • Edit statement (required; cannot be empty when transitioning to Argument Running).
  • Edit standard_of_review (legal only; visible only in legal mode; persists across mode changes as inactive metadata).
  • Edit options_box (per-instance SatisfactionPolicy override per A3).
  • Cannot delete (V-FR-1 requires exactly one RootQuestion). Replacing the RootQuestion is done by editing its statement and replacing its outgoing edges; the node identity persists.
  • Connect outgoing: DECOMPOSES_INTO (to SubQuestion), TURNS_ON (to Term), or LEADS_TO (to Conclusion, rare).

SubQuestion (B1.2)

  • Create as child of a RootQuestion or another SubQuestion via DECOMPOSES_INTO.
  • Edit statement, is_jurisdictional (legal only — visibility per D5), standard_of_review (legal only), options_box.
  • Connect outgoing: DECOMPOSES_INTO, TURNS_ON, or LEADS_TO.
  • Delete: cascades to descendants reachable only through this SubQuestion. Confirmation dialog lists the cascade set.

Term (B1.3)

  • Create as child of a Question via TURNS_ON.
  • Edit name, order, dispositive, linked_to, options_box.
  • Reorder among siblings: changes order values. Reordering matters for foreclosure resolution and default UI presentation only (per A7); it does not change which Interpretations are available downstream.
  • Link to another Term: sets linked_to. The UI surfaces this when a new Term is created with a name matching an existing Term in the same FrameVersion (per A10). Linking is one-way and cannot create cycles (V-NODE-3, enforced at edit time).
  • Connect outgoing: INTERPRETED_AS (to Interpretation).
  • Delete: cascades to all child Interpretations and any nodes reachable only through them. Disallowed if this Term is the target of any linked_to reference; the user is prompted to either remove the linking Term first or accept that the linking Terms become structurally invalid (V-NODE-4 will fire on subsequent validation).

Interpretation (B1.4)

  • Create as child of a Term via INTERPRETED_AS. A Term must end up with at least two Interpretations (V-FR-3) before transition to Argument Running, but during editing a Term can transiently have one or zero.
  • Edit statement, jurisdiction_override (legal only), options_box.
  • Connect outgoing: LEADS_TO (to Checkpoint, SubQuestion, or Conclusion), FORECLOSES (to Term, SubQuestion, or Checkpoint).
  • Connect incoming: CITES (from Authority), DISTINGUISHED_BY (from Authority).
  • Delete: removes the Interpretation and any FORECLOSES edges originating from it. The downstream nodes pointed at by its LEADS_TO edges are not deleted (they may be reached from other Interpretations).

Checkpoint (B1.5)

  • Create as a child of an Interpretation, SubQuestion, or LogicalGate via LEADS_TO (or as a GATES target). New Checkpoints default to answer_type: "boolean" with the default options pair from A2.
  • Edit question, requires_premise, requires_authority (legal only), burden_level (legal only), options_box.
  • Edit answer_type: switching between types is supported with the following rules:
    • boolean → multiple_choice: existing two options are preserved as the first two of the multiple_choice set.
    • boolean → graded: existing two options are mapped to "met" and "not_met"; an "unclear" option is appended with routes_to_status: "contested".
    • multiple_choice → boolean: only allowed when there are exactly two options; otherwise the user is asked to remove options first.
    • multiple_choice → graded: only allowed when there are exactly three options; the middle option in order gets routes_to_status: "contested".
    • graded → boolean: drops the middle option; the user is warned about routing implications.
    • graded → multiple_choice: middle option's routes_to_status is cleared; otherwise unchanged.
    • Switching while session-time CheckpointResponse records reference removed CheckpointOption ids creates orphans, which are surfaced for cleanup (see D3 — frame edits with live sessions).
  • Edit options: add, remove, edit label, edit satisfies, edit routes_to_status, set target_node_id per option ("wire option to target"). Subject to V-NODE-5/6/7 (option count per answer_type) and V-NODE-9 (option id uniqueness within Checkpoint).
  • Reorder options: rearranges the array; option ids are stable across reorders.
  • Connect incoming: LEADS_TO from Interpretation/SubQuestion/LogicalGate, GATES from LogicalGate.
  • "Connect outgoing" is not a direct action: routing happens by setting CheckpointOption.target_node_id. The graph view renders this as a virtual edge per decision #3 above.
  • Delete: cascades nothing on its own (option targets are referenced, not owned). Removes the Checkpoint and its incoming/outgoing edges. Any session CheckpointResponse referencing this Checkpoint becomes orphaned (see D3).

LogicalGate (B1.6)

  • Create with a chosen gate_type. The UI exposes five distinct creation actions: AND, OR, NOT, IF_THEN, UNLESS. Each opens an editor sized to the gate's slot count.
  • Edit inputs (per gate_type, per B1.6 named slots):
    • AND / OR: inputs[], minimum 2; add/remove/reorder.
    • NOT: input, exactly 1.
    • IF_THEN: antecedent, consequent.
    • UNLESS: main, exception.
  • Edit output_target (renders as the GATES edge in the graph).
  • Change gate_type: supported but treated as a destructive operation. The UI presents it as "convert gate," shows the inputs that will be lost or reinterpreted under the new type's slot semantics, and creates a confirmation step. AND ↔ OR is a clean swap (same input set, different evaluation). The other transitions reset to empty slots.
  • Connect outgoing: GATES (from this gate to its output_target).
  • Delete: removes the gate and its GATES edge; does not delete inputs or output_target nodes.
  • Per V-GATE-5 and V-GATE-6, the editor blocks selection of Term, Premise, or Authority as inputs at edit time, surfacing an inline message rather than waiting for validation to fire.

Conclusion (B1.7)

  • Create as a child of an Interpretation, SubQuestion, or LogicalGate via LEADS_TO (typically as a terminal node).
  • Edit statement, direction (mode-aware; the UI surfaces only the values valid for the current mode), reasoning_summary, tags, remedy (legal only), options_box.
  • Direction picker:
    • Legal mode: dropdown of the legal enum values (favors_plaintiff, favors_defendant, affirm, reverse, remand, dismiss, custom). Selecting custom reveals the custom_label text field.
    • General mode: dropdown of the Frame's positions table. If empty, the user is prompted to define at least one Position before saving the Conclusion direction. custom_label is also available for free-form override.
  • Connect outgoing: typically none (Conclusion is terminal).
  • Delete: removes the Conclusion and its incoming LEADS_TO edges. If a downstream entity references this Conclusion (best_inference cache, etc.), those caches are invalidated.

Authority (B1.8)

  • Create at frame-build time. Argument Running can also create session-layer Authorities; the action shape is the same.
  • Edit fields, conditional on mode/flavor visibility (D4/D5):
    • Universal: citation, url, short_label.
    • Legal mode: court, year, holding_summary, is_holding, is_binding (manual override; F6 will eventually compute), jurisdiction, binding_in (the embedded relation per B2's note).
    • Academic flavor: author, venue, position.
    • Personal flavor: as universal only by default; the per-frame "enable Source fields" opt-in surfaces academic-style fields when desired.
  • Connect outgoing: CITES (to Interpretation; carries strength), DISTINGUISHED_BY (to Interpretation; carries reasoning). BINDING_IN is not an edge (per B2 note); it is the binding_in field set on the Authority itself.
  • Delete: removes the Authority; CITES and DISTINGUISHED_BY edges with this source are removed; Premises with authority_ref pointing at it become orphaned (see V-ARG-4) and are flagged at session-time.

Premise (B1.9) — argument layer only

Premises are not editable in Frame Building. They are listed here for completeness. Their actions are specified in D2.

Versioning actions

Versioning is automatic per B3. The user-initiated actions are:

  • Save milestone: forces a new FrameVersion with is_milestone: true, with an optional user-provided change_summary. Milestones are the markers shown prominently in the version-history UI.
  • Open version history: surfaces the FrameVersion graph rooted at the current frame.
  • Preview prior version: opens a read-only view of an ancestor FrameVersion. No restore happens unless the user confirms.
  • Restore prior version: see frame-level actions above.

Frame Building entry and exit

Entry: opening a frame from the home screen, creating a new frame, or creating from template enters Frame Building by default. The application defaults to Frame Building when an existing frame is opened with no live ArgumentSession context.

Exit: navigating away (to home, to a session, to settings) implicitly exits. Switching to Argument Running explicitly exits via the D3 transition. There is no "save and exit" action; saves are automatic.

D2 — Argument Running actions and interview flow

Argument Running is the operating mode in which the user runs a specific set of Premises against a frozen FrameVersion to produce a conclusion. The frame is read-only here; the only mutations available are argument-layer ones (Premises, InterpretationSelections, CheckpointResponses, session Authorities, argument-layer edges).

Argument-layer node and edge actions

Premise creation and editing

  • Create a Premise: requires statement text and a kind drawn from the vocabulary corresponding to the Frame's mode/flavor (V-ARG-3). Optional: source (free-text), authority_ref (id of an Authority node — frame-layer or session-layer).
  • Edit a Premise: any of its fields. Edits propagate to all edges referencing the Premise.
  • Delete a Premise: removes the Premise node and all its argument-layer edges (ANSWERS, SUPPORTS, CONTRADICTS). The CheckpointResponses that referenced the deleted Premise are also removed; the corresponding Checkpoints become unanswered.
  • Reuse: when the user opens the Premise authoring UI for a target (e.g., answering a Checkpoint), the system offers existing Premises ranked by text similarity to the in-progress draft and by recency. Selecting an existing Premise creates a new edge to the same Premise; no duplicate is made.

Authority creation at session time

  • Create a session-layer Authority: same shape as B1.8. Stored in session_authorities rather than the FrameVersion. Visible to the session only; not promoted to the Frame.
  • Edit / delete same as frame-layer Authority. Deletion orphans Premises with authority_ref pointing at it (V-ARG-4 fires); the user is prompted to remove the reference or choose another Authority.
  • Promote session Authority to frame: not a v1 affordance. The user can manually copy the Authority into the frame in Frame Building if desired.

Answering a Checkpoint

  1. The user opens a Checkpoint (from the open-items list or by clicking the node in the graph).
  2. The system shows the Checkpoint's question and its options (boolean / multiple_choice / graded as configured).
  3. The user selects one option.
  4. The user authors a Premise (or selects an existing one for reuse). Premise statement is required if Checkpoint.requires_premise is true (default true).
  5. If Checkpoint.requires_authority is true (legal mode option), the user must also attach an authority_ref on the Premise — either pointing at an existing frame-layer or session-layer Authority, or by creating a new session Authority inline. V-ARG-2 enforces this.
  6. On save: a new Premise is created (or an existing one referenced), an ANSWERS edge is created with selected_option_id, and a CheckpointResponse is appended to the session. The status determiner (Stream C Phase 4) recomputes.
  7. Answering an already-answered Checkpoint replaces the prior CheckpointResponse and may delete the prior Premise if no other edge references it (configurable; default is to keep the orphaned Premise visible in the session's premise pool so the user can re-attach it elsewhere).

Selecting an Interpretation for a Term

  1. The user opens a Term (from the open-items list or by clicking the node).
  2. The system shows the Term's Interpretations.
  3. The user selects one or more Interpretations. Multiple selections are permitted (per the Stream C InterpretationSelection schema; this records "alternative paths to the same Conclusion").
  4. Optional: the user attaches supporting Premise references and Authority references, and writes notes.
  5. On save: an InterpretationSelection record is created (or replaced if one already exists for this term_id). The status determiner recomputes; foreclosure resolution may run if any selected Interpretation carries FORECLOSES edges or the Term is dispositive.
  6. Linked Terms (A10): if the user opens a linked Term (one whose linked_to points elsewhere), the UI redirects to the linked target's Term, with a notice. The linked target's selection takes precedence at runtime; per V-ARG-8, providing a selection on a linked Term raises a warning.

Adding evidence (SUPPORTS / CONTRADICTS) to an Interpretation or Conclusion

  1. The user opens the target node.
  2. The user authors a Premise or selects an existing one.
  3. The user chooses SUPPORTS or CONTRADICTS and an optional weight (strong / moderate / weak).
  4. On save: the edge is created. Status recomputes; the contested/satisfied logic in Stream C Phase 4 picks up the new edge.

Interview flow — the open-items algorithm

The interview is the user-facing orchestration of "what to address next." It is a derived view, not a stored entity: the system computes the open-items list from the current session state on every render. Determinism is preserved; the same state always produces the same list.

What counts as an open item

An open item is any node on the active or reachable path that is in a state requiring user input:

  • Term with no InterpretationSelection record (or a record selecting zero Interpretations), reachable from RootQuestion. Note: linked Terms are skipped if their target is reachable; the target's selection covers them.
  • Checkpoint with no CheckpointResponse, reachable from RootQuestion through current selections.
  • Checkpoint whose CheckpointResponse exists but whose answering Premise lacks a required authority_ref (V-ARG-2 fires).
  • SubQuestion or RootQuestion whose status is open with failed_conditions indicating user input is needed (e.g., children unresolved). These are derived from the children, so they appear in the list only as parent-grouping headers, not as separately-actionable items.
  • Interpretation lacking authority/premise support, when its options_box requires authority_required or similar. Listed with lower priority — surfaced as advisory rather than blocking.

Ordering algorithm

Inputs: FrameVersion, current ArgumentSession state. Output: an ordered list of open items.

def open_items(frame, session): reachable = compute_reachable_set(frame, session) # Stream C phase 3 active = compute_active_set(frame, session) # Stream C phase 3 status = compute_status_map(frame, session) # Stream C phase 4 items = [] # Walk the frame DFS pre-order from RootQuestion, filtering and grouping. for node in dfs_preorder(frame, root=root_question(frame), order_key=lambda n: (n.is_jurisdictional if hasattr(n, "is_jurisdictional") else False, getattr(n, "order", 0), n.id)): if node not in reachable: continue item = open_item_for(node, session, status) if item is not None: items.append(item) # Sort by priority key (jurisdictional first, then DFS order, then term order, then id). items.sort(key=lambda it: (0 if it.is_jurisdictional else 1, it.dfs_order, it.term_order if hasattr(it, "term_order") else 0, it.node_id)) return items

The DFS is a deterministic traversal that, at each branching point, descends into branches in a fixed order:

  • SubQuestion children: by is_jurisdictional (true first), then by node id lex.
  • Term children of a Question: by order ascending, then by node id lex.
  • Interpretation children of a Term: only the selected Interpretation(s) are descended into; if no selection exists, the Term itself is the open item and we do not descend.
  • CheckpointOption targets: only the chosen option's target is descended into; if no response exists, the Checkpoint itself is the open item.
  • LogicalGate inputs: descended into in their named-slot order (antecedent before consequent for IF_THEN, main before exception for UNLESS, list order for AND/OR, the single input for NOT).

Jurisdictional priority bubble

Per A8, jurisdictional SubQuestions render first regardless of their position in the frame's structural hierarchy. The DFS treats is_jurisdictional: true SubQuestions and their entire subtrees as a priority bucket: those subtrees are emitted into the open-items list first, before non-jurisdictional content. Within the jurisdictional bucket, ordering follows the same rules.

Recommended next item and free navigation

The first item in the ordered list is the recommended next. The UI highlights it. The user is not forced to address it; clicking any item in the list opens that item's editor. Resolving any item refreshes the list and re-derives the recommendation.

Closed-item revisits

The user can re-open a previously answered Checkpoint or a previously selected Term at any time. Doing so:

  • For Checkpoints: edits the existing CheckpointResponse. If the user changes the option, the prior Premise is detached from this Checkpoint (the Premise itself remains; the ANSWERS edge is replaced).
  • For Terms: edits the existing InterpretationSelection. Deselecting an Interpretation re-opens the Term as an open item and may re-introduce foreclosed nodes.

Output viewing during the interview

Per A9, the ConditionalOutput object drives three views: prose summary (default), decision tree (toggle), and path overlay on the frame (toggle). All three are available at any time during the interview, not just at the end. As the user resolves open items, the output updates live; partially-resolved sessions show conditional or incomplete shape outputs (per B5) until determinate.

Completion

A session is "complete" when the open-items list is empty. There is no separate "publish" or "lock" action in v1. The user can save a milestone version to mark a complete state, but the session remains live and editable.

Argument Running entry and exit

Entry: opening an existing session, or creating a new session against a valid frame. Both routes pass through the C5 strict pre-mode-transition check (D3).

Exit: navigating away. Mutations persist (debounced auto-save). The session remains in its current state on return.

D3 — Mode transition rules

"Mode transition" covers four distinct movements:

  1. Frame Building → Argument Running (operating-mode change with validation gate);
  2. Argument Running → Frame Building (free, with drift implications);
  3. Architectural mode change (legal ↔ general) on a frame;
  4. Flavor change (personal ↔ academic) within general mode.

Frame Building → Argument Running

The user clicks "Switch to Argument Running" or "Open session" or "New session." The system runs C5 in pre-mode-transition strictness against the current FrameVersion.

  • If any error-severity rule fires, the transition is blocked. The UI surfaces the list of errors with links to the implicated nodes/edges and suggested fixes (per Stream C's ValidationResult shape).
  • If only warning-severity rules fire, the transition proceeds. Warnings are shown as an acknowledgment dialog: "You can proceed; here's what's not yet ideal." The user clicks past.
  • If neither errors nor warnings fire, the transition is silent.

On successful transition: the user is offered the choice of (a) opening an existing session against the current FrameVersion if one exists, (b) opening an existing session against an older FrameVersion (with a drift warning), or (c) creating a new session. The session UI replaces the frame editor.

Per Stream C, the C5 rule list (B6) is the single source of truth for what counts as "ready." Stream D adds no separate readiness criteria.

Argument Running → Frame Building

This direction is always free. The user navigates away from the session — back to home, to the Frame view, to settings. The session is preserved; mutations persisted before navigation remain.

If the user opens the Frame for editing while a session against an older FrameVersion exists:

  • Frame edits proceed normally. Each persisted edit creates a new FrameVersion (B3 auto-save).
  • The active session, if visible in any tab/pane, surfaces a drift warning ("This session was built against version N; the frame is now at version N+1") with options: (a) keep working at version N (default), (b) migrate to N+1 (the migration action under D2).
  • Editing the frame does not break the session at version N. The session's frame_version_snapshot is an embedded copy and is unaffected by frame edits.

Frame edits with live sessions — orphan handling

Some Frame edits, when migrated forward, create orphans in argument-layer data. Orphans are not errors; they are work items the user resolves at migration time.

Architectural mode change (legal ↔ general)

Mode is set at frame creation. The user can change it later from Frame metadata. The change is significant and runs an immediate scan.

Mode-change scan

When the user requests a mode change, the system enumerates:

  • Every Conclusion.direction whose kind does not match the new mode. (Legal direction values become invalid in general mode and vice versa; the union shape requires a matching kind.)
  • Every Premise whose kind is not in the new mode's vocabulary. (Argument-layer; flagged for sessions, not frame-blocking — but listed for visibility.)
  • Every Authority node carrying mode-incompatible field values. Note: per B's "absent ≡ null" rule, fields like court and year simply become inactive metadata; this is not a blocker.
  • Every Checkpoint with requires_authority: true when switching to general mode. The flag is legal-only; it becomes inert in general mode (and an advisory warning surfaces).
  • Every SubQuestion with is_jurisdictional: true when switching to general mode. Same treatment as requires_authority.
  • Every standard_of_review field set on Question nodes. Legal-only; inert in general mode.

Resolution UI

The user is shown a list. The only items that block the change are direction mismatches on Conclusions, because those would leave a required field in an unrenderable state. The blocking items are presented with inline editors:

  • Legal → General: each Conclusion with direction.kind = "legal" needs reassignment to a Position. If the Frame has no positions table, the user is prompted to define one inline. The mode change is committed only when every blocking item is resolved.
  • General → Legal: each Conclusion with direction.kind = "general" needs reassignment to a legal direction value. Inline picker for the legal enum.

Non-blocking items are listed for awareness ("these fields will become inactive metadata; they will reactivate if you switch back"). The user can dismiss the list and proceed.

Data preservation

No field values are cleared on mode change. Inactive metadata stays on disk. Switching back restores the prior state of fields that were never overwritten in the interim. This is consistent with B's "absent ≡ null" semantics: fields that the new mode treats as absent are not deleted, just ignored.

Validation effects

After a mode change commits, C5 runs in editing strictness and surfaces any new advisories (e.g., V-FR-7 fires when a frame switches into legal mode without any jurisdictional SubQuestions). The user is not blocked from continuing in Frame Building; the warnings are advisory.

Flavor change (personal ↔ academic)

Flavor changes are lighter than mode changes. Schema is unchanged; only Premise vocabulary, Authority visibility/labeling, and UI strings shift.

  • The change commits immediately. No blocking items.
  • An advisory list shows: (a) Premises whose kind is outside the new vocabulary (these will be flagged as V-ARG-3 errors at session-time and need updating before Argument Running succeeds); (b) Authority fields that will become hidden or labeled differently.
  • Switching to personal hides Authority by default but does not delete any Authority nodes. The per-frame "enable Source fields" opt-in surfaces them again.
  • Switching to academic shows Authority with academic-flavor field visibility (author, venue, position). Legal-flavor fields stay inactive metadata.

D4 — Legal Mode toggle behavior

Legal Mode is the architectural mode legal (per A4). It is set at frame creation and changeable via D3's mode-change protocol. This section enumerates exactly what Legal Mode adds, hides, and changes.

Activation surface — what becomes available

Frame-level

  • jurisdiction_default field on Frame, with editor (level + region required; court and hierarchy_node_id optional).
  • The mode-change scan blocks direction-mismatch on Conclusions (D3).

Node-level

  • RootQuestion: standard_of_review field becomes visible and editable.
  • SubQuestion: is_jurisdictional checkbox visible; standard_of_review visible (with "inherit" default).
  • Interpretation: jurisdiction_override field visible.
  • Checkpoint: requires_authority checkbox visible; burden_level dropdown visible (per B1.5 enum: preponderance, clear_and_convincing, beyond_reasonable_doubt, scintilla, substantial_evidence).
  • Authority: full legal field set visible (court, year, holding_summary, is_holding, is_binding, jurisdiction, binding_in). Label is "Authority."
  • Conclusion: direction picker shows the legal enum; remedy field becomes visible.
  • Premise: kind picker shows the legal vocabulary (stipulated, found, disputed, procedural).

Edge-level

  • DISTINGUISHED_BY edge type becomes selectable from Authority to Interpretation.
  • BINDING_IN is rendered as a chip on the Authority node showing the jurisdictions where it binds (per B2's note that BINDING_IN is implemented as the embedded binding_in field).

Validation rule activation

  • V-FR-7 (warning): legal frame should have at least one jurisdictional SubQuestion.
  • V-FR-11 (error): every Authority with is_binding: true has a non-null jurisdiction.
  • V-ARG-2 (error): Premises answering requires_authority: true Checkpoints must have authority_ref.

Runtime condition activation

  • authority_required, authority_binding, not_distinguished, standard_of_review_applied conditions in the SatisfactionPolicy library become available in options_box editors.
  • The default Checkpoint policy includes burden_met(preponderance) by default (per B's default policies table) — this remains the default in legal mode and the user can override per-frame.
  • NodeStatus's via sub-flags (binding_authority, persuasive_authority) surface in the legal-mode display.
  • C6 authority binding determination runs (Stream C); F6 will eventually upgrade it.

Deactivation surface — what becomes hidden when Legal Mode turns off

"Turning off Legal Mode" means switching the frame to general. The schema retains all fields (nothing is deleted); the UI hides them and validation/runtime ignores them.

  • All fields listed under "Activation surface" are hidden in the editor.
  • The Authority node remains accessible if visible per general-mode rules (D5: visible in academic; hidden by default in personal). Legal-only Authority fields are inactive.
  • Legal-only validation rules (V-FR-7, V-FR-11) and the legal-only argument rule V-ARG-2's predicate become trivially satisfied (no Checkpoint has requires_authority: true in non-legal mode because the field is hidden and inert).
  • Legal-only condition library entries are not selectable in options_box editors. Existing selections persist as inactive policy entries; they are restored on switching back.

Mid-frame toggle implications

Per D3, the mode change runs the scan and blocks only on direction-mismatch. The user must reassign each Conclusion direction. All other legal-only data persists as inactive metadata.

D5 — Non-legal mode behavior

"Non-legal mode" is the architectural mode general (per A4). General mode requires a flavor: personal or academic. Flavor changes vocabulary, visibility, and labeling; it does not change schema (per A4: "Flavor is metadata, not schema"). This section enumerates the general-mode baseline and then the per-flavor specifics.

General-mode baseline (vs. legal)

In general mode, every field tagged "legal mode only" in the schema is inactive (validators and renderers ignore it). The user-visible removals from the legal mode surface are:

  • Frame.jurisdiction_default: hidden.
  • Frame.positions: required (general mode uses Position-based Conclusion directions). The Position editor surfaces alongside Frame metadata.
  • RootQuestion.standard_of_review and SubQuestion.standard_of_review: hidden.
  • SubQuestion.is_jurisdictional: hidden. Jurisdictional SubQuestions don't exist in general mode; the priority bubble in the interview ordering algorithm has no effect because no node carries is_jurisdictional: true.
  • Interpretation.jurisdiction_override: hidden.
  • Checkpoint.requires_authority and Checkpoint.burden_level: hidden by default; see "Academic flavor" below for flavor-specific exceptions.
  • Authority's legal field set (court, year, holding_summary, is_holding, is_binding, jurisdiction, binding_in): hidden.
  • DISTINGUISHED_BY edge type: hidden by default; selectable in academic flavor (see below).
  • BINDING_IN concept: not surfaced in either flavor.
  • Conclusion.direction picker: shows the Position-based variant only.
  • Conclusion.remedy: hidden.
  • Validation rules V-FR-7, V-FR-11, V-ARG-2: do not fire in practice (their predicates depend on legal-mode fields).
  • Condition library: authority_required, authority_binding, not_distinguished, standard_of_review_applied are not offered in options_box editors by default. Academic flavor surfaces authority_required as an opt-in (see below).

Personal flavor

Per A4, personal flavor is the lightest configuration. Used for analytical personal questions of an academic flavor that warrant fully fleshing out a logical argument; not for routine quick decisions.

  • Authority visibility: Authority node type is hidden by default in the node-creation palette. The frame author can opt in via "enable Source fields" in Frame metadata; once enabled, Authority becomes available with the universal field set (citation, url, short_label) plus the academic field set (author, venue, position) — same as academic flavor with the same fields. The legal field set remains hidden.
  • Authority labeling: when visible, the type label in UI is "Source" (not "Authority"). The schema name Authority is unchanged for code consistency. References ("supported by", etc.) use the term "Source" throughout the personal-flavor UI.
  • Premise kind vocabulary: observation, value, assumption, stipulated.
  • CITES edge: when Authority is enabled, CITES is selectable from Source to Interpretation. The label "cites" may be replaced in UI with "supports from source" — this is a Stream E concern.
  • DISTINGUISHED_BY edge: not surfaced in personal flavor (the concept of distinguishing an authority is academic/legal).
  • Condition library defaults: authority_required, authority_binding, not_distinguished, standard_of_review_applied are not offered in options_box editors. The burden_met condition uses a personal-flavor default of preponderance equivalent (see below).
  • Burden of proof in personal: burden_met is conceptually present but the burden levels' legal-flavor labels are non-idiomatic. The UI relabels preponderance as "more likely than not," clear_and_convincing as "highly likely," beyond_reasonable_doubt as "near-certain," and hides scintilla and substantial_evidence (administrative-law concepts) as advanced opt-ins. The schema enum is unchanged.

Academic flavor

Academic flavor is the configuration for academic argument mapping (philosophy, policy, theory). It treats Authority as a first-class citation object, with the academic field set active.

  • Authority visibility: Authority node type is fully visible in the node-creation palette by default. Universal fields plus academic fields (author, venue, position) are shown. Legal fields are hidden.
  • Authority labeling: type label is "Authority" (matching the schema name; "Authority" is idiomatic in academic citation contexts).
  • Premise kind vocabulary: empirical, definitional, normative, stipulated.
  • CITES edge: fully active.
  • DISTINGUISHED_BY edge: surfaced as "rejected by" or "challenged by." Used to mark an Authority as not applicable for a given Interpretation (e.g., the cited paper applies to a different domain or has been refuted).
  • Condition library: authority_required is offered as an opt-in (academic argument may want to require a citation for an Interpretation). authority_binding, not_distinguished, standard_of_review_applied remain hidden (they are legal-mode concepts).
  • Burden of proof in academic: same as personal — schema enum is unchanged; UI relabels for idiom. Academic flavor adds an additional advisory: the burden-of-proof framework is borrowed from legal practice and should be used with awareness that academic argument typically operates under different epistemic standards. This is an inline note in the burden-level picker, not a hard constraint.
  • Scholarly consensus: per A4 and B1.8, this is not a separate concept. It is an Authority node with position describing the consensus view, typically with a citation pointing to a survey or review article. The UI may suggest this pattern when the user types "consensus" or similar in a citation field.
