Stream I per-module spec — Argument Running UI (page + interview pane + output viewer + per-item editors + bottom panel + drift indicator + status summary)  ·  Stream I session I.9c  ·  May 11, 2026

This document is the implementation source of truth for src/ui/argument-running/. It commits per-file TypeScript signatures and behavioral contracts for: the ArgumentRunningPage top-level composition (which replaces the I.9a placeholder export of the same name); the two-pane layout shell with bottom-panel mount; the left pane (interview pane rendering the D2 deterministic open-items list with the jurisdictional priority bucket, recommended-next highlight, filter chips, search, and a recompute pulse indicator); the right pane (three-tab output viewer — path overlay default / decision tree / prose — with the per-item editor that narrows the output-viewer area when an interview item is selected); the per-node-type item editors (Checkpoint editor with answer + premise-authoring + optional Authority attachment; Term editor with Interpretation selection and optional supporting evidence; Interpretation evidence editor for SUPPORTS / CONTRADICTS authoring); the bottom panel (Premise pool + Session Authorities with attached-edge counts and proactive "Add Premise" / "Add Authority" buttons); the FrameVersion drift indicator and status-summary top-bar slots; and the session-side AI suggestion drawer mount (store_kind="session") consuming G6 (output prose rewrite) and G11 (premise drafting from fact pattern). The I.9a substrate (chrome + canvas + primitives + suggestion drawer + cross-cutting hooks) and I.9b's per-node-type editor patterns (dispatch-by-type registry, AI-attribution decoration, the useFieldAttribution hook) are the foundation this spec consumes; cross-module integration surfaces in stream_i_contracts_v1.html (Contracts 4, 6, 7, 8), stream_i_state_spec_v1.html, stream_i_modes_spec_v1.html, and stream_i_llm_hooks_spec_v1.html are cited by name and not restated.

All imports from @/runtime (value), @/persistence, @/modes (value), and value-level imports from @/llm-hooks are forbidden per Contract 8 and the I.9a substrate's no-restricted-imports wiring. The argument-running module's reads come from @/state selectors and snapshots; its writes go through session_store.applyEdit(patch) with SessionPatch from @/state; its type-only imports of HookInvocationRecord / SuggestionResult / ConfirmationDecision / HookId from @/llm-hooks are limited to the per-field AI-attribution decoration (reused from I.9b) and the suggestion drawer's typed payloads (the drawer itself lives in I.9a; this spec only consumes it).

Per-item editors (the run-through wizard's interaction surface)

The per-item editor opens when an interview item is selected. It occupies the right portion of the output viewer (narrowing the viewer rather than overlaying it), is dismissable (Esc / X / click outside the editor and viewer cluster), and dispatches a SessionPatch on save. The editor file layout mirrors I.9b's right-pane/editors/ pattern: a dispatch-by-node-type host plus per-node-type editor files. Unlike I.9b, where editors edit frame structure, these editors compose argument-side artifacts: a Checkpoint editor authors a Premise plus an ANSWERS edge plus a CheckpointResponse; a Term editor records an InterpretationSelection; an Interpretation evidence editor authors a Premise plus a SUPPORTS or CONTRADICTS edge.

Cross-document resolution notes (for self-review)

The I.9c spec carries small refinements relative to stream_i_state_spec_v1.html (I.5), stream_i_modes_spec_v1.html (I.6), stream_i_persistence_spec_v1.html (I.4), stream_i_ui_chrome_canvas_spec_v1.html (I.9a), and Contracts 7 and 8 in stream_i_contracts_v1.html. None is constitutionally novel; each is consolidated into a single in-session-escalated register flag (same family as F-003 and the I.3 / I.4 / I.5 / I.6 / I.7 / I.8 / I.9a / I.9b pending flags). Held in flag_pending_from_i9c.html for merge.

  1. State-spec selector named: selectInterviewItems. The I.5 state spec deferred the interview-ordering wrapper on the SessionStore as "I.6 territory," and the I.6 modes spec defined computeInterviewOrder(frame, session, compute_result) as a pure function in @/modes without pinning a state-layer wrapper. The I.9c UI cannot call computeInterviewOrder directly because the I.9a substrate's import boundary forbids value imports from @/modes in src/ui/ (the spirit of the boundary is "state acts as the broker for cross-module orchestration"). This spec resolves the gap: the state spec's selectors.ts gains selectInterviewItems(snapshot: SessionStoreSnapshot): InterviewItem[]. The selector returns [] when the snapshot has no current_version or no compute_result; otherwise it calls computeInterviewOrder(current_version.frame_version_snapshot, session, compute_result). The InterviewItem type is re-exported from @/state's barrel (type-only) so I.9c imports it through the state surface per Contract 8 boundary.
  2. State-spec selector named: selectStatusSummary. The top-bar status-summary chip ("{N}/{M} resolved · tentative: X") needs three derived numbers and a label: resolved count, total active-set count, conclusion-shape tier, and conclusion-label-when-applicable. All four derive from compute_result (status_map, active_set, output) — but the count semantics specifically ("how many active-set nodes have status === 'satisfied'" — not all reachable nodes, only active-set, and not all status values, only 'satisfied') warrant a named selector so the count is the same byte-for-byte across the top-bar, the empty-state's "{N}/{M} done" sub-line, and any tests. This spec adds selectStatusSummary(snapshot: SessionStoreSnapshot): StatusSummary to the state spec's selectors.ts, with StatusSummary = { shape: ConditionalOutput["shape"]; resolved_count: number; total_count: number; conclusion_label?: string }. The selector returns null when no compute_result is present.
  3. AppState extension: output_view_tab_choice_by_frame. Per Stream E E3, "Tab choice is per-user (A9) with per-frame override; the user's choice is preserved across sessions." The choice is persisted in app_state per Stream H H2 (1-second debounced; never versioned), keyed by FrameId so different frames can have different default views. The AppState interface in @/persistence gains output_view_tab_choice_by_frame: ReadonlyRecord<FrameId, "prose" | "decision_tree" | "path_overlay">. The AppStateStore (Contract 7) is extended with setOutputViewTabChoice(frame_id: FrameId, choice: OutputViewTab): Promise<void> that wraps set; the state spec's app-state-store.ts gains the implementation.   5. I.9d integration callback: on_open_migration_dialog. Determinism — UI-layer discipline

The I.9a chrome+canvas substrate established four UI-layer determinism obligations (canonical output verbatim; AI-attribution chips on AI-rewritten content; deterministic order with no client-side re-sort; canonical validation-finding order). Each obligation has specific surfaces in src/ui/argument-running/; this section enumerates them so the test plan and self-review can verify them.

D1 — Canonical output is rendered verbatim

The output viewer (path overlay, decision tree, prose tabs) renders content sourced from the ArgumentOverlayProjection (path overlay), the session-store compute_result (decision tree node-status colors), and the canonical prose generator (prose tab). None of the three tabs may rewrite, abbreviate, or stylistically transform the canonical text; the prose tab in particular renders the canonical generator's prose verbatim as the primary content. The optional G6-rewritten prose is shown only as a secondary surface and only with the AI-attribution chip (D2 below). This obligation is verified by the prose-tab Vitest case that asserts { canonical: "…", rewritten: null } rendering shows the canonical string unchanged.

D2 — AI-attributed content carries a chip

Two surfaces emit AI-attributed content in argument-running/:

  • The prose tab's "Show AI rewrite" secondary view (G6) renders with the I.9a AIAttributionChip primitive set to variant="rewritten" with a click-to-revert affordance per F-002's UI-constraint resolution.
  • The premise-reuse suggestions list in premise-reuse-suggestions.tsx, when populated by G7 in v1.5 (off in v1), renders with AIAttributionChip set to variant="ranked_by_llm"; in v1 the deterministic rankPremiseReuse path emits no chip (it is not AI-derived).

The G11 "draft premises from fact pattern" suggestion drawer is mounted via the I.9a SuggestionDrawer with store_kind="session"; the drawer applies its own AI-attribution per I.9a, and argument-running/ does not duplicate that decoration.

D3 — Deterministic order is preserved; no client-side re-sort

Five lists in this module are sorted by upstream selectors and must not be re-ordered in the UI:

  • The interview-pane list (interview-pane.tsx) — order is supplied by computeInterviewOrder via the selectInterviewItems selector. The pane filters (drops items not matching the filter chip or search query) but does not reorder.
  • The Premise pool — order is the canonical lex-id order from session.premises.
  • The Session Authorities list — order is the canonical lex-id order from session.session_authorities.
  • The premise-reuse suggestions list — in v1, rankPremiseReuse emits a deterministic order (similarity score desc, then premise.id lex asc as the tie-breaker). The component renders in that order. In v1.5, G7's ranked order is similarly emitted by the hook and rendered without modification.
  • The decision-tree tab's node listing in its layout pass — children of each node are laid out in canonical child-edge order from the frame (which is itself deterministic per I.9b D5).

D4 — Validation findings render in canonical order

The argument-running UI surfaces validation in two places: inline error decorations on interview-pane rows (V-ARG-* findings whose node_id matches the row's node) and inline error decorations within the per-item editors. Both surfaces consume session_store.getSnapshot().validation_findings, which the state layer guarantees in canonical order (severity desc, code asc, node_id asc). The UI must not re-order the findings list when grouping them by node; instead, it filters to the row's node_id and renders the filtered subarray in the order received.

I.9c self-review (Article VIII § 4)

The Constitution requires that every stream commit a self-review against the relevant Article II commitments and Article III quality criteria before persistence. This section records that review.

Article II — Architectural commitments

  • § 1 — Frame/Argument split. Honored. Every patch dispatched from src/ui/argument-running/ is a SessionPatch; no FramePatch is emitted from this module. The frame store is read-only here (consumed for parent-frame title in row breadcrumbs, drift comparison in the indicator, and frame-Authority listing in the authority-attachment section).
  • § 2 — Determinism. Honored. All output-viewer content is sourced from canonical selectors (selectOutputForView, selectStatusBadge, selectStatusSummary, selectInterviewItems, selectFrameVersionDrift). The UI does not call compute or computeInterviewOrder directly; it consumes the selectors that wrap them. The deterministic rankPremiseReuse is the v1 implementation for premise-reuse ranking; the function is pure and emits a stable order. The "Determinism — UI-layer discipline" section above enumerates the four D1–D4 obligations with their verification surfaces.
  • § 3 — Per-node satisfaction. Honored. selectStatusBadge is the UI's only path to render per-node satisfaction status; the badge is consumed in the interview-pane rows, in the path-overlay tab (via FrameCanvas's status_map prop), and in the decision-tree tab. The configurability of satisfaction criteria per node type is already a frame-side property (set in I.9b's frame-building UI) and is honored read-only here.
  • § 4 — Independent versioning. Honored. The FrameVersionDriftIndicator surfaces drift between the session's frame-version snapshot and the live frame version. No UI in this module couples the two streams' versions; the indicator is the only surface that even references both, and it reads each independently.
  • § 5 — Deterministic ordering. Honored. Section D3 above enumerates the five lists in this module that are upstream-ordered and not re-sorted by the UI.
  • § 6 — AI attribution. Honored. The two AI-emitting surfaces in this module (G6 prose rewrite, G11 premise drafting drawer) carry their attribution per section D2 above; the deterministic rankPremiseReuse emits no chip in v1 because it is not AI-derived; G7's v1.5 swap-in will carry the variant="ranked_by_llm" chip per the component spec.

Article III — Quality criteria

  • § 1 — Implementability. Every file in the file map carries a TypeScript signature and a behavior description sufficient for a coding agent to produce a first-pass implementation. No file is described in terms of "TBD" or "as needed."
  • § 2 — Testability. Every component file has a paired Vitest case in the test plan; the patch-shape contract is asserted explicitly against the state spec's SessionPatch union; determinism obligations are paired with specific test surfaces.
