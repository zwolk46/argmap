Stream G — LLM Hook Design

Companion to stream_a_resolutions_v1.html, stream_b_schema_v1.html, stream_c_runtime_v1.html, stream_d_modes_v1.html, stream_e_uiux_v1.html, and stream_h_persistence_v1.html  ·  Session date: May 10, 2026  ·  Version 1

This document specifies the application's LLM hook system end-to-end: the hook taxonomy and per-hook contracts, the common interfaces that every hook implements, the prompt-file format and storage path, the activation strategy (when each hook becomes live, opt-in defaults, mode/flavor visibility, graceful degradation), the determinism guarantees that govern how hooks coexist with Article II § 2, the user-confirmation contract for build-time hooks, and the testing strategy. It compiles directly against Stream A's A2 AI strategy decision (build-time AI suggestion-only / runtime LLM-free), Stream B's LlmSettings reservation on Frame and the LLM_HOOK candidate fields it identifies, Stream C's "LLM hook integration points" table fixing G1–G6, Stream D's candidate-surface additions (Premise reuse ranking, Conclusion direction suggestion, Position-table suggestion, frame-template selection ranking), Stream E's UI placement of suggestions and the "user always confirms" affordance, and Stream H's commitment of src/llm-hooks/prompts/ as the prompt-file storage path and the stub prompts IndexedDB store. Visual styling and key management are out of scope here.

How to read this

Conventions match the prior stream documents. TypeScript-flavored notation describes shapes and APIs; the actual implementation is plain TypeScript at the source level. Prompt bodies are markdown with YAML frontmatter; representative full prompts are written for two hooks (G1 and G3), and the remaining prompts are sketched at the same template level so an implementing agent can flesh them out without rediscovering structure. Hook ids G1–G6 are inherited from Stream C; G7–G10 number the Stream D candidate surfaces; G11–G12 number the Stream A runtime sidecar surfaces; G13 numbers the version change_summary hook flagged in Stream B.

Decisions made in this session

The following decisions are committed by this document. Each is a technical choice within agent discretion under Article V § 2(a)–(g); none reverses an architectural commitment under Article II or a prior stream's decision (verified in self-review below).

  1. Hook taxonomy: thirteen hooks across three activation modes. Build-time bake-and-commit (G1, G2, G3, G4, G5, G7, G8, G9, G10, G13), output-time opt-in (G6), and runtime sidecar (G11, G12). Build-time hooks bake their committed output into the FrameVersion at confirmation, after which subsequent runtime calls are LLM-free. Output-time produces a separate field while preserving the deterministic baseline. Runtime sidecar hooks are off by default in v1 per Stream B's runtime_hooks_enabled default and never touch routing.
  2. Uniform HookContract<TIn, TOut> interface. Every hook implements one shape: a typed input it consumes from frame/session state, a typed output it produces, a parser that validates the LLM response against the output schema, a fallback for parse failures and provider errors, and a commit function that bakes the user-confirmed output into the appropriate frame field(s). The contract makes hooks substitutable, testable in isolation, and trivially rewireable to new providers.
  3. Provider abstraction: LlmProvider interface, agnostic to vendor. A single interface (complete(prompt, schema): Promise<ParsedResponse>) sits between hooks and any LLM API. The v1 default implementation is Anthropic's Claude API (consistent with the user's working environment); an OpenAI implementation, a local-model implementation, and a deterministic mock implementation are all conformant. Hooks never import a vendor SDK directly.
  4. Prompt files are markdown with YAML frontmatter, stored at src/llm-hooks/prompts/<hook_name>/<version>.md. Frontmatter declares id, version, schema_in, schema_out, model_hint, and provider_hints; the body is the prompt template using {{variable}} Mustache-style interpolation. Versions are immutable filenames (v1.md, v2.md) so older FrameVersions whose HookInvocationRecords reference an old prompt remain reproducible. The prompts IndexedDB store from Stream H is the runtime cache, keyed by [hook_name+version] per H1.
  5. LlmSettings on Frame is refined from "opaque" to a concrete shape covering enabled-flags, prompt-version pinning per hook, calibrated thresholds (G5), and an append-only invocations: HookInvocationRecord[] log for provenance. This refines but does not amend Stream B B3, which explicitly reserved LlmSettings for Stream G ("opaque object in v1"). The shape is documented here; an implementing agent treats it as the canonical TypeScript definition.
  6. Provenance: every hook commit records a HookInvocationRecord on LlmSettings.invocations. Each record carries the hook id, prompt name+version, provider+model identifiers, input hash (for replay), the LLM's raw response, the parsed output, the user's commit decision (accepted / edited / rejected), the final committed value, and timestamps. The log is append-only; entries are never rewritten or removed in v1. This makes every AI-influenced field on a Frame auditably attributable.
  7. User-confirmation contract: every build-time hook produces a SuggestionResult the user must explicitly accept (optionally after editing) or reject before any frame field is written. The UI affordance (Stream E) is a three-button surface (Accept / Edit / Reject) with a visible AI-attribution chip; the contract is that no hook output reaches the FrameVersion without an explicit user act. Automatic-trigger hooks (G4, G7, G12, G13) still surface their output as a confirmable suggestion when the output writes to a versioned field; non-versioned outputs (e.g., G4 advisory warnings, G7 ranking) bypass confirmation because they do not modify the frame.
  8. Activation defaults: build-time hooks ON, runtime hooks OFF, output-time hooks ON-but-opt-in-per-invocation. Per Stream B's LlmSettings defaults (build_time_hooks_enabled: true, runtime_hooks_enabled: false). The user can disable all hooks per-frame and pin specific prompt versions for replay. Without an LLM provider configured, all hooks degrade silently: their UI affordances render disabled with a tooltip pointing to Settings.
  9. Determinism contract: build-time hooks are bake-and-commit, output-time hook (G6) preserves the deterministic baseline as a separate field, runtime sidecar hooks (G11, G12) are forbidden from writing to any field consumed by the Stream C compute pipeline. This is an enforcement of Article II § 2, not an addition to it. The runtime-module import boundary committed in Stream H (no src/runtime/ import from src/llm-hooks/) provides the structural guarantee; this document specifies the per-hook discipline.
  10. Testing strategy: every hook ships with a deterministic MockLlmProvider fixture that returns a hand-authored response per input; parse-failure tests assert the fallback path; round-trip tests assert that committing a hook output never makes the runtime pipeline non-deterministic. Hook tests live under tests/llm-hooks/; the iteration-order audit and frozen-clock discipline from Stream H apply to any code in src/llm-hooks/ that touches sorted iteration over frame data.

Common contracts

HookContract<TIn, TOut>

The single interface every hook implements. TIn is the typed input the hook consumes from frame/session state; TOut is the typed output the hook produces. The contract decouples the hook's role from any specific provider, prompt, or UI surface.

interface HookContract<TIn, TOut> { id: HookId // "G1" | "G2" | ... | "G13" name: string // "checkpoint_question_generation" etc. activation: "build_time" | "output_time" | "runtime_sidecar" trigger: "manual" | "automatic" | "manual_with_auto_offer" mode_visibility: ModeVisibility // see below buildInput(ctx: HookContext): TIn // pure: collects from frame/session renderPrompt(input: TIn, prompt: PromptFile): string parseOutput(raw: string, schema: JsonSchema): ParseResult<TOut> fallback(input: TIn, error: ParseError | ProviderError): FallbackResult<TOut> commit(output: TOut, ctx: HookContext): CommitPlan } interface HookContext { repository: Repository // Stream H frame_version: FrameVersion session?: ArgumentSession // only for argument-layer hooks selection?: NodeRef | EdgeRef // what the user has focused user_input?: unknown // e.g. pasted fact pattern for G11 } type ModeVisibility = { legal: boolean general: { personal: boolean; academic: boolean } } type ParseResult<T> = | { kind: "ok"; value: T } | { kind: "parse_error"; error: ParseError; raw: string } type FallbackResult<T> = | { kind: "deterministic_fallback"; value: T } // e.g., G7 default ranking | { kind: "advise_user"; message: string } // surface a non-blocking notice | { kind: "no_op" } // silently abort interface CommitPlan { writes: FrameFieldWrite[] // what fields to update on accept versioned: boolean // true if writes create a new FrameVersion }

LlmProvider interface

Hooks call out through this single interface. The implementer wires Anthropic Claude as the v1 default; OpenAI, Azure, local models, and the deterministic mock are all drop-in replacements.

interface LlmProvider { id: string // "anthropic" | "openai" | "mock" | ... complete(req: CompletionRequest): Promise<CompletionResponse> capabilities: ProviderCapabilities } interface CompletionRequest { prompt: string output_schema: JsonSchema // for providers supporting structured output model_hint?: string // from PromptFile frontmatter max_tokens?: number temperature?: number // default low (0.2) for build-time hooks } interface CompletionResponse { raw_text: string model_id: string // e.g. "claude-3.7-sonnet-2026-02" finish_reason: "stop" | "length" | "error" token_usage?: { input: number; output: number } } interface ProviderCapabilities { structured_output: boolean // schema-enforced; if false, hooks parse-then-validate streaming: boolean tool_use: boolean // reserved; v1 hooks do not use tools }

Build-time hooks default to temperature: 0.2 for stability. The mock provider is constructed with a per-test response map keyed by prompt-hash and is the testing harness for deterministic hook tests.

SuggestionResult and the confirmation flow

Every build-time hook produces a SuggestionResult<TOut> that the UI surface consumes. The UI affordance (Stream E) renders the suggestion with three actions; until one is taken, no frame field is written.

type SuggestionResult<TOut> = { hook_id: HookId prompt_version: string provider_id: string model_id: string input_hash: string // SHA-256 of the canonicalized input raw_response: string // retained for audit parsed: TOut // or fallback's value parse_status: "ok" | "fallback" | "error" generated_at: timestamp } type ConfirmationDecision<TOut> = | { kind: "accepted"; final: TOut } | { kind: "edited"; final: TOut; original: TOut } | { kind: "rejected" }

HookInvocationRecord

Every commit produces an audit record that lives on LlmSettings.invocations on the Frame. The record is the audit trail referenced in Article III § 3.

interface HookInvocationRecord { id: string // UUID v4 hook_id: HookId prompt_name: string prompt_version: string provider_id: string model_id: string input_hash: string raw_response?: string // optional; large responses may be omitted by config decision: "accepted" | "edited" | "rejected" final_value?: unknown // what was actually written; absent if rejected target_node_ids?: NodeRef[] // which nodes were affected target_field_paths?: string[] // e.g. ["question", "options[0].label"] invoked_at: timestamp committed_at?: timestamp // absent if rejected }

PromptFile schema

The on-disk and in-IndexedDB shape of a prompt. Filesystem files at src/llm-hooks/prompts/<hook_name>/<version>.md are parsed into this shape on app start (or on first hook invocation) and cached in the prompts IndexedDB store from Stream H1, keyed by [hook_name+version].

interface PromptFile { hook_name: string // must match the hook's contract.name version: string // e.g. "v1", "v2"; immutable once published schema_in: JsonSchema // describes TIn schema_out: JsonSchema // describes TOut model_hint?: string // e.g. "claude-3.7-sonnet" or "any" provider_hints?: { [provider_id: string]: object } body: string // markdown template with {{variable}} interpolation created_at: timestamp notes?: string // human-readable changelog vs. prior version }

Prompt file format

A prompt file is markdown with YAML frontmatter. Variable interpolation is Mustache-style ({{variable_name}}). The body addresses the LLM directly; structured output is requested explicitly with the JSON schema embedded in the prompt body for providers without native schema support.

--- hook_name: checkpoint_question_generation version: v1 schema_in: type: object required: [parent_subquestion, sibling_checkpoints, frame_mode] properties: parent_subquestion: { type: object } sibling_checkpoints: { type: array } frame_mode: { enum: ["legal", "general"] } frame_flavor: { enum: ["personal", "academic", null] } frame_jurisdiction: { type: ["string", "null"] } schema_out: type: object required: [question, options] properties: question: { type: string, minLength: 5 } options: type: array minItems: 2 maxItems: 6 items: type: object required: [label] properties: label: { type: string, minLength: 1 } target_hint: { type: ["string", "null"] } model_hint: claude-3.7-sonnet --- You are helping a {{frame_mode}} argument-mapper draft a Checkpoint question that resolves under the parent SubQuestion below. A Checkpoint is a single yes/no, multiple-choice, or graded question whose answer routes the argument forward. Parent SubQuestion: {{parent_subquestion.statement}} {{#if sibling_checkpoints.length}} Sibling Checkpoints under the same SubQuestion: {{#each sibling_checkpoints}} - {{this.question}} {{/each}} {{/if}} {{#if frame_jurisdiction}} Jurisdiction: {{frame_jurisdiction}}. {{/if}} Constraints: - The question must ask exactly ONE thing. If you find yourself wanting to write "X and Y" or "X, including Y," propose two Checkpoints instead and pick the most central one for this slot. - Phrase the question in the idiom of the {{frame_mode}} domain. - Each option must be a short label (a few words at most). - Do not invent target node ids; leave target_hint as null unless a sibling Checkpoint's question is structurally an answer to a specific option, in which case use that sibling's id. Return ONLY a JSON object matching this shape: { "question": string, "options": [ { "label": string, "target_hint": string | null }, ... ] }

The same template structure (frontmatter + addressed-to-LLM body + structured-output instruction) applies to every hook. The remaining hooks' bodies are sketched in their cards below; an implementing agent fleshes them out by following the same pattern.

LlmSettings refinement (refines Stream B B3's "opaque object")

Stream B B3 reserves LlmSettings on Frame as an opaque object pending Stream G. This document fixes its shape. The refinement is non-amending: the field's existence, optionality, and v1 default behavior are unchanged from Stream B; only its internal structure is now defined.

interface LlmSettings { build_time_hooks_enabled: boolean // default true (per Stream B) runtime_hooks_enabled: boolean // default false (per Stream B) output_time_hooks_enabled: boolean // default true; G6 still requires user click prompt_versions?: { [hook_name: string]: string } // pin a specific prompt version per hook per_hook_enabled?: { [hook_name: string]: boolean } // per-hook override of category default calibrated_thresholds?: BurdenThresholdMap // G5 output: baked numeric thresholds invocations: HookInvocationRecord[] // append-only audit log provider_id?: string // "anthropic" | "openai" | "mock" | ... } interface BurdenThresholdMap { preponderance: number clear_and_convincing: number beyond_reasonable_doubt: number scintilla: number substantial_evidence: number source: "default" | "g5_calibrated" // audit which table is in effect calibrated_from?: { standard_of_review: string; calibrated_at: timestamp } }

Defaults: when LlmSettings is absent on a Frame, the system behaves as if build_time_hooks_enabled: true, runtime_hooks_enabled: false, output_time_hooks_enabled: true, invocations: [], no version pins, no calibration override, and provider_id read from an app-level default in app_state. The Frame settings UI (Stream E E2) exposes per-frame overrides under "AI Settings."

Per-hook specifications

Each hook below is specified with the same fields: surface (where it lives in the UI), trigger, mode/flavor visibility, input shape, output shape, parser rules, fallback behavior, and commit semantics. Two prompt bodies (G1 above, G3 below) are written in full as exemplars; the rest specify the prompt's intent and key constraints, and the implementing agent writes the body following the same pattern.

G1 — Checkpoint question generation

build-time · manual · all modes/flavors · ships v1

Helps the user draft a Checkpoint's question text and answer options when adding a new Checkpoint or refining an existing one. Covers the three sub-surfaces named in A2 ("Checkpoint question wording, answer-option suggestions, routing suggestions") in a single hook because they share an input and produce a coherent output.

Surface and trigger

An "✦ Suggest Checkpoint" button in the Checkpoint inspector (Stream E E2). The button is disabled if the Checkpoint has no parent SubQuestion (a structural prerequisite). Clicking opens a SuggestionResult drawer with the three-button affordance.

Input

type G1Input = { parent_subquestion: { id: string; statement: string } sibling_checkpoints: { id: string; question: string }[] frame_mode: "legal" | "general" frame_flavor: "personal" | "academic" | null frame_jurisdiction: string | null }

Output

type G1Output = { question: string options: { label: string; target_hint: NodeRef | null }[] }

Parser rules

JSON parse, then schema validation. Reject if question is empty, contains the conjunctions " and "/" plus " at top level (compound-question heuristic from Stream C C5), or if options.length < 2 or > 6. target_hint, when present, must reference a sibling Checkpoint id from the input (otherwise null).

Fallback

On parse failure: return { kind: "advise_user", message: "AI suggestion was malformed; try again or write the question directly." }. The drawer surfaces the message and remains open. On provider error: same advice with provider-specific detail.

Commit

On accept (or edit-then-accept), writes question and options on the Checkpoint and creates a new FrameVersion via the standard auto-save path. target_hint values are written to the corresponding options[i].target_node_id; null hints leave the option's target unset (the user resolves later via the canvas connector). Records a HookInvocationRecord on LlmSettings.invocations.

Prompt

See the full body in "Prompt file format" above; G1 is the exemplar.

G2 — Interpretation suggestion

build-time · manual · all modes/flavors · ships v1

Helps the user populate Interpretations under a Term, with citation hints in legal mode. The hook proposes Interpretation statements; the user accepts, edits, or rejects each candidate independently (so a single suggestion may produce multiple Interpretation creations).

Surface and trigger

An "✦ Suggest Interpretations" button in the Term editor's Interpretation list (Stream E E2). Disabled if the Term has no name. The drawer shows a checklist of N candidates; the user selects which to commit (defaulting to all).

Input

type G2Input = { term: { id: string; name: string } root_question: { statement: string } ancestor_subquestions: { statement: string }[] // path from root to this Term's parent existing_interpretations: { id: string; statement: string }[] frame_mode: "legal" | "general" frame_flavor: "personal" | "academic" | null frame_jurisdiction: string | null }

Output

type G2Output = { interpretations: { statement: string citation_hint?: { authority_label: string; citation_string: string } // legal mode only rationale?: string // short explanation, shown in drawer; not committed }[] }

Parser rules

JSON parse + schema. Each statement must be non-empty and must not duplicate an existing Interpretation's statement (case-insensitive equality). At most 6 candidates. In legal mode, citation_hint is permitted; the prompt instructs the model to use only well-known case/statute citations and to omit the hint rather than invent one. Reject any candidate whose citation_hint.citation_string contains placeholder markers like "[citation needed]", "e.g.", or unbalanced brackets.

Fallback

On parse failure: advise_user. On provider error: same. The hook does not have a deterministic fallback because the input space is too open.

Commit

For each accepted candidate: create a new Interpretation node and an INTERPRETED_AS edge from the Term to it; if a citation_hint is present and the user accepts it, create a new Authority node (or reuse a matching existing one by case-insensitive label match) and a CITES edge from the Interpretation to it. All node and edge creations are written in a single FrameVersion auto-save tick.

Prompt sketch

System framing: legal/academic argument mapper drafting Interpretations. Inputs interpolated as the parent path (root → ancestor SubQuestions → Term). Constraints: each Interpretation must be a candidate reading of the Term, not a description of the Term; must be distinct from existing Interpretations; in legal mode, citations only when the model has high confidence in their accuracy; in personal flavor, no citations expected (Authority is hidden by default per Stream A A8/A4).

G3 — Conclusion reasoning_summary

build-time · manual_with_auto_offer · all modes/flavors · ships v1

Drafts the optional reasoning_summary field on a Conclusion (Stream B B1.7). The summary is a short paragraph explaining how the active path arrives at the Conclusion's direction; it complements but does not replace the deterministic prose summary from C4.

Surface and trigger

Two activation paths. (1) Manual: an "✦ Suggest summary" button on the reasoning_summary field in the Conclusion inspector. (2) Auto-offer: when a Conclusion is saved with reasoning_summary empty and the user has build_time_hooks_enabled, a non-blocking toast offers to draft one. The user clicks once to invoke; the suggestion still requires explicit accept/edit/reject.

Input

type G3Input = { conclusion: { id: string; statement: string; direction: ConclusionDirection } incoming_path: PathNode[] // from RootQuestion to this Conclusion via active selections frame_mode: "legal" | "general" frame_flavor: "personal" | "academic" | null frame_jurisdiction: string | null positions?: Position[] // general mode } type PathNode = { type: "RootQuestion" | "SubQuestion" | "Term" | "Interpretation" | "Checkpoint" | "LogicalGate" text: string // statement / question / name / interpretation, as appropriate resolution?: string // e.g. selected Interpretation, Checkpoint answer }

Output

type G3Output = { reasoning_summary: string // 1-3 sentences typical, max 120 words }

Parser rules

JSON parse + schema. Reject if empty, longer than 120 words, or if it contains a future-tense legal predictive claim (heuristic: "will likely," "is expected to") in legal mode (predictions are out of scope for a reasoning summary).

Fallback

On parse failure: advise_user. On provider error: no_op with a quiet console log; the auto-offer toast is suppressed for the rest of the session.

Commit

On accept, writes conclusion.reasoning_summary. Single-field write; auto-save tick advances the FrameVersion. Provenance recorded as usual.

Prompt

--- hook_name: conclusion_reasoning_summary version: v1 schema_in: { ... see G3Input above ... } schema_out: type: object required: [reasoning_summary] properties: reasoning_summary: { type: string, minLength: 10, maxLength: 800 } model_hint: claude-3.7-sonnet --- You are summarizing the reasoning that links a chain of resolved questions to a Conclusion in a {{frame_mode}} argument map. Conclusion statement: {{conclusion.statement}} Conclusion direction: {{conclusion.direction.value}}{{#if conclusion.direction.custom_label}} ({{conclusion.direction.custom_label}}){{/if}} Path from the root question to this Conclusion (in order): {{#each incoming_path}} - [{{this.type}}] {{this.text}}{{#if this.resolution}} → {{this.resolution}}{{/if}} {{/each}} Constraints: - Write 1 to 3 sentences (max 120 words). - Describe the path, not the validity of the path. The structural validity is verified separately by the application. - Use the {{frame_mode}} idiom. In legal mode, follow IRAC-adjacent prose; do not predict outcomes ("the court will likely") and do not hedge with confidence statements ("with high probability") — confidence is computed and displayed separately by the application. - Do not introduce facts not in the path. Return ONLY a JSON object: { "reasoning_summary": string }

G4 — Gap detection

build-time · automatic · all modes/flavors · ships v1

Stream C C5's compound-checkpoint warning is intentionally fuzzy ("a language judgment, not a structural one") and a precision upgrade was reserved as G4. The hook also extends to other gap-detection-flavored cases: missing branches under multi-option Checkpoints, missing Interpretations under Terms, Conclusions whose direction doesn't match the apparent thrust of the path. Outputs are advisory ValidationResult[] with severity "warning"; no field on the frame is rewritten.

Surface and trigger

Runs automatically as part of every C5 validation pass when build_time_hooks_enabled is true. Its outputs append to the validation drawer (Stream E E2) under a labeled "AI advisories" section so the user can distinguish them from structural rules.

Input

type G4Input = { frame_version: FrameVersion // full snapshot structural_validation_results: ValidationResult[] // from C5 deterministic rules frame_mode: "legal" | "general" frame_flavor: "personal" | "academic" | null }

Output

type G4Output = { advisories: ValidationResult[] // each with severity: "warning" }

Each advisory carries the same shape as Stream C's ValidationResult: rule_id (prefixed "AI-", e.g., "AI-COMPOUND-CHECKPOINT"), severity: "warning", node_ids, message, optional suggested_fix.

Parser rules

JSON parse + schema. Each advisory's node_ids must reference nodes that exist in the FrameVersion (else discard). The hook caps at 8 advisories per pass to bound noise.

Fallback

On parse failure or provider error: no_op. The deterministic structural rules continue to surface unaffected. A single subdued console line records the failure for debugging.

Commit

None — advisories are not stored on the frame; they are recomputed each validation pass. The hook still records a HookInvocationRecord on LlmSettings.invocations with final_value: undefined and decision: "accepted" (the user implicitly accepts by viewing) so audit trails can show "AI ran on FrameVersion N and produced these advisories." For storage economy, the raw_response field may be omitted from G4 records by configuration.

Prompt sketch

System framing: reviewer of a draft argument map looking for compound questions, missing branches, dangling Terms, and direction mismatches. Inputs: the full FrameVersion (or a structural digest for very large frames; the digest format is a v1.5 optimization). Output: an array of structured advisories with the AI- rule_id prefix, capped at 8.

G5 — Standard-of-review threshold calibration

build-time (calibration baked) · manual (with auto-offer) · legal only · ships v1

Per Stream C, the burden_met evaluation is categorical with a fixed scoring table; G5 is reserved as a calibration override. The hook reads the Frame's standard_of_review (e.g., "de novo," "abuse of discretion," "clear error") and proposes adjusted BURDEN_THRESHOLD values appropriate to that standard. The output is committed as LlmSettings.calibrated_thresholds; subsequent runtime calls read these as fixed inputs to Phase 4 (the runtime never invokes G5).

Surface and trigger

Manual: a "Calibrate from standard of review" button in Frame settings under "Burden thresholds." Auto-offer: when the user first sets a non-default standard_of_review on a Question/SubQuestion, a single inline coachmark suggests calibrating; dismissible per usual rules.

Input

type G5Input = { standard_of_review: string // e.g. "de_novo", "abuse_of_discretion" current_thresholds: BurdenThresholdMap // the table currently in effect (default or prior G5) frame_jurisdiction: string | null }

Output

type G5Output = { thresholds: BurdenThresholdMap // proposed; "source" set to "g5_calibrated" rationale: string // short paragraph for the confirmation drawer }

Parser rules

Each numeric threshold must be in range [0, 100] (the scoring table's units, per Stream C). The five named burden levels must all be present. Reject if any value is outside range or any level is missing.

Fallback

On parse failure: advise_user. On provider error: no_op; the user can still hand-edit thresholds in the same Frame-settings panel as a deterministic fallback.

Commit

Writes LlmSettings.calibrated_thresholds on the Frame. This does create a new FrameVersion (because LlmSettings sits on Frame, not on FrameVersion — but for runtime determinism the calibrated thresholds must be visible to compute). Resolution: the FrameVersion includes a llm_settings_snapshot projection that copies calibrated_thresholds at version-creation time. This is a small extension to Stream B's FrameVersion shape; see "Persistence and schema updates" at the end of this document for the precise change and its review-flag.

After commit, every subsequent Phase 4 status determination reads the calibrated thresholds via the FrameVersion snapshot. The runtime pipeline does not invoke any LLM and does not re-call G5; the hook's effect is fully baked.

Prompt sketch

System framing: a legal-domain expert mapping appellate review standards to relative weight thresholds for the application's burden_met scoring. Inputs interpolated. Constraints: outputs must respect ordinal monotonicity (scintilla < preponderance < clear_and_convincing < beyond_reasonable_doubt on threshold magnitude); substantial_evidence calibrated against the named standard of review when a literature consensus exists, otherwise left at the default.

G6 — Prose rewrite (fluency)

output-time · manual · all modes/flavors · ships v1

The deterministic prose summary from C4 is a template walk and reads as such. G6 rewrites it for fluency, opt-in per invocation. The deterministic baseline is preserved on the session so audits and replays remain stable.

Surface and trigger

An "✦ Rewrite for fluency" button in the prose tab of the Argument Running output viewer (Stream E E3). Each click invokes a fresh rewrite; the user can keep, copy, or discard the rewrite. The deterministic version is always one click away (a "Show baseline" toggle).

Input

type G6Input = { baseline_prose: string // the deterministic ConditionalOutput.prose_summary conclusion_direction?: ConclusionDirection frame_mode: "legal" | "general" frame_flavor: "personal" | "academic" | null }

Output

type G6Output = { rewritten_prose: string }

Parser rules

JSON parse + schema. Reject if the rewrite is more than 1.5× the baseline length (catches expansion that introduces material), or if it contains conditional-introduction phrases the baseline does not (catches reasoning hallucinations).

Fallback

On parse failure or provider error: advise_user; the baseline remains visible. The hook does not modify session state on failure.

Commit

On accept, writes session.output_overrides.rewritten_prose on the ArgumentSessionVersion (this is a session-level field, not a frame-level one — the rewrite is session-scoped because it derives from the session's specific Premises). Stream B B4's ArgumentSessionVersion schema is extended with an optional output_overrides: { rewritten_prose?: string; rewritten_at?: timestamp; rewritten_by_hook?: HookInvocationRecord["id"] } field; see "Persistence and schema updates" below.

The deterministic baseline is always available as the canonical ConditionalOutput.prose_summary; the rewrite is a parallel field.

Prompt sketch

System framing: legal/academic editor rewriting a deterministic prose summary for fluency. Constraints: do not add reasoning, conditions, citations, or claims not in the baseline; preserve every conditional ("if/then") structure; preserve the conclusion direction; output only the rewritten paragraph(s).

G7 — Premise reuse ranking

build-time of session · automatic · all modes/flavors · ships v1.5 (v1 ships heuristic)

Stream D D2 specifies that Premise reuse is supported and suggested; v1 ranks suggestions by text similarity and recency (deterministic). G7 layers an LLM-based relevance ranking that considers Premise kind, the target Checkpoint's question, and the frame's path context.

Surface and trigger

Inline reuse suggestions in the Premise authoring field (Stream E E3, line 269). Triggered automatically when the user begins typing a Premise. The hook's output is a re-ordered candidate list; the user picks one or continues typing to author a new Premise.

Input

type G7Input = { draft_text: string // what the user has typed so far target_checkpoint: { id: string; question: string; answer_type: string } candidate_premises: { id: string; statement: string; kind: string }[] // pre-filtered by similarity active_path_context: PathNode[] frame_mode: "legal" | "general" frame_flavor: "personal" | "academic" | null }

Output

type G7Output = { ranking: { premise_id: string; score: number }[] // score in [0, 1] }

Parser rules

JSON parse + schema. Every premise_id must come from the input candidate_premises. Scores must be numeric. Re-ordering does not require all candidates to be returned; missing ids are appended at the end of the displayed list at score 0.

Fallback

On parse failure or provider error: deterministic_fallback returns the input ordering (i.e., the v1 text-similarity-and-recency ordering). Failure is silent; the user sees suggestions ordered by the v1 heuristic without knowing the LLM was attempted.

Commit

None — ranking is a presentation effect. No frame field is written. A HookInvocationRecord is logged with decision: "accepted" and final_value: undefined; raw_response may be omitted by configuration.

Prompt sketch

System framing: legal/academic argument-mapper assistant choosing which existing premises are most relevant to the user's draft. Inputs interpolated. Constraints: rank by relevance to the target Checkpoint's question, with kind-vocabulary fit and active-path coherence as secondary factors; never fabricate premise ids; ties broken by input order.

G8 — Conclusion direction suggestion

build-time · manual · all modes/flavors · ships v1

Suggestion chips on the Conclusion direction picker (Stream E E2). Given the Conclusion's statement and the path leading to it, suggest a direction value (legal mode: from the fixed enum; general mode: from the Frame's positions table or a "custom" value).

Surface and trigger

Manual: clicking the "✦ Suggest" chip on the direction picker. Available only when the Conclusion has a statement set and (in general mode) the Frame has at least one Position defined.

Input

type G8Input = { conclusion_statement: string incoming_path: PathNode[] frame_mode: "legal" | "general" positions?: Position[] // general mode frame_jurisdiction: string | null }

Output

type G8Output = { direction: ConclusionDirection // per Stream B B1.7's discriminated union rationale?: string // shown in chip tooltip; not committed }

The output's direction shape must match the Frame's mode. Legal mode: kind: "legal" with value from the fixed enum (or "custom" with custom_label). General mode: kind: "general" with position_id referencing the input positions[] (or "custom"). Reject if the shape mismatches.

Fallback

On parse failure or provider error: advise_user. The picker remains untouched.

System framing: legal/academic reviewer suggesting a Conclusion direction. Inputs interpolated. G9 — Position-table suggestion

build-time · manual · general only · ships v1

General-mode Frames require a positions table (Stream B; Stream D D2). G9 helps populate it given a topic prompt. Particularly useful at frame creation when the user types a topic and gets back a suggested set of Positions like "for X" / "against X" / "alternative" / etc.

Surface and trigger

A "✦ Suggest positions" button in the new-frame wizard (Stream E E7) and in Frame settings under "Positions" for existing frames.

type G9Input = { topic: string // from the wizard's title/description, or user-entered frame_flavor: "personal" | "academic" existing_positions: Position[] // for the in-settings case }

Output

type G9Output = { positions: { label: string; description?: string }[] // 2-6 suggestions }

2 ≤ positions.length ≤ 6. Each label must be non-empty and not duplicate an existing position's label (case-insensitive).

Fallback

On parse failure or provider error: advise_user. The wizard's manual entry path remains available.

Prompt sketch

System framing: academic-or-personal argument-mapper suggesting positions for a topic. Constraints: 2–6 positions, mutually distinguishable, written as short noun phrases, neutral framing where possible.

G11 — Premise drafting from fact pattern

runtime sidecar · manual · all modes/flavors · ships v1, OFF by default

Stream A A2 explicitly authorized this surface: "drafting Premise text from a fact pattern the user pastes." The user pastes a passage of facts (case excerpt, statute language, deposition fragment, news clip), the hook drafts candidate Premises grounded in the passage; the user accepts, edits, or rejects each.

Surface and trigger

A "✦ Draft from text" button in the Premise authoring drawer during Argument Running (Stream E E3). Disabled when runtime_hooks_enabled is false (the v1 default). The button is also gated behind a per-frame opt-in confirmation the first time it is used (a single dialog explaining that the hook reads the user's pasted text and that the runtime routing remains LLM-free regardless of this hook's output).

Input

type G11Input = { fact_pattern_text: string // what the user pasted target_checkpoint: { id: string; question: string } frame_mode: "legal" | "general" frame_flavor: "personal" | "academic" | null premise_kind_vocabulary: string[] // per A11, mode/flavor-aware }

Output

type G11Output = { draft_premises: { statement: string kind: string // must be in premise_kind_vocabulary source_quote?: string // short verbatim excerpt from the input, optional }[] }

Parser rules

Each kind must come from the input vocabulary. Each source_quote, if present, must be a substring of the input fact_pattern_text (verbatim; this is checked deterministically, not heuristically). Reject any candidate whose source_quote is not a substring.

Fallback

On parse failure or provider error: advise_user. The drawer remains open; the user can author manually.

Commit

For each accepted candidate: create a Premise with the given statement and kind on the ArgumentSession; the source_quote, if accepted, is stored on the Premise's source field (Stream B's Premise.source) prefixed with "Excerpt:". The Premise's eventual contribution to routing flows through the deterministic Stream C pipeline; the LLM does not affect routing in any way.

Prompt sketch

System framing: legal/academic researcher drafting Premises from a passage. Constraints: each Premise must be supportable by the passage; no claims beyond what the passage says; kind from the supplied vocabulary; source_quote only when a single short verbatim segment grounds the Premise.

G12 — Cross-implication advisory notes

runtime sidecar · automatic · all modes/flavors · ships v1.5, OFF by default

Stream A A2 also authorized "flagging cross-implications as advisory notes." During Argument Running, given the active path and the Premises attached so far, the hook surfaces non-blocking notes when a Premise might cut against a Checkpoint elsewhere in the frame, when a selected Interpretation has analogues in another path with conflicting weight, or similar. The notes are advisory — they never alter routing or status.

Surface and trigger

Automatic during Argument Running, gated by runtime_hooks_enabled. Notes render as small "ⓘ AI advisory" chips on the implicated nodes in the canvas and as entries in a "Cross-implications" tab in the bottom panel of the Argument Running screen.

Input

type G12Input = { frame_version: FrameVersion // or a digest projection for large frames (v1.5) session_state: ArgumentSessionVersion recent_change: { kind: "premise_added" | "interpretation_selected" | "checkpoint_answered"; node_id: string } }

Output

type G12Output = { advisories: { implicated_node_ids: NodeRef[] message: string severity: "info" | "warning" // always "info" or "warning"; never blocks }[] }

Parser rules

implicated_node_ids must reference nodes in the FrameVersion. Severity must be one of the two allowed values. Cap at 3 advisories per invocation.

Fallback

On parse failure or provider error: no_op.

Commit

None — advisories are session-scoped and ephemeral; they re-derive on each invocation. Critically: nothing G12 produces is consumed by the Stream C compute pipeline. The runtime-module import boundary (Stream H determinism contract) makes this enforceable in CI.

Prompt sketch

System framing: assistant flagging cross-implications in an active argument map. Constraints: only advisory; severity capped at "warning"; do not produce advisories that would change routing decisions if the user acted on them; do not invent node ids.

G13 — Version change_summary generation

build-time · automatic · all modes/flavors · ships v1

Stream B B4 marks FrameVersion.change_summary as an LLM_HOOK candidate. G13 fills it for non-trivial saves (milestone saves and auto-saves whose structural diff exceeds a small-edit threshold). The deterministic alternative — the structural diff from Stream H H3 — is always available; G13's value is producing a one-line natural-language gloss like "Added two Interpretations under Negligence and connected them to the duty Checkpoint."

Surface and trigger

Fully automatic when build_time_hooks_enabled. Triggered on milestone saves and on any auto-save whose StructuralDiff (Stream H H3) shows ≥3 non-layout entries. Produces a string written directly to the FrameVersion's change_summary field; no user confirmation drawer (the field is otherwise empty by default and free-form, so AI-generated content is non-destructive).

Input

type G13Input = { diff: StructuralDiff // from H3 parent_version_change_summary?: string // for chronological context frame_mode: "legal" | "general" }

Output

type G13Output = { change_summary: string // 1 sentence, max 30 words }

Parser rules

Length ≤ 30 words. Reject if it contains a node id (heuristic: substring match against any id in the diff) — IDs in summaries are noise.

Fallback

On parse failure or provider error: deterministic_fallback returns the StructuralDiff's first-line summary verbatim ("Added 2 nodes, removed 1 edge.") so the field is never empty. Silent.

Commit

Writes frame_version.change_summary. The write happens inside the same Dexie transaction as the FrameVersion save itself (Stream H atomicity), so the summary is never out of sync. Provenance recorded as usual.

Prompt sketch

System framing: assistant writing a one-sentence summary of a structural diff. Constraints: 1 sentence, max 30 words, no ids, plain language, named entities (Term names, Checkpoint questions) from the diff are fine.

Activation strategy

v1 ship list

Hooks tagged "ships v1" in the taxonomy table are part of the v1 implementation: G1, G2, G3, G4, G5, G6, G8, G9, G11, G13. Hooks tagged "v1.5" (G7, G10, G12) ship their interfaces and deterministic fallbacks in v1; the LLM body is implemented when the implementing agent prioritizes it. The v1 ship list represents roughly 80% of the hooks' direct user value with substantially less implementation cost than the full set, since several v1.5 hooks (G7 ranking, G10 ranking) are improvements over already-acceptable deterministic baselines.

Opt-in defaults

From Stream B B3, refined here:

  • Build-time hooks: enabled by default. The user encounters them as opt-in moments at the suggestion drawer. Per-frame override is available in Frame settings → AI Settings.
  • Output-time hooks (G6 only): enabled by default, but each invocation requires a user click. The hook never produces unsolicited output.
  • Runtime sidecar hooks (G11, G12): disabled by default in v1. The user opts in per-frame in Frame settings; G11 also has a one-time per-frame confirmation dialog the first time the user invokes it (because the hook reads pasted user text).

Without an LLM provider configured, all hooks degrade silently: their UI affordances render disabled with a tooltip linking to Settings → AI Provider. The application is fully usable without any LLM provider.

Mode and flavor visibility

Per Stream D D4/D5, certain surfaces are mode- or flavor-restricted:

  • G5 (standard-of-review calibration): legal only. Hidden in general mode.
  • G9 (position-table suggestion): general only. Hidden in legal mode.
  • G2 (Interpretation suggestion): citation_hint output is gated on legal mode; in personal flavor (where Authority is hidden by default per A8/A4), the hook does not produce citation_hint entries.
  • All other hooks: all modes and flavors.

Per-hook enablement and version pinning

LlmSettings.per_hook_enabled and LlmSettings.prompt_versions let the user override the category default per hook and pin a specific prompt version per hook, respectively. The Frame-settings UI surfaces these as a per-hook table (hook name, enabled toggle, prompt version dropdown). Pinning is most useful for replay: when an old FrameVersion's HookInvocationRecord references prompt v1 but the application is now shipping v2, the user can pin the Frame to v1 to reproduce the historical behavior.

Provider configuration

The provider is selected at the app level (in the global Settings panel; not per-frame for v1). The default is Anthropic Claude; OpenAI is the second supported implementation. Mock is the test-only provider, never selectable at runtime by users. Provider selection is stored on app_state (singleton record per Stream H1). API keys are stored separately, by browser-native means appropriate to the host (e.g., the implementing agent picks between IndexedDB, browser keychain integration, or a small in-memory paste-on-each-session pattern; the choice is out of scope for Stream G).

Determinism guarantees

Article II § 2 controls. The contract is honored at three levels:

Module boundary (structural)

Per Stream H's determinism implementation contract, src/runtime/ imports nothing from src/llm-hooks/. ESLint no-restricted-imports enforces this in CI. Therefore no LLM, no provider call, no prompt loader can run from inside the runtime pipeline. This is a structural guarantee — independent of any hook author's discipline — that the routing computation is LLM-free.

Per-hook discipline

  • Build-time bake-and-commit hooks (G1, G2, G3, G5, G8, G9, G13): on accept, the hook's output becomes a fixed field on the FrameVersion (or, for G5, on the FrameVersion's llm_settings_snapshot projection). Subsequent runtime calls read these fields as ordinary inputs. The hook is not invoked at runtime; the hook's prior output cannot vary across runtime calls because the FrameVersion is immutable.
  • Build-time hooks that don't write to FrameVersion (G4, G7, G10): outputs are presentational only (validation drawer entries, ranking orders). They do not feed the runtime pipeline. A change in their output between two invocations does not change compute()'s result for any FrameVersion + ArgumentSession pair.
  • Output-time hook (G6): writes to session.output_overrides.rewritten_prose, a session-level field that is never read by compute(). The deterministic ConditionalOutput.prose_summary is the canonical output; the rewrite is a parallel field surfaced only at the prose tab.
  • Runtime sidecar hooks (G11, G12): G11 writes to Premise.source only when the user accepts a draft; the Premise itself flows through Stream C's deterministic pipeline as any other Premise. The LLM does not influence which routing the Premise causes. G12 produces only ephemeral session-scoped advisories that nothing in the compute pipeline reads.

Audit (provenance)

Every hook commit produces a HookInvocationRecord on LlmSettings.invocations with input_hash and raw_response. A FrameVersion can be replayed: load the FrameVersion's llm_settings_snapshot, fetch the matching PromptFile from the prompts store by [hook_name+prompt_version], hash the recorded input, and compare. If anyone disputes whether a given Conclusion's reasoning_summary was AI-generated and which prompt produced it, the record gives a definitive answer. Article III § 3 (auditability) is satisfied.

UI confirmation contract

Stream E E2 and E3 specify the UI placement of suggestion affordances. Stream G specifies the contract those affordances must honor.

The three-button affordance

Every build-time hook that writes to a FrameVersion field exposes a SuggestionResult in a drawer with three actions:

  • Accept — commits the suggestion verbatim. Records HookInvocationRecord with decision: "accepted".
  • Edit — opens the field for inline editing pre-filled with the suggestion. The user's edited value is what gets committed. Records HookInvocationRecord with decision: "edited" and both final_value (committed) and the original parsed output (in the record's audit projection).
  • Reject — closes the drawer, no field is written. Records HookInvocationRecord with decision: "rejected" and no final_value.

AI attribution

Fields whose current value was committed via a hook show a small "✦" attribution chip in the inspector when selected, with a tooltip showing the hook id, prompt version, and timestamp. The chip is informational and does not constrain editing — the user can hand-edit the field as usual after committing AI output, which records a separate provenance entry (a HookInvocationRecord with decision: "edited" on the next accept, or simply the user's manual edit in the FrameVersion diff with no hook record at all).

Bypassing confirmation (auto-trigger hooks)

Hooks that don't modify a versioned field (G4 advisories, G7/G10 ranking, G12 advisories) bypass the confirmation drawer because there is nothing to commit. Their outputs render directly in their UI placement (validation drawer, ranking lists, advisory tabs). G13 also bypasses confirmation, but for a different reason: it writes to change_summary, a field whose default value is empty and whose human alternative is also empty for the vast majority of saves; an automatic AI fill is non-destructive and the user can always overwrite it later in the version-history UI (Stream E E5 reserves the affordance).

Testing strategy

MockLlmProvider

A test-only provider implementing LlmProvider. Constructed with a Map<promptHash, response>. Returns the matching response for a given prompt hash; throws "unexpected prompt" for any unmatched call. Sufficient to make every hook test deterministic.

Per-hook tests

Every hook ships with these tests under tests/llm-hooks/<hook_name>/:

  • Happy-path test: input fixture → mocked LLM response → expected parsed output → expected CommitPlan. One per hook.
  • Parser-rejection tests: for each parser rule (e.g., G1's compound-question heuristic, G11's source_quote-must-be-substring rule), a fixture that violates the rule and asserts the hook returns the appropriate fallback.
  • Provider-error test: mocked provider returns an error; assert the hook produces the documented fallback.
  • Round-trip determinism test: run the hook on a fixture, accept the suggestion, then run compute() on the resulting FrameVersion under vi.useFakeTimers() with two different frozen times. Assert byte-identical compute outputs. This guards against any hook accidentally introducing time-dependent state into the FrameVersion.
  • Provenance test: assert that after each hook decision, LlmSettings.invocations contains exactly one new entry with the expected fields populated.

Cross-hook tests

  • Iteration-order test: runtime-pipeline tests already verify deterministic iteration in src/runtime/. A complementary test in tests/llm-hooks/ verifies that any hook code iterating over frame data uses sortedIter/sortedKeys from iteration-helpers.ts (Stream H). The custom no-unsorted-iteration ESLint rule is configured to apply to src/llm-hooks/ as well — even though the runtime ban is the only structural requirement, sorted iteration in hooks is a code-hygiene rule that prevents hooks from producing different prompts on different runs purely because of Map iteration order.
  • Module-boundary test: a CI assertion that src/runtime/ compiles without any reference to src/llm-hooks/. Implemented as an ESLint no-restricted-imports rule plus a Vitest smoke test that imports the runtime entry and verifies the bundle does not include any provider SDK.
  • Schema-conformance test: for each hook, the prompt file's schema_in and schema_out are validated against the TypeScript types TIn and TOut declared in contracts.ts at build time, using a JSON-schema-to-TS or TS-to-JSON-schema cross-check. Drift is a build error.

Provider integration tests

Out of v1 unit-test scope: end-to-end tests against a live Anthropic or OpenAI endpoint. The implementing agent may add these as smoke tests behind an env-flag for manual runs. The test suite that gates CI uses MockLlmProvider exclusively so tests are deterministic and offline.

Persistence and schema updates

Two small extensions to fields touched by Stream G are required. Both are below the threshold of an Article XI § 1 stream amendment because the field they touch was either reserved-as-opaque pending Stream G (LlmSettings) or introduced by Stream G itself (output_overrides). However, to be transparent in self-review:

LlmSettings shape (Stream B B3 refinement)

Stream B B3 reserves LlmSettings on Frame as an opaque object explicitly for Stream G to define ("opaque object in v1"). This document defines its shape. Refining a field that was reserved-pending-this-stream is the expected lifecycle, not an amendment. No flag.

FrameVersion llm_settings_snapshot projection

For G5 (calibrated thresholds) to be visible to compute() deterministically, the FrameVersion needs to carry a fixed snapshot of the relevant LlmSettings fields at version-creation time, because LlmSettings lives on Frame and Frame is mutable while FrameVersion is immutable. The snapshot is a small projection:

interface FrameVersion { // ... existing fields per Stream B B3 ... llm_settings_snapshot?: { calibrated_thresholds?: BurdenThresholdMap // the only field compute() reads } }

This is a small extension to FrameVersion. It does not break any existing read path: legacy FrameVersions without the field fall back to the default BURDEN_THRESHOLD table from Stream C. This is logged as an in-session escalated flag because it adds a field to a Stream B-defined entity, even though the addition is purely additive and Stream B's schema is forward-compatible (the JSON envelope per B7 ignores unknown fields). See "In-session flags" in the self-review below.

ArgumentSessionVersion output_overrides field

For G6 (prose rewrite) to attach to a session without polluting the canonical ConditionalOutput object, a session-level field is added:

interface ArgumentSessionVersion { // ... existing fields per Stream B B4 ... output_overrides?: { rewritten_prose?: string rewritten_at?: timestamp rewritten_by_hook?: string // HookInvocationRecord.id } }

Same logging rationale as FrameVersion above; logged as an in-session escalated flag.

Stream H IndexedDB store layout

Stream H's prompts store is the only Stream G-related store. Its key is [hook_name+version] per H1; its value is a PromptFile per the schema in this document. The store is populated lazily on first hook invocation by reading from src/llm-hooks/prompts/<hook_name>/<version>.md, parsing into PromptFile, and writing to the store. Subsequent invocations read from the store directly.

No new IndexedDB store is added by Stream G. HookInvocationRecords live on LlmSettings.invocations on the Frame, which already persists through the existing frames store. This was a deliberate choice during this session's design work to minimize Stream H surface change; the alternative (a dedicated hook_invocations store) was considered and rejected as over-engineering for v1, since invocation rates are bounded by user actions and the per-Frame storage cost is negligible.

Reserved / deferred design space

  • Provider abstraction layer beyond the four implementations. Anthropic, OpenAI, mock, and a "local model" implementer-defined extension cover v1. Adding Cohere, Google Gemini, etc. is mechanical once LlmProvider is in place. v2-as-needed
  • Streaming completions. v1 hooks are request/response. Streaming would let the suggestion drawer fill incrementally; a UX improvement for long outputs (G3, G6). Provider interface already includes a capabilities.streaming hint. v2
  • Tool-use hooks. v1 hooks are pure prompt-response. A future hook (e.g., "fetch a citation from Westlaw given a partial reference") would use tool-use. The LlmProvider.capabilities.tool_use hint reserves the surface. v2+
  • Per-hook learning from user edits. If users repeatedly edit G1's output in a similar way, a future system could learn the edit pattern and adjust the prompt. v1's invocation log captures edit deltas in principle (parsed output vs. final value), enabling this work later. v2+
  • Cross-frame Premise drafting. G11 operates within a single session. A future hook could consult Premises from related frames (predicated on A6 cross-frame references shipping in v2). v2 (with A6)
  • Frame-wide structural critique hook. A G14-style hook reviewing the entire frame for structural quality (not just gaps in C5's territory): "this frame would benefit from splitting Negligence into separate Subquestions for each element." Distinct from G4 (which is per-validation-pass). Out of v1 scope. v2
  • Configurable temperature per hook. v1 uses temperature 0.2 for all build-time hooks. Per-hook tuning is a small change (one PromptFile field) but not currently a felt need. v2-as-needed
  • Background pre-fetching. A future system could pre-fetch G1's suggestion when the user creates a Checkpoint, making the drawer feel instant. v1 fetches on click. v2-if-needed

Self-review against the Constitution and Protocol

Article II architectural commitments — line-by-line check

  • § 1 Frame/Argument Split. Honored. Build-time hooks operate on Frame fields; runtime sidecar hooks operate on ArgumentSession fields; G6 operates on ArgumentSessionVersion. The split is preserved at every commit boundary. No hook crosses the boundary in either direction (a build-time hook cannot read session state; a runtime hook cannot write Frame fields).
  • § 2 Determinism. Honored at three levels: structural (runtime-module import boundary forbids any LLM call from src/runtime/), per-hook (every committable output bakes into an immutable FrameVersion or non-routing-consumed session field, with G5's threshold calibration explicitly snapshot into FrameVersion to guarantee visibility), and audit (every commit records a HookInvocationRecord with input_hash and raw_response so any compute result can be reproduced from the FrameVersion alone). The "Determinism guarantees" section above details each layer.
  • § 3 Per-Node Satisfaction Criteria. Honored. No hook reaches into the SatisfactionPolicy structure or its evaluation. G5's calibrated_thresholds modify the BurdenThresholdMap that the burden_met condition consumes; the condition's signature, the SatisfactionPolicy schema, and the per-node satisfaction-criteria framework are unchanged.
  • § 4 Independent Versioning. Honored. Build-time hooks write to FrameVersion and follow Stream H's auto-save versioning rules without exception. G6 writes to ArgumentSessionVersion via output_overrides and follows the session-versioning rules. Every save is preserved per the existing infrastructure; no hook deletes versions or mutates ancestors.
  • § 5 Human-Readable Structure. Honored. Hook outputs commit to existing schema fields (Checkpoint.question, Conclusion.reasoning_summary, etc.); a reader of the JSON envelope sees plain content with no AI-special encoding. Provenance lives separately on LlmSettings.invocations and is opt-in to read; the frame's logic remains readable from the graph.
  • § 6 Use-Case Commitment. Honored. Mode/flavor visibility is honored per-hook (G5 legal-only, G9 general-only, G2 citation_hint legal-only-and-academic-acceptable-but-personal-suppressed, all others all-modes). No hook is exclusive to one use case; every hook respects the existing mode/flavor surfaces.

Article III quality criteria — score

  • § 1 Correctness. Each hook is specified against a known surface (Stream B, C, D, E identifying it) with a typed input drawn from those streams' schemas, a typed output that maps to a defined frame field or to a presentational artifact, and a parser whose rules are implementable. The hooks model what they claim to model: G1 drafts a single Checkpoint's content; G2 proposes Interpretations under a Term; G5 calibrates thresholds within the existing fixed-table structure; etc. None claims to do something outside its scope.
  • § 2 Legal Practice Fit. The hooks model how lawyers actually work with AI assistants today: paste a passage, get a draft, review and commit; click "suggest" on a field, edit, accept; never let the AI silently alter routing or status. The hooks that touch legal-flavored content (G1 with jurisdiction context, G2 with citation_hint, G5 with standard-of-review calibration) are scoped to the cases where AI assistance is plausibly useful to a practicing lawyer; the constraints in each prompt body (no fabricated citations, no future-tense predictive claims, no ID-leaking summaries) reflect known LLM failure modes that lawyers are increasingly sensitive to. The "user always confirms" affordance is consistent with model rules of professional conduct on AI-assisted output.
  • § 3 Determinism and Auditability. Maximally satisfied. The structural import boundary is checkable in CI. Per-hook bake-and-commit is the dominant pattern. Provenance is recorded per-commit with input hash + raw response, enabling exact replay. Every FrameVersion and ArgumentSessionVersion still computes deterministically because no hook output influences compute() beyond writing fixed fields onto the version snapshot.
  • § 4 Clarity. The hook system is debuggable and inspectable: open the Frame's llm_settings.invocations array and you see every AI commit ever made; click the ✦ chip on a field and you see which hook produced it. The contract is one interface implemented thirteen times; the prompt-file format is one schema with one parsing path. The cheat sheet at the end indexes by hook id.
  • § 5 Reversibility. Locked-in choices: the HookContract interface, the PromptFile schema, the LlmProvider interface, the LlmSettings shape, the bake-and-commit discipline. Each is structurally decoupled from any specific provider, prompt, or hook. Adding hooks is mechanical (write a HookContract implementation, write a prompt file, register in the contracts module). Removing hooks is mechanical (rejection in LlmSettings.per_hook_enabled hides the surface; existing FrameVersions still resolve via their snapshot data and stored prompt versions). Replacing the provider is contained behind LlmProvider. Replacing the prompt file format is contained behind PromptFile parsing. The provider abstraction was selected partly for this reason.
  • § 6 Interface Fit. A future session needing only the hook taxonomy reads the table at the top of this document; a session implementing G1 specifically reads the G1 card; a session changing the prompt-file format reads the "Prompt file format" section; a session adjusting determinism guarantees reads "Determinism guarantees." Cross-references to Streams A, B, C, D, E, H are explicit at every point of contact. The cheat sheet indexes by hook id.
