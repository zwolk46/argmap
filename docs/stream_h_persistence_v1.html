Stream H — Persistence and Infrastructure

Companion to stream_a_resolutions_v1.html, stream_b_schema_v1.html, stream_c_runtime_v1.html, stream_d_modes_v1.html, and stream_e_uiux_v1.html  ·  Session date: May 10, 2026  ·  Version 1

This document specifies how the application persists its data, how it auto-saves, how version history is implemented on top of that persistence, and what concrete tech stack the app runs on. It covers Stream H end-to-end: storage strategy and format (H1), auto-save mechanics (H2), version-history implementation (H3), and tech-stack selection including frontend framework, graph rendering library, and build tooling (H4). It compiles directly against Stream B's schema and JSON envelope, Stream C's determinism guarantee and incremental-computation strategy, Stream D's atomicity constraints (frame-template deep-copy and session migration), and Stream E's rendering and side-pane requirements. Visual aesthetics, prompt files, and the F5 court-hierarchy database remain out of scope here; what this document fixes is the engineering substrate the rest of the application sits on.

How to read this

Conventions match the prior stream documents. TypeScript-flavored notation describes shapes and APIs; the actual implementation is plain TypeScript at the source level and JSON on disk. Where a decision is final, it is locked; where a sub-question is intentionally deferred, it is tagged accordingly. Section ids (H1, H2, H3, H4) match the master plan's Stream H sub-items.

Decisions made in this session

The following decisions are committed by this document. Each is a technical choice within agent discretion under Article V § 2(c)–(g); none reverses an architectural commitment under Article II or a prior stream's decision (verified in self-review below).

  1. Local-first architecture for v1; cloud sync deferred to v2 with a sketched architecture. The client persists everything to browser-local storage and works fully offline. A sync layer can be added later without rewriting the app, because the persistence interface is abstracted (decision #4). Data movement between devices in v1 is via JSON file export/import, using the envelopes defined in Stream B B7.
  2. Browser storage backend: IndexedDB, accessed through Dexie.js. IndexedDB is the only browser-native store with the capacity, transactions, and indexed lookups the schema requires. Dexie is a thin wrapper that gives clean transaction semantics and typed queries without ceremony; it does not foreclose moving to raw IndexedDB later.
  3. On-disk export format is exactly the JSON envelope from Stream B B7. FrameExport and ArgumentSessionExport are the canonical interchange shapes. The IndexedDB layout decomposes those envelopes into per-record stores for transaction granularity, but every export round-trips losslessly.
  4. Repository pattern for persistence. A Repository interface defines the surface every persistence backend must implement (load Frame, save Frame, list versions, compute diff, etc.). The v1 implementation is IndexedDbRepository; the v2 cloud-sync implementation will be a wrapper composing local + remote repositories. Application code never touches IndexedDB directly.
  5. Auto-save debounce window: 5 seconds idle, 30 seconds maximum elapsed time. The window flushes on whichever fires first. Manual save and mode-transition saves bypass the debouncer and set is_milestone: true. The window is a tunable constant in one place; Stream C's incremental computation strategy keeps recompute cost bounded so the visible save cadence matches the user's edit pace.
  6. Versioning granularity: every frame-state edit forks a new FrameVersion (debounced); pure UI state does not. Frame-state includes nodes, edges, frame metadata, and node presentation hints (x, y, collapsed) — anything stored on the FrameVersion per Stream B. UI state (current selection, viewport pan/zoom, side-panel open/closed, current output-viewer tab) is per-user app state stored separately and is never versioned. The same rule applies to ArgumentSessionVersion.
  7. Single-transaction atomicity for frame-template deep-copy and session migration. Per Stream D's atomicity constraints, the deep-copy operation (regenerating every node and edge id and rewriting internal references) and the session-migration operation (forking an ArgumentSessionVersion plus applying orphan-resolution writes) are wrapped in a single Dexie transaction; either the entire operation commits or none of it does.
  8. Tech stack: React + TypeScript + Vite, with React Flow (xyflow) + ELK for the canvas, Zustand for application state, Dexie for persistence, Vitest + Playwright for testing. Each piece is the lowest-friction option that satisfies the requirements committed in Streams B–E; alternatives are noted under each H4 sub-section.
  9. Determinism is enforced at the runtime-module boundary, not at the framework boundary. Stream C's compute pipeline is a pure TypeScript module with explicit determinism contracts (stable iteration, no clock reads, no environment-dependent branches). The persistence layer, the React layer, and the Dexie layer all sit outside that boundary and are free to read clocks and rely on environment behavior. CI gates enforce the boundary.
  10. Schema migration is dispatched on the schema_version integer per Stream B B7. A single MigrationRegistry module holds N → N+1 transformer functions; on load, the reader walks from the file's schema_version to the current version, applying each transformer in turn before validating. The registry starts empty (v1 ships at schema_version: 1); the mechanism is in place so future upgrades are mechanical.

H1 — Storage strategy

Architectural approach: local-first

The application is local-first in v1. Every read and write hits a browser-local store; the app works fully offline; no network call sits on the critical path of any user action. This approach is chosen against the alternatives — a cloud-first SaaS, or a thick-client desktop app — for three reasons. First, it satisfies Article III § 5 (Reversibility): adding cloud sync later is a contained change because the persistence interface is abstracted; replacing local storage with cloud storage is the inverse change and would leave a dependency mass behind. Second, it satisfies Article III § 6 (Interface Fit): v1's infrastructure is small enough that a coding agent can stand it up without a server, an auth provider, a hosted database, or a deployment pipeline. Third, the user is a single person; multi-user collaboration is explicitly out of scope per Stream B B7. A cloud sync layer in v1 would be over-engineering for a use case that does not yet exist.

The cost of local-first is multi-device pain: the user's frames live in one browser profile on one machine. v1 mitigates this with file-based export/import using the JSON envelopes from Stream B B7. v2 replaces the mitigation with a real sync layer (sketched below). The Repository abstraction (decision #4) makes that replacement contained.

Cloud sync — v2 sketch and rationale for deferral

Cloud sync is deferred to v2. The intended v2 architecture is documented here so future agents do not re-derive it and so the v1 design can be checked against compatibility with the v2 path.

  • Auth. A managed auth provider (Supabase Auth, Clerk, or equivalent). The app gains a single "signed in / signed out" gate at startup; signed-out users see a "v1 local-only" banner and the app continues to function locally.
  • Server. Either Supabase (Postgres + REST + auth bundled) or a small Node/Express + Postgres server. The choice is a v2 decision; both fit the schema. Postgres stores Frame, FrameVersion, ArgumentSession, and ArgumentSessionVersion records as JSONB columns plus indexed metadata for query performance.
  • Sync model. Last-write-wins on each entity, scoped per-user (single-user means no real conflict). The client retains its full local copy and treats the server as an authoritative mirror; on reconnect, the client uploads any unsynced versions and downloads any newer versions. Because every version is immutable once written (per Article II § 4), conflict reduces to "which versions exist," not "what is the current content of an entity."
  • Hosting. Vercel for the SPA; Supabase or Render/Fly.io for the server. Specific choice is v2 work.

The v1 design preserves compatibility with this v2 sketch by routing every read and write through a single Repository interface (decision #4). The cloud-sync implementation will be a SyncingRepository that composes a local IndexedDbRepository with a RemoteRepository; v1 application code does not need to change.

Browser storage backend: IndexedDB via Dexie

IndexedDB is the only browser-native API that meets the requirements: storage capacity into the hundreds of megabytes (Frame versions are full snapshots, not deltas — per Stream B decision #4 — and a frame with active history can grow); structured records (each Frame, FrameVersion, and Session is a deeply nested object); indexed queries (we need to look up versions by frame_id and traverse parent pointers); transactions (per decision #7, atomicity matters for deep-copy and migration). LocalStorage is too small (5 MB) and synchronous; the File System Access API is unstable and gated by user permission per session.

Dexie.js wraps IndexedDB with a typed query builder, transaction helpers, and a schema-versioning system. The wrapper is small (≈40 KB minified) and replaceable; if Dexie ever becomes a constraint, the IndexedDbRepository can be reimplemented against raw IndexedDB without changing the rest of the application. Alternatives considered and rejected: idb (Jake Archibald's promisified wrapper — too thin; we end up rewriting Dexie's query helpers), RxDB (heavier, opinionated about reactivity), localForage (key-value only, no real transactions across keys).

On-disk export format

The export format is the JSON envelope defined in Stream B B7, repeated here for reference:

FrameExport { schema_version: integer app_version: string exported_at: timestamp frame: Frame current_version: FrameVersion history?: FrameVersion[] // optional; omitted by default for size } ArgumentSessionExport { schema_version: integer app_version: string exported_at: timestamp session: ArgumentSession current_version: ArgumentSessionVersion history?: ArgumentSessionVersion[] embedded_frame_export?: FrameExport // when sharing a session standalone }

Export operations build these envelopes from the IndexedDB stores. Import operations validate the envelope, run any schema migrations, and write the records into IndexedDB inside a single transaction. By default, export omits history for compactness; an "Export with full history" affordance includes it. app_version is the build's npm package version and is informational; schema_version is the gating value for migration.

Repository abstraction

All persistence operations route through a single interface. Application code (React components, Zustand stores, runtime modules) holds a reference to a Repository and never touches IndexedDB or Dexie directly. The interface is small and the surface intentionally narrow:

interface Repository { // Frames listFrames(): Promise<FrameSummary[]> loadFrame(frame_id: string): Promise<Frame> saveFrame(frame: Frame): Promise<void> deleteFrame(frame_id: string): Promise<void> // gated on no live sessions; checked by caller // FrameVersions listFrameVersions(frame_id: string): Promise<FrameVersion[]> loadFrameVersion(version_id: string): Promise<FrameVersion> saveFrameVersion(version: FrameVersion): Promise<void> // also updates Frame.current_version_id atomically // Argument sessions listSessionsForFrame(frame_id: string): Promise<ArgumentSessionSummary[]> loadSession(session_id: string): Promise<ArgumentSession> saveSession(session: ArgumentSession): Promise<void> deleteSession(session_id: string): Promise<void> listSessionVersions(session_id: string): Promise<ArgumentSessionVersion[]> loadSessionVersion(version_id: string): Promise<ArgumentSessionVersion> saveSessionVersion(version: ArgumentSessionVersion): Promise<void> // Composite operations — atomic per decision #7 createFrameFromTemplate(template_frame_id: string, new_title: string): Promise<Frame> migrateSession(session_id: string, target_frame_version_id: string, orphan_resolutions: OrphanResolution[]): Promise<ArgumentSessionVersion> restoreFrameVersion(frame_id: string, ancestor_version_id: string): Promise<FrameVersion> restoreSessionVersion(session_id: string, ancestor_version_id: string): Promise<ArgumentSessionVersion> // App state (per-user, not versioned) loadAppState(): Promise<AppState> saveAppState(state: AppState): Promise<void> // Search searchFrames(query: string): Promise<FrameSearchHit[]> // Export / import exportFrame(frame_id: string, opts: { include_history: boolean }): Promise<FrameExport> exportSession(session_id: string, opts: { include_frame: boolean; include_history: boolean }): Promise<ArgumentSessionExport> importFrame(envelope: FrameExport): Promise<Frame> importSession(envelope: ArgumentSessionExport): Promise<ArgumentSession> }

The interface deliberately exposes the composite operations (createFrameFromTemplate, migrateSession, restoreFrameVersion, restoreSessionVersion) rather than letting callers compose them from primitive saves. This places the atomicity contracts inside the implementation, where they can be enforced with a single transaction, instead of in caller code where they would inevitably leak.

FrameSummary, ArgumentSessionSummary, OrphanResolution, AppState, and FrameSearchHit are small types defined alongside the repository module; their shapes follow naturally from Streams A, B, and D and are documented in the source rather than fixed here.

Search index strategy

Per Stream A A5, search runs across title, description, node text, tags, and Conclusion statements. The implementation is a precomputed token index per Frame, rebuilt on every Frame save.

  • On saveFrame or saveFrameVersion, the repository computes a SearchIndexEntry: a flat token list extracted from the frame's title, description, tags, every node's primary text fields, and every Conclusion's statement. Tokenization is whitespace + lowercasing + punctuation stripping; no stemming in v1.
  • The entry is written to search_index in the same transaction as the frame save, so the index is never out of sync with the data.
  • searchFrames(query) tokenizes the query and intersects against entries. For v1's expected scale (single-user, dozens to low hundreds of frames), this can be done in JavaScript without a dedicated search library; the entire search_index store is read into memory once on app start and kept warm.
  • If the user count or frame count grows beyond what in-memory intersection handles cleanly, the index can be replaced with a real search library (FlexSearch, MiniSearch, Lunr) without changing Repository.searchFrames. v2-as-needed

Atomicity contracts

Three composite operations require single-transaction atomicity. Each is a method on Repository and is implemented inside a Dexie db.transaction(...) block.

  1. Frame-template deep-copy (Stream D D1, Article II § 4 reversibility hint). Reads the source Frame and its current_version FrameVersion. Walks every Node and Edge, regenerates UUIDs, builds an id-translation map, rewrites every internal reference (NodeRef in edges, output_target on LogicalGate, target_node_id on CheckpointOption, linked_to on Term, etc.) using the map. Writes a new Frame with a new id and a new FrameVersion (version_number 1, no parent). Either every record commits or none does.
  2. Session migration (Stream D D2, D3). Reads the session's current state, computes the orphan list against the target FrameVersion, applies the user-provided OrphanResolutions (delete-and-discard, reattach-to-replacement, etc.), forks a new ArgumentSessionVersion, updates the session's frame_version_id and frame_version_snapshot and current_version_id. Single transaction.
  3. Restore (frame or session version) (Stream E E5). Reads the chosen ancestor, deep-copies its content into a new version with parent_version_id pointing at the ancestor and change_summary: "restored from version N", updates the parent entity's current_version_id. Single transaction. Per Article II § 4, the ancestor is not modified.

Auto-save writes (decision #5) are also transactional: the new FrameVersion record + the parent Frame.current_version_id + Frame.updated_at commit together. A partial-save failure leaves the prior state intact.

Quota and failure handling

IndexedDB has a per-origin quota. On modern browsers this is generous (Chromium grants up to ~60% of total disk; Firefox similar; Safari more conservative around 1 GB), but failures are still possible. The repository handles them:

  • Quota-exceeded on save. The repository raises a typed QuotaExceededError. The auto-save controller catches it, surfaces a non-blocking "couldn't save — your changes are still in memory" warning in the UI, and retries on the next change. The user can click the warning to open an export dialog.
  • Generic transaction failure. The repository raises a RepositoryError with the failed operation name. The same UI affordance applies; the in-memory state is preserved.
  • Read failure on app start. If the database is corrupt or unreadable, the user is shown a recovery screen offering "Export raw data," "Try opening anyway in read-only mode," and "Reset database" (with a strong confirmation). Reset deletes the database entirely; export attempts to dump whatever records survive.

The persistent surface for these failures is intentionally a non-blocking warning rather than a modal; per Stream E's interaction patterns, the user keeps control of what to do next.

Schema migration mechanism

Per Stream B B7 and decision #10, schema migration is dispatched on schema_version. The MigrationRegistry module declares a sequence of (from_version, to_version, transformer) tuples; the reader walks the chain on import.

type Migration = (input: AnyJson) => AnyJson interface MigrationRegistry { readonly current_schema_version: integer migrations: { [from: integer]: Migration } // from_version → transformer that produces from_version + 1 } function migrate(envelope: AnyJson, registry: MigrationRegistry): AnyJson { let v = envelope.schema_version while (v < registry.current_schema_version) { envelope = registry.migrations[v](envelope) v += 1 envelope.schema_version = v } return envelope }

v1 ships with current_schema_version: 1 and an empty migrations table. The first time a non-backward-compatible change ships, the responsible session adds a single transformer and bumps the constant; the registry is the only file that needs to change. Running migrations on import is also the path used when an older version of the app's IndexedDB is opened by a newer build (Dexie's own version-bump hook calls migrate).

H2 — Auto-save mechanics

What constitutes a save

A save is the persistence of frame state or session state to IndexedDB through the Repository. There are two save kinds:

  • Auto-save: triggered by the debouncer (decision #5) after frame-state or session-state edits. Creates a new FrameVersion or ArgumentSessionVersion with is_milestone: false.
  • Milestone save: triggered by an explicit user action (the "save milestone" command, mode transitions, frame-template deep-copy, session migration, or restore). Bypasses the debouncer and creates a version with is_milestone: true.

App-state changes (selection, viewport, side-panel state, coachmark dismissals) are saved separately, debounced more aggressively (1-second window), and never create versions. They write to the singleton app_state record.

Debounce parameters

The auto-save debouncer follows the standard "idle + max" pattern.

  • Idle window: 5 seconds. If no frame-state edit has occurred for 5 seconds, flush.
  • Maximum elapsed window: 30 seconds. If 30 seconds have passed since the first un-flushed edit, flush regardless of subsequent activity.
  • Flush on navigation. If the user closes the tab, navigates away, or runs an action that needs the current state on disk (a milestone save, an export, a session migration), the debouncer flushes immediately. Tab-close is hooked via beforeunload; the flush is best-effort because browsers may not honor synchronous IndexedDB writes during unload, so the value is preserved primarily by the 30-second cap.
  • Single in-flight save. Only one save is in flight at a time per Frame or Session. If a flush fires while the prior save is still in progress, the new flush is queued and runs on completion.

5 seconds is short enough that data loss in a crash is bounded; 30 seconds is long enough that a heavy edit session does not produce a wall of versions in the history pane. Both are constants in one source file (autosave.ts) and tunable; user-facing exposure of these knobs is deferred unless it surfaces as a complaint.

Versioning rules

Per decision #6, the line between "versioned" and "not versioned" is drawn at the FrameVersion / ArgumentSessionVersion schema boundary. Anything stored on those types is versioned. Anything else is app state.

Presentation hints (x, y, collapsed) are versioned because they are stored on the Node within the FrameVersion (per Stream B B1) and because the layout engine treats them as fixed anchors (per Stream E E1). Restoring an older version is expected to restore the layout the user had at that point; that requires the hints to live with the version's content. The cost — a drag-drop creates a version after the 5-second idle — is acceptable; the version-history side pane filters auto-saves out of view by default per Stream E E5.

Milestone forcing

Per Stream B B3 ("manual saves and mode transitions force a new version with is_milestone: true") and Stream D D1 / D3, the following operations always set is_milestone: true:

  • The user clicks "save milestone" with optional change_summary.
  • Operating-mode transition Frame Building → Argument Running (after C5 strict passes).
  • Architectural-mode transition (legal ↔ general).
  • Flavor change (personal ↔ academic).
  • Frame-template deep-copy (creates the destination Frame's v1 milestone).
  • Session migration (creates the migrated session's milestone).
  • Restore (creates the new branch's first version as a milestone, with change_summary: "restored from version N").

Failure handling

Saves can fail in three ways: quota exceeded, transaction abort (browser closes the database mid-write), and unexpected exceptions. The auto-save controller's behavior in each case:

  • Failure preserves in-memory state. The application's Zustand store is not mutated by a save attempt; saves read from the store. A failed save does not roll back any user-visible state.
  • UI surface: non-blocking banner. A small banner appears at the top of the screen reading "Couldn't save your changes — they're still in memory. We'll keep trying." with a small "Export now" link. The banner clears on the next successful save.
  • Retry: on the next edit. The debouncer's next fire attempts the save again. There is no exponential backoff in v1; the interaction model is "try every time the user edits, and surface failure if it persists." If two consecutive saves fail, the banner upgrades to a stronger warning ("Saves are failing. Export to keep your work safe.").

Concurrency

v1 is single-user. Two browser tabs open on the same Frame is the only realistic concurrency case; it is handled by the BroadcastChannel API as a courtesy and the last-write-wins semantics of IndexedDB as the floor.

  • On startup, each tab joins a BroadcastChannel keyed to the database name. When a tab saves a new FrameVersion, it broadcasts {frame_id, new_version_id} to peer tabs.
  • Peer tabs receiving the broadcast reload the affected Frame from disk. If the user has unsaved changes in that tab, a small banner appears: "This frame was edited in another tab. Reload to see the latest version, or keep editing here (your changes will be saved as a new branch)." The user picks; either path is non-destructive.
  • Without BroadcastChannel (older browsers), the floor behavior is last-write-wins per IndexedDB transaction. A user editing in two tabs simultaneously may produce two FrameVersions whose parent_version_id points at the same ancestor — i.e., a forked branch — which the version-history UI already handles.

Because every version is preserved (Article II § 4), no data is lost in any of these cases; the worst outcome is a divergent branch the user resolves through the version-history pane.

H3 — Version history implementation

Storage layout

One IndexedDB record per FrameVersion and one per ArgumentSessionVersion. No deduplication. The frame_versions store is indexed on frame_id for "list versions of frame X" queries and on parent_version_id for "what forked from this version" queries. Same pattern for argument_session_versions.

Branch tree representation

The version graph is a tree rooted at version_number: 1. Each version has at most one parent_version_id; multiple versions may share the same parent (i.e., branches). The history side pane (Stream E E5) renders the tree by walking from the root forward and indenting children of each fork point.

Building the tree for the side pane:

async function buildVersionTree(frame_id: string): Promise<VersionTreeNode> { const versions = await repo.listFrameVersions(frame_id) // already sorted by version_number ascending const by_id = new Map(versions.map(v => [v.id, v])) const children: Map<string, FrameVersion[]> = new Map() let root: FrameVersion | null = null for (const v of versions) { if (!v.parent_version_id) { root = v; continue } const list = children.get(v.parent_version_id) ?? [] list.push(v) children.set(v.parent_version_id, list) } return buildNode(root!, children) }

The same algorithm runs for ArgumentSessionVersions. Cost is O(n) in the version count; for v1's expected scale this is well below any user-perceived threshold.

Diff algorithm

Per Stream E E5, the Compare view shows nodes added / removed / edited, edges added / removed / edited, and frame metadata changes. The diff is a pure function over two FrameVersions:

interface StructuralDiff { nodes_added: Node[] nodes_removed: Node[] nodes_edited: { before: Node, after: Node, fields_changed: string[] }[] edges_added: Edge[] edges_removed: Edge[] edges_edited: { before: Edge, after: Edge, fields_changed: string[] }[] metadata_changes: { field: string, before: any, after: any }[] } function diffFrameVersions(a: FrameVersion, b: FrameVersion): StructuralDiff { // 1. Bucket nodes by id from each version. // 2. nodes_added = ids in b not in a; nodes_removed = ids in a not in b. // 3. For shared ids, compare field-by-field (excluding x, y, collapsed which are layout hints — // layout-only changes are surfaced as a single \"layout changed\" entry, not field-level). // 4. Same for edges, keyed by id. // 5. Frame metadata: title, mode, flavor, jurisdiction_default, positions, default_satisfaction_policies, tags. ... }

The function is pure, deterministic, and unit-tested with snapshot fixtures. Layout-hint changes are bucketed separately because they are typically not what the user wants to see when comparing versions; the Compare UI exposes them as a single line ("Layout updated for 4 nodes") rather than four entries. This is a UX refinement consistent with Stream E E5's "shows structural and field-level changes" framing.

The same shape and algorithm apply to diffSessionVersions(a, b) for ArgumentSessionVersion comparison; the relevant collections are premises, argument_edges, checkpoint_responses, session_authorities, and interpretation_selections.

Restore implementation

Restoring an ancestor creates a new version off that ancestor. The implementation:

async function restoreFrameVersion(frame_id: string, ancestor_version_id: string): Promise<FrameVersion> { return await db.transaction('rw', db.frames, db.frame_versions, async () => { const frame = await db.frames.get(frame_id) const ancestor = await db.frame_versions.get(ancestor_version_id) if (!frame || !ancestor) throw new RepositoryError('not found') const max_version = await db.frame_versions .where('[frame_id+version_number]').between([frame_id, 0], [frame_id, Infinity]) .last() const new_version: FrameVersion = { id: uuid(), frame_id: frame_id, version_number: (max_version?.version_number ?? 0) + 1, parent_version_id: ancestor_version_id, created_at: now(), change_summary: `restored from version ${ancestor.version_number}`, nodes: deep_clone(ancestor.nodes), edges: deep_clone(ancestor.edges), is_milestone: true, } await db.frame_versions.add(new_version) frame.current_version_id = new_version.id frame.updated_at = now() await db.frames.put(frame) return new_version }) }

The transaction guarantees atomicity per decision #7. The ancestor is not modified (Article II § 4). The new version's version_number is the next integer in the frame's sequence regardless of which ancestor was restored — i.e., version numbers are monotonic across the entire frame's history, not per-branch. This matches the side pane's display, where v1, v2, v3, ... appear in chronological order with branches drawn as indents.

Session restore follows the same pattern over argument_session_versions.

Performance: lazy loading

The version-history side pane loads version metadata only — id, version_number, parent_version_id, created_at, is_milestone, change_summary — not the full nodes and edges payload. A FrameVersionSummary projection is a separate Repository method:

interface FrameVersionSummary { id: string frame_id: string version_number: integer parent_version_id?: string created_at: timestamp is_milestone: boolean change_summary?: string } repo.listFrameVersionSummaries(frame_id): Promise<FrameVersionSummary[]>

The summary projection reads the records but discards the heavy fields before returning. For Preview, Compare, and Restore, the full version is loaded on demand. For a frame with hundreds of versions this keeps the side pane responsive.

Compaction

Storage compaction (deduplicating identical nodes/edges across versions, or representing later versions as deltas against earlier ones) is deferred to v2, per Stream B decision #4 and B7 ("Storage size is a Stream H concern; deduplication can be layered in later without a schema change"). The on-disk format and the Repository surface are stable across that change. The trigger for v2 work is observed user complaint (or an internal storage-size metric exceeding a threshold), not a planned milestone. v2-as-needed

H4 — Tech stack selection

Frontend framework: React + TypeScript

React for the UI layer, TypeScript for everything. React is chosen for three reasons. First, it has the broadest ecosystem of mature, actively maintained graph-rendering libraries, and Stream E's requirements (custom node shapes, custom edge styles, drag-from-handle edge creation, Sugiyama auto-layout, side-pane interactions) map cleanly to React's component model. Second, future coding agents working on this project will be most productive in React; the constitution's Article III § 6 (Interface Fit) penalizes designs that require deep onboarding. Third, React's render model decouples UI updates from persistence operations, which matters for the auto-save controller — saves run on a debounced timeline, independent of React's render cycle.

TypeScript is required, not optional. Stream B's schema is written in a TypeScript-flavored notation; using TypeScript at the source level prevents schema drift from going undetected. Compile-time enforcement of node-type discrimination, edge-type validation, and condition-library exhaustiveness is the project's first line of defense against bugs that would otherwise surface only as broken sessions.

Alternatives considered: Svelte / SvelteKit (smaller, faster, but smaller graph-library ecosystem and less agent familiarity), Vue (similar tradeoffs to Svelte), vanilla TypeScript (would force the team to rebuild basic UI primitives). Solid.js (a strong technical fit, but ecosystem and agent familiarity are weaker than React).

Graph rendering library: React Flow + ELK

React Flow (the open-source library now branded xyflow) for the canvas, with ELK (specifically elkjs, the JavaScript port) for auto-layout. The combination satisfies every Stream E E1 requirement.

  • Custom node components. React Flow allows registering a React component per node type. Each of Stream B's nine node types gets its own component, rendering the shape (rounded rect, square rect, hexagon, diamond, double-bordered, document icon, pill, etc.) and the type-specific content. Status badges, sub-flag chips, and Premise-count chips are slots within those components.
  • Custom edge components. Edge styling per type (solid, dashed with stop-cap, dotted, thin gray with citation glyph) is implemented as React Flow custom edge components.
  • Drag-from-handle edge creation. React Flow's Connection system natively supports dragging from a node's handle to another node, then firing an onConnect callback. The callback opens the edge-type picker (Stream E E2) and creates the edge through the Repository.
  • Zoom, pan, minimap, controls. React Flow ships <Controls> and <MiniMap> components matching Stream E E1's affordances.
  • Auto-layout via ELK. React Flow does not ship a hierarchical layout. elkjs exposes ELK's "layered" algorithm (Sugiyama-style, top-down). Integration is straightforward: feed ELK the graph, get back x, y coordinates, push them onto React Flow nodes. ELK runs in a Web Worker for non-blocking layout on large frames.
  • Per-node x/y persistence. When the user drags a node, React Flow fires an event with the new coordinates; the handler writes them to the Node via the Repository. Subsequent ELK passes treat user-positioned nodes as fixed anchors per Stream E E1.
  • Collapse / expand subtrees. Implemented at the React layer: a SubQuestion with collapsed: true renders as a single node with a "+N nodes" badge; on expand, the children are added back to React Flow's node list and ELK re-runs.

Alternatives considered: Cytoscape.js (more graph-algorithm depth, but heavier and the React integration is more cumbersome — its renderer is canvas-based, which limits custom-component flexibility), D3 directly (maximal flexibility, but rebuilds basic affordances React Flow gives for free), vis-network (similar concerns to Cytoscape; aging API), JointJS (commercial license complications). For Sugiyama, dagre was considered against ELK; ELK produces visibly better layouts on the kinds of frames legal analysis tends to produce (deeper hierarchies, asymmetric branching) and is the choice when layout quality matters more than minimal install size.

Build tooling: Vite

Vite for development server and production build. Vite's dev server is fast, its config is minimal, and it has first-class React + TypeScript support. vitest (the test runner) shares Vite's pipeline, so unit and snapshot tests pick up the same module-resolution rules and TS settings as the application code — no second source of truth.

Alternatives: Webpack (heavier configuration), Parcel (less actively developed), Turbopack (tied to Next.js; this project does not need SSR), Rspack (newer; viable but no advantage over Vite for this scale).

State management: Zustand

Zustand for global state (current Frame, current ArgumentSession, undo/redo stack pointer, user-mode/flavor). React Context for cross-cutting concerns (Repository instance, theme tokens, user preferences) where Zustand would be overkill. The Zustand store is the source of truth for the in-memory copy of the user's working state; the Repository writes that state to disk on debounced flushes.

Alternatives: Redux (boilerplate-heavy, ceremony cost not justified at v1's scale), Jotai (atom-based; viable but Zustand's slice model is a closer match to "one Frame, one Session" mental model), MobX (proxy-based; works but harder to reason about than Zustand's plain functions), pure React Context for everything (re-render performance risk on large frames).

Persistence layer: Dexie.js

Per H1: Dexie.js wrapping IndexedDB. The dependency adds ~40 KB minified; the alternative (raw IndexedDB) is more code and more bug surface for no benefit at this scale.

Testing: Vitest + Playwright

Vitest for unit tests, snapshot tests, and Stream C's determinism test fixtures. Playwright for end-to-end browser tests covering the full stack (Repository → React → React Flow). The split mirrors the architecture: pure modules (the runtime pipeline, the Repository's pure helpers, the diff algorithm) are unit-tested with Vitest; user flows (build a frame, run a session, restore a version, migrate a session) are E2E-tested with Playwright.

Stream C's determinism test specifications land here as concrete files:

  • Snapshot tests: tests/runtime/fixtures/*.json — pairs of (input, expected output). Vitest snapshot mode compares byte-equal.
  • Property tests: tests/runtime/property/*.test.ts using fast-check. Random well-formed frames + random sessions, asserting compute(x) === compute(x).
  • Cache-equality tests: every snapshot test runs both the full-recompute and incremental paths and asserts byte-equal output.
  • Iteration-order audit: a custom ESLint rule (no-unsorted-iteration) running over the runtime modules; any for…of loop, map, or forEach over a Map, Set, or plain object that has not been preceded by a sort flagged as an error. The rule lives in eslint-plugin-argmap-determinism/ in the repo. Specified as a CI gate per Stream C.
  • Clock isolation: Vitest's vi.useFakeTimers() wraps every runtime test; the runtime modules are designed to take a clock as a parameter (or none at all), so a frozen clock surfaces any accidental Date.now() dependency immediately.

Linting, formatting, CI

ESLint with typescript-eslint, the React plugins, and the custom no-unsorted-iteration rule. Prettier for formatting. GitHub Actions (or whatever CI runs against the repo) executes, on every PR: typecheck, lint, unit tests, E2E smoke tests, and the iteration-order audit. The CI gate failing on the iteration-order audit is the practical enforcement of Stream C's determinism contract; this is the second line of defense after TypeScript's type system and is the line that catches the mistakes humans make under deadline.

Determinism implementation contract

The persistence and infrastructure layers do not undermine Stream C's determinism guarantee, but they create the surface where the guarantee is enforced. The contract has three parts.

  1. Module boundary. The runtime pipeline (src/runtime/) is a pure module: it imports nothing from src/persistence/, src/ui/, src/state/, src/llm-hooks/, or any third-party library that reads the clock or environment. Imports are restricted to src/schema/ and standard JavaScript built-ins. Enforced by an ESLint no-restricted-imports rule.
  2. Sort before iterate. Every iteration over a Set, Map, or plain object inside src/runtime/ is preceded by a sort by lexicographic id. The iteration-helpers.ts module provides sortedIter, sortedKeys, and sortedEntries as the canonical helpers; the custom ESLint rule no-unsorted-iteration flags any direct iteration that bypasses them.
  3. Frozen clock in tests. Every test in tests/runtime/ runs under vi.useFakeTimers() and a frozen Date.now(). Any incidental clock dependency in runtime code is detected by the property-test suite, which runs each test twice with different frozen times and asserts identical output.

The persistence layer reads the clock for created_at and updated_at fields. Those values flow into FrameVersion and ArgumentSessionVersion records, but they are inputs to compute only insofar as they live on the snapshot that compute reads — and the snapshot is by definition fixed at the moment compute runs. There is no clock read inside compute itself.

Article II commitments — compliance check

  • § 1 Frame/Argument Split. Honored. The IndexedDB layout has separate stores for Frames/FrameVersions and ArgumentSessions/ArgumentSessionVersions; the Repository interface treats them as independent entities; auto-save versioning rules apply identically and independently. The session embeds a frame_version_snapshot per Stream B, so the session's content is sealed against frame edits without coupling the two version streams.
  • § 2 Determinism. Honored. The "Determinism implementation contract" section codifies the runtime-module boundary, the sort-before-iterate discipline, and the frozen-clock test environment. The auto-save layer reads the clock for created_at/updated_at but those values are inputs to compute only as fixed snapshot fields, never as live reads inside the pipeline. ESLint's no-restricted-imports rule (Decision #9) and the custom no-unsorted-iteration rule make the boundary checkable in CI.
  • § 3 Per-Node Satisfaction Criteria. Honored. The persistence layer stores SatisfactionPolicy per Stream B B1 / SatisfactionPolicy section without modification; nothing in H1–H4 reaches into the policy structure or its evaluation.
  • § 4 Independent Versioning. Honored, and structurally enforced. FrameVersions and ArgumentSessionVersions live in separate IndexedDB stores; their sequences (version_number) are independent; the Repository surface has separate listFrameVersions and listSessionVersions methods. The branch-tree algorithm runs independently for each. Article II § 4's "every save is preserved" is enforced by the absence of any deletion path on individual versions; the only delete is at the Frame or Session level (gated per Stream D), and even that does not retroactively rewrite history within a frame's lifetime.
  • § 5 Human-Readable Structure. Honored. The on-disk format is JSON, the schema is the same one Streams A–E use, and an exported FrameExport file is human-readable text. Nothing in the persistence layer hides logic in ordering or coupling variables; the search index is a derived projection, not an authoritative store.
  • § 6 Use-Case Commitment. Honored. The persistence design treats Mode and Flavor as schema fields that flow through unchanged; nothing in H privileges legal mode at the storage layer. The export envelope round-trips a legal frame and a general frame identically.

Article III quality criteria — score

  • § 1 Correctness. Every persistence operation called for by Streams B–E has a defined Repository method or composite operation. Every atomicity constraint from Stream D (frame-template deep-copy, session migration, restore) is wrapped in a single transaction with explicit rollback semantics. Every versioning rule from Stream B (snapshots not deltas, milestone forcing, manual saves bypass debouncer) is implemented in H2. The model is correct in the sense that no upstream stream's contract is broken or weakened.
  • § 2 Legal Practice Fit. A practicing lawyer's expectation of working software is "I edit, it saves; I close the browser, my work is there; I open it on another machine, I can pick up where I left off." v1 satisfies the first two by default. The third is bridged by JSON export/import in v1 and by cloud sync in v2. The version-history model — every save preserved, restore creates a branch rather than overwriting — matches how lawyers expect a working document to behave (preferable to the "last save wins" pattern in older Office tools and consistent with Google Docs / Microsoft 365 expectations they will already have). The cost — multi-device pain in v1 — is honestly named and bounded.
  • § 3 Determinism and Auditability. The runtime-module boundary, the sort-before-iterate discipline, and the CI gates make determinism a checkable property rather than a wished-for one. Auditability: every entity is keyed on a UUID; every version carries provenance (created_at, created_by, parent_version_id, change_summary); every export is text JSON that can be diffed externally. The Repository interface logs in production are a v2 concern (deferred), but nothing structurally prevents adding them.
  • § 4 Clarity. The Repository interface is small (~25 methods), narrowly scoped, and replaces ad-hoc IndexedDB calls everywhere in the application. The module structure aligns with streams. The auto-save controller is one file with one tunable. The diff algorithm is a pure function with one entry point per entity type. A debugger looking for "where does state live and how does it get there" finds one answer per question.
  • § 5 Reversibility. Local-first → cloud sync is a contained change because the Repository interface is the seam. Dexie → raw IndexedDB is a contained change for the same reason. React Flow → Cytoscape (or any other graph library) is contained behind src/ui/canvas/. The MigrationRegistry mechanism keeps schema evolution mechanical. The choices that are most consequential and hardest to reverse — React, TypeScript, the JSON envelope — are also the choices that are most universal and least likely to need reversal.
  • § 6 Interface Fit. A future session needing only the auto-save mechanics can read H2; a session needing only the version-history implementation reads H3; a session bringing the project up in code reads H4 and the Module Structure section. The decisions are presented with rationale next to each, not in a separate appendix. The cheat sheet at the end of this document indexes by H-id.

In-session flags resolved

Two in-session flags were raised and resolved in this session. Neither is escalated to the register because neither reverses a prior stream decision and the resolution of each is internally constrained by the Constitution and prior streams.

  • Flag (in-session, resolved). Should presentation hints (x, y, collapsed) be versioned, given that they create FrameVersions on every drag-drop? Resolution: yes, version them. They live on Node within FrameVersion per Stream B B1; the layout engine treats user-set hints as fixed anchors per Stream E E1; restoring an older version is expected to restore that version's layout. The cost (a debounced version per drag) is bounded by the 5-second idle window and is filtered out of the side pane by Stream E E5's "milestones only" filter. The cleaner alternative — in-place updates to the current FrameVersion — would weaken the immutability guarantee on FrameVersion records and complicate test fixtures.
  • Flag (in-session, resolved). Should the diff algorithm surface layout-only changes (x/y, collapsed) at field-level granularity, or should it bucket them as a single line? Resolution: bucket. Stream E E5 frames the Compare view as "structural and field-level changes" with the user goal of "what changed semantically between v6 and v8." Showing four field-level entries for a four-node drag-drop adds noise without information; one line ("Layout updated for 4 nodes") preserves the signal. The diff algorithm is pure and can be re-tuned without affecting persistence.
